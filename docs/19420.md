# 如何从头开始创建一个诱导过程？

> 原文:[https://dev . to/alexantra/how-to-create-a-effective-induction-from-the-ground-up-30a](https://dev.to/alexantra/how-to-create-an-effective-induction-from-the-ground-up-3i0a)

Helen 在周五发布了一篇令人惊叹的[讨论文章，内容是关于你如何与员工共事。](https://dev.to/helenanders26/how-do-you-onboard-a-new-team-member-1khg)

它出现了一些非常棒的评论，并发现获得了很多见解。

因此，作为一名本周入职的新员工，我受到启发，分享了我如何制定端到端入职计划的过程。

我认为，在大多数地方，入职培训做得很差。

我看到的一些常见错误是:

*   一个他们以 4 倍的速度完成的入职计划，让你没有任何事情等着他们去做。
*   缺乏最新的或可访问的文档使他们没有任何参考资料，或者阅读一堆不再适用的东西。
*   忘记你刚开始的时候在纠结什么，并假设他们知道什么，学习东西有多容易。
*   以错误的顺序教他们东西，让他们困惑更久
*   假设每个人都以同样的方式学习。
*   阻止他们为团队贡献太长时间。

我一次又一次地看到上述内容，它们往往表明缺乏信息管理，有太多的筒仓，没有真正理解如何成为一名好的工作人员，更不用说理解角色到底是什么了。

人们倾向于生活在他们的工作中，无法后退一步，从树上看到森林。

真正搞清楚这一切的工作通常是压倒性的，所以管理层依靠他们可用的最大拐杖，时间，并期望你向上失败，直到你最终搞清楚。

你是否经常听到“你需要 6-9 个月的时间才能到达那里”？
简直是扯淡。你不傻。你在大学的一个学期里上了多少门不同科目的课？“你自学了多少东西？

这是因为他们根本不知道真正需要花多长时间，也没有投入时间去学习。所以他们把你的学习留给随机的机会，因为对于 99%的工作来说，6-9 个月的工作经历会教会任何人。

但是他们看不到做好这件事能得到什么。如果你确实知道你可以在 6 周内让某人快速上手，那么你不会接受 9 个月作为一个可接受的时间框架。这相当于把你的存款放在一边，在 12 个月内收取最低利息，因为你懒得考虑任何其他选择。

这就是世界上大多数就职典礼上发生的事情，他们懒得花时间去搞清楚。

我知道很难挤出时间，尤其是如果你要接替一个已经离职且资源不足的人，但是你难道不希望你的新员工在六周内而不是二十四周内就能有成效吗？这难道不是更好地利用你的时间吗？

是时候解释一些关键点了。

弄清楚教什么
知道如何培训一个人的基本方面是把你想让他们做的事情分解成更小的知识组成部分。

一个很好的简单例子是
任务:发送每周报告:
他们需要知道些什么？

*   需要知道将报告发送给谁
*   需要知道如何运行报告
*   需要知道如何做报告。

这些要点中的每一点都是你需要教授的，这样他们才能胜任工作。为每一点写一个训练计划，然后把它总结成一个更大的部分，瞧，你知道教他们什么和怎么教他们了。

不是所有事情都那么简单。不是每份工作都是一系列需要教授的任务。

所以让我们在宏观层面上应用它。我管理的团队没有一系列我可以教他们的具体任务，他们是分析师，所以更多的是告诉他们厨房里所有的食材在哪里，以及如何使用它们，这样当订单到来时，他们就可以立即决定使用什么以及如何使用。

所以我做了上面的，只是在更广的层面上。我写下了他们在工作中需要知道的每一种事情，然后列出了我需要教的更小的部分。

在某些场景中，相同的组件一次又一次地出现。例如，“需要了解 SQL”，这使我能够将其分解为满足他们在该角色中要做的事情所需的确切 SQL 级别。

当你像这样分解事情时，它帮助你理解你将如何教某人，它也帮助你识别你现在想当然的人们经常忘记教的事情。

它还提供了结构和顺序，这样你就可以在教其他东西之前确定要教什么，随着时间的推移，这将使你能够理解他们的进展。

通过归纳，你可以自信地说他们是 X%。运行几次之后，你就会知道要达到标准大概需要 X 周的时间。

辨认他们的梯子，数一数台阶。
如果你不知道他们是如何学习的，你的归纳就会失败或者效率低下。
我称之为梯子。每个人都有自己的梯子。
我之所以用梯子作为比喻，是因为向某人解释某事的关键是了解他们现在在哪里，并想出如何让他们去你需要他们去的地方。都是步骤的问题。他们需要爬多少级梯子。

例如，如果你必须向熟悉 iPad 及其功能的人展示如何使用 iPad 上的应用程序，你可能不需要教他们太多，他们只需要在梯子上移动一两步。

然而，如果你必须教从未使用过 iPad 或智能手机/平板电脑的人，那么你需要移动他们的步骤将会非常多，因为你必须教他们一些非常基础的东西，只是为了让他们知道你可以教他们如何使用你的应用程序。

能够让你的引导适应他们的学习风格，并且能够快速识别他们在阶梯上的位置，这是非常重要的。如果你的归纳比现在高几个步骤，那将很快失败。

**不信任，试探。**
这一个可能看起来违背直觉，但我不相信他们知道些什么。

即使他们的简历上很清楚他们已经做了很多年，确保这一点也是值得的。

你认为一种能力包括什么和他们认为什么可能是两回事。

例如，“我可以建立网站”可能意味着他们用精益引导和 Javascript 建立了整个 shibang 端到端。或者这可能意味着我已经建立了 wordpress 网站，我知道要改变什么 HTML 代码，将它定制成你想要的样子。这是两个截然相反的例子，但是中间有很多灰色地带。你可能是伟大的 JS，但你的 CSS 可能生锈，你不知道什么是 CNAME。

花时间去验证你认为他们已经知道的是真的，确保你的假设与他们的实际能力相符。假设某事的代价可能是相当有害的。

教他们如何解决问题。
如今，当这么多的工作都是解决问题时，给他们如何做开处方只是经验的一半。向他们展示你是如何解决问题的，这将大有帮助。大多数人都有一些保存在浏览器中的网站，当他们试图弄清楚事情，分享你的链接和解释你的思维过程时，他们会参考这些网站。

我做了一个“遇到困难时该怎么做”的指南，上面详细列出了解决网络内外某些问题的方法。

**设置场景，获取眼前的文化。**
成为一个新角色伴随着许多未知，主要是对你的期望。现在你可以让你的新员工蒙在鼓里，让他们通过一个反复试验的过程，弄清楚他们在加班、工作速度、在家工作等方面的情况。或者你可以直接告诉他们。

我对新员工做的第一件事就是带他们出去喝咖啡，讨论弹性工作制，他们不应该超过 40 小时的事实，以及我对他们的文化期望。

不好的事情，如[倦怠](https://dev.to/ronsoak/the-lies-and-lack-of-self-respect-that-lead-to-burnout-5007)，往往来自员工为自己设定的不切实际的期望，而不是来自管理层的具体期望。正是由于缺乏明确的文化和工作场所指导方针，人们才会失败。站在它的前面，口头设定期望，并在他们的整个雇佣期内继续。

**重复你教的**
最后要讲的是“什么都教三遍”没有人第一次就能得到任何东西，经常发生的是他们分三个阶段理解事物。

1.  起初，他们理解所教内容的大致背景。
2.  第二个阶段来自于他们实际接触或看到它的实际操作。
3.  理解的第三阶段是当他们看到它如何作为更广泛系统的一部分融入更大的画面时，这只有在他们在其他项目上到达第二阶段时才开始发生。

要么动动脑子，制定一个归纳计划，三次触及每一点，要么随着时间的推移，记下重新触及这些点，也许一天一次。有些事情，尤其是对工作至关重要的事情，会自然而然地发生几次，所以要确保你关注的是那些没有自然而然发生的事情。
