# 逐渐转向开源

> 原文:[https://dev.to/dbanty/easing-into-open-source-1fg9](https://dev.to/dbanty/easing-into-open-source-1fg9)

如果你以前从未参与过一个开源项目(或者只参与过很少)，那么开始可能会令人生畏、不舒服，甚至非常可怕。我自己对贡献者世界来说是相当陌生的，仅仅一个多月前，我才有了我的第一次(积极的)开源体验。我想以(我认为)最简单的方式讲述我是如何开始的。

## [](#1-dont-start)1。告诉…不要发牢骚

当目标是最终为开源项目做贡献时，这可能看起来是一个奇怪的第一步，但是请原谅我。我在大学里得到的建议是“只寻找有贡献的项目”，我相信这个建议是错误的(至少对我来说)。我不认为为开源做贡献应该是目标，我认为它应该是你实现其他目标的手段。仅仅带着做贡献的目的写代码是一个很好的方法，可以让你很快泄气或者筋疲力尽。

## [](#2-business-as-usual)2。照常营业

好吧，那么，如果我不应该去寻找一个开源项目来做贡献，我怎么才能成为一个贡献者呢？我认为第一步是停止将开源视为与其他事物根本不同。不是的。大多数开发人员在工作中每天都在使用开源库。自信地为这些库做出贡献的第一步是认识到它们只是软件，就像你正在写的东西一样。

## [](#3-find-problems-to-solve)3。找到要解决的问题

这是容易的部分。编写代码的全部目的是解决问题，这是我们的工作。最有可能的是，您正在编写自定义代码，因为还没有一个库可以解决上述问题，但是可能会有一个接近的库。当你意识到你正在使用(或想要使用)的库中有一些漏洞时，你已经找到了你的切入点。无论是您发现的缺陷还是您需要的特性，这个需要解决的问题都是通向开源世界的大门。

## [](#4-talk-to-the-maintainers)4。与维护人员交谈

在我看来，这是困难的部分。对一些人来说，向陌生人询问关于他们项目的问题可能是很自然的事，但对我来说肯定不是。这就是为什么我为这一部分提供几个次级建议。

### [](#41-read-the-docs)4.1。阅读文件

当你第一次开始使用这个库的时候，你可能已经这样做了，但是是时候再看看了，睁大你的眼睛寻找一些问题的答案。查看自述文件、贡献者指南、任何公共文档和现有问题，看看能否找到问题的答案。可能是这个项目的维护者出于某种原因没有实现您想要的特性，或者这个 bug 实际上是出于某种原因的预期行为。也要留意捐款的规则/指导方针。

### [](#42-ask-a-question)4.2。问个问题

这很重要，不要告诉维护人员任何关于他们项目的事情。这是他们的，他们知道的比你多，我保证。不管你的目标是什么，把它框成一个问题。“有办法做 X 吗？”"当我做 Y 时，我得到 Z，但我期望得到 Q，我做错了吗？"。你应该不带任何期望或预先的假设去找他们，只是一个需要解决的问题。这个问题基本上是你的，直到他们同意这个问题是项目的。

## [](#5-solve-your-problem)5。解决你的问题

不要无所事事地等待项目维护者的回应。如果你以写代码为生，你可能没有时间去做。写一个 bug 的解决方法或者一个能给你你想要的特性的函数。把它写在你的代码中，而不是开源库的副本中(你不会想从事维护分叉的工作)。尽可能地将这段代码从您的项目中分离出来，您以后可能会用到它。

## [](#6-go-with-the-flow)6。随波逐流

最终维护人员可能会回复您，也可能不会。如果有，希望他们会回答你的问题。如果他们愿意将解决方案添加到他们的项目中，你可以向他们提供你在步骤 5 中编写的代码作为帮助。你如何提供这个代码将取决于项目。有时他们会希望您经历一个分叉/克隆/分支/推/拉请求流程。有时他们可能想要一个 Git 补丁。也许他们只是想看看你在这个问题上是如何做的，这样他们就可以把它实现到他们的代码库中，无论他们认为合适与否。

## [](#7-write-a-comment)7。写评论

在这篇文章下面发帖告诉我你的成功故事。你是如何着手为一个新项目做贡献的？你使用了我上面的任何建议吗(即使你在阅读这篇文章之前已经使用了)？对于还没有为开源项目做出贡献的开发者，你还有什么其他的建议吗？
