# 测试人员不是最后一道防线

思考为什么测试人员需要超越他们的角色，进化成为整个团队的优质教练。

你好，

如果你正在阅读这篇文章，并且在软件工程领域主要是作为一名开发人员或测试人员，那么我有一些重要的想法要传达给你们两个人。

对于开发人员来说，Test 不是你的替罪羊，你可以将构建移交给它，并卸下对代码质量的任何担忧。

为了测试—*你的工作不是让上述功能失调的行为发生，而是一直充当看门人。*

你们中的一些人可能会立即反驳以上观点，因为这是一个完全颠倒的想法。

你甚至可能会说，<u>这是开发人员或测试人员必须做的事情</u>！然而，请允许我解释为什么这是一个坏主意。

在我的职业生涯中，我看到同样的事情发生了无数次，经常对产品质量产生不良后果。

## [](#bug-ping-pong-dysfunctional-loop)Bug 乒乓/回路失调:

下面的场景听起来熟悉吗？

1.  产品经理/业务分析师为业务想要解决的问题提出了一组用户故事/需求**(大多数情况下是紧急的或在截止日期前)**
2.  开发人员从一开始就与 UX 设计师/项目经理一起参与进来，并就构建什么和如何构建提出想法/实现。
3.  测试人员忙于其他的可交付成果(自动化堆积如山的测试),并且通常在产品/特性已经构建好的时候参与到游戏的后期
4.  测试人员相信他们是质量的唯一冠军**,查看应用程序并发现设计问题和错误(有时甚至是在基本的应用程序流程中，通常是在负面流程或边缘条件中)**
5.  开发团队很高兴也很惊讶能有这样高质量的把关者，修复这些错误，大部分忽略了设计问题，因为功能缺失或与未来版本不同，并把另一个版本交给测试人员
6.  测试人员又看了一眼，发现大多数问题都被修复了，但是仍然有回归错误并报告回来。(他们在这一阶段感到被认可，很开心)
7.  开发人员对于这么晚才发现这样的 bug 或高兴或恼火，尽管他们没有真正的选择来解决这些问题。

……如此循环往复，周而复始。。这类似于乒乓球比赛。

* * *

我已经记不清这种情况发生过多少次了。

你觉得这有什么问题吗？乍看之下，你可能会认为这是**一切如常**然而，这其中存在深层问题。

## [](#testing-anti-patterns)测试反模式:

这通常会导致以下反模式:

1.  开发人员变得自满，因为他们知道测试只是一种保护措施，而没有建立足够的测试覆盖面(单元/集成)
2.  开发人员大多对基本的健全感到满意，而不是严格地练习构建。
3.  测试经常充当这种往复循环的推动者，并且主要被多个执行循环所占据(在大多数情况下是手动的，以满足发布特性或 bug 修复的紧迫性)
4.  自动化程度很低的测试会导致堆积如山的自动化测试。
5.  设计师/项目经理经常困惑于为什么这么多边缘和负面的场景没有被更早的想到
6.  整个过程本身就成了低效的练习。

以下是一些值得思考的问题:

> 一个测试人员如何保证一段不是他编写的代码的质量？

好了，希望以上的论点能让你看到问题所在。然而，如果这是一个这样的问题，那么我们可以做些什么来解决这个问题呢？

我们当然可以在以下方面采取行动:

在乌托邦的世界里。

## [](#test-stops-acting-as-a-gatekeeper)测试停止充当看门人:

测试人员从产品设计的开始就参与进来，询问关键问题，并鼓励整个团队**开始思考**以下问题:

*   我们正在构建的功能是什么？
*   我们为什么要建造它？
*   这将在哪里为客户增加价值？代码将在哪里进行更改？
*   哪些应用领域会受到影响？
*   谁是我们的客户？
*   他们将如何使用该产品？
*   我们是在打造正确的产品吗？还是我们**在打造合适的产品**？(微妙但重要)

## [](#coach-developers-into-the-art-of-testing)蔻驰开发商进入测试的艺术:

测试人员指导开发人员戴上测试的帽子，帮助他们养成思考打破他们自己通过**结对制造的东西的习惯，成为代码审查的一部分，建立和促进暴民测试**或 bug bashes 和其他类似的形式，并从本质上帮助建立整个团队质量所有权的文化。

## 设置自动化测试并一起创作它们

测试人员通过构建易于使用的框架和工具，帮助开发团队在集成层或功能层设置自动化测试，开发人员还拥有创作功能测试的所有权。

## [](#be-part-of-code-review)成为代码评审的一部分

测试人员作为代码审查的一部分，调查测试覆盖范围，甚至指出遗漏了什么条件，以便在正确的层进行自动化或测试。

## [](#analyze-customer-data-to-learn-usage-patterns)分析客户数据，了解使用模式:

测试人员和 UX/项目管理/开发人员一起研究客户的使用模式，以理解和验证所创建的特性是否真正产生了预期的影响。

* * *

对测试人员来说，更好地利用时间和创造力并不重要。

我们最终都需要认识到的是，两个大脑比一个大脑更好，每个人(无论他/她在工程团队中扮演什么角色)都有天生的好奇心，并有能力弄清楚东西是如何工作或损坏的。

测试人员需要放下架子，更加务实地做出选择。开发者与其说是敌人，不如说是他们的犯罪伙伴。
