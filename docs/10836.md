# 给 JavaScript 初学者的 7 条坏建议

> 原文:[https://dev . to/canderson 93/7-给 javascript 初学者的坏建议-477c](https://dev.to/canderson93/7-pieces-of-bad-advice-for-javascript-beginners-477c)

让我们面对现实吧——不是所有的建议都是有帮助的，即使是善意的。

作为初学者，你特别容易受到伤害。你很难区分哪些建议会让你节省几个月的时间，哪些会让你损失几年的时间。

你可以避免接受任何建议，但自学成才的开发者的现实是，当你陷入困境时，你(T2)只能依靠网上的建议。你如何判断该遵循哪条建议？你的*进展如何？*

我见过各种各样的坏建议——其中最糟糕的还会重复出现。我整理了一些常见的坏建议，为什么你不应该遵循它，以及你应该做些什么。

## [](#1-you-should-switch-to-this-coursebookresource)**#1:你应该切换到这个课程/书/资源**

当有人已经完成了一门课程，但还在努力自己编写代码时，我看到了这种情况。他们仍然需要导师或向导的帮助来给他们答案。

这条建议很伤人，因为它暗示有一条“完美”的路线。一个能解决你所有问题的，你要做的就是找到它。

这是《教程地狱》的一个核心贡献者。初学者陷入了一个跟随新教程却从不解决根本问题的循环中。他们从来不知道如何通过教程。相反，他们重新开始，希望*这次会有所不同。*

[![7 Pieces of Bad Advice for JavaScript Beginners](../Images/be41b7fb640049446e64e42bedbc9beb.png)T2】](https://i.giphy.com/media/F8deaO3psSKiY/giphy.gif)

初学者的资源不会相差*足够的*来使它值得重新开始。

你没有一开始就搞砸。你不需要重新开始。你需要在你已经学过的材料上多花些时间。

我已经在这篇文章中详细阐述了这一点。软件开发有两个部分:语法和*学习如何构建软件*。

与其认输并重新开始，不如开始构建微小的项目。你已经知道了什么？试着想出[一些小的项目创意](https://dev.to/canderson93/how-to-find-ideas-for-good-practice-projects-5hcm)，你可以用你已经掌握的知识来创造。

## [](#2-you-should-use-this-frameworklibrary-instead)**#2:你应该用这个框架/库代替**

这不是“每次你看到这个，就走开”的建议。我特别指的是没有经过任何思考的答案。

> 嘿，我如何用一个按钮改变一个 div 的颜色？
> 
> *如果使用 React* 就简单多了

建议框架和库没有错。当框架被盲目地提出时，问题就来了。

React 根本不能帮助你改变一个 div 的颜色。_ 包含 React 所需的努力将远远超过用其他方法解决它所需的时间。评论者只是喜欢反应。

这并不意味着所有建议的技术都是不好的。重要的是你要做好调查，做出明智的决定。

每次你包含一个新工具，你必须学会如何使用它。您可能需要重写大部分代码。这是由你来决定是否值得花费这个成本来解决你的问题。

做你的研究。如果没有意义，那就继续前进，去找别的。不要相信任何人的话。

## [](#3-you-have-to-be-passionate-to-learn-javascript)**#3:学 JavaScript 要有激情**

我讨厌看到这个建议。

这个可以在任何关于 JavaScript 的线程中抬起它丑陋的头。有人讲述了他们如何在 6 个月内学会 JavaScript 的故事...每天花 12 个小时生活，呼吸，除了代码什么都不用。

如果你想成功，你也需要这么做。

太极端了。它美化了一种摧毁你生活方式的方法。它表明，如果你有一份全职工作，一个伴侣，一个年幼的孩子，或者*任何其他爱好*，你就不能学习 JS。

你不需要每天花 12 个小时在 JavaScript 上。不需要每天晚上找 3-6 个小时来编码。你甚至不需要开心。

你需要持续关注*花时间学习 JavaScript。*

 *你在学习编码。这是一项你将在余生中不断发展的技能，所以不要着急。如果你能以一种你能永远持续的方式找到时间，你会更成功。

只要你能坚持下去，以 1 小时为单位学习编码并没有错。

## [](#4-you-need-x-daysmonthsyears-to-learn-javascript)**#4:学 JavaScript 需要 X 天/月/年**

我能理解想知道事情需要多长时间的渴望。不幸的是，它不是那样工作的。

没有设定点让你“学会了 JavaScript”。任何说需要多长时间的努力都是有内在缺陷的。

这里的损害是暗示，有一个点，你会觉得“完成”。你可以努力达到那个点，当它永远不会到来时，你会越来越失望。

当别人说花了多长时间时，别理他们。这更有可能是邓宁-克鲁格效应，而不是真实的轶事证据。

相反，专注于切实的目标:

*   开始你的第一个个人项目
*   创建有趣的 SVG 动画
*   得到你的第一份工作

## [](#5-you-need-to-know-about-x-before-you-start-y)**#5:开始 Y 之前需要了解 X**

当有人想尝试新事物时，比如 React，就会出现这种情况。大约有一半的时间，它包含了 x 的一个很长的列表。

你*不可能*开始使用 React，除非你已经了解了:

*   班级
*   函数 JavaScript
*   库伯内特斯
*   量子计算
*   用钩针编织

/讽刺

这些列表倾向于涵盖新框架等所有内容。用途(还有一些)。它们是我们在指导初学者时似乎喜欢的“瀑布”方法的结果。“在尝试*任何*高级的东西之前，你必须学会*所有的基础*。”_

我在其他文章中谈到过这个问题，比如[何时开始学习前端框架。](https://dev.to/canderson93/when-to-start-learning-a-front-end-framework-9hk)进入高级概念所需的知识往往比你想象的要少得多。

这些列表很有用:它们是框架的大纲。如果你选择学习 React，这是你将会学到的东西的列表。

重要的是:不要让它阻止你。不要担心必须查找资料。

动力很难。如果你把时间花在你想做的事情上，那会更容易。

## [](#6-if-you-cant-even-do-x-you-dont-deserve-to-be-a-programmer)**#6:如果你连 X 都不会做，就不配做程序员**

好吧。你抓到我了。这不完全是建议。这是校园欺凌的编程版本。虽然不常见，但这种情况可能在任何地方出现。

这里的原因是它利用了许多挣扎中的初学者的不安全感。“我不明白——也许我不是*想要*成为一名程序员？”。

任何时候你看到这个，你都不会看到对“成为一名程序员需要什么”的批评。你看到有人在装腔作势——有人对自己的编程没有安全感。

注意到*从来没有*伴随着一个实际的答案吗？

你愿意花时间学习编码吗？你值得成为一名程序员。

按照自己的节奏学习。对于每一个像这样的白痴评论，有很多其他的实际上有帮助。

## [](#7-you-should-learn-this-other-language-instead)**#7:你应该学习另一种语言而不是**

这个建议到处都是，这似乎很奇怪。这经常出现在关于如何打破教程地狱的讨论中。

学习另一种语言是奇怪的建议，因为这种建议可能会奏效。它提供了一个新的编程视角，这可能是你打破教程地狱所需要的。

问题是不管你用什么语言，教程地狱都存在。学习 C 或 Ruby 不会解决你的问题。它提供了一套不同的教程，让你在地狱。

你最好把时间花在解决你写 JavaScript 时遇到的问题上。 [写一些小项目](https://dev.to/canderson93/how-to-find-ideas-for-good-practice-projects-5hcm)，做一些编码练习，写所有的排序算法。*练习*。
