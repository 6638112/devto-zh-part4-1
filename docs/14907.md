# 我第一次学到领导的 3 件事

> 原文:[https://dev . to/Emma/3-things-I-learn-feature-leading-for-first-time-289j](https://dev.to/emma/3-things-i-learnt-feature-leading-for-the-first-time-289j)

当我的团队开始开发一个新功能时，团队中会指定一名开发人员作为功能负责人。此人负责让其他人了解该功能的最新进展，与利益相关者的任何沟通，并创建/管理该功能的吉拉门票积压(有时会感觉永无止境！)

在过去的几个月里，我有机会将一个已存在的 Backbone.js 前端转换成一个闪亮的新 React 应用程序，并且在这个过程中我学到了一些东西。

## [](#get-feedback-on-your-implementation)获得对您的实施的反馈

在为您的特性创建标签时，您可能会遇到一些您对其实现细节有点不确定的任务。也许有一些方法可以解决这个问题，或者你不确定所有涉及到的边缘情况。在这种情况下，有必要整理一个简短的页面，描述一些不同的方法及其优缺点，并与团队中的其他开发人员分享。

然后，其他开发人员可以就他们认为应该采取的方法提供反馈，并提出他们想到的任何其他风险，你们可以一起决定一个解决方案。提前做这件事会让拿到票的开发人员的生活更轻松，因为他们确切地知道他们需要做什么。它还创建了一个更加平滑的拉请求过程，因为每个人在实现方面都已经在同一页面上了。

## 这可能比你想象的要长

当我估计完成这部电影所需的时间时，我认为我是慷慨的——但我最终却休假了一个多月。有各种各样的事情会占用开发人员的时间，比如团队活动、20%的时间和其他计划或功能会在某些周优先考虑，在你知道之前，你已经到了应该完成的周，还有很多事情要做！我们不能总是提前考虑这些事情，所以重要的是要知道什么可能会导致延迟，这样当你的功能发布日期改变时，你可以尽快与你的利益相关者沟通。

## [](#small-things-add-up)小事加起来

我没有意识到的是，为页面的主要功能编码并不是最难的部分——它是所有的小事情创造了这条长长的尾巴。处理某些错误情况，表单验证规则，修复讨厌的 UI 错误...在这种情况下，作为一个功能负责人，你的工作就是确保你已经为所有这些需要完成的事情做好了准备，并且毫不犹豫地决定这些事情是否真的在功能的范围之内。它可能对于特性的发布是不必要的，但是它仍然是一个“很好的东西”,你可以在之后回来做。
