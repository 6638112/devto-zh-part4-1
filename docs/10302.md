# 通过创建开源文档学习编码:一种有益的协同作用

> 原文:[https://dev . to/watkey/learning-to-code-by-creating-open-source-documentation-a-benefit-synergy-3e1p](https://dev.to/watkey/learning-to-code-by-creating-open-source-documentation-a-beneficial-synergy-3e1p)

### duo lingo 和 ReCAPTCHA 如何启发我们改进开源文档和编码教程的学习体验

对大多数人来说，验证码并不完全是他们的乐趣，而是一种必要的邪恶。Captcha 代表“完全自动化的公共图灵测试，以区分计算机和人类”，它们是令人讨厌的小任务，包括阅读和输入乱码，以识别你不是机器。我总是希望每当验证码弹出时，我能得到一个“简单的”。不是那种会让你从头再来并诅咒这种令人沮丧的浪费时间的发明者的难以辨认的单词，特别是如果它是你在 30 分钟内的第 10 个验证码。

具有讽刺意味的是，验证码背后的想法是通过设置机器不擅长的任务来混淆机器，同时对人类来说是自然而容易解决的，例如图像识别或阅读手写。过了一会儿，reCAPTCHA，也就是“Captcha reloaded”，出现了，突然你不得不键入两个单词而不是一个。太好了，更疯狂！

诚然，我夸大了一点，虽然不可否认，我有一个点。当然，这与疯狂无关，后来我发现在引擎盖下发生了一些变化，不再让验证码浪费时间，而是将它们转化为一个强大的众包工具。

## [](#captcha-reloaded)验证码重新加载

随着越来越多的网站使用验证码，越来越多的人每天完成这些小任务，验证码的开发者们想出了一个聪明的主意，利用大量的人力来更有建设性地解决验证码。他们成立了一家名为 reCAPTCHA 的公司，并重新设计了验证码，使得求解和验证过程基本保持不变，但不同的是，用户给出的答案现在用于解决大数据问题。

[![](../Images/e5af6f5698390a93b1395111f40f4e90.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--dAqI_6n_--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/r5gyiyo6i8kgrb2v2yvi.png)

顾名思义，大数据问题需要大量的数据才能解决，而且数据集通常必须手动创建。这种任务的例子是数字化手写书籍或为机器学习数据集标记图片。用于数字化手写内容的方法包括让人们阅读手写单词并输入，然后将结果存储为手写单词的数字化版本，而图片识别任务只是要求你点击包含某些对象的图像。

当然，有几种机制可以确保只存储正确的答案，但除此之外也就如此了。极其简单高效。reCAPTCHA 的开发者和创始人之一 Luis von Ahn 表示，他们每年成功地将 250 万本图书数字化，事实上，这种效率如此之高，以至于到 2011 年《纽约时报》的档案和谷歌图书上的图书已经全部数字化。

你可以在 Luis von Ahn 关于大规模在线合作的 Ted 演讲中了解更多信息。现在，我想转到他在演讲中提到的另一个项目，这个项目近年来越来越受欢迎:

## [](#duolingo)多林哥

由 von Ahn 共同创建的 Duolingo 是一个在线语言学习平台，它通过让你每天完成简短的练习来教你选择的语言。它可以免费使用，并为每个人提供了学习一门新语言的可能性，同时，引用《麻省理工技术评论》的话来说，“是一种为自己买单的教育”。

Duolingo 是如何做到这一点的？除了更多的“经典”选项，如语言测试和获得证书，Duolingo 还提供了一项非常聪明的服务:众包翻译。公司可以向 Duolingo 提交文本，如每日新闻，并请求翻译成特定语言。然后，Duolingo 会将文本分发给平台上的学生，他们会在将翻译后的文本返还给公司并收取费用之前，将文本作为学习经历的一部分进行处理。

很容易看出 Duolingo 和 reCAPTCHA 的商业模式背后的原则非常相似:

*   **智能众包**
*   **组合工作流程**
*   **巧妙利用现有资源**

## [](#synergetic-benefits)协同效益

让我们仔细看看这些组合原则的高明之处。显然，识别人类用户的问题和解决方案，以及数字化书籍的问题在 reCAPTCHA 之前就已经存在了。然而，只有通过将这两个问题结合起来，以便同时解决它们，您才能获得一个真正优雅和高效的解决方案，该解决方案通过简单地结合现有的流程，从根本上无中生有地创造附加值。Duolingo 也是如此，在那里，翻译新闻等必要任务在作为语言学习者的榜样的同时完成。翻译和学习这两个方面之前就存在了，但是只有把它们结合起来，你才能创造一种对双方都有益的协同作用。翻译变得更便宜，因为它们是通过众包完成的，语言学习者可以获得一些实践经验，并可以免费学习语言。

reCAPTCHA 和 Duolingo 都非常受欢迎，在我看来，这是因为它们基于巧妙的原理或“协同优势”:

*   **2 对 1——用一个动作解决两个问题**
*   **双赢——双方都从协同效应中受益**
*   **回收利用——重新利用现有流程和解决方案**

## [](#background)背景

你也可以将 reCAPTCHA 和 Duolingo 描述为“交互式知识传递平台”，如果你想到更传统的传递知识的方式，你就不能绕开图书馆。事实上，我一直在研究“协同效益”，作为我为 ETH 图书馆实验室进行的开源软件开发研究的一部分。图书馆实验室旨在重新思考和推进科学、研究和教育的信息基础设施和信息周期，这就是为什么我密切关注开源软件及其作为数字公共基础设施日益增长的重要性。

这里有一些背景信息，所以我们都在同一页上:最著名的代码共享平台 Github 上的用户和存储库的数量在过去十年中几乎呈指数级增长，根据 Octoverse 2018 的[状态，已经达到超过 3100 万开发者和超过 9600 万存储库。越来越多的公司使用或发布开源软件，根据](https://octoverse.github.com) [2015 年 BlackDuck/Northbridge“开源的未来”调查](https://www.slideshare.net/blackducksoftware/2015-future-of-open-source-survey-results)，78%的公司在开源软件上运行他们的部分或全部系统。此外， [StackOverflow 开发者调查 2019](https://insights.stackoverflow.com/survey/2019) 指出，41%的受访开发者通过对开源软件的贡献提高了他们的技能或学习了新的技能。

这种开源软件开发量的增加只能通过足够的开发人员和维护人员来实现。统计数据显示，随着技术在我们的生活中变得越来越重要，想要学习编码的人数也在增加，这与对开源软件的需求类似。在一直运行和执行代码的设备的包围下，理解和设计这样的系统的愿望和需要变得越来越普遍。如今，代码学习的首选是互联网。根据前面提到的 StackOverflow 调查，超过 60%的开发人员参加了在线课程或通过在线教程学习，以便获得新的编码技能。

## [](#all-about%C2%A0code)所有关于代码

随着越来越多的软件被开发，两个因素变得越来越重要:

*   **阅读、导航和理解外国代码的能力**
*   **代码文档**

随着软件项目在代码行方面变得越来越大，越来越多的人在项目上合作，开发人员需要越来越能够理解其他人的代码。这种技能远非无足轻重，需要练习，就像你刚开始学习阅读时一样。有一些有趣的文章和论坛帖子是关于如何浏览和阅读外国代码以掌握其结构和理解其功能的技术和方法的。不幸的是，由于大多数在线课程和教程缺乏关于如何正确进行的深入练习和指导，从在线教程到真正的编码项目的过渡可能会很艰难。在我看来，将代码阅读任务和练习集成到编码教程中是有益的，但是稍后会有更多的介绍。

关于代码文档的话题，[开源调查](https://opensourcesurvey.org/2017/)指出，开源最常见的问题是不完整或令人困惑的文档。由于开源思想，越来越多的软件变得可用，对软件的关注已经从某个软件解决方案的存在转移到所述软件解决方案的可用性。糟糕的文档提出了软件变得不可用或以错误或不安全的方式使用的问题，因为它没有被正确理解。随着开源软件成为公共基础设施，软件项目被很好地记录和维护是很重要的。这对于很多人和公司依赖的关键项目来说尤其重要。提高文档的质量和数量也提高了项目的可维护性，因为开发人员可以更快地参与和理解项目。不幸的是，由于开发人员将大部分时间花在编码、调试和解决问题上，文档通常是最不重要的，因此许多项目要么没有很好的文档记录，要么根本没有。

## [](#the-beneficial-synergy)有利协同

让我们总结一下，你会看到所有这些话题是如何联系在一起的。我相信您已经注意到，前面提到的学习阅读代码和改进开源代码文档的问题是双重的，您希望同时解决它们，这样双方都能受益。作为我们实验室工作的一部分，我们提出了一个基于“协同效益”的系统，旨在解决上述两个问题，我想与大家分享这个例子，说明如何将不同的工作和知识流结合起来，以产生附加值。

该系统由一个分发平台组成，开源项目和在线编码教程可以连接到该平台。类似于 Duolingo 提供的众包翻译服务，开源项目可以向平台上传代码，请求众包文档。该平台为在线编码教程提供了一个标准的 API，通过该 API，他们可以提取代码并将其集成到学习体验中，作为其用户的代码阅读练习。一旦理解了代码，人们就会被要求提供理解给定代码所需的文档。通过内部众包审查过程，文档得到改进，然后返回到发布平台，在那里它可以被开源项目检索和集成。

[![](../Images/5357e2b1c34566d5d08e876f04b58378.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--wh1B7nhv--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/ysgwyhp38q0go6z8j3ox.png)

以上述方式设置系统具有以下直接优势:

*   从事开源代码工作的代码学习者学习阅读代码，并习惯于使用他人的代码。
*   通过为给定的一段代码制定文档，他们可以测试并巩固他们对代码的理解。
*   未记录的开源项目将被提供文档，使它们更容易访问和使用。
*   文档将通过由不属于项目的人编写或扩展而得到改进，因此他们对这个问题有一个外部的观点。
*   文档将变得更加方便用户，尤其是初学者。

科学界正在认真讨论在研究论文中整合用于获取结果的代码并将其公之于众的可能性，以使结果可验证并加快研究进程。所展示的系统将间接受益于此，因为花在处理糟糕的或无文档记录的代码上的时间会更少，而花在实际工作和试验代码上的时间会更多。

这就引出了一个问题，为什么在线学习教程不直接从 GitHub 库获取开源代码并返回，为什么你需要一个额外的中间平台。无论是开源项目还是编码教程都不是专门针对这种交互的，所以它们可能是小规模的，相对低效的。一个独立的中介允许多平台交互，因此任何编码平台都有可能将开源代码集成到它的学习体验中。此外，该平台完成对接收到的代码进行排序的任务，以便提供不同难度级别的选择，并确保在线教程接收到与其主题匹配的代码(例如，机器学习代码到机器学习教程，java 代码到 java 教程)。我认为最大的潜力在于提供一个标准化的接入点，以确保知识交流尽可能简单，并使该方法具有良好的可扩展性。

## [](#all-about-collaboration)一切关于协作

只有在已经存在的大量资源的基础上，这种方法才能充分发挥其潜力，从而使系统完全基于扩展现有结构并与开源项目和在线教程合作。对于传统机构来说，这似乎是一个充满希望的领域，以便将它们的触角延伸到传统服务之外，并在未来保持相关性。我相信，还有许多其他问题可以通过使用“协同效益”得到有效解决，我期待着看到其他人提出的想法。
