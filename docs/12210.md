# 35 天的代码进度

> 原文：<https://dev.to/haleyelder/35-days-of-code-progress-eln>

嘿，第一个新帖子，我现在已经建立了一个新的博客空间，由[盖茨比](https://www.gatsbyjs.org)和 [Netlify](https://www.netlify.com) 主持！在过去的一年左右的时间里，Gatsby 在开发人员圈子里获得了一些关注，他想看看到底是怎么回事。现在，我有了自己的博客，开始运行并为全世界服务。我仍然有我的旧媒体帖子，这些帖子很可能会成为一个档案，直到我把帖子转移过来，重新开始。

说到重新开始，我现在已经“完成”了著名的[代码新手](https://www.codenewbie.org/)和[免费代码营员](https://www.freecodecamp.org/)、[代码 100 天](https://www.100daysofcode.com/)。这个挑战适合新的学习者和经验丰富的开发人员，通过记录他们在 100 天内每天工作的内容来创建学习代码、新框架或辅助项目的一致实践。还有一些其他细节，包括在 Twitter 上公开发布你做了什么，不能是教程，等等，这些细节在上面的网站上，将会更详细；我把我的稍微改了一下，以适应我自己的生活习惯。

几年前，我曾尝试过这个挑战，大约在一个月后的夏天结束，创建这个挑战是为了在冬天保持练习的一致性，这样更容易呆在家里打发学习时间，这就是为什么我的 [Github repo](https://github.com/haleyelder/100-days-of-codev2) 被称为“v2”我把我的特殊资格改为每天 30 分钟，认为这在我的时间表中很容易实现，可以一路上做辅导，并在 Twitter 上发布一天的工作。在听到其他程序员的成功后，我想这样做，并回到制作一些附带项目和持续学习中；我已经有一段时间没有写新代码了，感觉这是一个更容易的提升。

在这段时间里，我参与并记录在我的 Github 资源库中的一些项目是 freeCodeCamp React challenges(drum kit 和 markdown previewer)，以一个完整的 Udemy 教程开始，以一个简短的 MERN (Mongo、ExpressJS、React、Node)教程结束，作为未来项目的基础。从仲春开始到这个月结束的 100 天里，计算我记录的时间，我提前一个小时出来了！不，我甚至没有接近 100 天，所以这些天对我来说是一分钟的胜利。你可以在我的日志中看到，肯定有一些时候我要么很忙，要么不在城里，要么不想做什么事情。

我个人承认一些焦点和注意力问题，大部分源于社交媒体的使用和其他坏习惯(努力解决它们！)，此外，我听说公开发表一些东西会让你更负责任。这是真的，每当我在推特上发布我的作品时，100 个 daysofcode 机器人都会转发它们，这大概是唯一的“你做了一件事！”我得到了认可。也许范围应该更远？或者也许这是一个好的开始，反过来，它可能会让我更紧张不发布？我不知道，我的大脑很奇怪😅和大多数新项目一样，我确实在开始时注意到了更高的积极性，这可能归因于精疲力竭的感觉和中断。

随着挑战的进行，我一天中的优先事项发生了变化，我今年的另一个目标是自我训练以完成 10k 跑，这也成为了一致性问题的竞争者。在我确信我们都想做的事情中，优先级总是会改变的，为当前重要的事情腾出时间是最好的技能。拥有多项高优先级任务而不为其中一项耗尽精力是非常困难的，尤其是当它是你喜欢的事情时。

我对精疲力尽和冒名顶替综合症的感觉并不陌生，我不确定这次的挑战是有帮助还是有负面影响。挑战结束已经有几天了，我已经把注意力集中在了其他项目上，对记录和工作不那么焦虑了。我确实有额外的项目要添加到一个板上，但是我不希望每天记录进展的卑微任务成为例行公事和无聊。承认我已经成长了多少很好，我喜欢跟踪前一天的统计数据，但这种跟踪似乎变了，这可能是没有一套明确目标的副产品(嘿，缺乏重点！).我很想在未来再做一次，并从这一轮中学到了在 100 天里做什么和不做什么，以记录更多的日子和项目来分享。
