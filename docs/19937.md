# 使用 Brevis CSS 从组件优先到实用程序优先

> 原文:[https://dev . to/dlzi/from-component-first-to-utility-first-with brevis-CSS-5g LH](https://dev.to/dlzi/from-component-first-to-utility-first-with-brevis-css-5glh)

CSS 工具包的选择是由信念所包围的，这些信念可能并不总是支持逻辑论证。大多数时候，这是个人的决定。不管是你还是你的老板。为了帮助那些需要决定下一步使用或尝试哪个 CSS 工具包的人，以下是我认为你应该尝试 Brevis 的四个理由:

1.  也许你不需要灯笼裤。如果您对当前的工具包感到满意，并且能够完成任务，为什么还要费神呢？如果你的项目不需要任何特殊的技术细节，为什么要花时间在你不需要的东西上，对吗？但是，如果你有一点好奇心，想作为一名开发人员提高你的 CSS，那么，也许你应该给 Brevis 一个机会。Brevis 将向你展示一种新的做事方法。更好、更快、更简单的方法。

2.  Brevis 最大的特性之一是生成精益代码的能力。意味着不需要担心未使用的 CSS 负担。很简单，因为你是创建 ui 的人，而不是工具箱。使用 Brevis，您可以完全控制您正在构建的内容。显然，这个特性只与那些关心性能和页面加载时间的人相关。

3.  我们都知道给事物命名很难。组件优先(CF)工具包也不容易。即使像 BEM 这样的方法论也帮不了你多少(实际上可能会变得更糟)，因为现在你需要说出三样东西！所以，Brevis 的伟大之处在于根本不需要命名。这对于像 Brevis 这样的实用优先(UF)工具包来说是一个巨大的胜利，超过了 CF 工具包。

4.  一旦你开始编码，Brevis 相对于 CF 工具包的好处将变得更加清晰。对于 CF 工具包，大部分重复的代码发生在您需要创建定制样式的时候。一个简单的导航条就是一个很好的例子。如果您不想使用默认的 navbar 样式，您将不得不用自己的样式覆盖代码。当你这样做的时候，你创建了重复的代码。这适用于您想要定制的 CF 工具包中的每个组件。还有更多。随着时间的推移，这种方法将导致您不敢删除未使用的 CSS。Brevis 不一样。它通过保持中立把你从疯狂中拯救出来。没有默认的样式来规定事物应该是什么样子。也就是说，永远不会发生覆盖！

当你需要速度、健壮性和精简代码时，Brevis 是 CSS 工具包中的领跑者。从 CF 工具包(如 Bootstrap、布尔玛或 Materialize)转移到 Brevis 等 UF 工具包的好处是有形的。但是你需要自己尝试 Brevis，因为这是你充分利用好处的唯一方法。Brevis 就像一个 CSS 文件一样简单。不需要依赖或设置。就用吧！

你可以在[getbrevis.com](https://getbrevis.com)或 [GitHub](https://github.com/dlzi/brevis) 仓库找到更多关于 Brevis 的信息。
