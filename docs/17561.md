## [](#analyzing-your-stacks-real-costs)分析您筹码的实际成本

几周前，有人请我写一篇关于一个服务于非常大规模的小型服务的不同平台的相对成本的文章(我在这里的模型是一个非常简单的 ipify，一个月有 300 亿个请求)。我不断回到一些基本的东西，经济学家称之为“外部性”:资产负债表上没有列出的一个过程的成本。典型的外部性是污染。只要不计算因喷出烟雾而伤害的所有人的成本，一家公司就是盈利的。我经常看到技术高管没有正确考虑的外部因素是团队压力的成本。

让我们举一个简单的例子:CTO 被推荐采用[蜂巢](https://www.honeycomb.io/)来测试和分析性能。有了 Honeycomb，像存储耗尽这样的几类问题将在导致故障之前生成警告。CTO 说没有，年度预算比蜂巢的费用少 100%。团队工作的所有夜晚、周末和额外的白天，试图在东西坏了之后修理它们，根本不会出现在预算中。也许首席技术官会因为降低成本而获得奖金。几年后，这家总能实现预算目标的公司每年花费数十万美元来替换那些总是在两年内辞职的团队成员。

因此，由于未能采用工具，我们最终对我们的团队要求过高，这在短期内节省了资金，但从长期来看成本更高。但是我要说，即使是这个* * * * _ 也把问题的深度简单化了。

让我们从技术市场中劳动力可用性的一些基本事实开始:

没有足够的计算机科学专业来满足计算行业的需求，这种差距在不久的将来只会变得更糟。虽然计算机科学课程是为编写代码做准备的一个好方法，但没有进入运营工作的一般中等教育途径。

无休止的会议讨论和论文讨论了影响所有技术公司的人才短缺问题。科技公司不会出于好心提供啤酒龙头和育儿假。事实是，一个好的软件团队不仅仅是昂贵的替代品，它通常是不可能的。

我最喜欢的关于这个话题的演讲，[没有人才短缺](https://www.youtube.com/watch?v=P_sWGl7MzhU)实际上并没有否认这篇文章中的观点。它证实了雇佣一个有技能的团队是极其困难的，我们应该专注于发展技能来创建我们需要的团队。这是一个优点！两年的角度经验无法与你在过去两年中开发的开发人员相提并论。

## 所有这些与 Heroku 有什么关系

关于 Heroku 的一个最大的误解是，它要么不能扩展，要么当它扩展时会变得太贵而无法使用。事实是，所有平台都需要成本，Heroku 通常与其他服务相比具有竞争力。但是我认为即使讨论这个问题也是在问错误的问题。_

我认识很多喜欢写代码的人，我认识喜欢运行代码的人，我认识喜欢聪明的服务器配置的 Linux 书呆子。即使是热衷于设计最佳服务拓扑的架构师。我从未见过有人在凌晨 4 点调试负载平衡器的想法时高兴地拍手。

对于任何平台、任何服务架构，你应该问的真正问题是‘我的开发人员喜欢使用这个工具吗，它能帮他们解决问题吗？’

Heroku 提供了一个简单的部署路径，以及扩展容器的简单接口。更好的是，它提供了深度可配置性，允许您满足复杂的需求。当你的团队想要尝试不同的架构，或者为你的代码尝试不同的[类型的容器](https://devcenter.heroku.com/articles/dyno-types)时，使用 Heroku，这是一个简单的配置，而不是花费数小时来设置新的硬件或虚拟机。或许最关键的是:有了自动扩展，你的工程师不太可能在半夜被叫醒去处理危机。

回想一下所有的 meetup 会谈，所有的会议研讨会，所有你读过的文章。想想他们中有多少人说找不到服务器空间。有多少人表示无法购买 RAM、稳定的数据连接、升级操作系统以应用安全补丁、安装数据库，或者配置负载平衡器。现在，比较一下你见过多少团队领导不能雇佣他们需要的人才。

因此，当你考虑像 Heroku 这样的工具的成本时，让你的开发人员专注于业务逻辑，这使他们的生活更容易，并导致更少的深夜灾难；你可能没有权衡最重要的好处:它对团队快乐和留住优秀工程师的影响。一个干净的、容器化的工作流将帮助你建立你需要的团队来运营你的业务，维护一个热爱他们的工具的团队的好处可能很难写在资产负债表上，但它是巨大的！

关于这一主题的进一步阅读，请查看谢恩·罗斯塔德的文章《T2》对 Heroku 和 AWS 的成本比较，该文章也得出了类似的结论。
