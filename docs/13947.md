# [](#the-requirements-domain)需求域

在每一个知识领域，无论是医学、制造业还是软件开发，在我们开始考虑在该领域提供解决方案之前，我们都需要有一个涉及领域实体及其交互的心智模型。领域模型是特定领域中所有重要实体、活动及其关系的表示。

例如，在比萨饼制作领域(我最喜欢的领域之一)，我们会期望找到诸如面团、浇头、餐馆、送货员等实体。类似地，在需求领域，我们期望找到帮助我们概念化需求的实体和关系。

需求领域的实体是:

*   利益相关者
*   商业目标
*   能力
*   特征

## [](#stakeholders-and-actors)利益相关者和行动者

*利益相关者*是从我们的系统中获取价值、受益或影响我们的系统的人。一个*参与者*是直接或间接与我们的系统交互的涉众。所有的参与者都是利益相关者，但是利益相关者不一定是参与者

[![](../Images/457c8a47b176cd6d8cb7ae8cf2274152.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--22m_kk_z--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/qwfmja9908wsbx7nh505.jpg)

利益相关者有一个很小但很重要的子集，这是利益相关者和参与者之间的集合差异。我们称这些为“不作为的利益相关者”。这些人通常是董事、企业主、企业架构师或高级经理。他们将在系统中拥有股份，他们可能会做出影响系统的决定，但不要期望看到他们坐在屏幕前使用系统。

演员可以分为两类:

1.  主要参与者通过 UI 或 API 直接与系统交互，以实现特定的目标。
2.  次要参与者是系统为了实现主要参与者的目标而需要帮助的参与者。

## [](#business-goals)业务目标

利益相关者受益的目标。业务目标通常是由非代理的利益相关者定义的，例如董事、高级经理等。业务目标的激励和最终结果应最终归入以下类别之一:

*   增加收入
*   降低成本
*   保护收入
*   避免未来成本

确定商业目标动机的一个有用的技巧是 5 个为什么技巧。如果在 5 个问题之后，答案不是上面的一个，那么你应该严肃地质疑这个目标给企业增加的价值。

*举例*:奖励忠实客户
为什么？这样回头客就能感受到被重视，也能省下一些钱
为什么？这样顾客会更频繁地回来
为什么？以便他们在我们的产品上花更多的钱(增加收入)

提示:好的业务目标通过指定与系统固有的能力和功能无关或无关的目标，为涉众增加价值。例如，“我想让顾客买书”对于网上书店来说不是一个有效的目标。我希望顾客从我们这里购买更多的书，而不是竞争对手 X '更好，尽管它仍然缺乏一个战略层面。说到这个...

### [](#strategic-goals-are-the-best-goals)战略目标是最好的目标

最好的商业目标是那些不太抽象也不太具体的目标，那些概述了战略而不是战术的目标。例如，“奖励忠诚客户”的目标可以被框定为“通过奖励忠诚客户来增加销售额”。“增加销售”是最终目标，“奖励忠诚客户”是策略。如果目标仅仅是“增加销售”或“降低成本”，那么最终将由实现这一目标的人来决定最佳策略，不管是业务分析师还是软件开发人员。确定目标背后的战略的最佳人选是非行动利益相关者、董事、企业主和高级经理。

另一方面，如果目标变成了“如果忠诚的顾客单笔交易消费超过 500 美元，就给他们免费的礼品袋，以此来奖励他们，从而增加销售额”，那么我们就是在混合策略和战术。这可能是一件坏事，因为创建策略需要一定水平的领域、系统和架构知识，而这些知识通常是非行动利益相关者所缺乏的(我是以尽可能好的方式说的)。例如，指定“免费礼品袋”策略的主管或销售经理可能没有意识到在交付这么多礼品袋时会出现任何物流或配送问题。其他利益相关者的参与可以更好地开发策略，并且应该作为系统能力单独捕获。

## [](#capabilities)能力

能力是一种系统能力，它使涉众能够实现业务目标。能力定义了系统做什么，而不是 T2 如何做。它们是高级的、粗粒度的活动描述，使涉众能够从系统中获益。只有在业务目标及其相关涉众的上下文中才能看到功能。

*例题*

*   人力资源人员能够使用该系统为新员工入职
*   卖家可以使用该系统查看实时库存水平
*   最终用户能够使用他们的社交媒体帐户登录系统

作为系统构建者，我们的最终工作是向我们的利益相关者交付能力。

## [](#features)特性

特征描述了系统行为。它与一项能力密切相关，并回答了“我们如何交付这项能力？”

至少，一个特性由一个特性标题和一个场景组成。场景是从一个特定的角度来看特性的功能。例如，如果我们的功能名为“从提款机中提取现金”，我们可能会遇到“现金需求超过账户余额”或“钞票面额与要求的金额不匹配”的情况。

然而，只有标题和场景并不能帮助我们描述功能。一个全功能特性(请原谅双关语)将包括:

*   **功能标题**:简要描述，如提现
*   **用户故事**:我们使用以下模板

    ```
    As an [Actor]
    I want [specific System Behavior]
    So as to achieve [a goal contributing to a Capability] 
    ```

*   **影响**:这个特性相关的影响图的链接(在下一篇文章中会有更多)

*   **注释**:有助于读者更好地理解该功能的任何其他文本

*   背景:(如果适用):共同前提

*   **场景**:以结构化的方式编写

    ```
    Given [a Condition]
    When [an Action is invoked]
    Then [an expected Outcome occurs] 
    ```

有效功能必须至少有一个已定义的方案。

为了获得特性，我们必须考虑我们的系统能力并回答这个问题:“我们必须实现什么功能来提供这种能力？”。

让我们看看“卖家可以使用系统查看实时库存水平”功能。我们如何提供这种能力？我们可以从实现这些功能开始:

*   特性 1:将仓库 API 集成到我们的系统中
*   功能 2:为仓库库存变化创建通知机制
*   特性 3:构建显示当前库存水平的 UI

很明显，在现实生活的系统中会有比这些更多的特性，但是希望你能明白要点:特性是关于实现功能的，能力是关于利益相关者实现目标的能力。我们将在本系列的后续文章中对此进行更多的讨论。
