# 何时不使用工具/范例/工作流程

> 原文：<https://dev.to/niorad/when-not-to-use-a-tool-paradigm-workflow-3dcd>

为一项工作使用正确的工具/范例/解决方案/库/工作流被广泛认为是一个好主意。

这与货物养殖相反，货物养殖意味着盲目地使用一种工具，因为其他人也这样做，而不一定知道他们的原因。

< / obvious >
<点>

每一个销售工具的人通常会称赞他们的工具是适用于广泛用例的银弹。

缺少的是信息，什么时候**不**使用它，或者该工具不是一个好的选择。

最近，我开始欣赏工具作者、供应商、专家和教师，他们清楚地表明…

…我们的书可能不适合您。你应该使用更简单的方法。对于你的项目规模来说，这个库引入了太多的开销。

这种行为可能会阻止一些购买或下载，但是…

…它会立即建立信任。它提供了有用的信息。
…从长远来看可以防止挫败感。

我最近或多或少遇到的正面例子:

*   [《你可能不需要 Redux》](https://medium.com/@dan_abramov/you-might-not-need-redux-be46360cf367)作者 Redux

*   [“何时及为何使用 Docker”](https://www.linode.com/docs/applications/containers/when-and-why-to-use-docker/#when-not-to-use-docker)

我很想知道更多“如果…请不要使用我们的产品”的例子！
