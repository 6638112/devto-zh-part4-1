# 软技能在技术开发人员关系中重要吗？

> 原文:[https://dev . to/drnugent/are-soft-skills-important-in-technical-developer-relations-3105](https://dev.to/drnugent/are-soft-skills-important-in-technical-developer-relations-3105)

上个季度，Slack 直接上市。与其他一些最近的科技公司 IPO 相比，分析师认为 Slack 的上市执行得很好。今天我们和朋美·伊穆拉对话，他是我以前的同事，现在是 Slack 的员工，也是开发者倡导者。

朋美和我一起在一家名为 [PubNub](https://www.pubnub.com) 的初创公司工作，在我认识她的这段时间里，她已经在全球数十个技术会议上发表过演讲或做了主题演讲。朋美能说流利的日语和英语，在任何她能去的地方骑自行车，并创建杀手级的长篇技术博客帖子和演示文稿，简单明了地解释复杂的工程问题。我和朋美谈到了技术技能在 DevRel 工作中的重要性，以及这与软技能的关系。

### 问:技术技能在开发人员关系工作中有多重要？

技术技能的重要性因你的角色和日常任务而异。对我正在做的事来说，这很重要。我确实与 DevRel 内部和附近的人(比如社区经理)一起工作，他们不需要编码技能。然而，我在 Slack 的主要职责是对开发人员进行一般性的教育，让开发人员了解我们正在开发的技术。这些不是像我作为一个开放网络倡导者与 W3C 一起工作时所提倡的一般技术，而是我们希望开发者能够识别更具体的技术并深入理解它们。在我们的情况下，我们需要理解开发人员的喜悦和痛苦。不了解开发者的痛点，就无法提升自己的 API 和平台。为了理解这种痛苦，你必须理解底层的技术，从这个意义上来说，你需要技术化。

这可能特别适用于我和我目前的角色。并不是每个在 DevRel 中寻找工作的人都必须是高度技术性的:在开发者关系生态系统中有许多不同的工作，人们来自不同的背景。就我个人而言，我来自工程部门，但我团队中的其他人来自技术支持、产品和非工程操作角色。拥有一个多元化和富有成效的团队很重要，因为你无法独自运营整个团队，你拥有的任何技能都应该由队友来补充。

### 问:你在 Slack 和之前工作过的公司发布了很多高技术含量的内容，从移动开发到 JavaScript 再到 API 管理。你是如何利用你的专业技能的？

编写教程、创建研讨会材料、网络研讨会、黑客马拉松辅导、编写拉动式请求。我曾帮助编写 SDK 和工具。当然，在 DevRel 中，我们每个人都必须在编程技能之上发展额外的技能。你得有教人的能力。你在 GitHub 上的代码样本不一定是最优化、性能最好或“最好”的，因为在这种情况下“最好”的代码是教人们利用你的服务的代码。编写高性能代码很重要，但有时您需要简化代码，以确保您的受众理解基础知识，然后继续确保您的受众理解性能、优化和安全性。编写人类可读的代码是最重要的因素。与人类一起工作是我们在 DevRel 的工作！

在会议上发言是一个不同的故事:除非你是现场编码，否则你必须用最简单的方式解释你的应用和代码。这绝对是一种不同的技能，一种几乎没有人教的技能，与工程学相比，这是 DevRel 所独有的。

然而，你不一定要精通多种语言。有些人几乎知道每一种编程语言，这很好:但是，嘿，我不知道！我决定专注于 JavaScript，因为我的团队可以补充我的技能，并用其他语言编写。

### [](#q-youve-worked-at-enterprise-companies-and-we-worked-together-at-a-30person-startup-now-that-slack-is-getting-bigger-and-has-gone-public-are-soft-skills-more-important-in-your-job)问:你在企业公司工作过，我们一起在一个 30 人的创业公司工作过。现在 Slack 越做越大，已经上市，软技能在你的工作中更重要吗？

是也不是。Slack 发展得非常快，我不在管理层，所以随着我们团队的扩大，我的职责变得更加集中。我在很多不同的公司工作过，每个公司的运作方式都不一样。不同公司对 DevRel 没有教科书式的定义，因为角色取决于公司规模和产品类型。当我在 PubNub 这样的平台公司工作时，该公司严重依赖 DevRel，我们的风投实际上在观察我们的所作所为——我会收到电子邮件说“我们的一个风投喜欢你的博客帖子！”

当公司成长时，它们也可以变得更加以企业为中心。这造成了巨大的差异，与所有开发人员打交道，而不是专注于企业客户。

无论公司规模大小，软技能都很重要。你的工作不是编码或做你被告知要做的事情——很多时候你必须在与团队成员合作的同时启动、运行和完成整个项目，比如销售或营销，以及其他公司的人。这些都是软技能。不知道没有这些软技能能不能在 DevRel。

就我个人而言，我意识到软技能的一面，因为我觉得我可能会花太多时间编码和创建代码样本。你必须要有一个平衡，有时我会养成写太多代码的习惯，而 DevRel 的工作远不止这些！

随着 Slack 的发展，我们也有越来越多的机会向同行学习，无论是非正式的还是通过公司提供的课程。事实上，我目前正在举办一个实践研讨会，以构建一个 Slack bot，我们所有的新工程师都将在入职期间观看它！我还参加了 Slack 高管培训，我们在那里学习谈判技巧和进行艰难的对话。不仅仅是导师，我还从 DevRel 外部获得指导，并看到这些软技能能对 DevRel 团队产生多大的影响。

### 问:你与聊天机器人广泛合作，处于人工智能和人机交互的交汇点。你在这些领域的工作如何影响你对 DevRel 的态度？

作为我的许多聊天机器人演讲的结论，我说“工程师在机器上编写机器人代码，机器理解你的代码，但最终你是在为人类创造这些机器人。”这也是一个设计原则:理解代码中的人为因素。你为工作发展的软技能不同于这些人类的 UX 技能，但它们是相关的。永远记住你的用户是人。当我在人机界面团队工作时，我的老板是一名设计师，我有很好的机会了解人类心理学并参与研究实验室。我开始了解人们是如何思考的，以及作为工程师，我们并不总是优先考虑用户的想法。

一个关键的提醒:不要做假设。如果你的工程师更关心逻辑和数据，而不是人们如何使用你的产品，那么获取一些数据:与 UX 的研究人员合作，看看你的产品如何在测试指标中失败，如果你的团队更喜欢看数字的话。即使作为一名工程师，您也一定在使用应用程序时遇到过挫折。你必须同样理解用户的沮丧。

在 DevRel 中，如果你不是 API 的用户，你就不应该编写 API。我知道这很难，但是在您真正开始使用它之前，您通常不会看到 API 中的不一致或属性名称的改变。最好的方法是让团队中的每个人都使用它，同时与使用它的人一起工作。API 都是关于用户体验- *开发者体验* -这是一回事！你的产品必须让你的用户/开发者满意。无论你是在编写应用程序、服务还是 API，你都必须关心你的用户，从端点到文档，创造一个可用的系统。

### [](#q-how-did-you-get-into-devrel)问:你是怎么进入 DevRel 的？

起初，我甚至不知道什么是开发者关系。当我在 8、9 年前进入 DevRel 时，我是一名 UX 工程师，在我提到的人机界面团队中编写代码。我在 Palm 和 webOS 一起工作，还记得他们吗？两个家伙加入公司组建了一个开发人员关系团队:Ben Galbraith 和 Dion Almaer，他们曾经写过一个名为 Ajaxian 的博客。(天哪，那是很久以前的事了，没人再说 Ajax 了！)

总之，我喜欢那个博客。我从那里获得了所有最新最棒的 JavaScript 信息和片段。当他们来到 Palm 时，我知道我想和他们一起工作，所以我加入了 DevRel 团队，尽管我不知道我的工作是什么！

那时，我已经在我们的 JavaScript 框架的“厨房水槽”项目上工作，以展示 API 的所有特性。我甚至没有意识到这是一个 DevRel 任务，但 Dion 看了看，说这正是他们在团队中想要的。我还作为用户界面团队的工程师写了一些教程，这在公司内部变得非常受欢迎。从我自己公司的开发人员那里得到这些教程和文章的感谢是令人满足的。我喜欢在 Palm 的工作，当公司倒闭、我失业时，我搬到了诺基亚，从事 Windows Phone (OMG。问我任何关于死亡的移动操作系统！).令人高兴的是，我的很多同事也搬到了那里。从那以后，我在许多初创公司工作过，现在我很高兴能在 Slack 工作。

为了留在 DevRel，我不得不拒绝了一些可能非常有趣(也非常赚钱)的工程角色。)然而，我想做我想做的事情，和我想一起工作的人一起工作，而不是追逐金钱。我喜欢通过 DevRel 帮助人们热爱技术。

### 问:有没有人在 DevRel 中做得很好，你想表扬他？

我马上想到，你在 IBM 的同事，太地町。他以前是个理发师，他在一个乐队里——他和我的背景完全不同，现在我们都在 DevRel！我之前说过 DevRel 中不是每个人都来自工程，他有着*最有趣的*背景！

接下来几个人是在早期 DevRel 职业生涯中影响我的人

Dion Alamer 在我的过去给了我很多关于 DevRel 的机会和想法，也是我第一次开始做 DevRel 的地方。

Estelle Weyl 给了我很多在会议上发言的鼓励——我不想，我非常害怕公开演讲，而且英语不是我的母语。我甚至不知道“冒名顶替综合症”这个术语，即使我有！但是埃斯特尔告诉我“你写的博客很棒，你肯定能做到。”

微软的 Doris Chen 博士，她是我刚开始时唯一的女性少数民族 DevRel 专业人员，所以我们能够分享我们的奋斗。当我刚刚起步的时候，这是一个巨大的帮助。

出于类似的原因，我很高兴我认识了王宣予，她是我在她组织的 SFHTML5 meetup 上认识的，她也是一名 DevRel 专业人士，目前在谷歌工作，还有 T2，在 Mozilla 工作。他们是我灵感的来源。我们有着共同的目标和奋斗目标，是我谈论问题的最佳人选。是啊，做女人，尤其是 PoC，一点都不容易！

Aysegul Yonet 是一名出色的工程师，她可能不在 DevRel org，但她所做的事情非常以人为本，因为她在许多关心贫困背景的人的组织中担任志愿者和组织者，帮助人们获得编码技能。

还有，[贝尔·道格拉斯](%E2%80%9Dhttps://twitter.com/beardigsit%E2%80%9D)，我目前在 Slack 和他一起工作。与我提到的其他人不同，我认识她才几年，然而，她目前是我的 DevRel 生活中最有影响力的人之一，就好像她来自 DevRel 的一个平行宇宙，我以前从未在那里遇到过！我知道这听起来很奇怪，但从她的大脑和视角出发让我着迷。

最近几年我认识了更多的人，我想提一下，但是名单越来越长，所以我就不说了！

朋美，谢谢你和我们坐在一起，分享你的知识。
