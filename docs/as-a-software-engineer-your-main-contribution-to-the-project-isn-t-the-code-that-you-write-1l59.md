# 作为一名软件工程师，你对项目的主要贡献不是你写的代码

> 原文：<https://dev.to/adugamayuba/as-a-software-engineer-your-main-contribution-to-the-project-isn-t-the-code-that-you-write-1l59>

作为一名软件工程师，你对项目的主要贡献不是你写的代码

您的主要任务是塑造软件设计、构建和测试的方式，以便所有子系统能够很好地协同工作，并最终在产品中平稳运行。这需要一个跨越不同项目阶段边界的鸟瞰图，以及对过程、架构、组织和技术的批判态度，这些最终都会对所生产的软件的质量产生影响。当然，你会花相当多的时间编码。这是软件工程的核心。然而，如果你没有花足够的时间来分享知识，设定设计和文档的标准，并优化用于 CI/CD 的工具链，那么这些时间将是徒劳的。

对于软件工程师来说，真正的测试发生在生产中

当我们说真正的测试发生在产品中时，我们并不是说代码不应该在前面进行适当的测试。不言而喻，应该进行强大的测试自动化。然而，自动化这些测试会花费大量的时间和金钱，所以最初确定应用程序的关键程度是很重要的。你不能在每个应用程序上花费无限的资源来编写测试自动化。因此，重要的是平衡客户成本与客户价值和时间盒——与您的项目团队和客户——我们同意在自动化测试上花费多少时间是可行的，以及我们在生产中允许多少时间用于“真实测试”和快速修复。

在实践中，自动化测试可以过滤掉许多错误，但是最复杂的情况出现在野外:当代码在生产中运行时。因此，如果您正在努力开发一个高质量的应用程序，您应该确保当您的代码在生产环境中运行时，您能够对错误做出快速反应。这就是开发运维方法至关重要的原因。通过持续监控和记录输入数据，可以在早期发现意外行为，团队可以在这些行为成为真正的问题之前采取行动。DevOps 允许较短的开发和测试周期，使您的应用程序可以灵活地进行更改和改进。换句话说:如果你的工作方式是敏捷的，并且你已经为快速的变化周期做好了准备，那么在生产中进行测试可能是一种公认的最佳方式。

作为一名软件工程师，你的工作就是让自己成为多余的人

这意味着你交付的工作应该有这样的质量，未来的工作要么自动离开，要么随时转移给同事。当然，以一种可能让你失业的方式工作听起来很有争议，但这也意味着你的雇主和/或客户可以在不同的项目和解决方案中灵活运用你的技能。在另一个项目中突然需要你的专业知识，任何优先事项的转移——将当前工作转移到待办事项中——甚至任何突然的离开或缺勤都不应该阻止同事从你离开的地方继续工作。

这对你的日常活动意味着什么？除了交流和记录你所做的决定，你所交付的软件应该是经过深思熟虑的和可以理解的设计和制作的。它应该清晰地编写和记录，应该有一致的风格、格式和语法，应该有适当的结构和版本。简而言之:这意味着编写文档完善且易读的代码。你越擅长成为多余的人，你对你的雇主或客户就越有价值。
