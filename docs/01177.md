当编写软件时，很容易假设发布的任何东西都能解决问题。毕竟，这就是代码的全部思想——自动解决特定的问题。但是代码本身并不这样做。当你将代码产品化时，它会变得更加复杂。所有这些都适用于围绕新技术设计的代码。人们很容易对技术感到兴奋，并构建人们实际上不想要或不需要的解决方案。即使他们确实需要解决方案，也有可能人们不想要它。在这种情况下，您的代码不仅要解决问题，还要弥合需求和期望之间的差距。

对于我的公司，simple id——一个 Web 3.0 开发工具套件——来说，我们陷入了最后一个困境。我们需要解决一个问题，但我们需要以一种让最终用户能够集体采用通过 SimpleID 构建的解决方案的方式来解决。在这个问题上，我们并不孤单。这个问题适用于 Web 2.0 和 Web 3.0 的产品创作者。

让我们来看看我所工作的市场:Web 3.0。Web 3.0 本质上是去中心化的 Web。用户控制数据的网络，而不是大型科技公司。在 Web 3.0 领域，有许多解决方案提供非托管、完全分散的体验。他们是伟大的，他们是需要的。但是对于这些解决方案，我们要问的问题是:

#### [](#who-is-the-customer)谁是客户？

如果您构建的解决方案适用于极少数愿意经历使用您的解决方案的复杂性的 cypherpunks 和开发人员，那就太好了。但是，如果你像 Web 3.0 领域的许多人一样，想要吸引更广泛的互联网用户群，你必须让你的解决方案对那个市场有吸引力。

换句话说，你必须缩小差距。

我们在 Web 3.0 领域中的许多人试图解决的问题是相似的:

*   隐私
*   安全性
*   数据谱系关系
*   身份控制

不同的是解决这些问题的方式。在今天的几乎所有情况下，围绕构建满足上述需求的应用程序的工具给现代 web 用户带来了前所未有的负担。我在这里说现代网络，是因为肯定有人能够指出万维网早期的一个场景，在那里人们为了使用新技术承受着极大的认知负担。

所以，再一次，当在 Web 3.0 空间中构建时，问问你自己你在为谁构建。如果你是在为一个愿意承担使用协议或产品的额外负担的未来客户构建，那么就这样做吧。那是必要的。如果你正在为那些最终会找到使底层协议和工具变得方便的方法的开发人员构建，那么就这样做吧。那也是必要的。但是，如果您今天正在为企业和消费者构建，请考虑您提供的解决方案。

它真的需要非监禁吗？它真的需要没有信任吗？你提供的东西好到你可以合理地要求人们做更多的工作并付给你报酬吗？或者，有没有一种方法可以给人们足够的权力下放和获得权力下放的技术来解决上面要点中的问题？

无论是在 Web 3.0 领域还是 Web 2.0 领域，都有必要经常问自己谁是你的客户。为他们而建，而不是为自己。你的理想很重要，但是如果你试图为观众建立，他们的理想更重要。

你的产品只有被采用才是好的。当然，收养是相对的。但无论如何，你都要看到人们在使用它。否则，很可能你没有解决客户需要你解决的问题。
