# 进入开源维护的五点思考

通常情况下，我不会在周日写博客，更不用说发布了，因为我通常会花时间和家人在一起或者去教堂。今天有点特别，因为天气原因我滞留在达拉斯，导致我错过了昨晚回家的转接航班。如你所见，我现在有点累了:

当我在这里的时候，我注意到今天是 8 月 25 日，在大多数情况下，这个日子并不特别引人注目。不过有一点值得我注意——6 个月前的今天，我在 github 上打开了[我的第一个拉请求](https://github.com/mikaelbr/node-notifier/pull/271)到 [`node-notifier`](https://github.com/mikaelbr/node-notifier/) 项目！在合并拉取请求的过程中，后来又返回并推送到我们现在正在开发的 v6 分支，这促使 Mikael(项目负责人)让我作为贡献者提供帮助，这是我以前从未考虑过的。由于这是我第一次涉足真正的开源，我想记录一些想法，希望对其他人有所帮助。我所说的真正的开源是指一个实际上在自身之外被消费的项目(根据 github 的数据，有 150 万人依赖于 node-notifier)，而不是像我的个人网站那样的东西，它在 github 上是开源的，但除了我之外不影响任何人。

进入开源维护领域后，我有 5 个主要的想法:

1.  问题时有发生，不可能总是立即解决
2.  花时间清楚地交流可以带来巨大的不同
3.  项目和我的时间的优先级
4.  跳出来帮忙
5.  移情是关键

## Github 问题

当 Mikael 提到他需要一些帮助时，我试图解决的第一件事是人们在 github 上提交的开放内容的问题队列。[正如你在存储库](https://github.com/mikaelbr/node-notifier/issues)上看到的，还有很多未解决的问题。我通常努力让我的电子邮件达到零收件箱，因为我已经这样做了一段时间，所以每天保持这种状态是相对容易的。但是，当我进入开源领域时，我不得不接受的第一件事就是它的无限性。在大多数情况下，消费者可能会提出无穷无尽的问题，特别是在各种情况下使用的项目中。如果在我写这篇文章的时候，你刚好有 52 个问题中的一个尚未解决，并且它仍然影响着你，请随时[在 twitter](https://twitter.com/jnielson01) 上联系我，我会在有时间的时候看看它。一旦我意识到，如果没有一些认真的专门时间和大量的沟通，这些问题可能永远不会出现在“零收件箱”中，我就决定快速响应新的(或更新的)问题，因为这些人更有可能做出回应，并且相关的上下文仍然可用。

## 沟通清晰

我很快意识到的下一件事是，当我对公开的问题进行分类时(直到我决定采取我上面决定的方法)，花一些额外的时间重读我写的文字并确保它们传达了我想要的东西是多么重要(这是我通常不会为博客帖子/推文/电子邮件等做的事情)。因为我发现在我使用的其他项目中遇到老问题有很多好处，所以我希望我写的回复对那些可能偶然发现这个问题的人有用。但是，更重要的是，很多在 Github 上交流的人并不是把英语作为他们的第一语言，而是尽他们所能。有时会有误解，但尝试使用清晰简洁的语言会帮助别人，虽然他们可能不会为此感谢你，但知道你付出了一点点努力让你的回应更清晰会给你留下美好的感觉(希望如此，因为我也是)。让我的沟通更加清晰是我需要继续努力的地方，但是我觉得我已经取得了一些进步。

## 区分工作的轻重缓急

自从被邀请合作这个项目以来，我一直在想的另一件事是优先化的想法。这如何适应家庭/工作/博客/等等？提出的问题与那些事情相比如何？这些问题相互之间的排名如何？一个开源项目如何决定接受什么或者做什么？我不知道这些问题的答案，但当我试图在这个新的生活领域寻找出路时，我发现我不一定需要明确的答案。优先级是一个不断变化的选项和权衡的海洋，今天可能是最重要的事情明天会变得不那么重要。因此，我发现每周花一两个小时在这上面有助于我保持脉搏，而不会感到不知所措。我目前工作的另一个好处是，我可以从事我们公司使用的开源项目，因为我们在整个 JavaScript 项目中使用`Jest`，而`Jest`使用`node-notifier`，我可以很容易地证明偶尔从其他工作任务中抽出时间来处理这个项目的一些事情是合理的。到目前为止，在比较其他问题以确定实际上应该做什么时，我倾向于优先考虑最近的问题，因为这些问题最有可能仍然是问题。这一领域将继续改进，但我认为一个可以带来重大回报的方法是向`node-notifier`澄清我们希望支持哪些平台/节点版本，并确保我们尽可能通过统一和一致的 API 达到最佳效果。

## 直接跳进去

与优先化相关的是跳进去的想法。有时候，尽力而为是有用的。在这种情况下，我试图直接参与并帮助解决未解决的问题，以了解项目中可以改进的地方，因为在我使用它时，我正在处理的东西看起来状态良好。但是，我决定我能做的最有用的事情就是深入研究这个问题。我通常会尝试在我的一台机器上重现这个问题，这并不是超级可持续的，但对于一些初始测试来说是有效的。最困难的事情之一是确保我的沟通(如上所述)清楚地表明我不是专家，所以这只是一个建议。虽然我在这一点上对这个项目有一些扎实的知识，但我不知道过去所做选择的历史或背景。所以，我尽我所能为开题人提供一些指导，并在我遇到问题的时候把问题留给 Mikael。

## 换位思考是关键

当我思考这种开源项目的增加时，我的最后一个想法是，有人维护几十甚至几百甚至可能几千个开源项目是多么令人印象深刻。我个人在任何时候都不渴望做比一对夫妇更多的事情，特别是现在我更熟悉积极维护一个项目的工作量。虽然这不是一个巨大的项目，但它仍然是一个我发现在我的编程中有用的项目，并且我想为它的其他用户做得更好。在这一努力中，我对其他开源项目的维护者产生了更多的共鸣，以及他们付出了多少难以置信的努力，才让这个社区有了由了不起的人创造的一些不可思议的资源。

随着我继续在开源社区工作，我有许多目标和事情要继续努力，但这可能会成为以后的帖子:)感谢阅读！如果您有任何反馈，请告诉我，我很乐意听到您进入开源的经历(希望是积极的)!
