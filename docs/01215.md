# 科技面试流程有什么问题？

> 原文:[https://dev . to/remote synth/what-s-mist-with-the-tech-interview-process-3b3m](https://dev.to/remotesynth/what-s-wrong-with-the-tech-interview-process-3b3m)

越来越多的人似乎认为，科技公司在招聘技术人员(尤其是开发人员)时通常采用的面试程序被打破了。大约一年前，我在一篇讨论当今科技面临的问题的[帖子中把它作为一个主要问题，可以说，从那以后它就没有被解决过。](https://dev.to/remotesynth/is-today-the-best-time-to-be-a-developer-nobut-were-getting-better-5ik)

人们甚至不需要找很远就可以在 Twitter 或各种博客的黑客新闻中找到对它的讨论。这些问题似乎可以归结为三点:

1.  编码测试是任意的，不必要的困难，并且与工作实际需要的技能脱节。
2.  面试的轮数和时间要求很难管理。
3.  招聘决定往往看起来很武断，而关于某人为何未能通过某个阶段的沟通，即使有沟通，也往往沟通不畅。

让我们仔细看看其中的每一项。

## [](#coding-tests-and-whiteboards)编码测试和白板

让我们从编码测试和白板开始，因为这可能是开发人员引用的最常见的挫折。

开发人员对编码测试表达的主要不满与它们不必要的困难和经常的武断有关。许多人描述了被问到复杂的计算机科学概念和算法，他们可能在大学里学过(假设他们是 CS 专业的)，但此后没有参考的需要(或者如果他们有，他们只是查一下)。

> 我不应该在面试中依靠算法问题抽奖来展示我的技能和能力。运气和机会不应该成为技术面试过程的一部分...这就像测试你的牙医的技能，基于他们使用大学化学研究中的氧化数方法平衡氧化还原方程式的能力。
> 
> *   萨哈特·雅尔卡博夫，[如果***你，我辞职——招聘中断](https://medium.com/@evnowandforever/f-you-i-quit-hiring-is-broken-bb8f3a48d324)

代码测试或白板的部分问题是，它似乎与设计筛选的工作不相关，并且似乎经常作为一种把关方式被包括在内。

> 这些人是如何确定我在技术上不够强这一确定的事实的呢？想猜猜吗？如果你说，“在 codewars.com 上找到的带有随机脑筋急转弯类型问题的现场编码测试”，你就对了。不要误解我，我也通过了这类测试的面试。嗯（表示踌躇等）..这不是很奇怪吗？
> 
> *   安布罗斯·利特尔[技术面试坏了，但我们能修好它](https://dev.to/ambroselittle/technical-interviewing-is-broken-but-we-can-fix-it-4964)

对我来说，编码测试有两个主要问题。第一个是这些编码测试和设计它们的假设目标之间的脱节，假设目标是确定受访者的技能和知识。在我看来，大量的工作——无论是他们的教育、在公司的工作、对开放源码的贡献还是附带项目——比完全脱离上下文的随机代码挑战更能说明一个人的技术技能。作为一名开发人员，回答随机的技术琐事并没有什么特别独特的地方，但这似乎是这个角色特有的做法。

更重要的是，第二个问题是实践优先考虑随机的技术知识(相对容易学习的东西)，而不是其他方面，如性格适合度、与他人合作的能力、职业道德、学习的渴望(不容易学习的东西)。你可以是一个出色的程序员，也可以是一个蹩脚的人，但我可以告诉你，等式的哪一部分将对你在开发工作中的成功产生更大的影响。

## [](#extensive-time-demands)广泛的时间需求

技术面试通常是马拉松。除了耗时的“家庭作业”之外，许多角色还需要大量的面试。

这似乎并不是那些家喻户晓的大型科技公司的专利。总的来说，你会听到科技公司对待这一过程的故事，就好像它旨在淘汰那些根本无法在激烈的竞争中坚持到最后的候选人。

> 越来越多的雇主正在采用无意义、昂贵且耗时的额外流程，使得他们的招聘渠道变得更加缓慢，让应聘者更加失望。
> 
> *   丽兹·瑞安，[不，你没有疯——招聘程序已经被破坏了](https://www.forbes.com/sites/lizryan/2018/04/12/no-youre-not-crazy-the-hiring-process-is-broken/#1109c2db4751)

缓慢或完全中断的沟通过程加剧了这种情况。

> 我经历了半天到一整天的名为“挑战”的面试，只等了一个月他们说不或者什么都不说。这是过程的一部分，我们都会克服它。(耸耸肩)然而，当你不得不利用宝贵的假期时间去参加毫无结果的面试时，这是令人沮丧的。
> 
> *   疯狂的求职过程

通常情况下，对于没有继续面试的候选人来说，沟通障碍似乎已经消失了，这对被晾在一边的候选人来说肯定是不公平的。然而，即使对于进入下一阶段的候选人来说，沟通也很差或根本不存在，但往往要等几周才能沟通。

对于至少以两种方式招聘的公司来说，这似乎完全是弄巧成拙。首先，它毫无意义地将你的潜在候选人限制在那些财力和/或工作允许他们度过无数个必要的小时和天的人。第二，它错过了退出这个过程的候选人，因为他们要么对缺乏沟通和过程的长度感到沮丧，另一个当然是在一个不需要这么长时间的过程中找到不同工作的人。

## [](#arbitrary-hiring-decisions)武断的招聘决定

最大的问题似乎来自前两个问题，即拒绝可能看起来完全是任意的——以至于有一个专门分享拒绝故事的网站，里面充斥着开发人员社区中可识别的名字。

液体错误:内部

甚至许多参与招聘过程的人似乎也意识到这是有缺陷的。

部分问题源于筛选过程(这是[上的故事拒绝的。us](https://rejected.us) 经常验证的)，根据下面引用的关于 Y Combinator 招聘的研究，这占了申请人拒绝的近一半。

> 大多数公司在招聘电话(或简历筛选)中会拒绝很高比例的申请人。在我们采访的 25 家公司中，平均有 47%的申请人以这种方式被拒(个别公司的这一比例高达 80%，低至 0%)。招聘人员这样拒绝是非技术性的。他们所能做的就是拒绝那些不符合他们被教导要寻找的特征的候选人。
> 
> *   Ammon Bartram，[Y Combinator 公司想要谁](https://data.triplebyte.com/who-y-combinator-companies-want-c1880a08ac88#.8dspa9u3p)

正如这项研究指出的，在许多情况下，这是因为筛选者通常不是技术人员，因此必须依赖他们提供的个人资料。这并不是说筛选者应该是技术型的，我相信包括受过培训并专注于招聘的人有很多优点，但这意味着我们需要做得更好，以确保他们了解概况，并获得他们正确评估员工所需的信息。

另一个问题是回到编码测试。它们看似琐碎的本质似乎会导致不可预测的结果。基本上，正如下面的研究指出的，好的候选人失败的几率很高。

> 大约 25%的受访者表现稳定，其余的则各不相同...
> 
> 对我来说，看着这些数据，然后假装我不得不根据一个面试结果做出招聘决定，感觉就像从一个钥匙孔窥视一些漂亮、布置豪华的客厅。有时你会看到墙上的一件艺术品，有时你会看到酒的选择，有时你只会看到沙发的背面。
> 
> *   技术面试的表现有点武断。这是数据。

筛选、测试和时间的结合可能意味着公司的招聘流程不一定是为找到最佳候选人而优化的。

## 这不是一个容易解决的问题

> 听着，我明白。面试一个人需要时间和努力，而你们中的大多数人只想回去做些事情。想出一个标准的问题可以让你事半功倍，并且给你一点比较不同候选人的能力。
> 
> 但是真的要花很长时间看看这是否选择了正确的候选人。
> 
> *   扎克·霍尔曼，[开始面试是 F****d](http://zachholman.com/posts/startup-interviewing-is-fucked/)

正如扎克指出的，这是一个难以解决的问题，这可能解释了为什么该系统目前似乎是由“我们一直这样做”和“其他人也这样做”的组合来支持的。这种推理给了公司借口，他们需要避免仔细检查他们的过程，看看他们是否真的实现了他们应该实现的目标。大多数公司似乎认为，在最坏的情况下，这个系统给了他们大量的假阴性，而不是假阳性。

看起来公司会受益于问自己:

*   我们的代码测试或白板练习真的能提高我们识别优秀候选人的能力吗？或者有没有更有用的相关练习(与日常工作要求相关的合理的家庭作业)或标准(可以代替测验或家庭作业进行评判的项目经验或开源工作)？
*   我们真的需要一整天的面试马拉松或面试过程的 5 个以上阶段吗？有没有一些阶段我们可以省去，但仍然同样有效？我们正在失去优秀的候选人，仅仅是因为他们没有能力完成我们的流程吗？
*   我们的筛选人员是否获得了正确筛选人员所需的信息，或者我们是否有可能失去高素质的候选人？我们的标准是否过于具体(例如当 JavaScript 或 JavaScript 框架经验足够时，特别寻找 Angular)？我们的流程是否会导致高度的假阴性？

毫无疑问，这些问题并不容易回答，而且很明显，许多优秀的候选人仍然设法通过了考试。但是我是这样想的:作为开发人员，我们开发了单元测试系统来帮助防止我们部署错误的代码。这些测试并不总是检查会导致整个应用程序崩溃的问题。根据特定的标准，有些可能会导致软件在相当长的时间内或对相当大比例的用户不正常工作。然而，我们不会允许我们的开发人员部署没有通过这些测试的代码，仅仅因为应用程序可能在某个百分比的时间里对某些用户有效。我们的面试和招聘系统目前没有通过大量的单元测试。让我们修理它。
