# 来自 Vue 新功能 API RFC 的 3 个关键见解

> 原文：<https://dev.to/kball/3-key-insights-from-vue-s-new-functional-api-rfc-5467>

6 月 7 日，尤雨溪(Vue.js 的创始人)发布了一篇关于 Vue.js 的基于 Hooks 的函数式 API 的 RFC。

这也引发了相当多的讨论和一些反弹。我不打算深入讨论，但从我的角度来看，Vue 团队在听取反馈并利用反馈改进 RFC 方面做得很好。

我要说的唯一一件事是**我希望那些选择称重的人能更尊重一点**。开源维护者为了公众利益做了大量的工作，像 RFC 这样的东西在那里给你一个机会来权衡并帮助指导他们什么是最有用的。他们是*而不是*让你在那些维护者的工作或努力上拉屎。

猖獗的负面情绪在我们的社区中无处不在(HN 和 Reddit 都被笼罩在其中)，在我看来这是有害的。如果你发现自己要做出非常负面的评论，停下来，想想如何建设性地表达你的反馈，正如老话所说，“如果你不能说些好话，那就什么也别说”。这并不是说不要不同意，而是要尊重那些和你说话的人。

总之，回到 RFC。

我认为这个 RFC 值得探索，不仅仅是因为它本身，还因为它揭示了关于前端开发的未来的 3 个关键见解。

## 1。可重用状态管理的未来是可组合的功能

通过允许无缝组合独立开发的部分，面向组件开发的趋势对于前端世界的生产力是显著的。这使我们能够创建可重用的构建模块，我们可以在整个应用程序中根据需要插入这些模块。

然而，将这种方法应用于状态管理的努力是笨拙而冗长的。像高阶组件(HOCS)这样的工具引入了极度的嵌套复杂性，而混合很难组合，可能导致名称空间冲突，并且在试图理解属性来自哪里时不透明。

React Hooks 引入了一种新的方法，极大地提高了使用函数的有状态逻辑的可组合性，这个 RFC 看起来可能会更进一步，因为它设法消除了 Hooks 中一些更不直观的挑战(比如调用顺序敏感性和不能在条件中使用它们)。

## 2。类型对于框架的设计越来越重要

提案中强调并贯穿大部分讨论的激励因素之一是，基于函数的 API 使得类型检查和类型推断比其他方法简单得多。

npm2019 年的一项调查发现，高达 62%的 NPM 用户(事实上几乎所有 JavaScript 用户)都在使用某种形式的 TypeScript，其中 42%的人直接编写 TypeScript，其他人使用 TypeScript 编写的库和代码。这比前一年的 48%有所上升。

趋势线很强，各种类型的人似乎都会留下来。

## 3。框架沿着共同的方向共同发展

我喜欢读这本书的一个原因是，想法是如何从一个框架转移到另一个框架的。

这是我认为选择哪个前端框架并不重要的原因之一——最好的想法会被所有的框架采用，并且采用曲线会越来越快。

对此有一些批评；反对 RFC 的部分原因是它与 Vue 传统的工作方式太不一样了。但在我看来，交叉极化是一件好事。

这也有助于减轻选择框架的压力。只要你学得足够深，掌握了底层的概念，你就能根据需要快速跳到其他框架。如果你特别喜欢其中一种的“味道”，你就不会被任何重大创新遗漏，因为它们很快就会传播开来。

## 包装完毕

这个 RFC 让我对前端开发的未来感到无比兴奋。这不仅仅是因为我喜欢 Vue.js(尽管我确实喜欢)，还因为它强调了我们作为一个行业如何在自我改进方面做得越来越好。创新突破在整个生态系统中迅速传播、改进和采用。未来的确非常光明。
