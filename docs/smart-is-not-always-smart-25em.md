# 聪明并不总是聪明

> 原文：<https://dev.to/golangch/smart-is-not-always-smart-25em>

今天我查看了我在前任 codementor 页面上的一个或多或少有点旧的个人资料，现在是 arc()。

我看到我把我的经历列成了这样:
[![Alt Text](img/469b1ac2117c9e20a75a9aafd514d7d0.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--eY-fp679--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/7aerqyr0c02vwlof2hyf.png)

好的...没错，但我不想在这里列出所有的项目，因为我更关注 Go 和 JS 项目，而不是更多。NET/C#

好吧，所以我挖编辑个人资料部分和技能设置，并发现，我不能定义这 5 个列出的经验技能。

但他们有一个非常聪明的解决方案，这是技能的列表，其中配置文件所有者拥有最丰富的经验。

这有道理吗？
嗯。我有 4 年多的围棋经验。Go 存在 7 年左右(v1+)。但是我从 HTML 开始然后。净 20 年前和 12 年前...

因此，这种智能解决方案可能是正确的，开发人员或采购订单认为它应该工作。但这可能不是我这个老家伙想在这里看到的。想象一下，我列出了 20 年的 Cobol，25 年的 Pascal 等等，但我知道并想使用的“新”语言都不会在这里列出。

这并不是对 arc()解决方案的指责。他们试图做一些聪明的事情，这可能对年轻人有用，但对老年人就没那么聪明了(在这种情况下)。

有时我们试图让事情变得像开发者或采购人员一样聪明。因此，当我们做一件聪明的事情时，我们真的必须在所有可能的情况下检查它是否聪明。

现在...就我个人而言，我也在考虑那些试图让一切都变得非常智能的框架。这基本上是可行的，但是它可能是危险的，或者会成为一个问题，因为你必须马上处理它，这对于特定的情况来说并不太明智。

斯特凡，干杯
