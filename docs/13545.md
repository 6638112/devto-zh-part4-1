# 嘿，现在我知道反应了，那接下来呢？🧐

> 原文:[https://dev . to/mi Jim/hey-now-I-know-react-so-s-next-1k 7](https://dev.to/mijim/hey-now-i-know-react-so-what-s-next-1k7)

作为一名初级软件工程师，我最近一直在专业领域内从事一些 React 项目，我想分享我学到的一个重要教训。

有时，当我们是初学者时，我们通过跟随页面教程开始学习一个新的现代框架，创建我们的第一个项目，咨询 API，创建新项目，搜索优秀的组件，咨询 API，创建新项目，搜索优秀的组件，咨询 API，创建新项目，搜索优秀的组件，搜索优秀的组件...内部错误:太多递归。

那么下一步是什么？最大的问题是认为我们将要创造的比其他解决方案更糟糕。

-🤨好的，我们的应用程序需要一个日历，我应该创建一个新的日历组件吗？
-😎🍹嗯，不，我搜索了一下，发现了一个非常好的日历，非常完整而且容易使用。这会节省我们很多时间。

仅此而已，使用其他组件将节省您的时间，至少在开始时是这样。

-🤨嘿，客户需要在应用程序的这一部分选择一个日期范围。
-🤯哦，我们的日历不允许，我会试着找一个新的。

我们都知道什么是时间压力和截止日期，但是强迫你、你的团队或者你的公司花多一点时间来创建通用的可重用组件，这些组件通过项目扩展，关注新的需求和解决用例，将会节省你的时间。这还不是全部，你的思维会以一种更抽象的视角看待解决方案。我们必须遵循这些简单的组件设计规则:

*在每个新项目上扩展组件。
*始终向后兼容。
*始终带测试。
*风格必须尽可能修改。
*保存好文档。
*与社区分享。

最后一条规则是我认为编程最美妙的部分，看到其他开发人员使用你的方法/解决方案/组件，并从中受益。

-🤬等等，我们不会免费分享我们的代码，我付钱给你。

这是典型的没有大商业视角的老板思维。通过分享你的代码，你正在建立一个品牌，你扩大了你的联系，你在你的公司内外创造了知识，你得到了更少的错误，当然，你没有损失金钱(相信我)。因此，如果你的公司在这方面有点过时，试着尽可能改变它，好处是多方面的。
