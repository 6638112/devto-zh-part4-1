# DevOps 是一种文化，而不是一个角色

> 原文:[https://dev.to/k4ml/devops-is-a-culture-not-a-role-27nk](https://dev.to/k4ml/devops-is-a-culture-not-a-role-27nk)

开发人员和运营人员之间的传统问题是，为了实现组织的目标，他们的目标相互冲突。开发人员需要快速构建和发布新功能，以支持用户的需求，而运营人员需要通过最小化系统更改来确保系统可靠性。

DevOps 文化试图解决这个问题，这样两个角色就可以一起工作，而不是相互对立。为了更好地理解这种文化，我们需要先看看这两个角色的职责是什么。

软件工程(SE)的主要角色是设计、构建、测试和部署应用程序，以便我们能够向客户交付 it 即服务。

而运营工程(OE)角色可描述如下

1.  OE 的主要职责是确保我们的服务以可靠和安全的方式正常运行，以便我们的客户能够使用它来满足他们的业务需求。
    *   定义和测量可用性指标。
2.  OE 还负责为 SE 团队完成开发任务提供支持环境。这包括:-
    *   提供高效的开发和测试环境。
    *   提供了一种从开发到生产的无摩擦和连续的方法。
3.  跟踪停机和事故，以及它们对业务的相关影响。确保所有停机和事故都及时妥善解决，并充分报告对我们的客户产生影响的内容、时间、原因和方式。

要了解有效运行经验的更多内容，可以参考这篇文章- [高度成功的运营从业者的 8 个习惯](https://blog.newrelic.com/2017/06/07/successful-operations-practitioners-habits/)。一般来说，运行经验应该是这 3 个技术领域的专家

1.  低级系统知识。
    *   知道如何调试低级系统问题。
    *   了解网络的基本原理。
    *   了解 Linux 内核的复杂性，了解地球上的每一个 Linux 命令(好吧，也许不是全部，但你明白了)。
    *   交叉操作系统知识，了解权衡以及它们更适合什么。
2.  高级分布式系统知识。
    *   了解整个生产系统如何配合和协同工作。
    *   知道如何设计可伸缩性和可靠性。
    *   知道事情应该如何部署、监控和记录。
3.  有扎实的编程功底。
    *   对多种语言相当了解，能轻松编写应用和工具供开发者使用。

我更愿意将“devops”视为一种文化，而不是具体的角色。正是在这种文化中，我们将角色定义为软件工程师和运营工程师，以适应文化的意图。其中一种文化可以从 SE 和 OE 的职责中看出。

如上所述，SE 角色包括将应用程序部署到生产环境中。这是与传统设置的一个区别，在传统设置中，部署是一个操作角色。但与此同时，OE 的角色还包括“提供一种无摩擦且持续的方式，将应用程序从开发推向生产”。因此，虽然将代码部署到生产环境是 SE 的责任，但是 OE 必须提供最有效的方式来完成这项工作。在这里，我们应用 80/20 法则，每个角色负责 80% + 20%的关键领域。

## [](#conclusion)结论

没有特定的 DevOps 角色。相反，我们建立了一种文化，在这种文化中，开发人员和运营工程师一起工作来实现组织的目标。这就是 DevOps。
