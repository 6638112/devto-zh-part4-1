# 克服为开源做贡献的障碍

> 原文:[https://dev . to/isabelcmdcosta/overriding-blocker-about-contribution-to-open-source-2ojf](https://dev.to/isabelcmdcosta/overcoming-blockers-about-contributing-to-open-source-2ojf)

[![woman on pc emoji](../Images/de4e4b0853deebfe5f194edbb408bcec.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--wQLhZAHe--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/efe3l1juu0tkp3h1cyzg.png)

为开源做贡献可能会非常令人生畏。早在 2016 年，我就已经知道了贡献开源软件的想法。但直到 2017 年 11 月才真正开始。这是我第一次投稿的时候。尽管我那时就开始了，但在被谷歌 2018 年代码之夏接受后，我学到了很多关于开源的知识。

## [](#my-first-open-source-contributions)我的第一次开源投稿

[![woman juggling emoji](../Images/39d651956b378ed88a975ea3a8a72bb8.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--KUHHcNmI--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/18hhxxlv2aivws9uq6vv.png)

有一段时间，我想为开源做贡献，但从未真正采取行动，直到我看到一个名为 [24 Pull Requests](https://24pullrequests.com/) 的倡议，该倡议是关于在 12 月份回馈开源。

在 2017 年的这个虚拟活动中，我做了我的第一个开源贡献，将一个求职板网站添加到一个由 markdown 文件组成的资源库中，该资源库为寻找工作的人提供资源。我的贡献在我为 [fvcproductions/hire-me](https://github.com/fvcproductions/hire-me) 项目提交的[拉动请求#22](https://github.com/fvcproductions/hire-me/pull/9) 中。

然后在 2018 年初，我开始为一些项目做出贡献，因为我试图申请[谷歌代码之夏](https://summerofcode.withgoogle.com/) (GSoC)，做出这两个贡献:

*   我向 [opendatakit/collect](https://github.com/opendatakit/collect) 项目提交了 [pull request #1986](https://github.com/opendatakit/collect/pull/1986) ，在那里我从另一个贡献者的贡献中获得了工作并完成了这个问题。这涉及到修改 Android 应用程序上的一行文本。

*   我创建了向 [systers.io](http://systers.io) 网站报告改进的[问题#167](https://github.com/systers/systers.github.io/issues/167) ，然后提交了[拉动请求#168](https://github.com/systers/systers.github.io/pull/168) ，更改了“联系我们”页面上 Slack 社区的文本。

所以我从小步开始。然后在 2018 年 4 月，我加入了谷歌代码之夏 [Systers 社区](https://github.com/systers)，在那里我能够了解更多关于开源的知识。

## [](#blocks-to-contribute-to-open-source)区块贡献开源

[![woman facepalming emoji](../Images/919d8531446262ed2e2c14bbb864d9e5.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--9Aba9Oku--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/h8qt0ymxvlrk2svigq8s.png)

因此，当我参加谷歌代码之夏并试图找到让社区参与我正在从事的[导师系统项目](https://summerofcode.withgoogle.com/archive/2018/projects/6592097335377920/)的方法时，我想起了最初是什么阻止了我为开源做出贡献。

然后我为 Systers 社区的新成员做了一个简短的调查，这样我就可以了解其他成员对开源做出贡献的一些阻碍。

在阅读了给出的答案和我个人经历中的阻碍因素后，我收集了这些阻碍因素:

*   如何以及从哪里开始贡献；

*   没有看到许多初学者的问题；

*   对自己的技能不自信；

*   有些项目看起来太吓人了；

*   没有时间为开源做贡献；

*   如果我不能完成自己承诺的事情怎么办？

## [](#overcoming-blocks-on-contributing)克服阻碍贡献

[![woman raising hand emoji](../Images/40d82bfd54de34919e765c58a8176b37.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--R8FxueDS--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/5h7y0xp1u9u55pcjjg7v.png)

因此，在研究了收集到的常见障碍后，我开始思考它们，看我是否能找到克服它们的方法。这是我对这些阻碍因素的看法。

### [](#how-and-where-to-start-contributing)如何以及从哪里开始贡献

你可以从一些计划开始，就像我对 [24 个拉取请求](https://24pullrequests.com/)所做的那样。其他一些例子有[谷歌代码之夏](https://summerofcode.withgoogle.com/)、[outreach](https://www.outreachy.org/)和 [Hacktoberfest](https://hacktoberfest.digitalocean.com/) 。

您可以找到开源社区，这些社区的项目有一致的组织和上下文，还有一个贡献者社区可以帮助您入门。找到这些的一个好方法是在谷歌代码之夏查看被接受的组织。这些组织可能已经为你准备好了项目。我在这里找到了像[开放资料包](https://github.com/opendatakit/)、[开放食品信息](https://github.com/openfoodfacts)和[系统](https://github.com/systers)这样的社区。

关于寻找项目，你可以查看你经常使用的项目(例如:开源库、社会影响项目、你使用的任何软件等等)。你可以查看标有“仅限新手”、“良好的第一期”和“需要帮助”的问题，因为这些通常用来表示这些问题对初学者友好。你也可以查看[待价而沽](https://up-for-grabs.net)，它有一个针对新贡献者的项目列表，还有[代码分类](https://www.codetriage.com/)。

在 [opensource.guide](https://opensource.guide/) 的[“如何为开源做贡献”](https://opensource.guide/how-to-contribute/)上有一个很牛逼的指南，你也可以看看。

### [](#not-seeing-many-beginner-issues)新手问题不多

正如我之前所说的，留意那些被标记为“仅首次发布”和“好的首次发布”的问题，因为这些是专门针对新贡献者的。例如，在[systems/mentorship-back end](https://github.com/systers/mentorship-backend/issues?q=is%3Aissue+is%3Aopen+label%3A%22First+Timers+Only%22)和[systems/mentorship-Android](https://github.com/systers/mentorship-android/issues?q=is%3Aissue+is%3Aopen+label%3A%22First+Timers+Only%22)项目中，只要有可能，我就会给新人贴上这样的标签。尽管这些标签很有帮助，但不要把自己局限于它们，因为有些问题可能没有这些标签，但对你来说仍然是可行的。

你总是可以提出新的问题，如果它们是合理的，那么就为它们做出贡献。这正是我对 Systers.io 网站所做的贡献。我创建了一个问题，然后解决了它。

如果你真的找不到你认为可以解决你正在研究的项目的问题，试着从其他项目开始。

### [](#not-being-confident-with-your-skills)对自己的技能不自信

自信是随着时间而来的…你不需要知道所有的事情。作为一名软件开发人员，我想开始编写实际的代码，但是我的第一个贡献是编辑 markdown 文件。尽管这不是以代码形式添加到项目中的特性，我的贡献可以帮助人们寻找找工作的资源。我非常高兴这是开始为开源做贡献的第一步。

我的下一个贡献是改变 Android 应用程序上使用的文本。尽管这看起来是一个小变化，但它完全是一个有效的贡献。所以说了这些，我认为你可以从小事做起，在你开始做出贡献的同时建立你的自信。即使很小的变化也是对推进项目的有效贡献。

在 GSoC 项目和 Systers 社区结束的时候，我明白了，在我的旅程中，当我为社区做出贡献的时候，我感觉更加自信了。

### 有些项目看起来太吓人了

我完全理解这一点，尽管我可能有好奇心去探究，但我并不准备探究开源项目。我能想到我所使用的开源编程语言或工具，它们有很大的代码库。

要知道项目有文档可以帮助你，所以要注意这一点。请确保您查看了项目的文档，这些文档可以在 Wiki(例如:GitHub Wiki)上，与代码一起在文档文件夹中，在论坛上或在问题评论部分的讨论中。文档可以分散在多个地方。此外，理解一个你从未做过的项目需要时间，你可能需要花一些时间研究代码并设置它来理解这个项目(如果你是用代码做贡献的话)。

除了查看可用的文档，如果你真的感到困难，你可以联系社区和项目的维护者，这样他们可以提供资源来更好地理解项目。

### [](#lack-of-time-to-contribute)缺乏贡献的时间

每个人都有自己的生活，有自己的承诺。你必须考虑你有多想参与开源，以及“为什么”。这值得你为一个特定的项目贡献时间吗？是为了回馈一个自己关心的项目吗？你能为此付出多少时间？你的动机是什么？当你理解了这些，你就必须考虑如何将它纳入你的时间表，以及如何对它进行优先排序。

为什么不从一小步一小步开始，无论是在时间上，还是在你要做的任务上。你可以试着每周分配一段时间从事开源项目(例如:周日 1 到 2 小时，每天 1 小时等等)。你也可以像我第一次投稿时那样，开始解决低努力的问题。这可以帮助你熟悉通常的开源工作流程(例如:fork-> clone-> work-> submit pull request)，并帮助你养成习惯。

对我来说，这真的有助于为我关心并希望看到增长的东西做出贡献。此外，我非常喜欢维护一个项目，并使它对开源的新来者开放(例如:通过在头脑中创建带有“仅首次使用”标签的问题)。我也喜欢回馈我在 GSoC 工作过的社区。

### 如果你不能完成自己承诺的事情怎么办

如果你致力于一个问题，然后你不能完成它，要么是因为你对这个问题失去了兴趣，要么是因为你不能在你设定的时间内完成它，要么是因为它不适合你目前的技能水平…不管是什么原因，这都可能发生。

如果您最初陷入了您试图解决的问题，向社区提问并阅读您可用的文档会有所帮助。

如果你真的觉得你必须放弃它，不要只是抛弃它。通知项目所有者、维护者和社区，你将不再从事这个项目，从而给其他人解决这个问题的机会。你可以在你承诺的问题上留下评论，让他们知道。这完全没问题！同样，如果你已经做了一些可以帮助下一个贡献者的工作，分享一下。通过这样做，您将使维护者的生活更容易，因为他们将知道另一个问题可以分配给另一个贡献者。
