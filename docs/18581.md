# 为什么你不应该(只)做暴民编程

> 原文:[https://dev . to/peternycander/why-you-should-not-only-do-mob-programming-4jo 9](https://dev.to/peternycander/why-you-should-not-only-do-mob-programming-4jo9)

我接触并尝试了很多不同形式的编程；单人、双人、小团队、大团队、远程团队、跨公司团队以及本文的主题 mob 编程。

Mob 编程是指整个团队在一台计算机上编程——一个键盘，一个鼠标，3-7 个(或者更多？)人。一群乌合之众不一定只由开发人员组成；一个 mob 可以由一个设计师、一个产品负责人和两个 web 开发人员组成。

虽然我对标题的选择有点挑衅，并暗示暴民编程不好——但这并不是我感受的全部真相。乌合之众编程*可以*伟大。例如在前述的 mob 设置中，在新特征前端的布局和用户旅程的设计实现期间。

在这种情况下；设计师可以在开发过程中对设计进行快速调整(并且不必花费时间来获得完美的设计文件)，两个 web 开发人员可以比一个人更快地解决 CSS 问题，产品负责人可以确保客户之旅如预期一样进行。

我认为有问题的想法是由 3-7 个具有相似技能的人组成的永久群体编程的想法。

## [](#some-tasks-are-not-suited-to-mob-programming)有些任务不适合 mob 编程

作为开发人员，我们的任务不仅仅是编写闪亮的新特性。有大量的沟通、调试、重构，可能还有一些“手工”劳动，比如配置服务器或编写文档。

我听到的关于在一群人中做更卑微的工作的最常见的论点是，你可能更容易在“无聊”的工作中犯错误——一群人比一个人更容易发现错误。

对某些人来说可能是这样，但我发现通常会发生的情况是，一两个人可能会专注于任务，而其他人可能会走神。考虑下周晚餐吃什么的人没有捕捉到细微的错误。到那时，他们可能会做一些更有成效的事情，而不是让头脑麻木。

如果你对暴民编程很了解，你可能会坐在那里，愤怒地想着，如果他只做真正的暴民编程——用一个旋转的导航器和一个司机，那么这一切都不会发生。相信我，这种事总会发生。也许对你来说不是，亲爱的个人，但也许你的队友。

## [](#people-converge)人们汇聚一堂

支持 mob 编程的一个理由是，你永远不会停滞不前——你有这么多的人同时在处理同一个问题。所有人都有不同的知识背景，*暴民中的某个人*可能知道解决方案。

在你们在一起的头几个月，这可能是真的。但是随着时间的推移，暴徒的成员会将他们的大量知识相互传递。不同的知识集和背景将变得越来越不同。

知识转移是一件很棒的事情，尤其是对于经验较少的个人(以及公共汽车因素)，但是暴民/团队中的知识和经验总量几乎保持不变。它只会随着黑帮电脑上发生的事情而增长。从长远来看，这对团队来说并不太好。

## [](#it-does-not-deliver-on-its-promises)不兑现承诺

值得重申的是，我说的是一群拥有相似技能的人，他们已经在同一群人中工作了几个月或更长时间。比如说 4 个拥有高超 javascript 技能和一般 CSS 知识的 React 开发者。

我听到的一些关于暴民编程的承诺，我认为并不适用于这些暴民:

### [](#fewer-mistakes)错误少

这在很大程度上取决于个人，但我发现很常见的是，如果某个备受尊敬的高级开发人员犯了错误——很可能不会被发现。我们是群居动物，召唤一个充满尊敬的人是很难的。你对错误的确定程度必须相当高。

另一个起作用的因素是[群体思维](https://en.wikipedia.org/wiki/Groupthink)。这不是一个积极的术语，当你们一起工作一段时间后，它会变成一个真正的问题。它本质上减少了对想法的批判性评价。当你设法把你的乌合之众聚集成一个舒适的单位时，你可能会失去少犯错误的好处。

### [](#work-always-continues)工作总是在继续

据说，即使有人不得不去开会或生病在家，暴民编程也会一直持续下去。虽然这通常是真的(一个例外是团队/公司范围的会议)，但你正在将工作暴露给我们人类的另一个错误:[责任的分散](https://en.wikipedia.org/wiki/Diffusion_of_responsibility#Social_loafing)。其要点是，在团队环境中，你不太可能自己采取行动或承担责任。

比方说，你们聊到了旅游或其他与工作无关的话题。在那段时间里，你可能不会觉得自己在工作时间查私人旅行计划那么糟糕。我的猜测是，只要周围环境允许，mob 版本也能持续更长时间。

另一个因素是，你需要的任何休息都有升级为整个暴民的趋势。人们需要多长时间的休息，休息时间间隔有多长，这是因人而异的。如果声音最大的人比一般的团队成员需要更多更长的休息时间，那么净产出就会低于预期。

### [](#communication-is-fast)沟通快

你可以在人群中快速交流——因为这本质上是一次永恒的会面。这可能是一个很好的工具，可以避免向利益相关者封锁问题，但在一群同样熟练的开发人员中却不是这样。

开发人员之间的沟通障碍往往是架构性的，或者是代码审查的形式。在架构讨论的情况下；我不知道你怎么想，但是这些讨论对我来说并不是每天都发生的。当它们发生时，聚集在白板前通常并不困难。

不过，mob 编程中的代码审查速度很快，我不会反对这一点。然而，我认为代码审查的有效性降低了。这是因为你们都是引领你们找到解决方案的同一思路的一部分。也许有一个更简单的方法来解决这个问题，一些有新眼光的人会发现的。

* * *

Mob 编程可能是一个很好的工具，但不是一个很好的默认工作方式。我认为任何暴民都应该是短命的，以尽量减少群体思维。

我还认为，为了获得最大的交流利益，他们应该包含具有不同技能和不同职责的人。所有这些技能都需要与正在进行的特性开发相关。我和后端开发人员一起参加过纯粹的前端工作。

我认为尝试 mob 编程的早期成功可能会吸引一些团队长期致力于此。我认为这个决定会在你不知不觉中对你的工作产生负面影响。

如果我谈到的设置中的 mob 编程对您有用，那就太好了。希望我帮助你看到了一些可能发生在新团队中的潜在陷阱。
