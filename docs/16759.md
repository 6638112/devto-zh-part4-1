#### [](#a-perfect-sast-tool)一个“**完美”的 SAST 工具**

对于组织来说，购买第三方工具不仅昂贵，而且是一项巨大的投资。根据组织的需求定制工具会花费大量的精力，反之亦然。围绕它建立一个生态系统同样困难。因为我们已经用 Veracode 烧过手了，所以第二次选择正确是非常重要的。没有犯错的余地。我们意识到，如果我们不做尽职调查，那么它可能是致命的，并且在未来几年里，它将需要承担我们的错误。

考虑到这一点，我和我的同事 SecOps(安全运营部)的 Chris 提出了一个选择标准来筛选 SAST 工具。根据选择标准，我们对市场上的各种 SAST 工具进行了独立研究。选择标准帮助我们消除了无意识的偏见，同时试图筛选出 SAST 工具。

#### [](#our-selection-criteria)我们的选择标准

我们将我们的标准分为必须拥有、必须拥有和应该拥有。我们还根据开发、安全、IT 管理等利益相关方对我们的需求进行了分类。以下是我们在选择过程中高度重视的几点:

*   支持安全风险和软件错误，如 SANS 25、CERT、OWASP Top 10
*   提供根据开发、安全、IT 管理等利益相关方定制的详细的执行报告摘要。
*   对用户的更多控制–允许用户根据项目配置扫描策略。
*   允许用户通过众所周知的格式(如电子邮件、pdf，而不仅仅是专有格式)将报告导出到门户之外。
*   高准确率和低假阳性率。
*   对代码上下文中漏洞的良好解释。
*   提供示例代码和指南来修复该问题。
*   与 JIRA、Visual Studio、Visual Studio code 等工具的良好集成。
*   与我们的构建管道和 CI/CD 流程很好地集成。
*   易于理解、导航和现代化的用户界面
*   扫描周转时间
*   本地扫描的能力
*   支持 JavaScript 框架，如 Angular、VueJS 等以及最新版本。Net 核心框架。
*   报告通过 NuGet 和 NPM 添加的第三方依赖关系的漏洞。
*   提供丰富的 API 集来执行查看扫描、下载报告等操作
*   优秀的文档
*   良好的支持
*   成本效率
*   安全性之外的功能，如代码效率、重复代码等

#### [](#what-we-did-not-consider)我们没有考虑的事情

*   来自 [ITcentralstation](https://www.itcentralstation.com/) 等网站的评论。我们发现关于该产品的评论可能会产生误导，他们更关注 IT 运营。
*   组织的规模。我们从 Veracode 获得的经验让我们意识到，越大并不总是越好。因此，我们将其排除在选择标准之外。
*   我们试图与产品营销团队保持距离，因为从我们的经验来看，这对于获得我们问题的答案帮助不大。

上述标准帮助我们将搜索范围缩小到以下产品:

*   检查马克思
*   按需加固
*   魁湾
*   索纳库贝
*   Coverity +黑鸭
