# 我们是如何使用 CSS-in-JS 的？为什么有些开发者不喜欢 CSS

> 原文：<https://dev.to/katsanos_george/the-real-reason-why-new-developers-don-t-like-css-jb8>

我最近读了[为什么大多数开发人员害怕 CSS](https://dev.to/steffenpedersen/why-most-developers-fear-css-3h99) ，我决定花点时间来介绍一些我在过去 4-5 年中积累的想法。

人们对 CSS 失望的大多数“为什么”是:

## 浏览器不兼容

*声明*
浏览器，主要是 IE，在 CSS 规范的实现上并不一致。

现实
这种情况偶尔会发生，但如果你在 2014/2015 年之前没有做过 CSS，那么你就不会看到我们必须解决的黑客、技巧和错误，以使事情看起来更好。你用 png 的，4 个额外的 HTML 元素和绝对定位做过圆角吗？没有吗？那你就没有权利抱怨。在前 flexbox 时代尝试垂直对齐。这种情况已经有了很大的改善，随着 IE Edge 采用 Webkit，这个问题就永远解决了。

## 范围/层叠/继承

*认领*
我在这里改了这条规则，那边坏了东西。

*解决方案*

*   功能 CSS([https://tailwindcss.com/](https://tailwindcss.com/)或[https://tachyons.io/](https://tachyons.io/)
*   基于组件的架构，带有一个(只有一个)父选择器，为您创建一个范围

在我看来，一些程序员对 CSS 有问题的真正原因要复杂一些。

## Ex-backend devs (PHP，Java，。NET)转向前端是因为大约 6 年前开始的 JS 炒作。(慢慢用脊椎骨，接着是有棱角的，不久后用 React)

平均而言，这些工程师更擅长处理逻辑/编程/算法，比那些在职业生涯中主要从事 HTML/CSS/jQuery 的人更擅长。他们选择 React 是因为它仍然是最流行的 JS 框架。

但是，他们从来没有时间，意志和实践经验来学习 CSS。

他们认为 CSS 是“低劣的”，是一种必要的邪恶，“它不是真正的编程”(也许是真的，但并不意味着它很容易做到)，他们只是一头扎进去，这里改几行，那里改几行，然后祈祷它能工作。喷，为 CSS 祈祷。

我见过优秀的程序员写出我这辈子见过最丑的 CSS。不必要的重复，深度嵌套的选择器，幻数，很少甚至没有架构等等。

## 一般低门槛进入

CSS 作为一种描述性语言，一套规则等等，感觉很简单。谁都可以做。人们希望不用任何学习就能理解一切。

"这东西不能完全按照我的意愿运行"？->“位置:绝对；”
“这个规则还不够强大”？——>“！重要"

诸如此类。

## 框架

CSS 框架即使是非常有用的工具，也不能用普通的 CSS 来代替真正的实践经验。如果运气好的话，你可能会将一些引导模块组合在一起，但是你不会走得太远。当你遇到实现中的第一个障碍时，你会想“哦，这太复杂了，这是语言的错”。React 的 UI 框架也有类似的结果。你可以用 Material UI 来制作一个 webapp，以方便反应。但是当你需要定制的时候呢？

## 结论

对 CSS 的厌恶和困惑，以及不用真正做 CSS 就能做所有 web 的可能性，导致了像 CSS-in-JS 这样的模式和 Webflow 这样的工具。大多数反对 CSS-in-JS 的技术观点已经被 Chris Ferdinandi [在这里](https://gomakethings.com/whats-wrong-with-css-in-js/)驳斥了。
