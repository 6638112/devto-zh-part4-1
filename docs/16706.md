### 什么是图形 API？

图形 API 的一个简单定义是这样一种 API，它根据节点和边(对象和关系)对数据进行建模，并允许客户端在一个请求中与多个节点进行交互。例如，假设一个服务器保存着关于作者、博客文章和评论的数据。在 REST API 中，为了获取特定博客文章的作者和评论，客户端可能会发出三个 HTTP 请求，如/posts/123、/authors/455、/posts/123/comments。

在 graph API 中，客户端制定调用，因此来自所有三个资源的数据被一次拉入。客户端也能够指定它所关心的字段，对响应模式给予更多的控制，*”zsóFIA "*

### 什么是 Rasa？

Rasa 是一个开源的对话式人工智能框架。Rasa 不是一个基于规则的框架，你不需要像 Dialog flow、微软 LUIS 或亚马逊 Lex 或 IBM Watson 那样担心将数据放在别人的云中。

Rasa 有两个主要组成部分——Rasa NLU 和 Rasa 核心。
NLU 是理解自然语言的。假设用户说“我想订一本书”。NLU 的工作就是接受这些输入，理解用户的意图，并找到输入中的实体。比如上面这句话，意图是订购，实体是书。Rasa NLU 内部使用词袋(BoW)算法来查找意图，使用条件随机场(CRF)来查找实体。尽管您可以使用其他算法通过 Rasa 来查找意图和实体。

Rasa 核心的工作主要是为聊天机器人生成回复消息。它采用 Rasa NLU(意图和实体)的输出，并应用机器学习模型来生成回复。这种对话管理采用来自 RASA NLU 的结构化输入，并使用概率模型(如 LSTM 神经网络)预测下一个最佳行动。

RASA stack 是一个开源的人工智能工具，作为一个开源框架，它很容易定制。事实上，在许多情况下，客户不希望共享他们的数据，大多数可用的工具都是基于云的，并提供软件即服务。您不能在您的环境中内部运行它们。所以你需要把你的数据发给第三方。有了 RASA，就没有这个问题了。您可以在服务器或环境内部构建、部署或托管 Rasa，并对其进行完全控制。

NLU 和核心是独立的，你可以使用没有核心的 NLU，反之亦然。尽管 Rasa 建议两者都用。

在本地进行核心处理、完全可定制和开源的概念使 RASA 与其他对话式人工智能平台不同。👍

### 什么是 TimescaleDB？

[TimescaleDB](https://www.timescale.com/) 是安装了 TimescaleDB 扩展的 PostgreSQL，它为 DBMS 添加了一些与时间序列相关的“超能力”。因此，PostgreSQL 生态系统中的工具也可以与 TimescaleDB 配合使用，例如备份工具(例如 pg_dump)、CLI 和 UI 客户端(例如 psql 和 pgAdmin)、可视化工具(例如 Grafana)、其他 PostgreSQL 扩展(例如 PostGIS)等。而这大概也是 TimescaleDB 的主要优势和卖点。此外，您不必学习新的查询语言，只需使用好的旧 SQL 和 TimescaleDB 扩展中的几个函数。

TimescaleDB 的中心概念是一个超表。超表是一个虚拟视图，由许多单独的小表组成，这些小表保存称为块的数据。通过按时间间隔和可能的一些附加关键字(例如，GPS 坐标、温度、位置名称、速度等)划分数据来创建块。).

因为超表是一个视图，所以您可以用它做任何事情，就像您通常用 DB 视图做的那样:您可以做选择、连接、过滤(通过 WHERE 子句)、排序(通过 ORDER BY 子句)等。超表被分割成块的事实不会使你与它交互的方式变得复杂。

TimescaleDB 将自己推销为一个适合处理时间序列数据的关系数据库，当数据量大大增加时，它可以很好地垂直伸缩。它声称加载 10 亿行数据库的速度是普通 PostgreSQL 安装的 15 倍，并且提供的吞吐量是普通版本的 20 多倍。
