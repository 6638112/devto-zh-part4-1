# SAST 工具——第 1 部分:为什么我们抛弃了 Veracode

> 原文:[https://dev . to/ankitvijay/sast-tooling-part-1-why-we-burged-vera code-21ba](https://dev.to/ankitvijay/sast-tooling-part-1-why-we-ditched-veracode-21ba)

这篇文章是系列文章的第 1 部分，描述了我们抛弃流行的静态应用程序安全测试(SAST)工具 [Veracode](https://www.veracode.com/) 以及我们寻求更好的*安全工具*的旅程。

### [](#background)背景

直到最近，我们的组织使用 Veracode 对我们的一些应用进行安全分析。Veracode 名声在外。Gartner 认为它是应用安全领域的 [**领导者**](https://info.veracode.com/analyst-report-gartner-mq-appsec-testing-2019.html) ，并被全球数百家组织使用。不幸的是，我们对 Veracode 的体验完全相反。

> **免责声明**:这篇文章讲述了*我们去年根据需求使用 Veracode 和其他 SAST 工具的*经验。从那时起，情况可能发生了变化。不应将其视为对任何 SAST 工具的支持/建议。

### [](#veracode-painpoints)**Veracode 痛点**

*   Veracode 文档没有跟上他们的更新步伐(无论如何，这是千载难逢的)。甚至在一些地方，我们发现它有误导性。
*   我们发现 Veracode APIs 很难使用并与我们的构建管道集成。
*   Veracode UI 门户网站还有很多需要改进的地方。它的用户界面看起来过时了，而且很不直观。它不支持一些基本的集成。例如，报告、扫描结果等。无法通过电子邮件发送。
*   Veracode 支持部门花了几周时间才做出回应，即使他们做出了回应，回应也很简短，没有任何帮助。
    *   来自 tickets 的支持几乎总是包含一个过时文档的链接。甚至当提出的罚单涉及相同的过时文件时也是如此。🙂
    *   通常需要后续的电子邮件才能得到他们的回复。
*   与 JIRA 融合是不可行的，也没有解决我们的目的。由于以下原因，我们最终回退到在 JIRA 手动创建票证:
    *   如果在不止一个区域(比如一个文本框)中发现了相同的发现，Veracode 会为该发现开出几十甚至几百张罚单。例如，一个客户端验证问题可以通过对验证库进行一个简单的修改来解决，它被标记为每个文本框创建一个新标签。这导致数百张新的罚单在 JIRA 开出，造成一片混乱。
    *   它的 JIRA 插件本身并没有定期更新。我们的一个 JIRA 更新破坏了插件。
*   扫描错误地将第三方库识别为我们的内部组织代码。结果，扫描显示了不正确的结果。这导致我们的代码扫描失败。开发者不能为第三方开发者重写代码。Veracode 花了几个月的时间确认这是一个 bug，然后又花了几个月的时间修复它。
*   许多依赖我们的应用程序，如城堡温莎和 Autofaq 不被支持。Veracode 提到他们没有计划支持它。
*   Veracode API 密钥有一个有限的有效期。我们发现 Veracode 似乎只发送了一个关于即将到期的通知，就在 API 密钥到期的前一天。门户网站中没有其他跟进电子邮件或通知来警告 API 密钥过期。结果，由于 API 密钥过期，我们的 Veracode 扫描失败了。忽略一封电子邮件会导致所有相关扫描无法完成。
*   如果在上传 dll 时出现任何问题(例如，由于 API 密钥过期)，那么我们在门户上得到的错误是非常隐晦的，并且很难理解根本原因和可能的解决方案。
*   Veracode 既没有与我们的源代码控制集成，也没有提供本地分析。扫描代码的唯一方法是将 dll 上传到 VeraCode 门户网站。
*   扫描时间相当长。小型应用程序大约需要 30 分钟。如果对代码进行了微小的更改，则需要上传所有的 dll，并且需要再次运行完全扫描。我们无法找到将它与我们的构建管道完全集成的方法。
*   Veracode 增加的功能似乎专注于通过新的付费产品创造新的收入来源，而没有解决该平台的任何已知问题。
*   VeraCode 推出了两个 Visual Studio 插件:Veracode 和 Veracode Greenlight。Veracode Greenlight 是单独许可的。但更重要的是，这两个插件同样没用。
    *   Veracode 插件:在编辑器中没有提供静态扫描。我们仍然需要上传 dll，然后下载报告。当开发人员从他们的 IDE 上传文件时，机器无法使用并且没有响应。随着每个文件的上传，焦点返回到上传窗口，使得不可能同时执行任何额外的任务。同样，上传可能需要几分钟才能完成。
    *   Veracode Greenlight:当这个插件对内联扫描源代码时。它只能在文件级而不是 dll 级完成。根据我们的经验，它甚至不能识别文件中的漏洞。

我们每天都感受到这些痛点，迫使我们寻找替代方案。在下一篇文章中，我将谈论我们评估市场上不同 SAST 工具的方法。
