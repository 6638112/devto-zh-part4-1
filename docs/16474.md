# 敏捷再探:简单性、自组织和反思

> 原文：<https://dev.to/miniscruff/agile-revisited-simplicity-self-organizing-and-reflections-36p6>

一个开发人员回顾了敏捷的 [12 条原则并对其进行了小的修改。](https://agilemanifesto.org/principles.html)

欢迎在评论中不同意我的观点。

# 简朴

> 简单——最大化未完成工作量的艺术——至关重要。

很难确定作者到底想说什么。所以，我要在这里做一个假设，这个假设可能正确，也可能不正确。

我认为，这一原则的意图是暗示创造简单的产品是非常重要的。而且，这样做的最好方法是尽可能少做工作。

## 修正后

> 没有变化

我非常同意这个原则，我没有任何可以增加价值的改变。

然而，我确实认为原作者指的只是最终产品。因此，虽然我不会改变原则的措辞，但我想改变对它的解释。我想将这一原则扩展到整个工作流程中，而不是仅仅局限于面向用户的产品。

在我解释为什么和它意味着什么之前，我想提一下这可能是原作者的意图。

现在，我认为通过考虑相反的情况来讨论简单性要容易得多，**复杂性**。也就是说，更简单等同于不太复杂。此外，我想把复杂的任务分成困难的和简单耗时的。

因此，为了使事情变得简单，要么不困难，要么不耗时(小警告，耗时是基于主动工作，而不是等待下载完成，看你的 npm 安装),或者两者兼而有之。

我提出这一切的原因是，我相信诸如持续集成和 DevOps 之类的工具和模式本质上使您的开发人员工作流更复杂，因此更不简单。另一方面，它节省了大量的时间和精力，超过了复杂性。

# 自组织

> 最好的架构、需求和设计来自自组织团队。

在我看来，这是最有影响力的原则。乍一看，这似乎很简单，但好处很多，我可以单独写一整个系列。

简而言之，这表明自组织团队能创造出最好的东西。事实上，据我了解，所有的敏捷衍生产品如 Scrum 和 SAFe 都在自己的规则中包含了这一点。

我不想解释为什么自组织团队如此重要的所有原因，我只想让你想象一下你最喜欢的体育比赛，其中团队中没有人喜欢彼此，他们都是同一职位的专家。比如，如果一个足球队在进攻中有 11 个四分卫。

## 修正后

> 最好的产品来自自组织的团队。

我只是让它更加通用和包容。除此之外，我认为这仍然是最重要的原则。

事实上，我认为，如果你的公司在某种程度上不能采用这一原则，你会发现其余的原则要困难得多，甚至是不可能的。

# 倒影

> 团队定期反思如何变得更有效，然后相应地调整自己的行为。

除了举行这些讨论，团队还必须做两件事。

1.  能够给彼此诚实的建设性反馈
2.  能够根据反馈调整自己的工作方式和流程

培养一种能够做到这两点的文化和团队，至少可以说是很难的。

如果你熟悉 Scrum，这些是在每个 Sprint 结束时召开的回顾会议。

我想指出的是，该原则并未指定会议，但可以假定会议是根据先前的原则进行讨论的默认方式。

## 修正后

> 作为一个团队，不断思考如何变得更有效，然后相应地调整。

我对此唯一的不满是定期间隔条款。对你的团队来说，最好的方法可能是定期召开会议，但按需组织团队午餐也是可行的。

另外，这可以减少你的会议次数，感觉很好。

我还要补充一点，在某些情况下，你可能不需要召开一个完整的团队会议，而只需要包括一些与问题最相关的成员。

除此之外，团队参与对这种有效性至关重要。我也注意到一个趋势，那些不喜欢敏捷的人倾向于跳过，不参与，或者在某些情况下完全从他们的工作流程中移除。如果有足够多的人停止参与，那么这个会议将会完全从日程表中删除。

没有这种反思，团队将会一遍又一遍地重复同样的错误。大多数情况下会导致员工流失和精疲力竭。
