# 跟踪读取了多少不同的代码(字面上)

> 原文：<https://dev.to/kirillrogovoy/tracking-how-much-different-code-is-read-literally-1e4o>

嘿，希望你有一个伟大的星期四！

我参与了各种关于消除技术债务和重构的讨论。

这种会议的目标之一是对问题进行优先排序，所以我们把宝贵的重构时间花在修复最重要的东西上。

这个过程的一个有用的规则是“改变人们工作和阅读的代码，不要为偶尔阅读的代码而烦恼，即使在某些方面有缺陷，这些代码也相对简单。”

这不是唯一的规则，但绝对是一个先看什么的好指标。至少从我的经验来看。

但是问题就在这里。当谈到关于我们更频繁地阅读和修改哪些代码的争论时，大部分过程只是*猜测*。我们试图从我们目前的直观环境中衡量这一点:最近的特征、主要特征、计划、观点等等。

尽管我们通常以一个不太坏的估计结束，但是真的没有好的方法来度量或反映这个过程。

* * *

这让我开始思考:难道这不仅仅是一个我们可以用工具解决的技术问题吗？

1.  我们可以使用 Git 中的历史记录来跟踪不同文件被更改的数量(和频率)。数据就在我们面前，我们将使用类似于[https://github.com/jez/git-heatmap](https://github.com/jez/git-heatmap)的东西来评估。
2.  但第一点只是故事的一半。跟踪开发人员*阅读了*代码的不同部分不是也很棒吗？这可能需要一个编辑器扩展来跟踪所有的光标移动和文件间的跳转+导航——对于大多数流行的编辑器来说，这可能是一件微不足道的事情。然后，我们将构建一个类似的代码读取热图。

**问题:**

1.  整个重构问题也是你在组织中经常解决的吗？你的方法是什么，你的奋斗是什么？
2.  那个代码导航跟踪器对你有用吗？你能想到一个已经可以做到这一点的项目吗？
