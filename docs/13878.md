谢天谢地，会议上的东西对我来说并不陌生——如果我没有听说过其中的大部分，我会担心我的职业生涯！-但是我已经在 Python 中工作了很长时间，我已经很久没有在 JS 上花太多时间了，所以这是一个很好的复习。下面是我学到的/觉得有趣的事情的简要概述。

* * *

**主线程**

有几个人谈到了主线。这还不是我们必须要担心的事情，因为我们的大部分繁重工作都是用 Python 完成的，而 JavaScript 就像是 99%的 UI。然而，我们已经在搜索功能上做了一些工作，这可能是使用 worker 的一个很好的测试案例。我们也有几个可以外包的外部 API 调用。虽然我怀疑这是否会对性能产生巨大影响，因为我们的大部分数据都是静态的，并且是在后端管理的，但是值得考虑一下我们可以将哪些数据卸载到后台进程中。

* * *

**打字稿**

现在还有人写 JavaScript 吗？这里有太多关于 TS 的讨论，我实际上已经开始查看我们的代码，看看我们是否可以转换。有些人不喜欢 Python 的随意性，所以也许选择一个好的静态类型的 JS 版本会让事情看起来更好！

* * *

**安全**

在小团队中，将安全性嵌入到开发过程中是很重要的，这样您就不必考虑它了。这肯定是我对此事的首选方法，因为我对此一无所知，更不关心。

一些可以考虑的工具:

*   自动安全扫描仪，有点像 pentest
*   制作风筝 -詹金斯的替代品，因为这里有一个有趣的事实:詹金斯在“最高数量的昆虫赏金”名单中排名第二。现在我有一个谷歌，我找不到这方面的来源，但这家伙在他的谈话中说，所以一定是真的？
*   [幽灵探长](https://ghostinspector.com/) -云基硒替代品？
*   [哨兵](https://sentry.io/welcome/) -错误记录
*   [Terraform](https://www.terraform.io/) -基础设施即代码，这意味着如果您破坏了某些东西，您可以轻松地回滚到上一个良好的配置
*   AWS:
    *   [Cloudtrail](https://aws.amazon.com/cloudtrail/) -查看哪个用户做了哪些更改
    *   [云观察](https://aws.amazon.com/cloudwatch/) -日志记录
    *   [守卫职责][[https://aws.amazon.com/guardduty/](https://aws.amazon.com/guardduty/))-识别异常活动
    *   补丁——让 AWS 来做这一切，尽管这在过去已经对我们产生了反作用
*   自动保持你的依赖关系最新
*   [Scout Suite](https://github.com/nccgroup/ScoutSuite) -云安全扫描器
*   [土匪](https://github.com/PyCQA/bandit) - Python OWASP 安全检查
*   [Synk Docker… stuff](https://snyk.io/docs/container-security-overview/) -监控 Docker 映像安全注意事项

**良好的组件设计**

我原以为这个演讲会非常反应激烈，就像实际设计反应组件一样，但实际上她谈论的是一种更通用的软件设计方法，它完全反映了我一直在谈论的东西，但更加定量。

良好(组件或其他)设计的指导原则:

1.  可访问性，又名“开发人员工效学”
    *   这里我们指的是开发者体验，而不是用户体验，这是一个独立的话题。你可以合理地安排和命名，这样人们就可以很容易地完成它。
2.  可量测性
    *   再说一遍，开发者体验，而不是代码/服务器/任何东西。
3.  单一责任
    *   使您的代码更容易测试。合理地使用命名有助于识别哪些组件可以放在一起
4.  可试验的
    *   好的 obvs
