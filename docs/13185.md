# Ruby on Rails 和 Django——为你的 Web 项目选择哪一个？

> 原文:[https://dev . to/idea motive _ co/ruby-on-rails-vs-django-the-one-to-pick-for-your-web-project-10g 5](https://dev.to/ideamotive_co/ruby-on-rails-vs-django-which-one-to-pick-for-your-web-project-10g5)

当谈到为在线应用程序开发选择正确的框架时，选择正确的技术既可以是火箭燃料，也可以是膝盖上的箭。是时候对 Ruby on Rails 和 Django 进行比较，找出它们各自的优缺点了。

与 Ruby on Rails 支持的 387，696 个网站相比，Django 这两种技术都相对受欢迎，Django 支持 74，405 个网站，这两种技术都可以免费使用，不仅用于交付网站，还用于交付所有隐藏的机器，支持各种业务的基于网络的应用程序，包括电子商务网站和不同的 SaaS 解决方案。

尽管有许多相似之处，这两种技术还是有很大的不同。

什么是 Ruby on Rails？Ruby on Rails 是一个为 Ruby 编程语言设计的框架。作为一个框架，它迫使编码者遵循一组预定义的规则，这些规则有助于交付更好的结果。对于 Ruby on Rails，最重要的规则是:

不要重复自己——该框架通过支持程序员尽可能多地重用代码来解决隐藏相似性的问题。因此，不存在只使用一次并且很难在众多代码行中找到的隐藏的独立类。约定优于配置——框架提供了一组预定义的规则，使用户摆脱了配置环境的痛苦，因此，代码更清晰、更可移植，并且由于从更多的管理工作中解脱出来，编码人员也更放松。
Ruby on Rails 是一项广泛使用的技术，由科技巨头支持，包括 Hulu(网络流媒体平台)、base camp(Ruby on Rails 发明的项目管理工具)、GitHub(最强大的版本控制工具)和 Airbnb(在线预订服务)。

姜戈是什么？Django 是最流行的现代编程语言之一——Python 的后代。它是一个设计用来在 web 开发和 web 应用程序中使用这种通用语言的框架。由于 Python 与机器学习设备的联系，该框架在用于数据库密集型应用时大放异彩。与 Ruby on Rails 相反，框架本身不支持任何特定的编码风格，而是被设计为一种工具，只用于编码人员认为最好的方式，而不支持任何特定的风格或启发。

Django 被包括 Mozilla 和《华盛顿时报》在内的重量级互联网公司使用。

为什么要比较？尽管解决流行问题的编程范例和方法不同，但两个框架都擅长类似的应用。两者都是开源的，速度快，支持强大的互联网应用。两者都是面向对象和动态类型的，并提供持久支持，

那么，这些差异是什么，比如说，造成了差异？

Ruby on Rails vs Django–用户界面
Django 和 Ruby on Rails 都是以网络为中心的框架，旨在提供高级体验。这两个框架都使公司能够建立出色的网站，并用各种附加组件和插件丰富它们。

随着两种技术的成熟和发展，很难挑选出更好看的一种。关键在于编码者的技能。

判决:平局。

Ruby on Rails vs Django–架构
[Ruby on Rails web 开发](https://ideamotive.co/services/ruby-on-rails-development)因其采用模型-视图-控制器(MVC)范式而受到称赞，能够在要执行的任务和用户的思维方式之间找到匹配，使程序员能够避免早期部署和遗留技术中的模糊结构。

Django 在交付架构时减少了压力，信任程序员的技能。有时，当编码团队在设计应用程序方面拥有高超的技能，并且能够找到交付超一流体验的方法时，这是一种更好的方法。

但这真的是一个观点吗？这真的是你的观点吗？

结论:通过迫使团队遵循一套严格的预设规则，Ruby on Rails 提供了一个更好的应用架构。

Ruby on Rails vs Django–编码速度
Ruby(Ruby on Rails web 开发的支柱)和 Python(用于 Django)都是现代编程语言，它们吸收了过去的最佳实践，避免了它们的错误。

Ruby on Rails 受益于丰富的库和插件库(称为 gems)来丰富项目，大大减少了编码人员需要投入的精力。另一方面，Ruby 是一种典型的编程语言，它支持编码者以他或她认为最好的方式解决问题的自由。

与 Ruby 相反，Python 遵循的哲学坚持认为应该有一种(最好是唯一的)明显的方法来解决遇到的问题。因此，代码通常更易读，也更容易调试。另一方面，团队必须知道方法。

结论:取决于项目，Ruby on Rails 由 Gems 支持，Django 由 Python 的一种解决问题的范例支持。

ruby on Rails vs Django–性能
当比较性能时，两个框架都做得很好，几乎不可能指出谁是赢家。这两种技术都利用了现代编程语言，并提供了优化代码的工具。

这种差异取决于要交付的应用程序。在为电子商务或类似项目构建通用应用程序时，Ruby on Rails 大放异彩。另一方面，Django 利用 Python 的多功能性来构建性能更好的特定用途应用程序。

判决:僵局。没有明显的赢家，围绕微不足道的性能提升有多种利弊。

Ruby on Rails vs Django——社区和支持
文本介绍中显示的统计数据清楚地显示了 Ruby on Rails 开发者在交付 web 应用和在线系统方面的优势。因此，有一个蓬勃发展的社区支持新用户，并为他们提供解决问题的建议。

Django 本身是非常不受欢迎的技术，因此遇到以前未知问题的可能性明显更大。另一方面，Python 的人气暴涨，使得这种语言[成为 2018 年](https://www.infoworld.com/article/3331603/pythons-popularity-surges-as-a-mainstay-language.html)的最大收益者。然而，Python 是一种多面手类型的语言，在机器学习、后端和网络开发中应用广泛，因此并非所有 Python 用户都能帮助解决问题。Ruby 更加以网络为中心，因此社区更加完整，专注于一个特定的用例，被打磨和掌握。

结论:说到 web 项目，Ruby 在比较中胜出。

ruby on Rails vs Django——稳定性
在创造性和稳定性之间有一场持续的斗争。寻找解决问题的新方法对进步至关重要。另一方面，任何新事物都会带来不稳定性。Ruby on Rails 通过鼓励用户重用代码来减少隐藏的依赖性，从而解决了这个问题。此外，它利用了配置范例之上的约定，因此编码人员摆脱了许多棘手的问题。

另一方面，Python 带来了更保守的编码方法，并鼓励开发人员坚持使用一种经过验证的方法来解决问题。谈到稳定性，这是一件好事。

事实上，这两个框架都很成熟，并且经过了足够的测试，可以稳定地为任何项目提供可靠的基础。

判决:平局。没有显著差异。

老实说，没有理由抱怨。负责框架开发的两个团队都做了很好的工作，为他们交付的项目准备文档。界面清晰易读，语言清晰明了，没有隐藏的谜语。

因此，在文档方面没有真正的区别。

判决:在没有真正比赛前打平。

Ruby on Rails vs Django——平台的成熟
Ruby on Rails 于 2004 年诞生，并在 web 开发社区中被广泛采用。互联网上有成千上万的网站利用这种[技术](https://ideamotive.co/blog/40-best-ruby-on-rails-companies-websites)的力量和灵活性。它还会定期更新，最新版本发布于 2018 年 8 月。

Django 也不例外，出生于 2005 年，最新版本于 2019 年 4 月发布。这项技术已经无数次地证明了自己，其可靠性的最好证明就是它与 Ruby on Rails 在同等权利下的比较。

判决:平局。

总结–选择正确的工具以获得最佳结果
这两个框架都很棒，并且在正确使用的情况下都能提供卓越的性能。关键是要知道什么时候使用每一个。

何时使用 Django
Django 利用了 Python 的灵活性，并依赖于编码人员的经验和对项目的责任感。它闪耀于:

要交付的应用程序是高度可定制和特定的——如果客户需要交付新的社交网络，Django 无疑是更好的选择。当你真的知道自己在做什么，并且百分百确定的时候。
这款应用将变得异常复杂——交付执行复杂数据操作的系统，并致力于成为一个单片多工具巨兽，当用 Django 构建时会更好。
何时使用 Ruby on Rails
Ruby on Rails 是一个自动化编码过程的框架，让编码人员的工作变得更加容易。该框架在以下情况下大放异彩:

当时间压力很大时——Ruby on Rails 利用了丰富的插件库和更丰富的 Ruby on Rails 程序员社区。因此，以更快的速度和更少的努力交付项目的可能性(并且最终以更高的质量！)要高很多。当你的项目相对可重复时——这个框架对于交付一个不需要重新发明轮子的项目来说是很棒的——如果客户需要一个可靠的技术来启动电子商务系统或交付应用程序来支持内部业务流程，Ruby 将是一个更好的选择。
当项目不是世界级的游戏规则改变者时——尽管 Ruby on Rails 在支持将打破所有规则并改变世界的项目时很棒，但它的整体架构可能太过时、太笨重，无法支持更小的项目。和由更小的生物组成的万维网。关于网络应用，尤达说得很对——大小并不重要。

考虑到上面提到的所有因素，大多数情况下 Ruby on Rails 会是一个更好的选择——它可以为较小规模的项目提供更快的交付，有许多现成的解决方案。
