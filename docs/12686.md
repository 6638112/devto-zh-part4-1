# 技术主管的五大职责

> 原文:[https://dev . to/teaching TLS/top-5-of-a-tech-lead-170j](https://dev.to/teachingtls/top-5-responsibilities-of-a-tech-lead-170j)

昨天，您是一名高级开发人员，愉快地在分配给您的场景或用户故事上工作。当你和你的经理坐下来的时候，他们要求你承担技术主管的新角色，你接受了，因为，为什么不…

你的日常生活中增加了什么新的责任？

## [](#gathering-requirements)采集要求

每个项目都是从有人意识到他们有问题开始的。那个人向其他人抱怨，迟早他们会意识到它需要来解决一切。

你将会见企业或客户，听听他们的问题是什么。你倾听，你提问，你在考虑提出解决方案之前尽可能多的获取信息。然后你提出来。

“以下是我们解决您的问题的方法。”一旦他们同意，就该开始工作了。

仅仅让他们同意整体解决方案是不够的，你还需要让他们同意更小的细微差别。你最不希望的事情就是让他们看到所有的事情都完成了，却不满意。

小型反馈回路是关键。

## [](#designing-the-solution)设计解决方案

一旦您签字同意，就该开始整合该解决方案的工作方式了。在这一点上，根据您的组织/团队的规模，您可能单独工作，或者与某种架构师一起工作。您将从“我们如何让这个东西工作？”的抽象部分转移到更详细的分析部分。

一旦你有了解决方案及其所有活动部分的草图，你的内部团队流程就开始了。我的团队使用 BDD 来实现我们的规则，你可能使用 TDD 来完成你的看板任务，或者 Scrum 来迭代用户故事；这取决于你。但是，在这一点上，您可以开始将工作单元移交给您的团队。

## [](#interfacing-with-business)与业务对接

当团队在发展时，你很可能不会。我们将在以后的文章中讨论这个问题，但是你已经不再是开发人员了。习惯就好。

你是什么，是一个接口。客户会有一个问题或顾虑，他们会想谈谈一个特性是如何实现的。或者他们想知道下一个发布周期是什么时候。他们会尝试与团队中的某个人交谈。

那个人就是你。没有别人。

你所做的是解放你的团队去工作。

## [](#guarding-the-code)守护代码

啊是的，代码。你不会写它，但你肯定会读很多。

代码库需要维护。而你是黑骑士，守卫着通往“主”或“主干”或特性分支的桥梁…无论你的分支策略是什么。

每个合并请求都应该被看作是与正在实现的特性和实现它的开发人员的对话。

他们满足检查风格和 PMD 的要求吗？他们提供了必要的代码覆盖率吗？
是否有一个验收测试来证明该功能按设计运行？
开发人员在设计时实现了你们讨论的模式吗？
它们符合你代码库的整体风格指南吗？
代码评审的主题需要自己的文章，所以我们稍后会深入探讨。但是可以说，你会读很多书。

## [](#releasing-the-product)发布产品

一旦完成了之前达成一致的要求，您就可以开始工作了！只有编写发行说明、与 QA 联系、提醒开发人员或系统管理员或任何负责在管道中移动事物的人的任务。

当这些人有问题时，你就是他们的联系人。他们是否需要从某处获取一个 WAR 文件，或者将他们的 docker 映像更新为一个新的发布标签？QA 是否需要对所有需求进行一次总结，以及他们如何测试每一个需求？

然后，一旦产品发布，当客户发现他们需要的增强功能，或者应该在下一个版本中修复的错误时，我们就重新开始这个循环。

# [](#conclusion)结论:

作为技术主管，您将负责:

*   找出问题所在
*   设计解决方案
*   解释该解决方案
*   委派任务并确保完成
*   将溶液释放到野外

这是一个非常简单的总结软件开发生命周期的清单，我知道这很令人震惊。但是，当你在一个项目中担任技术领导的角色时，你将从头到尾亲自参与。

但是不要担心，因为你不会独自做这件事。在以后的文章中，我们将谈论你的团队以及他们将如何帮助你。因为，如果他们实际上没有“领导”任何东西，那么什么是技术领导呢？
