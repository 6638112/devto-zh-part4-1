# 测试驱动的会议

> 原文:[https://dev.to/jcutrell/test-driven-meetings-4l3o](https://dev.to/jcutrell/test-driven-meetings-4l3o)

写一个测试，让它失败。编写代码使其工作。重构。

这种“红绿”循环是大多数现代开发人员都理解的。我们中的许多人都将这个概念放在前面和中心来构建我们的整个工作流程。

在开会的时候，我们也认同彼此的痛苦。

为什么我们又见面了？为什么这个会议要持续两个小时？大家不是已经有这些信息了吗？如果我们都回去工作不是更好吗？这不是真正的工作...这只是让我们觉得我们在工作。

有些会议感觉比其他会议更有成效。也许你真的喜欢每天的站立会议，但几乎不能容忍“市政厅”会议。

无论你在职业生涯的哪个阶段，通过将测试驱动的思维模式应用到你的会议日程中，你都可以让会议变得更加有用。

我们来分解一下！

## [](#1-look-at-your-calendar-and-identify-all-events-that-are-meetings)1。查看您的日历，找出所有会议事件。

不是每个事件都是会议。为您的会议事件制作专门的日历。如果你有非正式会议，也把它们放在日历上。

## [](#2-in-the-calendar-event-write-out-the-expected-output-of-the-meeting)2。在日历事件中，写出会议的*预期产出*。

可以说，伟大的函数总是会产生输出。你会从这次会议中得到什么？以下是一些输出类型:

*   任务-你的会议是为了产生需要完成的工作吗？
*   “决策记录”——无论你在哪里记录你正在进行的工作，会议的结果可能就像对某事做出决策并记录下来一样简单。
*   笔记——至少，你的会议应该能够输出一个关于讨论内容的简单总结。

## [](#3-in-the-calendar-event-also-write-out-the-desired-side-effects-of-the-meeting)3。在日历事件中，也写出期望的会议的*副作用*。

虽然你的大多数会议应该产生功能性的输出，但你也会有一些会议产生副作用。这里列出了你可能关心的副作用。

*   友情——会议结束后，你的团队感觉彼此更亲近了吗？
*   归属/知情-你的团队是否觉得他们被包含在重要信息中？
*   情感支持和表达——你的团队成员感受到情感支持了吗？每个人都有机会表达自己的意见吗？

### [](#dont-fly-blindly-into-a-meeting)不要盲目飞入会场。

很容易忽略会议准备作为一种练习。很难立即看到*停止*你正在为看似不太重要的事情做的“重要”工作的价值。

通过创造这些期望，并对照它们进行测试，让你的会议变得有意义。如果会议未能产生预期，做一个好的开发人员在测试失败时做的事情:重构会议！
