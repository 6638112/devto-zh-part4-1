# 测试代码很棒的 5 个理由

> 原文：<https://dev.to/flippedcoding/5-reasons-testing-code-is-great-8dj>

开发人员是一群奇怪的人。我们同意应始终遵循最佳实践。对吗？然后我们开始测试。一些开发人员喜欢编写测试，而一些开发人员则刻意避免编写测试。我只是想列出一些你绝对应该写测试的理由。

## 它们节省了你调试的时间。

当你有你的代码需要通过的测试时，你是在防止错误的蔓延。直到你写出通过测试的代码，你才能完成列表上的任务。这样，当到了提出拉请求的时候，你就不会受到团队中其他开发人员的抱怨，并且你可以提交更高质量的代码。

您将发现所有这些小问题，比如语法错误或传递了错误的值，这些问题有时会进入管道的下一个阶段。在你为应用程序写了一堆代码之后，与用你的单元测试立即找到它们相比，你会花更长的时间来找到那些小东西。

## 他们让你在写代码之前/写的时候考虑你的代码。

因为你必须编写每个单元测试，你真的必须理解功能应该做什么。它消除了你从用户故事或一般需求中得到的大部分模糊性。当你为单元测试编写代码时，你必须了解你测试的所有东西。

你应该期待什么参数？你为什么用一种特定的方式写代码？你认为用户如何破解密码？当你写单元测试时，所有这些问题都会出现，你必须有一个答案。当你完成某项任务的时候，你会知道它背后的所有细节。

## 他们让你第一次写出高效的代码。

最终你的测试会让你停止编写懒惰的代码，因为你厌倦了它们的失败，你需要一直重构代码。单元测试迫使你从一开始就写出好的代码，这意味着你的第一轮代码是相对高效的。你不必花那么多时间去调试，你会有更多可读的代码。

编写单元测试有助于保持代码简单，因为您只需编写足够通过测试的代码。在你的第一轮编程中拥有简单、易读、抗错的代码会让你作为一个开发者更加自信，也会让你的雇主对你更有信心。

## 他们给你申请的文件。

单元测试是一小段描述用户故事或某种需求的代码。随着时间的推移，所有这些小段代码加起来就讲述了整个应用程序是如何工作的。当一个不同的开发人员跳到这个项目上时，他们将能够通过测试并看到代码的每一部分都做了什么以及为什么。

这可能是你所希望的最好的文档来源。当新的 bug 爬出来或者你需要添加新的特性时，你将有办法确保你的改变不会破坏任何当前的功能。您还将有办法找到应该从哪里开始调试或构建新功能。

## 它们有助于保持部署顺畅。

因为您已经为应用程序中的大多数功能编写了测试，所以您可以在使用单元测试和集成测试部署它们之前捕捉到任何最后的问题。有时你会将每个人的代码合并在一起，而代码深处的某个地方会出现问题。当您没有这些测试中的任何一个，并且您应该在第二天之前部署变更时，您可能会花费几个小时惊慌失措地寻找问题。

当你进行测试时，情况就完全不同了。如果在合并后代码中有什么地方出错了，测试很可能会找到它并告诉你哪里出错了。那么你的恐慌时间可能是几分钟而不是几小时。这最后一分钟的检查可以让你避免很多客户和经理的问题。最好能够告诉他们你知道问题出在哪里，而不是什么都不知道，说它在你的机器上工作。

这是我尽可能多地编写测试的几个原因。当然，开始的时候会花更多的时间，但是这让周五晚上的部署变得更好。你对编写单元测试有什么想法？
