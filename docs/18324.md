# 工程管理:概述

> 原文:[https://dev . to/obahareth/engineering-management-an-overview-1814](https://dev.to/obahareth/engineering-management-an-overview-1814)

我职业生涯的大部分时间都是一名工程师，有一天我成为了一名经理，开始时我非常失落和害怕，我意识到我没有分享我那时的经历。在过去的几年里，我一直在断断续续地做工程管理，我决定分享我所学到的一个简短的总结。

很多这些都是我从书籍、文章、网站中学到的，还有一些是我在工作中学到的。在“伟大的资源”部分的底部是我所学到的所有资源的全部功劳，如果这篇文章引起了你的兴趣，我强烈推荐你去探索它们。我喜欢用“制造者”(工程师、设计师或任何制造东西的人)和“经理”这两个词，这两个词都是我从《引领雪花》一书中学到的。我是从一个人的角度写这篇文章的，这个人在他职业生涯的大部分时间里都是一名创造者，几年前才转向管理层。

# [](#what-to-expect)期待什么

## [](#you-need-to-write-a-lot)你需要写的，很多

会议记录、同事反馈、给你的反馈、在交给工程师之前记录任务等等。把你自己想象成你的组织或团队中文档的标准，因为你现在代表了这一点。你的盘子里会有很多东西，你不可能全部记住，写作是你最好的朋友。

## 你会与人交谈，参加更多的会议

管理就是人与人之间的互动，你将比以前更多地与人打交道，这意味着会议。对你的团队来说，过多的会议是生产力杀手，如果这成为一个问题，你也需要找出如何减少会议的方法。

## [](#learn-to-measure-your-success-by-your-teams-success-not-what-you-make-yourself)学会用你团队的成功来衡量你的成功，而不是你自己创造的。

当我开始成为一名领导者时，我习惯于把自己作为一名工程师或一名“创造者”(用我制造的东西来衡量)。当我刚开始成为一名经理的时候，我常常觉得我在一天中没有完成任何事情，这很困难。

## 你将会面临更多的冲突

你会看到很多人之间的冲突，有时是因为很小的问题。你会花很多时间来处理它们。可能不是每个经理都这样，但对我来说这是工作中最难的部分。

## [](#hard-choices)艰难的抉择

你将不得不让一些人离开，你将不得不介入以化解冲突，有时你会遇到那些尽了最大努力却仍然伤害队友的人。这不是一条容易的路，充满了人的问题和冲突。即使你有一个很好的招聘流程，它也不会完美，有时你会招错人。几乎在任何情况下都没有正确或错误的答案，你需要可以信任的同伴和导师，并寻求指导。不要独自做艰难的决定，更不要在你不冷静的时候做。

## 你需要做很多人不想或不愿意做的事情

改进代码标准、错误报告结构、手工做一些事情直到它们自动化、改进团队和其他部门之间的交流等等。你的目标是让团队保持专注，并消除他们前进道路上的障碍，而这些障碍通常是没有人想做的简单/无聊的任务。

# [](#what-to-strive-to-do)努力做什么

## [](#11-meetings)一对一会议

定期与你的团队成员坐下来，与他们交流，了解他们的目标和他们想要达到和学习的东西，了解什么在困扰他们。将此列入时间表，定期做这件事是非常重要的。写下一对一会谈中的要点，这对你在这项工作中需要做的所有事情都很重要。

## [](#mentorship)师友

你需要帮助你的队友变得更好，有时是在他们想变得更好的方面，有时是在你认为他们为了他们的团队需要学习的方面。这可以通过将他们与在他们希望提高的领域更有经验的人配对，让他们访问 PluralSight 等网站来实现。有时通过亲自指导他们。你必须给他们目标，让他们尝试新事物，帮他们写文章，做讲座，等等。

## [](#scheduling)调度

你可能会参加很多会议，人们需要一种简单的方式坐下来和你交谈。让他们有一个简单的方法来预订你的时间，谷歌日历、 [Freebusy.io](https://freebusy.io/) 、[calendary](https://calendly.com/)等等。都是很好的工具。

你还需要学会在你的制造者和管理者模式之间切换，定期安排在一周的某几天或某几天的某几个小时是很好的。你需要时间投入并回顾一些工作(拉请求、设计等)。)也为他们做贡献。

## 保持你的“创造者”技能

你不会像以前那样有那么多时间去“制作”，你可能会生疏。花点时间让自己跟上你的团队正在做的事情，以及你所在领域的新发展。你需要从总体上了解你的团队在做什么，这样你就可以帮助他们区分优先级。

## [](#keep-a-list-of-tasks)保存任务列表

不要忘记任何事情。把事情写下来。任何人要求你做的事情，记在某个应用程序上，并在上面写上截止日期(我用[来表示](https://todoist.com))。这个列表也非常适合授权。有一个复杂的任务即将到来，你希望队友有经验吗？委派它。今天参加了很多会议，你的队友要求使用一些东西或尝试一种新工具，写下任务，不要忘记。

## [](#hiring)招聘

招聘时你需要考虑的第一件事是这些候选人是否适合你的团队。与你的队友和同事一起坐下来，决定什么是你工作场所的必需品，什么是好东西。你关心在你的组织里写得好吗？你能接受人们在工作中学习它吗？还是连加入都是必须的？

避免脑筋急转弯面试问题，避免人们在工作中永远不会做的理论场景和白板问题，确保你的招聘过程尽可能涉及最真实的场景。让候选人和你的团队一起完成一项实际任务，并给他们报酬。[微软的新流程](https://blog.usejournal.com/rethinking-how-we-interview-in-microsofts-developer-division-8f404cfd075a)是我最喜欢的招聘方式，强烈推荐阅读。不是所有的组织都有资源做这些事情，这是可以理解的，但是在招聘的时候要尽你最大的努力做到这两点:

1.  让候选人做一些真正的工作。
2.  在做实际工作时，让你的队友和候选人互动。

### [](#hire-juniors)雇用晚辈

很多时候，组织只寻找那些已经知道他们需要什么的人，而不会雇佣足够多的初级员工。初级员工在任何组织中都很好，因为他们想学习，而且有很多这样的人。很多时候，在初级员工成长并在您的组织中获得更多技能后，他们会开始从其他地方获得更好的机会，所以有些人认为这不是一项好的投资，但我相信这是组织必须尽最大努力去做的事情，特别是在发展中国家，因为这些初级员工需要成长，他们需要帮助和培训，因为您将有更多的人力来处理许多经常被忽视的小问题。

## [](#let-your-team-grow-delegate)让你的团队成长/授权

作为经理，你需要授权。进入创造者模式去完成一些事情是你以前可能已经习惯的事情，但是你现在需要让你的团队负责大部分的“创造”。

边做边学是最好的学习方法。作为创客，如果有一个领域你是专家，不要马上跳到创客模式并尝试去做，让你的队友去体验并成长。给他们写一个简短的指南，告诉他们需要做什么和需要注意的事情，并相信他们会完成。

## [](#encourage-knowledge-sharing)鼓励知识共享

确保你的队友有一个容易找到和分享经验教训的地方，不管他们有多小(类似于 [StackOverflow for teams](https://stackoverflow.com/teams) 等)。).确保你的团队中没有人害怕或不好意思问问题，定期鼓励你的团队问他们能想到的任何问题，并确保你的团队由对此没有意见的人组成。

## [](#always-be-open-for-feedback-and-change)随时欢迎反馈和改变

我喜欢在我的组织中有一个 GitHub repo，用于与我如何进行管理、1:1 会议如何进行、我作为经理努力做什么、队友对我有什么期望、我对他们有什么期望等等相关的一切。我把它放在 GitHub repo 中，因为工程师们通常习惯于使用 GitHub，习惯于用 pull 请求来改变事情，所以我想成为那样。提出拉动请求，以改变您的经理的做事方式😛

## 让你的团队在没有你的情况下也能正常运作

这并不是说你应该一无是处或者让自己被淘汰，而是说你不应该成为团队的阻碍者。想象自己去度假一个月，会出什么问题？你是什么任务的拦路虎？代替你领导的人知道该怎么做吗？你给他们留下必要的信息了吗？团队知道要做什么吗？如果灾难发生了，他们知道该做什么或者该和谁说话吗？试着偶尔去度个短假，看看哪里出了问题。在开始的时候，你很可能会发现，你作为“创造者”处理的许多事情已经不再被做了，你需要让团队意识到这些事情，并委派给他们。
