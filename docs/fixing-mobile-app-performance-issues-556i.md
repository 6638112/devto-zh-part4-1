# 修复移动应用程序性能问题

> 原文：<https://dev.to/headspin_io/fixing-mobile-app-performance-issues-556i>

[![abandoned-app-abstract](img/bfb764acd8ac14058bb3b52917403964.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--69KmmeIE--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://www.headspin.io/hs-fs/hubfs/abandoned-app-abstract.jpg%3Fwidth%3D800%26name%3Dabandoned-app-abstract.jpg)

“有一个应用程序可以做到这一点。”

自从苹果[在近十年前注册了这五个小字](https://www.cultofmac.com/62892/apple-gets-a-trademark-theres-an-app-for-that/)以来，移动应用变得更加无处不在。普通人每天花在手机上的时间超过 4 个小时(T3)，其中大部分时间花在手机应用上。

毫不奇怪，应用程序已经成为一个巨大的行业。2018 年，移动应用的总收入超过了[921 亿美元](https://www.businessofapps.com/news/app-revenue-reaches-92-1-billion-in-2018-driven-by-mobile-gaming-apps/)。桌面上有那么多钱，竞争是激烈的；Google Play 提供超过 240 万个应用程序，而苹果的应用程序商店包含不到 200 万个应用程序。

在这种背景下，解决移动应用程序的性能问题变得前所未有的重要。用户的采用可能是变化无常和短暂的，移动应用程序的性能问题可能会导致有人完全放弃你的应用程序。一旦用户点击卸载，他们就不太可能再回来。

因此，虽然总下载量或转化率等指标很重要，但它们肯定不是唯一重要的东西。如果你的应用程序滞后、崩溃或表现不佳，你就站在一艘沉船的甲板上。如果你想为自己的长期成功定位，考虑其他重要的绩效指标。

**端到端延迟**

“超速杀人”是父母和警察常说的一句话。但是对于移动应用程序开发者来说，这只是部分正确——速度不够是致命的。

据[谷歌](https://www.thinkwithgoogle.com/marketing-resources/experience-design/speed-is-key-optimize-your-mobile-experience/)报道，移动应用的生死取决于启动和加载时间。如果你的应用很慢，用户会删除它。大多数开发者都知道这一点，所以许多公司专注于开发良好的 API 来提高应用程序的速度。

然而，API 延迟测量并不能描绘全貌。开发人员还应该跟踪支持 API 的应用程序的端到端响应时间。一个好的经验法则是争取一秒钟的反应时间。使用标准化的 API 可以降低延迟增加的风险。开发人员应该尽早并经常运行更新，以利用潜在的改进。

**用户会话**

用户会话衡量一个人使用一个应用程序的持续时间——从他们打开应用程序的那一刻开始，直到他们点击关闭——这是任何应用程序生命中的重要信息。会期长度可以帮助公司争取更长的会期。用户使用你的应用的时间越长，你的收入就越多。用户会话可以帮助开发人员发现客户端-服务器问题。如果用户的位置离服务器太远，或者网络连接不好，会话长度就会受到影响。

会话之间的间隔(应用程序使用的频率)可以帮助验证实验，以提高用户保留率。尝试新事物，鼓励用户再次使用你的应用。开发一个有吸引力的优惠或折扣，发送一个关于新功能的更新，或类似的。然后查看用户会话数据来告诉你什么在起作用。

**崩溃处理**

很少有事情比崩溃更快地赶走用户，因此了解你的应用程序崩溃的频率对提高其性能至关重要。早在 2013 年，一项研究发现，只有 16%的用户会第二次尝试失败的应用程序——用户的期望变得不那么宽容了。但并不是所有的崩溃都是一样的——根本原因取决于许多变量。为了修复崩溃，开发人员需要知道应用崩溃时用户在做什么，设备/操作系统组合，有多少用户受到影响，等等。

为了防止崩溃(和用户流失)，最好在开发过程中彻底测试你的应用。在你的应用可用的地方，在尽可能多的网络上运行真实的测试。自动化您的测试，并设置警报，以便您可以快速响应性能问题。区分崩溃的优先级，首先应对最关键的崩溃。无论如何，继续在发布后测试你的应用，尤其是当你添加或更新功能的时候。
