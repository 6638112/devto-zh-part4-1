# 当我同时学习 Java 和 Python 时发生了什么

> 原文：<https://dev.to/alli/what-happened-when-i-learned-java-and-python-at-the-same-time-1haa>

这是一个关于我如何同时学习 Python 和 Java 的故事，在这个过程中，我学到了一些关于我自己的东西。

## 出发踏上旅程

当我开始目前的工作时，我主要是一名前端开发人员。我在大学里使用过一些 PHP，也涉猎过一点 C# -我可以写一个简单的 SQL 查询-但仅此而已。在我目前的公司，我们的后端主要是 Java 和 PHP。虽然我仍然主要在我们的 React 前端工作，但我一开始就用 PHP 编码，然后逐渐开始用 Java 编码。

事情是这样开始的:一个端点没有给我想要的东西。我可以等待其他人来修复它，或者我可以只是看一看代码...没错。有一部分需要修理。没过多久，我就创建了我的第一个 Java 拉请求。

一些人正在创建一个 Python 学习小组。我一直想学习 Python，但从未开始。为什么是 Python？首先，我认识的 Python 开发者都很爱他们的 Python。其次，Python 听起来很糟糕。当我告诉人们我是一名软件工程师，他们问我用什么编程时，Python 听起来比 JavaScript 酷多了。所以我加入了这个团体。

在小组中，我们正在经历 Zed Shaw 的 *[艰难地学习 Python 的](https://www.oreilly.com/library/view/learn-python-3/9780134693866/)* 。我拿起那本书，开始翻阅。

然后我需要选择一个 6 个月的目标。我考虑过 Python，但我知道 Java 会更相关。我可以到处修改一些东西，但是我对语言基础没有扎实的掌握。我请一位同事推荐一本书，他推荐了凯西·塞拉和伯特·贝茨的《头先 Java》。我制定了一个 6 个月的目标来完成前六章。

我就是这样结束了两本不同的编程书，同时学习两种不同的语言。我仍然主要为我的日常工作编写 JavaScript 代码。我在想什么？

我不知道。

## 我现在知道什么

目前我已经完成了 Java 书的前六章，Python 书只剩下几章了。(Java 书的章节长得不可思议，而 Python 书的章节才几页。)

我绝对不会称自己是这两种语言的专家，不像我对 JavaScript 那样。(你不能把使用一门语言的十年和学习一门新语言的六个月相提并论。)我对 Python 有足够的了解，因此我决定用这种语言来构建一个 Slack bot，作为我未来六个月的目标。我精通 Java，足以在我们的代码库中构建新的特性，并对 Java 代码评审做出更多贡献，而不仅仅是语法上的挑剔。

我也非常了解我的学习风格，以及我喜欢什么，不喜欢什么编程语言。

## 我是如何学习的

我发现比起包含大量信息的长篇章节，我更喜欢短小精悍的章节。我了解到我更喜欢写小程序，而不是做小学生般的练习。

*艰难地学 Python*和 *Head First Java* 是两本截然不同的书。 *Head First Java* 有很长的章节，试图用不同的方式教授不同学习风格的概念。我最喜欢的是结尾的受欺负名单摘要。我发现这一章太长了(我不能一口气读完一整章并做完所有的练习)，而且大部分练习看起来毫无意义。他们让你将代码片段组合在一起创建程序，做填字游戏，甚至进行匹配，而不是真的写 Java。

我知道这本书是有目的地试图让读者通过练习记住这些概念，但我发现它们单调乏味，重复，只是一些我可以满足我的目标的东西。我确实喜欢这本书使用的休闲风格。但问题是:我不打算完成它。我不认为我已经“完成”了 Java 的学习(我永远也不会“完成”JavaScript 的学习)，但是我会找到一些其他的方法。

*艰难地学 Python*每章都有一个很小的概念，章节通常有 2-3 页长。我通常可以在不到一个小时的时间内完成它们，这对于我在 Python 学习小组会议的午休时间完成它们来说是完美的。

不过，最棒的是，我实际上是在编写和运行小 Python 程序，而不是做填字游戏。Zed 让你从书中打出了很多代码，这看起来很傻，但它帮助我习惯了语法。然后 Zed 让你修改代码，破坏东西，确保你理解每一行在做什么，并添加你自己的东西。最后，你正在构建自己的文本冒险游戏，并在游戏中学习新的概念。

*敬酒不吃吃罚酒*的语气比*头头爪哇*和*的泽德肖可要严厉得多，对事情相当*固执己见。(我承认，我们有时候会在 Python 学习小组里取笑他。)然而，这还不足以让我离开这本书，即使泽德建议你记下一系列事情，而你决定，不，你不会这么做。

我被这种所谓的"艰难方式"说服了。我认为有第二本 Python 的硬路书，我可能会选择那一本。我不确定这个小组是否愿意做这件事，但是我们可能需要先休息一下。如果我永远不需要学习另一种语言，我可能会找一本难学的书。有一本《艰难地学 Java》的书。这是另一位作者写的，但使用了基于练习的简短章节，所以我可能会捡起来。

## 我如何编码

比起 Java，我更喜欢 Python。

我真的很喜欢 Python 好吗？没有 JavaScript 多，至少现在还没有，但我可以看到它是一种可能性。起初我对 Python 关于间距的特定规则很反感，但现在我完全同意了。Python 代码看起来不错。Python 代码总是看起来很好，因为不管是谁写的，什么时候你都要努力让它变得难看。最终你会停止得到关于间距的错误。

你知道我还会犯什么错误吗？Java 中的类型。在实际代码中比在我们工作中的实际 Java repo 中更少。我理解强类型语言的原因。我用过打字稿。我和一群 C#开发人员一起工作了四年。我总是在我的 React 组件中使用 PropTypes。我明白了。但是我不喜欢被*强迫*使用类型。我尤其不喜欢必须决定我想要使用哪种数字类型。我知道在 Java 中，你关心内存分配，你想选择使用最少内存的类型。不过，我不想太担心内存分配。*我不觉得这很有趣。就像我不喜欢在电子游戏中处理库存管理一样。

我不喜欢 Java 的另一点是所有的重新编译。我记得有一次我一直在修改我们的 Java 代码库，但是它们不起作用。我告诉它`System.out.print`的事情，什么也没有印出来。然后有人提醒我，我必须停止并重启应用程序。布雷克。我想我会写一些自动化的东西，这样当我保存一个更改时，它会自动为我重启。我看看能不能用 Python 写出来。

## 一次两个

一次学一种语言可能是更好的方法。我不会推荐它。它确实给我带来了意想不到的好处，让我能够在学习过程中比较不同的语言。我想给 Java 书一些信任——它在解释一些概念方面做得更好，比如面向对象的设计和编程。看到不同的语言如何处理诸如继承之类的事情是很有趣的。

同时学习两种语言肯定会使我出错。如果某一天我正在阅读 Python 书籍，第二天又试图编写一些 Java 代码，我就会开始使用错误的注释语法或者忘记使用分号。它甚至让我的 JavaScript 出错:有几次我忘记了我不是在写 Python，忘记了在 if 语句条件两边加上括号，并且想知道为什么 VS 代码给了我红色的曲线。

除了完成 Python 这本书，我还打算继续通过兼职项目练习这门语言。我经常用 Java 编写代码，希望我能找到一些更适合我的资源。

我意识到这比我想象的要复杂得多。这不是一个关于不同语言如何处理内存分配的帖子。这只是我对编写 Java 的一种本能的厌恶。
