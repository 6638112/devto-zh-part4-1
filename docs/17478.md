### 1。介绍:T2

有一句话一直萦绕在我的脑海中，据说是林肯说的:“给我六个小时去砍树，我会用前四个小时去磨斧子。”基于这些可爱的名言的历史，他可能没有真的说过，尽管如此，这种观点是正确的:花适当的时间“磨利”和“切割”会有很多收获。在这种情况下，“磨砺”就是阅读。

我非常支持程序员读书。我知道 Google、StackOverflow 和 GitHub 为我们提供了神圣的三位一体的帮助，但是我们也很容易变得自满和依赖它们。他们只能帮你找到你能认识到的问题和别人已经遇到的问题的解决方案。如果你甚至不知道要搜索什么，或者你甚至不知道你的 20k line“Helper”类有 0%的测试覆盖率的问题，Google 就帮不了你。

用 RPG 术语来说:书籍让我们从作者那里免费获得经验提升，并提升我们自己！就像不指望每个土木工程师都能独立地重新发现混凝土一样，我们也不应该把编程当作一种“砍砍”式的实践。真正学习 git 比阅读更多“git 工作流”文章花费的时间更少。

这个列表可能看起来令人望而生畏，但这只是因为我给出了几个“或”选项，并给出了几个特定于语言的例子。所以对于刚开始的人来说，这真的只是十几本书。事实上，如果你作弊到底，你会看到我的 tldr 部分，其中包含了绝对最基本的书籍，不管你的编程语言。

内务处理:

我知道书籍不是磨利你的斧子的唯一途径，甚至不是 T2 最好的途径。在线参考和文档是非常宝贵的。谷歌是免费的，全天候为你服务。但是对于那些**不会随着每个小版本的改变而改变的重要的东西，比如职业建议、算法和 Big-O 注意事项，有时一个编辑过的页面是最好的。如果你只是“知道”你不能从书本中学习，那么这篇文章可能不适合你，但是我希望你能再给书本一次机会。**

这都是为了*补充*你的其他自我教育手段。我知道林肯的名言建议你花更多的时间打磨而不是切割，但这是不现实的。我建议，当你处于“稳定状态”时，分配 10%左右的工作时间用于学习。如果你在一个新的团队或项目中，那就向上提升。如果你要发射，可以降到零，但不要把它留在那里。

纸质还是电子书？:没关系。无论你喜欢哪个，你都会去读。

**有声读物？**:好吧，这一条很重要。我已经把一个🔊(演讲者表情符号)放在有声读物旁边。

### 2。重要的专业建议

这些是最重要的，因为如果你读了这些，并遵循他们的建议，你就不需要读这篇文章的其余部分。你已经在寻找新的信息和磨练你的技艺。

我也称之为“至关重要”,因为它们是我读过的最有影响力的书籍之一。当我说这些书改变了我的职业生涯时，我并不是老生常谈。

*   [工程的不成文法则](https://www.amazon.com/dp/0791801624)🔊这是这份清单上最古老的书之一。它原本是为机械工程师准备的，提供具体和一般的非技术性职业建议。它侧重于我们今天所说的“软”技能。这个领域如此重视技术能力，以至于我们经常认为这些“软”技能在某种程度上低于“硬”技能。没有什么比这更偏离事实了。如果你不花时间学习如何驾驭你的职业生涯，你将会像一个穷途末路的人一样:你不会很快到达任何地方。我只希望我能早点读完这本书；这会让我在早期省去很多麻烦。
*   [成为技术领导者](https://www.amazon.com/dp/0932633021)💎像许多这样的书一样，不要让标题欺骗了你:这不仅仅是为那些计划成为“技术领袖”的人准备的。这是为科技领域的任何人准备的。如果你拿起这本书，你必须完成练习，以获得完整的效果。这是值得的。这就像有了自己的治疗师、生活教练和导师，除了只有你和一个笔记本在回答非常重要的问题。
*   [务实的程序员](https://www.amazon.com/dp/020161622X)💎这真的是整个列表中的“创世纪”书。我认为这本书比《干净的代码》和《完整的代码》加起来要好 10 倍！(虽然那可能只是因为我先看了 PragProg？)顾名思义，这本书提供了更多的*战术*建议，但也给出了很好的职业建议。最著名的是“每年学习一门新语言”这种建议似乎有点多，但在我的职业生涯中，我不得不用十几种不同的语言编写代码，尽管我编写的 90%的代码都是用一种语言编写的，但快速轻松地掌握新语言的能力是一项必须具备的坚实技能。这只是这本书的一个特别的提示。

### 4。实际有用的算法和数据结构

既然乱七八糟的都出来了，那就来说说*真正的*算法，而不是愚蠢的琐事谜题。

实际算法很重要。它们是 O(2^n 和 O(n)的区别。如果你还没有享受过将程序运行时间从几个小时缩短到几秒钟的乐趣，那么这是非常了不起的。如果你对欧拉项目感兴趣，可以尝试一下。

算法可以给你 1000 倍的性能提升，但更重要的是，它们可以帮助你分析 T2 为什么 T3，并在未来避免再次发生。

几乎适用于所有语言的一个技巧是“不要在循环中追加字符串”

这是为什么呢？因为如果追加是一个 O(n)运算，而不是 O(1)，那么在循环中做是 O(n^2).即使如此，如果你只是遵循 Big-O 符号的“循环计数”方法，你也不知道为什么。

我真正喜欢这两本书的地方是它们对其他实现的参考。在现实世界中，如果你正在手写，从头开始，你将会有一段糟糕的时间。

“但我不能谷歌一下这些吗？”当然，但是你怎么知道你在找什么呢？你怎么知道你正在寻找一个有向无环图的最小凸包解呢？

*   [算法简介 CLRS](https://www.amazon.com/gp/product/0262033844)T2 算法教科书。我推迟买这本书，因为它价格昂贵，但它确实很好。它对最重要的算法和数据结构提供了如此深入和平易近人的解释。我特别喜欢这种方式，即使这是一本教科书，你也可以专注于你想学的东西。
*   [算法设计手册](https://www.amazon.com/gp/product/1848000693)💎在目前列出的所有书籍中，我想把这本书放在我的书桌里。我说“在”而不是“在”，因为我不会经常需要它。当你真的需要它的时候，它可以成为一个项目救星。它有令人难以置信的描述和与算法相关的指导性问题(例如，“边是有向的吗？”).一些算法是可读的 C 语言文本，所有都是在线的【algorist.com T2】。你的团队应该至少有一本。我总是在我的本地机器上保留一份源代码的副本作为参考。

### 6。每种语言都有一个

最后，我们有大多数人认为是“编程书籍”的书籍

我喜欢这些，因为它们有几个优点:

*   高质量的工作代码示例
*   深入研究这种语言
*   综合教程
*   习语和风格的例子
*   语言设计和注意事项
*   易处理的问题

对于新语言来说，将教程作为语言网站的一部分更为常见，但这仍然不常见。

更常见的情况是，如果你试图搜索“如何学习 x 语言”或一些类似的查询，你会被语言库参考和数以千计的可怕介绍淹没。

考虑一下什么时候你的时间比你的钱更有价值，或者至少确定一个界限。考虑到当你第一次学习一门语言时，保持严格遵守纪律是很难的，更不用说*和*做寻找好材料的跑腿工作了。

我不是说要买每一本关于语言的书；事实上，我的经验法则是每种语言每年不超过 1 本书。尤其是等待“有效”的书籍，直到你至少熟悉一种语言，否则这些提示会让你难以理解。(当你开始一份新工作，需要快速掌握一门新语言时，请打破这条规则)。

示例:

*   K & R C 💎即使不用 c 写也推荐。
*   [加速 C++](https://www.amazon.com/gp/product/020170353X)
*   [有效 Java](https://www.amazon.com/dp/0134685997)
*   [Clojure 编程](https://www.amazon.com/dp/1449394701/)
*   [围棋之旅](https://tour.golang.org/welcome/1)🆓

### 7。中级到高级

最后，有了职业生涯的坚实基础，基本的相关计算机科学，和你的主要语言，接下来是什么？

嗯，实际上很多。我们的知识库越来越深。

我建议在你有几年时间之前不要继续读书。不是出于某种“狠狠的敲打”的心态。这样你就能看到足够多的问题，知道解决方案在哪里，而不是有一堆解决方案渴望解决问题。

首先，您已经获得了一般的代码质量和模式。不要太宠他们，但是“代码质量”的书基本上只是测试驱动设计的伪装。你会发现能够使用代码，不管它是不是“遗留的”，都归结于测试。如果你没有测试，你的代码是脆弱的。不管它是最近提交的，如果它没有经过测试，它就已经是遗留的了。

*   [有效地使用遗留代码](https://www.amazon.com/gp/product/0131177052)💎我真希望他们没有把“遗产”放在标题里。这本书在一个项目的第一天和第一千天一样重要。我曾经遇到过初级开发人员问我的问题，这些问题就是本书中的章节名称；太诡异了。
*   老实说，这本书在现代 ide 中已经过时了，但是对于没有企业级 ide 的语言来说，这很方便。

第二，随着开发人员的进步，您会注意到代码中的模式，尤其是在面向对象代码的设计中。我们称之为设计模式。

我已经看到了一些围绕设计模式的愚蠢行为，比如不得不在你的代码中使用一些模式来获得提升。

最重要的是，记住:它们只是普通模式的名称。它们不是需要编码的东西(过度工程化)，但是你也不应该重新发现它们(欠工程化)。

一些语言可以通过像 Ruby 这样的元编程或者像 Common Lisp 这样的宏来完全避免它们。

*   [头先设计图案](https://www.amazon.com/gp/product/B00AA36RZY)
*   [设计模式](https://www.amazon.com/gp/product/0201633612)

最后，向“高级”工程师或“技术领导”角色的重大转变都与元工作有关:处理工作的工作。
简单来说:项目策划。

这里存在悖论，比如:

*   周五部署是极端风险或极端信心的标志
*   作为一名“摇滚明星/牛仔/忍者/超级英雄”开发者意味着你实际上工作做得很差
*   让开发人员加入一个迟到的项目只会让它更晚

### 8。奖金类别

如果你想破产和痛苦，跳过这一节。

我只是半开玩笑地说，如果你不知道“财务”是什么，或者“哲学”与现实世界有什么关系，“那么是的，你可能会有点破产，有点痛苦。公平地说，无论发生什么，你仍然可能破产和痛苦，所以这也不是一个好的论点。

好消息是，你不需要花太多(或真的任何)钱来接受基本的个人理财教育。我会让下面的消息来源为我说话，但要知道，当你管理好自己的钱，你就是在给自己加薪，这并不是老生常谈。

*   [reddit.com/r/personalfinance](https://www.reddit.com/r/personalfinance)🆓💎尤其是[总理指令](https://old.reddit.com/r/personalfinance/wiki/commontopics)

我认为，因为我们的工作通常更内向(我们经常独自工作)和大脑(我们更多地在大脑中工作，而不是在手中工作)，我觉得开发人员可以从哲学中受益。(剧透:大家也可以！).
