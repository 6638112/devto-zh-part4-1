在我的职业生涯中，我有幸与来自世界各地的团队一起工作了几年，每个团队都有不同的方式、文化和经历。当然，从个人和技术角度来看，都有高潮和低谷。

这让我经常反思每个团队的软技能能力不仅影响技术成果，还影响个人健康和工作满意度。这就是为什么我每天都努力提高和反思自己的软技能，并与他人分享我的想法和建议。

这篇文章最终写下了我对成为一个好队友的想法，以及我希望我未来的每一个同事都知道的事情。

## [](#before-we-start-what-are-soft-skills)开始之前:什么是软技能？

软技能是沟通、情感、社交和人际交往技能的组合，它使我们能够与其他人很好地合作。

这些经常被忽视，然而，在软件开发世界中，一组关键的技能，在我看来，每个软件开发人员都应该理解并努力发展他们职业生涯的开端。

当然，写好代码很重要，但是我倾向于把硬技能和软技能放在对每个程序员同等重要的位置上。在我看来，一个伟大的程序员(或者臭名昭著的 10x 工程师)也需要是一个伟大的队友。这两件事密切相关。

如果你即将开始你的第一份编程工作，或者即使你已经是一名老手，请继续阅读并思考以下几点。

> # [](#a-great-programmer-or-the-infamous-10x-engineer-also-needs-to-be-a-great-teammate-the-two-things-go-handinhand)A great programmer (or notorious 10x engineer) also needs to be a great teammate. These two things are closely related

## [](#develop-empathy)培养移情能力

同理心是理解另一个人感受的能力。暂时把自己想象成另一个人，试着想象成为那个人是什么感觉。在我看来，这是每一次互动的基础。

作为必须与许多人合作的开发人员，有些人您可能喜欢或不喜欢，我们的同理心每天都会在各种情况下受到多次挑战，例如:

*   配对和复习时

*   收集需求时

*   调试别人的代码时

*   当领导其他程序员时

### [](#the-importance-of-language)语言的重要性

语言和交流在我们的日常生活中有着巨大的影响:很多时候，我注意到人们表现出居高临下或冷漠是因为文化和语言相关的技能，而不是他们缺乏同理心；特别是对于一些没有 100%掌握某种语言的人，往往比母语人士更容易被误解。

由于我工作的团队的多样性，这样的情况每天都在发生；有时候不沮丧是不容易的。当一个人说另一种语言时，你会面临两个问题:词汇选择和文化方面。

当然，用词的选择会受到文化方面的影响:例如，在英语中(尤其是在英国)，我们经常在句子中使用“对不起”、“谢谢”和“请”。

当然，我们希望其他人知道这一点，并在适当的时候使用相同的词语。但是如果我们的对话者的语言不经常使用这些词(或者根本不使用！)?

我本人不是一个以英语为母语的人，这就是为什么我能在这种情况下与其他人相处的原因:我对他们感同身受。

简而言之，这就是程序员感同身受如此有用的原因。这是一项我们每天都需要使用的技能，即使我们并不真的知道。

## [](#listen-then-talk)听，然后说

这大概是放之四海而皆准的常识，但需要列出来。毕竟，作为开发人员，我们努力在生活的各个方面都保持高效。

打断你的同事是我在会议中看到的最常见的行为之一。如果你想知道的话，这肯定会让你的同事感到沮丧。

首先认真倾听是非常重要的，不仅是出于尊重，即使你不同意每一个单词。你可能认为这是浪费时间，然而这是应该做的事情。

> # [](#listen-carefully-then-talk)Listen carefully, then speak

## [](#teach-but-also-learn-at-the-same-time)教，同时也学

当你试图表明自己的观点、解释某事或指导他人时，很容易让人觉得你居高临下。

沟通是极其困难的，所以你应该预料到这种情况会发生，即使你完全不是故意的。

在讨论时，我们有一些方法可以避免听起来居高临下:

*   当然，谨慎选择你的用词是很重要的，我的建议是尽可能保持中立:例如，我会说它“不是最佳的”或者“它可以更好”，而不是“一个糟糕的方法”

*   停止命令，开始建议:不要说“你应该那样做”，你可以说“我的建议是那样做”

*   不要只谈你的观点；定期提问，并检查你的对话者对你正在讨论的问题的看法

*   不要以为你对这个主题了如指掌

    > # Communication is extremely difficult, so you should expect this to happen, even if you didn't mean it at all

## [](#accept-criticism-amp-be-open-to-change)接受批评&虚心求变

我们的行业是创意行业。

对我们来说，我们的代码就是艺术，就像一幅画对于一个艺术家或者一首歌对于一个作曲家一样。

作为开发人员意味着每天都要接受这样或那样的评判。当我们倾向于认同自己的产出时，**我们在内心深处对我们的代码、想法和观点做出任何判断**。

我们工作中最难的部分之一是接受他人的批评，无论是在面试、会议、公关评估等过程中。

我不认为我自己已经抓住了这一点，但是从我成为职业球员的第一年开始，我确实有了很大的进步。

我们花了数年时间在特定的技术上打造我们的技能，而久而久之，我们已经习惯了我们做事的方式，或者我们团队的方式。因此，**我们变得越来越抗拒改变**，尤其是当它们来自第三方的时候。

还记得那个想开始重构整个代码库的新人吗？还是那个想引入 Go 替代你的 Java 栈的？我敢肯定，我们都在某个时候经历过这样的情况:我不认识你们，但对我来说，他们(非理性地)认为这些是人身攻击。

可以理解的是，**开发者讨厌变化**。

如果你没有注意到，这是双向的:

*   这位新员工喜欢他以前公司的架构，并迫不及待地想用他习惯的方式重写一切，因为他 100%确定这样更好

*   相反，团队习惯于他们编写代码库的方式(或者他们继承代码库的方式),不打算让新员工改变一切

当然，这是一个普通的场景(尽管非常普遍)，任何一方都可能是对的。也许当前的架构真的很糟糕，需要重构，也许这个人只是非常固执己见。

关键是，如果当别人提出改变某事的可能性时，你立刻感到受到了攻击，那么问题出在你身上，而不是想法本身。在你完全理解一种新方法之前，你会不理智地对它产生抵触情绪。

我想这种事经常发生在我身上。我就是这样，其他人也是这样。我如何确保我不会因为自己而反对这个想法？

*   我仔细地听它

*   我请我的同事给我一些时间来做一些研究并考虑一下

*   我带着诚实的想法回来了

    > # If you immediately feel attacked when someone else raises the possibility of changing something, then **the problem lies with you** , not the idea itself.

## [](#general-advice)一般建议

*   如果你认为一个同事做得很好，不要退缩，真诚地赞美他们

*   到期时给予他人信任

*   尽可能透明:与你的同事谈论澄清、改变和意见。例如，当某人不在办公室时偷偷提交并不是重构他人代码的好方法。一起开诚布公地谈论和思考是

*   有时候，如果你什么都没做错，并且听从了所有可能的建议，冲突就会发生；记住这是绝对正常的，我们无法控制一切。公司和人类是复杂的，这只是对每天实际发生的事情的简化

## [](#takeaways)外卖

当然，这是我的个人观点，关于什么是好的团队伙伴，以及一个全面发展的软件工程师应该如何在工作场所与同事相处。这些都不是科学方面——只是我基于经验的看法。

总之，以下是我希望在下一位同事身上看到的一些方面:

*   同理心让你成为更好的人，尤其是更好的同事

*   试着理解对方的观点，考虑文化、语言以及他们的社会规范与你的有何不同

*   在听到别人的充分论点之前，不要拒绝别人的意见

*   在容易产生误解和冲突的情况下，尽量不要居高临下

*   试着更开放地接受批评和改变，理解什么时候你会非理性地为自己的观点辩护

*   有时候，从你的角度来看，即使做了所有正确的事情，糟糕的情况还是会发生；不要气馁，这是正常的，这些将成为你长期职业生涯的经验
