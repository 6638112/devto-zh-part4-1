# (奇妙的)XY 问题以及如何避免它们

> 原文:[https://dev . to/briwa/fantastic-xy-problems-and-how-to-avoid-avoid-them-11 C4](https://dev.to/briwa/fantastic-xy-problems-and-how-to-avoid-them-11c4)

你有没有遇到过一个问题，让你想知道它背后的目的是什么？也许是这样的:

> 你知道实现这个的正则表达式吗？`https://dev.to/new`=>= T1】

(老实说，我不知道这个问题的答案，有人知道吗？)

由于~~我的弱正则表达式游戏~~我的好奇心，我会改为问，“上下文是什么？”。没有适当的背景，很难定义解决方案。正则表达式必须包含查询参数吗？URL 哈希怎么样？URL 的深度是什么水平？它是当前页面的 URL 还是仅仅是任何一个 URL？诸如此类。

如果上下文是关于当前页面的 URL，可以用 [`URL.pathname`](https://developer.mozilla.org/en-US/docs/Web/API/URL/pathname) 代替:

```
// When the URL is "https://dev.to/new"
window.location.pathname // Returns "/new" 
```

如果这确实是上下文，我们都知道正则表达式根本就不需要。谁知道如果从来没有提供上下文，反而会出现一个错误的解决方案。时间可能已经被浪费了。

* * *

根据我的理解，前面提到的例子是一个简化的 XY 问题。简而言之，当一个问题试图用手边的固定解决方案来解决时，会使其他可能的(也许更好的)解决方案无效，同时忽略问题的背景。我曾经认为这是一个只发生在过去的经典问题，而事实上这在今天非常普遍，甚至在编程用例之外。

再拿一些(极端的，假设的，也许是荒诞的？)示例:

*   由于空调带来的冷空气，办公室将为员工提供冬衣；他们不应该首先问为什么空调要那么冷吗？
*   由于测试覆盖率的惊人下降，所有没有任何测试的拉请求将被机器人自动关闭；难道不应该问开发人员为什么他们一开始不写测试吗？或者有没有更好的方法来做到这一点，也许阻止公关合并代替？
*   开发人员将被要求在他们修改的代码上方写下他们名字的首字母作为代码注释，因为在产品中会出现错误，而且没有人会把错误归咎于谁；我的意思是，当然，但是我们不是已经有了吗？或者是跟踪部署中的错误并指出漏洞版本的服务？难道不应该改进 QA/代码审查过程吗？

我希望这些都不相关。如果我们这样看，这些解决方案的缺陷是如此明显，以至于不可能忽略它们。事实上，没有其他的解决方案会让你从一开始就感到困惑。但是，当它发生在一个会议室里，挤满了具有相同层次背景(或缺乏背景)的人时，它可能不会那么明显，结果可能会出现一个不那么深思熟虑的解决方案。

* * *

如果你(或任何人)将要解决一个问题，并且你想避免 XY 问题，不管它是一个编程问题还是任何一般问题，我建议在解决它们之前尝试以下步骤:

1.  后退一步，再次观察这个问题。
    不管你是否已经有了解决方案，尽可能多地从问题中获取背景信息。考虑解决问题所需的所有情况。检验你是否对问题做了足够的尽职调查的一个方法是，如果有人问你关于这个问题的任何问题，你已经有了强有力的答案。如果没有，记下并继续观察问题。
    如果你目前没有解决方案，马上提出这些背景。此时，他们应该已经具备了制定解决方案所需的一切。也许你已经尝试过 StackOverflow 或者 StackExchange 了。如果这是关于决策，并且你已经有了一些解决方案的想法...

2.  想出不止一个，而是多个解决方案。
    有一个最喜欢的解决方案没关系，也许是你已经是专家的东西，或者是你在实现方面更喜欢的东西。然而，对参与该过程的其他人(以及其他解决方案)公平是很重要的。想出一个以上的解决方案，这样就有东西可以比较了。如果你什么都想不出来，那也没关系，只要你能坦诚面对。有了足够的背景，你可以发起一个头脑风暴会议来探索其余的解决方案。

3.  权衡所有的利弊。
    根据利弊对所有的解决方案进行排序会使决策过程更有效率。这将把好的放在聚光灯下，并消除坏的，而不是一个接一个地随机检查每个解决方案。此外，利弊必须在所有方面得到验证:资源、对性能的影响、可维护性、长期支持，我想你明白了。
    有时，这个过程可以判断候选解决方案是否真正合法，而不是一个最喜欢的解决方案和一个故意混合的坏解决方案。如果发生的是后者，这是一个糟糕的头脑风暴会议的标志，重复步骤 2。

4.  尽可能保持开放性。
    最后但并非最不重要的一点是，记住你不是在告诉人们同意你的观点，并使用你喜欢的解决方案来解决问题。整个过程就是为了*找到*解决问题的办法。这从来不是关于谁是对的或者谁的解决方案是最好的。所以，保持它的开放性是必要的；唯一被修正的是问题本身，而不是解决方案。
