# PHP 和 Ruby on Rails——选择哪个作为你的下一个网络产品？

> 原文：<https://dev.to/ideamotive_co/php-or-ruby-on-rails-which-one-to-pick-your-next-web-product-5gno>

德高望重的老兵还是神童？PHP 和 Ruby on Rails——web 开发应该选择哪一个？答案并不明显。

当谈到互联网开发时，有两种技术已经把用健壮的 HTML 制作的静态网页转变成了今天的交互式多媒体技术。第一个是 JavaScript，使用户能够完善解决方案的前端。第二个是 PHP，它位于前端和后端之间，是包括 WordPress 和 Magento 在内的大多数流行解决方案的后台。

但是，最著名的技术之一在 2019 年仍然相关吗？它能与现代语言和框架如 Ruby on Rails 相比吗？

什么是 Ruby on Rails
Ruby on Rails 是一个 Ruby 语言框架，旨在让 web 开发人员更容易、更快、更方便地编写代码。该技术强加了许多好的实践，包括鼓励开发人员重用代码块的 DRY(不要重复自己)。

此外，该工具遵循配置路径的约定，使开发人员能够通过建议预定义的环境来最小化配置工作。

Ruby on Rails 是世界上最受欢迎的技术之一，支持超过 824，000 个网站。它被列入了由 [TIOBE Index](https://www.tiobe.com/tiobe-index) 发布的世界上最受欢迎的 20 项技术的名单中，并在 [StackOverflow 最受欢迎的编程语言排名中排名第 13 位](https://insights.stackoverflow.com/survey/2018#technology)。而且有很多[牛逼的网站在用这种技术](https://ideamotive.co/blog/40-best-ruby-on-rails-companies-websites)。

什么是 PHP
PHP 是一种脚本语言，旨在实时交付网站和 web 应用。它是网络中最著名的技术之一，被现存的 79%的网站所使用。PHP 支持流行的开源解决方案，WordPress 和 Magento 是其中最流行的。脸书最初是用 PHP 开发的，后来被翻译成了 C++。

关于 PHP 有趣的事实是，每年评论员和专家都在争论这种语言是已经死亡还是即将死亡。但这一次 PHP 与马克·吐温有很多相似之处，因为关于其死亡的报道是夸张的。

为什么要比较呢？
事实上，拿 PHP 和 Ruby on Rails 做比较并没有那么公平。PHP 是一种编程语言，而 Ruby on Rails 是一个框架。因此，将 RoR 与一些 PHP 开发框架进行比较要公平得多，比如 Zend 或 Symfony2。

但是对于 PHP 编程来说就不是了。大多数开发人员更喜欢挑选自己的库和插件来在他们的项目中利用，有效地创建自己的框架。由于 PHP 的遗留问题，大多数流行的框架使得编码变得更加复杂，Ruby on Rails web 开发框架也是如此。所以实际上，在框架和框架基础上进行比较是有缺陷的。

那么这些技术相比之下表现如何呢？

ruby on Rails vs PHP——用户界面
PHP 作为连接网站和数据库的工具有着强大的背景。它的设计不是为了让网页光彩照人，而是为了让用户能够访问大量的内容。在这一点上，它工作得很好，但是为了提供令人惊讶的用户体验，它需要 JavaScript 等其他技术的支持。

Ruby on Rails 以提供能增强用户体验的闪亮而精致的网站而闻名。这是一项现代技术，诞生于 UX 原则之后，完全支持建立以用户为中心的网站。

结论:Ruby on Rails 超过了 PHP。双倍时间。

Ruby on Rails vs PHP——架构
Ruby on Rails 因采用模型-视图-控制器范式而受到称赞，它有效地迫使用户遵循构建应用程序的最佳实践。因此，当用户正确使用时，框架会带来性能良好且稳定的架构。

如上所述，PHP 通常用于 saute，每个开发人员都构建了自己的技术栈。它给予自由——包括交付混乱代码和不稳定架构的自由。质量只取决于开发人员的技能。

结论:所以，Ruby on Rails 再次大大优于 PHP。

Ruby on Rails vs PHP——编码速度
在 DRY 原则和强制配置之上的惯例下，Ruby on Rails web 开发使开发人员能够做他们最擅长的事情——编码。它减少了通常浪费在配置和交付编程环境的日常工作上的大量时间。

PHP 通常又来 saute，没有工具来加速配置或任何其他方面。因此，尽管这种语言相对容易，但在最终解决方案的实现过程中，它可能会让人头疼。

判决:对不起，PHP。这次不会了

Ruby on Rails vs PHP–性能
Ruby on Rails 既快又优雅。它交付由良好的编程实践和可靠的架构支持的高性能代码。

PHP 是一种编程语言，所以很难确定它的性能是否优于 RoR。完全自由地交付精彩的(或蹩脚的)代码。

但是假设只有开发明星在船上，PHP 在 Rails 上的表现优于 Ruby。但是说实话，真的有吗？无论你是在寻找 Ruby on Rails web 开发人员还是 PHP 程序员，要找到一个明星都不是那么容易的。

判决:PHP。但只有做得好

ruby on Rails vs PHP——社区和支持
PHP 是著名的、古老的、防锈的和永不沉没的。考虑到这一点，对 PHP 程序员的需求很大。此外，自 1995 年 PHP 问世以来，已经有了一个庞大的技巧库，Q & As，以及已经解决的问题。此外，GitHub 或 Stack Overflow 上有许多 PHP 开发人员在语言本身和 WordPress 或 Magento 等特定技术方面提供帮助。

Ruby on Rails 是一项非常年轻的技术，但是已经很成熟，有一个蓬勃发展的社区。但是没有办法超越 1995 年以来的经验。

结论:PHP，有时候什么都比不上经验。事实就是如此。

Ruby on Rails vs PHP——稳定性
在 MVC 和 DRY 范例的强迫下，Ruby on Rails 在稳定性方面几乎没有问题。通过重用代码块，开发人员可以确保没有隐藏的依赖项或应该作为变量插入的硬编码信息。

有了 PHP，就有了完全的自由。这意味着机器人创造或毁灭的自由。没有强加的规则，但程序员的技能，稳定性严重依赖于编码的技能。这可能意味着任何事情——从完全牢不可破到雷区。

结论:RoR 提供某种程度的稳定是理所当然的。PHP 的范围从“牢不可破”到“雪花”。当接近一片未知的土地时，Ruby on Rails 似乎是一个更好的选择。

Ruby on Rails vs PHP–文档
Ruby on Rails 附带了清晰、易读且符合逻辑的文档，支持 Ruby on Rails 程序员工作的各个方面。

相反，PHP 附带的文档是在 1995 年设计的，那是良好实践出现之前的时代。有时千疮百孔或不一致，通常由于互联网上广泛的帮助而被忽略。

结论:Ruby 提供了更好的文档。PHP 缺一个，但是这次真的重要吗？

Ruby on Rails vs PHP——平台的成熟性 Ruby on Rails 是这些技术中足够成熟的一种，可以解决初期的问题，同时又足够现代，可以与最新的趋势兼容。

判决:PHP。也许 Fortran 在这方面更加成熟。

总结:
什么时候用 PHP？
PHP 是目前最流行的 web 语言。它的流行来自大量使用它的开源软件。因此，如果问题是完全标准的，没有任何角度可以调整，PHP 可能是一个不错的选择。毫无疑问，一个完全标准的博客可以在 WordPress 上运行，就像 Magento 可以支持电子商务一样。

但是如果你寻找更复杂的东西，上个千年的技术可能是一个冒险的选择。

什么时候使用 Ruby on Rails？
[Ruby on Rails 开发](https://ideamotive.co/services/ruby-on-rails-development)是一项成熟稳定的技术，支持更复杂的解决方案。此外，作为一个以网络为中心的框架，RoR 不仅能让公司数字化，还能为业务流程增加新的可能性。当公司需要的不仅仅是“标准”或“典型”的解决方案时，Ruby on Rails 是一个更好的选择，尤其是当你需要一些[复杂的电子商务产品或 SaaS 产品](https://ideamotive.co/blog/ruby-on-rails-saas-product)时。

老实说，你希望你的公司成为“典型”和“标准”的公司吗？
