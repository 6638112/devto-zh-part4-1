# 为什么我默认旧的方式

> 原文:[https://dev . to/ferricoxide/why-I-default-to-the-old-ways-5a fa](https://dev.to/ferricoxide/why-i-default-to-the-old-ways-5afa)

我和一个不断壮大的自动化工程师团队一起工作。大多数是纯粹的开发类型。那些生活在操作世界中的人，完全倾向于 Windows，或者只需要非常轻松地处理 UNIX 或 Linux。

另一方面，我从 1989 年开始使用 UNIX 版本。我的第一个 Linux 系统是 1992 年从麻省理工镜像下载发行版的结果。结果，我有很多*老*的习惯(说真的:我的一些习惯比我的一些队友还老)。而且，因为这些年来，我不得不多次深入到所有这些操作系统的杂草中，*很多次，这些习惯相当根深蒂固(“血学习”和所有那些腐烂)。*

大约一年前，我提交了一份 PR，其中包含了一些带有大量正则表达式的 shell 脚本。审查 PR 的人曾问“为什么你在正则表达式中使用`[<SPACE><TAB>]*`而不仅仅是`\s`？”。当时，我想我的回答是半油嘴滑舌的:“A)旧习难改；B)我知道前一种方法总是有效的”。

也就是说，我是一个懒惰的打字员。键入“`\s`”比“`[<SPACE><TAB>]*`”的击键次数要少得多。类似地，“`\s`”占用的列宽比“`[<SPACE><TAB>]*`”少得多(我/我们通常喜欢按照相当标准的页面宽度编码)。因此，出于懒惰和专栏保守的原因，我开始更多地使用“`\s`”，而不是使用“`[<SPACE><TAB>]*`”。我想在过去的 12 个月里，我几乎完全转向了“`\s`”。

今天，这一举动让我很不爽。嗯，实际上是昨天，因为那时我开始收到报告，说我在 EL7 上创作的工具在 EL6 上安装/使用时不工作。最终，我将问题追溯到一个`awk`调用。具体来说，我有一段代码(过滤 DNS 输出),看起来像:

```
awk '/\sIN SRV\s/{ printf("%s;%s\n",$7,$8)}' 
```

这在 EL7 上很管用，但在 EL6 上“不太管用”当我把它改成老式的调用时:

```
awk '/[]\*IN[]\*SRV[]\*/{ printf("%s;%s\n",$7,$8)}' 
```

它在 EL7 *和* EL6 上都运行良好。原来 EL6 上的古代版`awk` (3.1.7)不知道如何正确解读“`\s`令牌。奇怪的是(我写其他工具时的回忆)EL6 版本的`grep`很好地理解了“`\s`”标记。

当我偷懒的人，我有一个链接到 PR 的原始对话，带有一个“见:*这*是为什么”的注意事项，他回答说:“哦:我从来没有真正使用过 awk，所以从来没有遇到过它”。
