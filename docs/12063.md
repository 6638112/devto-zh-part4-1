# [](#some-time-ago)前段时间

这条推特消息引发了一场关于如何衡量软件开发团队生产力的讨论。我将重点介绍一些最常见的答案:

*   已打开/已解决的票证数量
*   站点正常运行时间
*   站点错误百分比
*   常规发布节奏
*   感觉生产率提高(通过调查/民意测验)
*   原始增加收入
*   满足业务成果
*   积极的用户反馈
*   冲刺故事点/时间的总和
*   达到预期的估计
*   工作软件的数量
*   客户满意度
*   是时候合并 PRs 了

# 这是一件棘手的事情

这些大部分都是我所说的高效团队的症状。因此，它们在某种程度上是有价值的。

然而，其中一些确实提出了更多的问题。

> “常规发布节奏”

好的。从技术上来说，一年一次是有规律的节奏🤷‍♂️.

> “原始增加收入”

好的。你怎么知道这是由一个高效的开发团队引起的，而不是营销工作引起的，等等。🤷‍♂️?

> “积极的用户反馈”

好的。但是，那些测试了你的产品并发现它很差或不符合他们需求的客户通常不会留下任何反馈。那么，您如何知道 80%的客户由于缺少某些功能或选项而没有加入🤷‍♂️?

我希望您能明白，在选择重要的度量标准时，您需要小心谨慎。

# [](#productive-businesses)生产性企业

我认为富有成效的开发团队确实为公司的成功做出了很大贡献！

最近，[《加速:精益软件和 DevOps 的科学》](https://amzn.to/2No88fV)这本书好评如潮。

它通过询问某些软件开发实践如何使企业变得更有生产力和更成功，以一种更全面的方式处理这个话题。

比如“提交到部署”之间的时间很短、持续集成/部署、小批量工作、允许试验等等。

虽然这些对我们来说听起来显而易见，但这本书的真正功绩在于它使用合法的研究和数据来证明这些。

# [](#what-can-i-do-to-help)有什么可以帮忙的？

太好了。但是您可能没有能力简单地告诉您的组织开始做所有这些事情！我们大多数人都不知道。

我们有今年计划的项目，需要尽快完成。

您的组织中可能存在阻止您实施甚至建议这些变革的政治因素？

你的企业可能正在经历某种整体重组，这会儿阻止这种讨论？

上面提到的许多项目(CI、CD 等。)是需要你的开发团队作为一个整体来采用的实践。

作为起点，你可以做的事情之一是成为这些实践的倡导者。每当出现由于缺乏自动化测试而导致的问题时(或者一开始就没有任何测试！)-提出事实。指出这种做法的重要性。

如果你有像上面提到的那本书一样的资源——使用真实的数据和真实的故事。找出其他经历过类似困境的公司，并与你的团队/组织分享。

第一步是简单地指出这些改进将解决的痛点。

# [](#pro-tip)亲提示

这里有一个很棒的建议，对我来说非常有效。如果你能够原型化、实施一个过程或建立一个实践，你知道这将改善你的团队/组织的事情，并且不会花那么长时间，那么就去做吧。

当我开始为我现在所在的公司工作时，所有的软件部署都是 100%手动的，很容易出现人为错误。这个过程有多个步骤(也没有在任何地方记录...)如果不遵守，将会导致生产应用程序出现问题🤪。

我开始提出的第一件事就是这个问题。然而，实际上没有人对此做任何事情。某些政治/期望的过去历史在起作用。反正我就是做了。

这就是现状:**我们喜欢谈论和抱怨事情——而不是去解决它们。**

一旦我发现什么都没有修复，我就写了一个脚本，为我们的开发环境自动化整个构建/部署过程。这大大简化了过程。

但是，有人用了。有些没有🤦‍♂️.

下一步，我使用一个流行的 CI 服务构建了一个连续的构建过程。在几个星期的过程中，我慢慢地添加了更多的内容，向团队中的每个人展示了它是如何帮助我们的(尤其是项目经理！).最终，一步一步地，我们采用它作为完整的构建/部署机制。

有可能！

这需要真正的努力。有人需要做这件事，并从字面上证明它可以节省时间或金钱。

# [](#what-can-you-change)你能改变什么？

**问自己:** *“我可以开始着手解决什么业务流程或改进？”*

确保你有可靠的研究来源。从小的开始，而不是大的。

就我而言，我从一个剧本开始。长成更大的东西。我也有一个简短的关于自动化如何显著减少软件中的错误的调查和研究列表，等等。

如果你能实现新的过程，引入新的技术，教授新的写软件的方法等等。这让你的直接经理的日子更轻松——那么你将把自己定位为这样一个人:

*   a.完成工作
*   b.正在改进业务——而不仅仅是抛出代码。

任何人都可以甩代码。那些改善企业的人是值得提倡和珍惜的！
