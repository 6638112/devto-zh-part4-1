我一直在思考 web 开发社区中的一个老问题，这个问题一直是🔥火焰战争🔥基本上自从 web 浏览器存在以来:**“浏览器单一文化”**。由于[微软放弃他们的 EdgeHTML 浏览器引擎，将 Edge 转移到谷歌的 Chromium 引擎](https://blogs.windows.com/windowsexperience/2018/12/06/microsoft-edge-making-the-web-better-through-more-open-source-collaboration/)，这个话题最近又火了起来。

# [](#what-does-browser-monoculture-even-mean-and-why-should-we-care)“浏览器单一文化”到底是什么意思，我们为什么要在意？

“浏览器单一文化”是当一个浏览器变得如此主导，以至于引发连锁反应:它实际上是唯一的选择，所以它是唯一有人使用的浏览器，所以它是唯一有人关心的浏览器，所以它是唯一一个*开发者为*编写代码的浏览器。

这是一个主要的担心:担心的是如果一个浏览器变得太占优势，开发人员可能会开始忽略其他浏览器，而只是针对占优势的浏览器支持的 CSS 和 JavaScript 功能集，从不费心测试其他浏览器的交叉兼容性。

这并不是没有根据的:这正是 21 世纪初发生的事情，在浏览器战争的高峰期:IE 浏览器变得如此绝对主导，以至于开发者经常专门针对 IE，许多网站根本无法在任何其他浏览器上运行。最糟糕的部分是 IE 是[超级古怪](http://jkorpela.fi/quirks-mode.html)，不遵循标准，并且*非常*缓慢地改变或改进。

这是另一个主要的担忧:当市场上真的只有一个玩家时，浏览器厂商遵循标准和游戏规则的动力就会下降；他们可以有效地为所欲为。你可能认为开发者社区会变得疯狂，开始远离浏览器，对他们的能力进行检查，但事实是，大多数浏览器用户不是 web 开发者。大多数用户不知道或不关心这些东西，他们也不会因为一些人🤓*书呆子*🤓都在抱怨“API”和“标准”。

也就是说，还有第三个非开发人员确实有点关心的技术问题:当每个人都使用一个供应商时，该供应商就可以访问每个人的用户习惯、数据等。这也不是完全没有根据的；许多人对谷歌过去处理用户隐私问题的方式有很大的疑问。

# [](#an-emerging-chromium-monoculture)一种新兴的铬单作？

这就把我们带到了最近的事件。谷歌 Chrome 的底层浏览器引擎是作为一个名为 [Chromium](https://www.chromium.org/Home) 的开源项目开发的。它被设计成可以作为新浏览器的基础，许多 [*和*浏览器都是基于它而构建的](https://en.wikipedia.org/wiki/Chromium_(web_browser)#Browsers_based_on_Chromium)，包括你听说过的浏览器。 [Opera](https://web.archive.org/web/20130607164906/http://my.opera.com/ODIN/blog/2013/05/28/a-first-peek-at-opera-15-for-computers) 和[三星互联网](https://www.smashingmagazine.com/2016/10/whats-the-deal-with-the-samsung-internet-browser/)都在 2013 年迁移到基于 Chromium 的版本，当然，如前所述，今年的大新闻是[微软 Edge 也将迁移到 Chromium](https://blogs.windows.com/windowsexperience/2018/12/06/microsoft-edge-making-the-web-better-through-more-open-source-collaboration/)。

所以这是一个问题:如果足够多的市场份额都集中在基于 Chromium 的浏览器上，我们最终会变成一个单一的 Chromium 文化会怎么样？

从目前的统计数据来看，我们似乎正朝着这个方向前进:

仅 Chrome 就占据了全球浏览器市场 60%以上的份额。再加上三星互联网、Opera 和 Edge 这些顶级的基于 Chromium(或即将基于 Chromium)的浏览器，这个数字上升到了 70.8%！！！想象一下，如果苹果公司厌倦了每个人都抱怨 Safari 的缺失功能(因为 Safari 现在在实现酷的新网络平台 API 方面往往落后最远),并决定走与微软相同的道路。这不太可能，但我可以预见它会发生。

过去几年中许多新的实验性浏览器项目也是基于 Chromium 的，比如维瓦尔第、勇敢的 T2 和史诗级的 T4。这对我来说很有意义:Chromium 是一个非常完善的、积极维护的、易于扩展的浏览器引擎。从头开始创建完整的 JavaScript、HTML 和 CSS 引擎的任务绝对是巨大的；Chromium 实际上有几十年的工作经验(考虑到它是基于 WebKit fork 的)，赶上它似乎是不可能的，尤其是对于一个独立的初创公司。另外，这些浏览器主要不是试图改善网络语言的状态；他们关注更高层次的特性，如安全性、隐私等。我不责怪他们使用 Chromium 作为基础，专注于他们关心的功能。

这就是我想说的要点:Chromium 开始被定位为浏览器引擎的事实上的标准，是 W3C 规范的一种规范实现。但是我们会把它钉在上面。📌

# [](#what-about-firefox)火狐怎么样？🔥🦊

回到浏览器战争的日子，Firefox 是我们都需要的英雄，它打破了 Internet Explorer 的垄断，光荣地进入了浏览器多样性和合作网络标准的新时代。

但不幸的是，不幸的是，自从 Chrome 出现以来，Firefox 的使用率一直在下降。我认为 Mozilla 最近在 Quantum 上做了英勇的工作，他们经常带头实现新的网络平台特性，尤其是在 CSS ( [subgrid！！！😭](https://twitter.com/ken_bellows/status/1135930281444204544))。

但底线是:谷歌是 huuuuuuuuge。他们只是有太多的人在 T2 工作。我不认为 Mozilla 或围绕 Firefox 的开源社区有足够的人力或机构支持来与 Google 保持同步。

我也不认为正确的解决方案是谷歌解雇他们的浏览器团队，这样 Firefox 也能跟上。我喜欢他们不断尝试很酷的新事物。那我们该怎么办？

# 我们有单一文化吗？

从重要的意义上来说，是的。在未来的 5-10 年里，浏览器引擎单一文化似乎至少有一个很好的机会出现。目前，Chromium 的两个主要对手是 Safari 和 Firefox，这两家公司共拥有约 21%的用户。多年来，Chrome 的使用数量一直在缓慢但稳定地增长，并且可能会继续增长。现在 Edge 是基于 Chromium 的，我想它的数量也会增加一些。

**Buuuut...**这就是我要引起争议的地方。我想假设这是一种不同于我们在 2000 年代看到的互联网浏览器 6 的单一文化，它有可能成为一种*非常*不同的类型，一种可能不真正配得上“单一文化”这个名字的类型。跟着我！

# [](#is-it-bad)不好吗？

好吧，让我们涉足有争议的领域。这是我一直在开发的一个镜头:

我之前提出的所有三个问题都基于一个场景，即单一的*浏览器*成为主导，包括所有与具有单独业务优先级的特定公司结构的挂钩，以及与公司专有生态系统的挂钩(例如，将你的谷歌账户与你的浏览器挂钩)。但是铬已经不是那样了，它有可能完全相反。

概括来说，以下是我在浏览器单一文化讨论中听到的三个主要问题:

*   一个占主导地位的浏览器会导致开发者只关注它，而忽略与较小浏览器的兼容性。
*   当一个单一的浏览器变得太占优势时，它就失去了遵循和贡献共享网络标准的动力。
*   使用单一浏览器的人越多，拥有该浏览器的公司可以获得的用户数据和习惯就越多，这就引发了隐私问题。

当你意识到我们不是在谈论一种单一文化时，所有这些都会有所改变；我们说的是 T2 Chromium T3，它是底层的浏览器引擎。目前，Chromium 是各种浏览器的基础，而不仅仅是谷歌的旗舰产品。据我所知(作为 Chromium 来源的非专家，所以请纠正我)，谷歌那些让隐私倡导者紧张的东西是独立于核心 Chromium 浏览器引擎的。我的意思是，微软显然不会将任何优势与谷歌的生态系统挂钩，所以它应该很容易分离。

## [](#privacy)隐私

对我来说，“浏览器”和“浏览器引擎”之间的区别已经回答了关于隐私的主要问题。隐私问题不在于引擎本身。见鬼，我前面提到的两个基于 Chromium 的实验性浏览器， [Brave](https://brave.com/) 和 [Epic](https://www.epicbrowser.com/) ，都是为了提供一个更安全的隐私保护选项。

## [](#single-dominant-platform)单一主导平台

至于担心开发者针对单一的网络引擎，如果它变得太受欢迎，这肯定是一个真实的事情。一些人声称这已经在 Chrome 上发生了。

但事实是:由于它是几个不同浏览器的基础，当 Chromium 获得一个新的 JavaScript 或 CSS 特性时，它不是将 Chrome 推向其他浏览器，而是将所有这些浏览器推向前进。因此，当 the Verge(在上面链接的文章中)说 Chrome 正在成为新的 IE 浏览器，“网络开发人员主要针对 Chrome 进行优化，然后针对竞争对手进行调整”时，他们真的应该用“基于 Chrome 的浏览器”取代“Chrome”。“为 Chrome 优化”也是为 Opera，Vivaldi，Samsung Browser，以及 soon Edge 优化，因为它们都使用相同的引擎。

也就是说，这对于 Firefox、Safari 以及任何试图维护并行实现的人来说都是一个问题。让我一会儿再回到这个话题。

## [](#web-standards)Web 标准

这是一件大事。谷歌在历史上一直是推动和开发供所有人使用的网络平台标准的大玩家，但正如 the Verge 正确指出的那样，他们在过去几年里有点偏离了这一信息:

> 无论你责怪谷歌还是经常行动缓慢的万维网联盟(W3C)，结果在整个 2017 年都特别明显。我们开始看到网络上出现了许多“与 Chrome 配合使用效果最佳”的信息，谷歌一直处于这些信息的中心。Google Meet、Allo、YouTube TV、Google Earth 和 YouTube Studio Beta 都阻止 Windows 10 的默认浏览器 Microsoft Edge 访问它们，并且它们都指示用户下载 Chrome。火狐浏览器也不支持 Google Meet、Google Earth 和 YouTube TV 下载 Chrome 浏览器。谷歌已经公开承诺支持 Earth on Edge 和 Firefox，该公司正在“努力在未来将 YouTube 电视带到更多的浏览器上。”
> 
> *   [The Verge，“Chrome 正在变成新的 Internet Explorer 6”](https://www.theverge.com/2018/1/4/16805216/google-chrome-only-sites-internet-explorer-6-web-standards)

这不太好。但据我所知，严格来说，这并不总是谷歌忽视标准的问题；这通常是因为其他浏览器没有跟上谷歌实现尖端 API 的步伐。同样，谷歌有大量的资源可以利用，所以在我看来这是不可避免的。但是，再说一次，因为谷歌正在为 Chrome 做贡献，而不仅仅是 Chrome，所有基于 Chrome 的浏览器都获得了这些好处。

# [](#my-crazy-idea)我疯狂的想法

好吧，我们做到了，我们打下了所有的基础。至此，我的想法可能很明显了。所以我们开始吧:

如果整个网络在一个浏览器引擎上实现标准化会怎样？如果 Chromium 成为 web 标准的参考实现的基础，并且所有的浏览器都转换成基于它，会怎么样？

在你拿出干草叉之前，让我解释一下。

## [](#a-collaborative-engine)协同引擎

我希望这是显而易见的，但我还是要说:**我是*而不是*建议我们将网络控制权让给谷歌。**事实上恰恰相反。我的愿景是，目前花费无数时间在不同代码库中并行编写相同代码的开发人员将被统一在一个单一的公共代码库下。

理想情况下，这个代码库不会被任何一家公司控制。我希望看到一个由独立的非盈利基金会控制的通用浏览器引擎，比如 Python 软件基金会，包括来自每个参与的浏览器厂商的一些代表。

请和我一起想象一下:如果 Safari 和 Firefox 以及其他所有人不再维护独立的代码库，不再重复大量的工作，不再拼命跟上谷歌开发团队这个庞然大物的步伐，而是开始为共享的代码库做出贡献，会怎么样？如果每个人都从其他人的工作中获益会怎样？

我们已经看到了这样的例子。Opera 开发人员[过去已经这样做了](https://blogs.opera.com/desktop/2015/10/opera-33-our-contribution-to-chromium/)，Edge 团队[也已经开始这样做了](https://www.windowslatest.com/2019/05/08/microsoft-details-scrolling-improvements-for-edge-and-chromium/)，两个团队都带来了他们的专业知识，使 Chromium 受益。

我现在非常沮丧，虽然 CSS 子网格已经在 Firefox Nightly 中实现了，Chromium T2 还没有开始工作。想象一下，如果火狐开发者的工作也对 Chrome 有所贡献！这个问题会消失！

这通常是 web 平台上的一个经典问题:我们都为某个浏览器中某个新的 Web API 的精彩演示而兴奋，但第一反应是，“当然，但我多久才能真正使用它呢？其他浏览器多久会实现它？”也许这里有一个永远摆脱这个问题的机会。

## [](#what-about-the-lost-diversity-of-implementation)实现的多样性丧失怎么办？

这里有一个我担心的可能的缺点:我喜欢目前流行的各种浏览器引擎的一点是，它们经常以不同的方式实现相同的功能，并且其中一个通常比其他的更快。我不得不想象(虽然老实说我不知道)Chrome 开发人员看到 Firefox 如何解决问题并借鉴了一两个想法，反之亦然。我们会放弃吗？这会是多大的损失？

我有两个想法。

首先，我注意到这对其他平台来说没什么大不了的；我没有听到多少抱怨说 CPython(Python 参考实现)在鼓励实现特性的替代方法方面没有大的竞争。但也许这不是反驳。也许这对 Python 社区来说是一件坏事，他们会从一些竞争中受益。我不知道。

其次，也许有一种方法可以在某种程度上保留网络的这一特性。是的，所有的浏览器都将建立在同一个引擎上。但也许引擎可以(或者已经是？再说一遍，我不是专家)以这样一种方式构建，即每个浏览器都可以在核心引擎之上构建功能的初始实现。也许这可以发展成一个常规的过程:当一个新的 API 被讨论时，或者甚至在规范最初被发布后，也许每个感兴趣的浏览器都写下他们的特性实现并试用它。也许我们花了 6 个月左右的时间在 Firefox、Chrome 和 Opera 中开发不同版本的功能，然后基金会的一个委员会坐在一起，讨论应该将哪个实现合并到主分支中。

这是我希望得到具体反馈的一点。我不知道事情会如何发展，但感觉必须有一种方法来完成它。
