# 我的第一个带回家的编码挑战

> 原文:[https://dev . to/calier/my-first-take-home-coding-challenge-10l 2](https://dev.to/calier/my-first-take-home-coding-challenge-10l2)

作为编码训练营的一名应届毕业生，我开始找工作，申请研究生和初级开发人员的职位。我看到了人们对带回家的编码挑战的看法，并且刚刚提交了我的第一份，我想从我自己的角度给存钱罐增加 2 便士。

几个 reddit 帖子和博客帖子表明，一些开发人员不喜欢带回家的挑战，原因很多，并因情况而异。一些人哀叹这项任务本身不过是耗时太长的无偿工作，另一些人则认为有更好的方法来判断某人是否适合职位配对编程。作为一个拥有大量投资组合和良好记录的中级或高级开发人员，我会倾向于分享其中的一些观点吗？几年后再问我....

然而，作为一个寻找我的第一个开发角色的大三学生，我可以看到这种类型的任务对于处于我这个位置的人的有效性。我有一组有限的项目可以展示给未来的雇主，不是所有的项目都经过了时间的精心制作，并给予了我认为他们应该得到的爱和关注。到目前为止，我所做的大部分工作(尤其是在训练营)都是在一周内完成的，而我只花了两倍的时间学习这个主题。就其本身而言，它们提供了一个有用的快照，展示了我是如何快速地学会每个新概念的，但它们都需要大量的重构。

据我理解，我完成的作业是初级开发人员角色的一个相当标准的带回家的编码挑战的例子。就我个人而言，我在做这件事的时候学到了很多——这项任务要求程序是测试驱动和面向对象的。它可以用您选择的语言编写，所以我选择 JavaScript 有几个原因:

*   我真的很喜欢，我真的很想学好它！
*   在 bootcamp 中，我只是在功能上使用它，但是我知道理解面向对象编程的基础是很重要的——没有比现在更重要的了！

TDD 是另一件我知道很重要的事情，我想去做，但是还没有时间去做。我的整个程序都是由我写的测试驱动的，这花了我一点时间，但是最终的产品已经演变成了我引以为豪的东西——不可否认，直到测试向我表明某些类属性不再是必要的，或者我确实需要一个额外的方法来分解一个做得太多的方法，我才意识到 TDD 的实际价值。现在我发现在我的代码实践中，我经常使用 TDD 解决问题，这感觉很棒！

我很感激这个代码挑战把我从舒适区推了出来，因为它加速了我在两个主要开发领域的学习。我已经提交了它，很高兴得到一些反馈，并了解审查我的代码的开发人员对它的看法。对于第一次尝试，我认为我做得很好:我多次阅读了任务，计划并彻底测试了代码，它满足了简要要求，并且我对它还能做什么有很多想法，所有这些都记录在一个仔细思考的自述文件中。这可能是我第一次带回家的挑战，但我肯定不会是最后一次。(用于命令)等待下面发表的消息....
