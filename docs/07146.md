Kent 的职业生涯是为其他开发人员提供资源，在我们聊天期间，他给了我一些很好的建议。

### [](#1-youre-an-engineer-not-a-developer)1。你是工程师，不是开发者

在我的第一份工作中，我听说一些不太道德的经理故意给他们的工程师贴上开发人员的标签，以降低他们的工资。

Kent 强调，你应该避免把自己描述成一个开发人员，而应该倾向于“工程师”这个术语为了证实他的观点，我做了一些谷歌搜索。

这张截图显示，Glassdoor 和 PayScale 报告的估计工资下限相差 10-20，000 美元。因此，即使你的雇主拒绝授予你工程师的头衔，你也应该在简历中使用工程师这个词。

肯特在采访中表达了一些关于标题的有趣想法。

### [](#2-part-of-seniority-is-faking-it)2。资历的一部分是装出来的

Kent 谈到了他在准备好(以大多数标准衡量)之前第一次进入“高级”角色的经历。那次经历让他明白，至少晋升之路的某一部分被最有效地描述为一种行为。

这并不是说你的许多资深同事和同行没有获得这个头衔，但它确实暗示了将自己展示为专家对你的职业有着重要的价值。

他们说你应该为你想要的工作着装；我想你也应该表现得像你应得的那样。

### [](#3-perception-is-essential-but-can-fuel-imposter-syndrome)3。感知是必不可少的，但它会加剧冒名顶替综合症

作为关于塑造自信的建议的后续，过度自信可能会适得其反。

如果你像 Kent 在 JavaScript 社区中所做的那样，创造了一个专家的名声，这可能会很可怕。人们会看到你说的每一句话，并可能对其进行分析。

肯特描述了他在做像写推文这样平凡的事情时的恐惧。

用你在审查中的自信来缓和你被认为是专家的渴望。尤其是如果你像肯特一样，非常自信的话！

### [](#4-speak-at-meetups)4。在聚会上发言

肯特认为在聚会上演讲是一种演讲技巧，因为这比试图在会议上演讲的承诺要低得多。

总的来说，肯特说在聚会上获得发言机会相当容易。他谈到随机联系组织者，甚至要求在活动的那一周发言。

在聚会上演讲是增加你可信度的一个好方法，而不会有在大型会议上演讲的压力。

### [](#5-consume-a-lot-of-information)5。消耗大量信息

肯特给出的另一条建议与跟上行业发展有关。

Kent 提到了消费大量内容，甚至在他不在乎建立专业知识的领域。肯特建议说，能够就技术话题展开对话是明智的，即使你对这些话题并不感兴趣。

### [](#6-build-side-projects)6。构建辅助项目

人们经常注意到的关于肯特的一件事是他超凡脱俗的生产力水平。他写了大量的内容，录制了大量的视频，做了很多演讲，并开发了大量的开源软件。

他的一条建议就蕴含在这种生产力中。稳步建立小的副业项目会让你接触到更多的实用知识，并对你正在学习的技术有更深的理解。

专业知识不能只是理论上的；它必须有一个实用的元素！

### [](#7-produce-more-than-you-consume)7。生产多于消费

尽管肯特渴望新知识，但他声称，与他花在创造上的时间相比，他花在消费上的时间微不足道。

生产的行为才是真正巩固你对一个主题的理解。肯特用他的教育内容和他的开源作品体现了这一点。

把这看作是你学习过程的开源。

### [](#8-take-care-of-yourself-and-your-relationships)8。照顾好你自己和你的人际关系

回到肯特疯狂的生产力水平。在我们的谈话中，肯特警告说，在追求事业的过程中，牺牲健康和人际关系是非常危险的。

他说尽管他喜欢他的工作，但与他从家庭生活中得到的相比，他并没有真正的满足感。

当事情到了紧要关头，如果你不能在身体上、精神上和心理上照顾好自己，那么其余的事情都是徒劳的。

Kent 不仅是 JavaScript 专家，也是建立世界级职业生涯的专家，他的建议是无价的。我会把这份清单给任何想从新工程师晋升到高管或领导者的人，因为它诚实而全面。遵循这些建议不会让你误入歧途。
