在我的第一篇文章中，我决定我应该给出一些我如何走到今天这一步的背景。

# 开始

大约 3 年前，我开始在一家小工程公司工作，只有一些很少的编码知识。虽然编码应该是我职位的一部分，但我对推进我们的数据可视化仪表板更感兴趣。

最初的堆栈非常简单，数据从。csv，通过一个小的 php 后端进行一些小的数据库交互。前端最初是使用 DC.js 和 jQuery 构建的，没有 webpack 或 build/toolchain。

我们很快开始强调 DC 的表演极限。就这样开始了我的第一个主要项目:编写一个定制的图表库。

# 第一个项目

首先，我需要学习如何创建一个可以在公共名称空间下引用的库。所以我看了 d3.js 源码！进入并复制了 umd 标题部分并开始运行。

然后是回调，如何让函数在构造的函数中更新一些东西(后见之明应该让它成为一个类，但后见之明是 20/20，我们也没有结束)。

最初的代码非常难看，一点也不干燥。没有延伸的基础。唯一的好处是我们摆脱了 DC.js 的限制，并清理了相当多的渲染 jank。

缺点:仍然导入 jQuery(30kb)、d3.js (70kb)和最精简的 my library @ 70kb。这还只是从属关系。我们剩下的代码超过 200kb。

这导致了下一步:移除 jQuery。

# 依赖性的去除和 webpack 的启动

jQuery 相对容易移除。d3 做了很多和 jQ 一样的事情。一个依赖消失了！

我之前看到过 webpack，但是到目前为止我们对它的使用很少，而且由于它使我们的构建链变得复杂，我们花了一些时间才采用它。一旦我们想通了，生活就改变了。在这一点上仍然没有使用 npm，但它是一个开始。

# lithtml

然后我们转向使用 lithtml。我曾经想过迁移到一个 ui 框架，当时这是最不令人畏惧的选择。

从这里我开始了我的第二个项目，一个使用 lithtml 作为渲染代理的准框架。

在努力实现之后，我决定，这已经完成了，为什么不使用一个已建立的库。所以我开始摆弄 React。但是我不想和它与脸书的关系捆绑在一起，所以我选择了 preact！

# preact + TypeScript

一旦我们开始使用 preact，我选择同时使用 ts 来进行适当的属性检查。

这也伴随着正确使用 npm 和相当深入的 webpack 配置。

在这一点上，我们已经完全过渡到 preact，但是图表库几乎没有改变，只是用了一些 preact 包装器来进行适当的集成。

然后是我最近的开源项目 [preact-charts](https://github.com/pmkroeker/preact-charts) ！

# preact-图表和 OSS 投稿

这是从创建一个供我们内部使用的小库开始的，但是我开源了它，因为所有当前的图表库都是基于 react 的，没有纯粹的 preact 等价物。它仍然是一个 WIP，但它很稳定，目前支持 preact 8 和即将推出的 preact X。

然后我开始支持我使用的库。从一些 bug 报告开始，慢慢变成了拉请求。

我现在喜欢尽我所能帮助 preact typescript 定义！

希望在未来，我将继续更多地参与开发社区，并在此过程中帮助更多的项目。
