# 给新手的建议

> 原文：<https://dev.to/stereobooster/advice-for-the-newbie-11oj>

所有的建议都适用于软件开发的初学者，如果你已经是经验丰富的开发人员，我想你可以找到自己的路。

## 前言

最近我参加了初学者聚会。作为介绍，人们说他们的名字和他们今天想学的东西。

人们想学的东西的例子:

*   我想学苗条，因为它很快
*   我正在用 Next.js 构建我的第一个 React 应用程序

## 什么是现代软件开发？

让我们暂停一下，问一个问题:现代软件开发是关于什么的？我认为开发人员的主要任务是**管理[复杂性](https://stereobooster.com/posts/complexity/)** 。你需要一些逻辑(商业规则或一些算法)并把它编码成一个程序。要做到这一点，你需要了解系统行为的所有方面，因为机器只能执行精确的命令，所以你不能手工向计算机解释你想要什么，你需要精确。

当你把一些手写的解释转化成计算机的精确指令时，你会发现你需要比手写解释更多的精确指令来描述它。

指令数量越多，复杂性就越大。此外，一些规则(要求)会随着时间的推移而改变，您需要更改指令，并确保这些更改与之前的指令一致。

当我们创建软件时，我们希望它有一些独特的复杂行为(功能)。

但是软件开发的复杂性很容易失控，这意味着开发软件会越来越难。在某些时候，它会变得如此艰难，以至于你不得不扔掉你所拥有的，从头开始。

软件中的复杂性就像盐，你加入一点点，食物味道会更好，你再加入一点点，甚至更好，但是在某个时候，你会发现盐开始变苦了。你需要平衡，不要太多，但是不要太复杂，只有程序什么也不做。

所以作为开发人员，你的主要任务是管理复杂性。控制它不会扼杀你的项目。

你会怎么做？例如，分而治之——将复杂性分成几个小的桶，这样万一你需要它，你可以扔掉整个桶，而不需要重写整个项目。

或者，举个例子，你可以去掉所有不重要的复杂性——你需要问:我们的项目需要这个工具吗，我们需要这个库吗，我们需要这个功能吗。(是否会激起欢乐的火花？)如果不是，那么这不是本质复杂性，它可以被无损耗地删除。

## 永远选择简单

建议很简单:永远选择简单而不是炒作，选择性能问题(你没有)，选择建议那些看起来很酷、超级聪明、使用很多行话的人。

让它工作，让它正确，让它快速。按此顺序。

复杂性总是随着时间的推移而增长，所以你将有机会与你的怪物战斗。总是从简单的解决方案开始。

## 关注要点

在你深入一些高级的东西之前，总是选择初学者友好的工具先学习。

寻找以下关键词:零配置、约定优于配置、[开发者体验(DX)](https://stereobooster.com/posts/evaluating-dx-of-a-programming-language) 。不关注:速度极快，捆绑尺寸小，其他炒作词。

选择旧的更稳定的编程语言，库，框架，因为它会更完善，更健壮，更文档化。

当你开始学习的时候，你需要一次学习很多东西，所以你想从你的方式中去除不必要的知识，以达到你感觉的自信水平，因为你可以完成一些事情。

例如，要学习 web 开发，你需要了解 HTML 和一些 CSS，浏览器和服务器之间的通信是如何工作的，一点 JS，git，也许是 GitHub，如何进行部署，浏览器的一些基础知识，如何调试(web 工具)，如何使用 IDE。已经很多了。不需要设计模式，不需要类图，不需要 GraphQL，不需要硬核函数式编程。你可能想用框架代替原始的 JS/DOM，因为那些 API 在**我的观点**中过于复杂，框架可以在一定程度上隐藏它。

## 注意工具中的意外复杂性

并非所有技术都生来平等。有些技术是为初学者设计的，去掉了所有不必要的(偶然的)复杂性，有小而干净的 API，有容易学习的一致约定，有好的教程。有些人更关注性能或新的难题或其他问题。

有时工具中会有意外的复杂性，因为该工具是第一个进入该领域的，它对其 API 做出了一些错误的决定，但现在改变已经太晚了。

**所有的 API 都是人类创造的，所有的人都会犯错**。因此，如果您发现一些困难或令人困惑的东西，您就有可能面临该工具的意外复杂性。

例如，C.A .霍尔称他的空引用思想是一个十亿美元的错误。

## 不遵循咒文

有一堆“最佳”实践、编程咒语、设计模式、范例、原则等等。

千万不要盲从他们。总是问这样一个问题:这种做法是否会让我的生活更轻松。通过增加一些实践或工具，你很可能会增加复杂性，所以你需要确保你得到一些有用的回报。

比如 TDD 口头禅:你应该先写测试。而不这么做的人会觉得很难受。但是请记住，测试对于你的软件来说并不是必不可少的，测试的唯一目的是让你更容易对软件进行快速的修改。如果编写测试非常困难，例如，你没有稳定的公共 API，或者你没有清晰的规范或者一夜之间的规范变化，测试就没有任何好处。你将花更多的时间写测试，而不是你的代码。但是如果你觉得你需要一次又一次地手工测试同样的东西，花一些时间在测试上可能是有意义的，如果你不是一个人在项目上，测试也是有用的。

还是不要重复自己(干)的口头禅。人们试图编写更密集的代码，但他们引入了一些难以理解的实体、函数或类，以消除所有的重复。这会让他们的生活更轻松吗？一点也不，他们有令人困惑的代码。一点复制粘贴不会杀死你的代码。如果你觉得我复制粘贴相同的代码 3 次或更多次，你有一个好名字，然后可能提取它。适量干敷。

## 不要听自作聪明的人的

不要被过于自信的语气所充斥。受邓宁-克鲁格效应影响的人听起来可能非常自信。如果有人提倡技术，只说优点，而不说缺点，这意味着他们使用技术的时间还不够长。总有取舍(我还没看到什么理想的)。

不要对行话和术语印象深刻。想要帮助你并知道他们谈论什么的人会尽可能少地使用术语，并能够解释他们使用的术语。如果他们不能解释，他们就不知道自己在说什么。或者他们想比你感觉更好，因为他们知道“聪明”的词。不管怎样，不要在这里浪费时间。

## 不要分心

你可能会被不重要的事情分心，例如，选择哪个 IDE，或者使用哪个插件，或者使用哪个字体。谁在乎呢。选择有用的东西，学你想学的东西。工具很重要，但是如果你花 4 个小时配置理想的 IDE，花 1 个小时实际学习——这里就有问题了。

## 不注意守门人

有人会说，例如，技术 A 是狗屎，你不应该使用它，或者你应该遵循一些咒语，或者 CSS 不是编程语言(暗示你不是真正的程序员，因为你使用它)。别听他们的。

守门人不想对你好。他们不是来传播真理或终极知识的。他们这样做只是为了看门本身。
