# 为什么不应该在 Dev.to 上发布内容？

> 原文:[https://dev . to/slashgear _/why-you-should-not-publish-content-on-dev-to-117g](https://dev.to/slashgear_/why-you-should-not-publish-content-on-dev-to-117g)

首先，我不是网络战士。我不想在这里作出 Dev.to 的判断，也不想试图说服你这个平台是对还是错。

相反，我想警告人们一种正在危及我们所知的网络的趋势。

# TL；速度三角形定位法(dead reckoning)

我不认为 Dev.to 是一个问题，而是作者如何使用它。作者不应仅在开发者网站上发布内容。您不应在开发者网站上发布内容。您应该在开发者网站上**分享**内容。

## [](#actually-whats-your-problem-with-devto-)“其实你跟 Dev.to 有什么问题？”

作为一名作者，我过去常常在我的个人博客上分享文章。我很快面临一个明显的问题。在社交网络占据越来越多空间的网络世界里，很难“制作”一个小型个人博客。

通过在我的个人博客上发表文章，我的受众很少。我的文章很难通过我的一级接触范围。

因此，我很快就想找到一个解决这个可见性问题的方法。我的一位同事提出的解决方案似乎对我很有吸引力。

> “你为什么不试试新媒体平台？在 Medium 上发表的作者拥有成千上万的读者！还有，编辑界面也挺酷的！”

我就是这么做的。我不再在我的个人博客上发表文章，开始只在媒体上写作和发表文章。我今天后悔了吗？

不幸的是，故事出了差错！

Medium，在仅仅几个月的时间里，并遵循一些商业决策，已经真正成为一个应该避免的平台。

用户跟踪，广告，付费墙，一杯真正恶心的鸡尾酒，会让 ["la quadrature du net"](https://www.laquadrature.net/en/) 尖叫。由于这些决定，许多人决定离开这个平台。

我不是说 Dev.to 会遭受同样的命运。我不是灵媒，我没有预见未来的能力。但实际上，Dev.to 仍然是一家公司，因此与财务问题密切相关。而且 Dev.to 的*商业模式*的稳定性相当模糊。

你真的必须信任团队，这样 Dev.to 才不会走错方向。我很想听听本·哈尔彭对这个问题的看法。

# [](#so-we-should-stop-writing-on-devto-)所以我们应该停止写 Dev.to 了？

答案是否定的

Dev.to 很棒，有一个很棒的社区。**但是**，作者不应该只在 Dev.to 上发布他们的博客内容。

我们应该遵循 **P.O.S.S.E** 原则。

> 在你自己的网站上发布，在其他地方联合发布

尊重它将有助于作者对他们在互联网上的出版物负责。这将:

*   保护网络免受过度集中的影响
*   避免你的版权被盗
*   推广你的博客

所有的作者都应该有他们自己的个人博客来发布内容。然后，他们可以通过放置一个指向自己博客页面的`canonical url`在其他平台上分享这些内容。

Dev.to 有这个功能，非常难得。我真的希望这个特性能被强调出来，这样作者就能意识到这个问题。

例如，这篇文章可以在[我的个人博客](https://slashgear.github.io/posts/stop-dev-to/)上找到，它的内容可以在 Dev.to 上分享。

我还参与了一个名为[“开放博客”](https://open-blog.dev/)的开源联合平台项目，目的是让人们意识到自己不是内容的所有者。如果你喜欢这篇文章，来看看吧。
