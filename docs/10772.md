# 新手和有经验的开发者犯的 10 个最致命的 CSS 错误

> 原文:[https://dev . to/bill mei/top-10-most-deadly-CSS-由新手和有经验的开发者犯的错误-100d](https://dev.to/billmei/top-10-most-deadly-css-mistakes-made-by-new-and-experienced-developers-100d)

多年来，我看到我的客户的许多工程师在级联样式表(CSS)方面犯了同样的常见错误。即使是才华横溢、经验丰富的工程师也会犯这些错误，尽管他们在广泛的技术领域拥有专业知识，因为他们没有像在大学里花时间学习算法一样花时间学习 CSS。

我认为 CSS 被认为是一种不直观和难以使用的语言，因为这些常见的错误让大多数开发人员在编写 CSS 时感到沮丧。如果你能避免这些错误，你会发现 CSS 实际上是一种简单易懂的语言，它名不副实。

**以下是我在咨询过程中见过的工程师犯的十大致命错误**，按严重程度递增排列。错误越严重，不完全重写代码库就越难恢复。

## [](#mistake-10-following-the-dom-when-writing-selectors)错误#10:编写选择器时遵循 [DOM](https://en.wikipedia.org/wiki/Document_Object_Model)

让我们以这个 HTML 页面为例:

```
<body>
  <div class="container">
    <div class="main-content">
      <div class="blog-row">
        <div class="blog-col">
          <section>
            <article>
              <a href="#">This link is not bold</a>
              <p><a href="#" class="bold">This link is bold</a></p>
            </article>
          </section>
          <section>
            <a href="#">This link is not bold</a>
          </section>
        </div>
      </div>
    </div>
  </div>
</body> 
```

一些开发人员喜欢保持一种极简主义的风气，他们试图不要用大量的类来混淆 HTML，通过使用 DOM 结构作为编写 CSS 选择器的指南，以便他们的选择器遵循 DOM 树的结构，如下所示:

##### [](#mistake)🚫错误

```
body > .container > .main-content > .blog-row > .blog-col > section > article > p > a.bold {
  font-weight: bold;
} 
```

像 [SASS](https://sass-lang.com/) 这样的预处理程序让这变得特别诱人，因为它们使得嵌套选择器变得简单而自然:

##### [](#mistake)🚫错误

```
body
  .container
    .main-content
      .blog-row
        .blog-col
          section
            article
              p
                a
                  font-weight: bold 
```

然而，这种“极简主义”是被误导的，因为通过保持 HTML 的“干净”,你最终会弄乱 CSS，使其更难理解、调试和更改。你的 CSS 变得不灵活，因为长的组合子链迫使 CSS 承担复制 HTML 结构的工作，但这不是 CSS 应该做的工作。

CSS 的工作是提供样式，HTML 的工作是提供结构。仅仅因为 DOM 树看起来是单向的，并不意味着 CSS 应该与之匹配——你编写 CSS 的方式应该完全独立于 HTML。如果你决定改变 HTML 的结构，那么你将不得不返回并更新你在 CSS 中创建的每一个组合子链，这是痛苦的，容易出错的，并且增加了不必要的工作。

相反，我建议尽可能在元素上使用目标类。如果您需要从页面中挑选特定的元素，类选择器可以很好地完成这项工作:

##### [](#correct)✅纠正

```
.bold {
  font-weight: bold;
} 
```

<small>这些代码样本来自[无痛 CSS](https://www.painlesscss.com) ，这是一个从基本原理开始教你 CSS 的书籍和视频课程。你可以在[课程](https://www.painlesscss.com)中下载包含所有 60 个代码样本和解决方案的完整报告。</small>

现在，如果您改变了页面上链接的顺序，您不需要重新编写整个 CSS 规则。您可以简单地将`.bold`类移动到您想要的元素上。

幸运的是，如果您使用 CSS-in-JS 实现，这个错误就不是问题了。然而，CSS-in-JS 尚未被广泛采用(截至 2019 年)，我仍然在不使用任何预处理器的代码库中看到不必要的链式选择器。

更一般地说，您应该避免目标性过强或过度指定的选择器，也要避免使用`!important`。CSS 特异性锦标赛给了我们一个直观的感觉，为什么过于强势的选择者是不好的。如果一个选择器在锦标赛中非常强大，它会在更早的回合中赢得更快，这意味着击败强大选择器的唯一方法是编写另一个更强大的选择器。

这种特异性不断升级的趋势被称为*特异性战争*。就像储备核武器一样，这场战争没有谁会赢——随着特殊性的增加，降级只会变得更加困难。避免全面的特异性战争的唯一方法是首先不要储备高性能的选择器。

## [](#mistake-9-ignoring-seo-when-writing-html)错误 9:写 HTML 时忽略 SEO

一旦你完成编码，搜索引擎优化(SEO)就不仅仅是你卸载给你的离岸营销团队的任务。SEO 应该是你在编码时要考虑的东西，特别是因为一些影响 SEO 的因素一旦在生产中实现就很难改变；例如，您站点的 URL 方案或您的服务架构。

在 HTML 中影响 SEO 的一件事是确保你使用语义标签，因为你选择的标签会影响搜索引擎如何理解和排列你的内容。**如果你想出现在搜索结果的顶部，选择正确的标签是第一步**。

例如，假设你正在写一篇关于多伦多房地产价格的博文。你可以把所有东西放入`<div>`元素中；然而，`<div>`是一个普通的、不具实体性的标签，在内容中并不暗示任何固有的含义。相反，你应该选择一个更具体的语义标签，比如用`<article>`来包含文章，`<nav>`来包含一些到其他博客文章的链接，`<table>`来包含关于房地产价格的表格数据。

选择更具体的语义标签而不是普通的非语义标签的好处是，你给搜索引擎更多关于你的网站的信息，这使得搜索引擎爬虫更好地理解和传递与读者的搜索查询相关的内容。

例如，让我们来看看标题。按照从最重要的标题到最不重要的标题的顺序，有六个潜在的标题标签:

*   `<h1>`
*   `<h2>`
*   `<h3>`
*   `<h4>`
*   `<h5>`
*   `<h6>`

`<h1>`是你最重要的标题，因为对于搜索引擎来说，这是一个特殊的信号，表明这是你内容的标题(也有`<title>`标签，但大多数搜索引擎都会考虑这两者)。此外，如果你的用户有视觉障碍，并且使用屏幕阅读器，大多数屏幕阅读器知道直接大声朗读`<h1>` s，因为这被认为是你内容的标题。

这意味着，如果你使用一个`<h1>`来表示任何不是标题的内容，或者如果你在一个页面上有不止一个 `<h1>`的*，搜索引擎会感到困惑，你的“真正的”标题可能不会出现在搜索结果中。*

例外情况是在不同的部分中嵌套多个`<h1>`。举下面这个例子:

```
<article>
  <h1>My cool blog post</h1>
  <p>This blog is so cool!</p>
</article>
<footer>
  <h1>Contact Us</h1>
  <p>Tel: 867-5309</p>
</footer> 
```

即使这个页面上有两个`<h1>`，第一个有一个父`<article>`，另一个有一个父`<footer>`。由于搜索引擎知道`<article>`天生比`<footer>`更有趣、更重要，它可以适当地使用`<article>`的`<h1>`标签，而不是`<footer>`的`<h1>`标签来确定标题。所以在某种意义上，你的父母是谁很重要，决定了你有多重要。就像现实生活中一样。😉

除了上面提到的因素，还有很多因素会影响你的搜索引擎排名。为了抓住本文的主题，如果您有兴趣更详细地了解这个主题，我鼓励您阅读一本关于 SEO 的书。(如何确定什么是 SEO 方面最好的书？我想知道在谷歌上随便找一个先出现的是否有意义。xkcd.com/125)

## [](#mistake-8-using-raw-px-endraw-units)错误之八:使用`px`单位

实际上，在某些情况下使用`px`单位是没问题的。真正的错误是**使用绝对单位而不是相对单位**。

我们更喜欢使用相对度量，如`em`(一个 [Em](https://en.wikipedia.org/wiki/Em_(typography)) 的长度)、`%`(百分比)、`rem` (root-Em)，以及其他任何可能的度量。这可以确保网站根据您选择的字体和用户选择的缩放级别按比例缩放。

##### [](#mistake)🚫错误

```
p {
  font-size: 16px;
  line-height: 20px;
  margin-bottom: 8px;
} 
```

##### [](#correct)✅纠正

```
body {
  font-size: 16px;
}
p {
  font-size: 1rem;
  line-height: 1.25;
  margin-bottom: 0.5em;
} 
```

<small>这些代码样本来自[无痛 CSS](https://www.painlesscss.com) ，这是一个从基本原理开始教你 CSS 的书籍和视频课程。你可以在[课程](https://www.painlesscss.com)中下载包含所有 60 个代码样本和解决方案的完整报告。</small>

使用像`px`这样的绝对单位的问题是每个用户有不同的显示器尺寸，他们的浏览器窗口(视窗)也有不同的尺寸。这是违反直觉的，因为在我们生活的几乎每一个其他领域，像室内装饰、城市规划或烘焙，我们总是与绝对措施打交道。但是在网上，我们正在画的画布会突然改变比例，每次都有新的人看着它。

这就使得你的网站不可能处处都是一样的比例，所以你必须尽量用相对项(砖块 B 应该和砖块 A 一样大)而不是绝对项(砖块 B 是 200px 乘 400px 的长方形)来放置你的元素；否则，屏幕上的可用空间不会适当地缩放到每个窗口大小。相对单位使这变得简单— `px`将您的设计锁定在特定的缩放级别，使您的设计难以扩展到不同的设备。

## [](#mistake-7-trying-to-achieve-a-pixel-perfect-design)错误 7:试图实现“像素完美”的设计

更一般地说，在编写 CSS 时，“像素完美”的设计是一个糟糕的目标。

现代网站必须在各种设备(移动设备、桌面设备、平板电脑和手表)上工作，其中有大量的屏幕尺寸、屏幕分辨率、操作系统、用户设置和 JavaScript 引擎，它们会干扰您呈现“像素完美”设计的能力。

在我看来，对“像素完美”设计的狂热追求将使你付出巨大的额外代码复杂性，来处理这些不一致的渲染方案(以及复杂代码导致的错误风险的增加)，而一旦下一代设备、操作系统版本和浏览器版本发布，这种小小的好处可能会被淘汰。

在实践中，只有当你是苹果或谷歌这样的大公司，并且有资源(钱)支付大型设计团队对设计进行 0.01%的改进时，追求这种设计完美才有商业意义。但是，除非你的业务也能每分钟赚 40 万美元，否则 0.01%的改进不会让你每年多赚几百万美元——相反，这只会增加不必要的工程复杂性。

不要混淆“像素完美”的设计和伟大的设计。真正伟大的设计是给用户留下持久的情感影响。这是他们打开你的产品时得到的快乐，也是他们从参与你的网站中得到的快乐。这不是对你选择 16pt 字体而不是 15pt 字体的崇拜。

## [](#mistake-6-breaking-the-document-flow)错误 6:打破文件流程

CSS 的巧妙之处在于，你可以编写不同的 CSS 代码来获得相同的视觉效果。例如，要在页面上水平居中一个元素，您可以使用以下任何一种技术，它们对用户来说看起来都是一样的:

1.  用`&nbsp;`包围该元素，直到它看起来居中
2.  显式地将元素的`width`和 left `margin`设置为一个固定值，使其看起来居中。
3.  使用`text-align: center`
4.  使用`margin: 0 auto`
5.  使用一个组合的兄弟和一个 clearfix。
6.  使用`position`、`left`和`transform`的组合。

哪种技术最好？根据你的目标，使用上述任何一种方法都是完全正确的。

一般来说，有一些我们试图遵循的经验法则来选择最佳的布局方法。

首先，我们排除任何使用绝对度量而不是相对度量的技术。这意味着#1 `&nbsp;`和#2 显式设置`width`不可用。

第二，我们更喜欢*而不是*修改文档流的技术，而不是破坏文档流的技术。这是因为默认情况下，元素遵循文档流，所以如果你打破了这个默认假设，你的同事(或者你自己，6 个月后)会更难理解发生了什么。这意味着#5 `float`和#6 `position`退出。

更一般地说，我们更喜欢对其他东西要求尽可能少改变的技术。保持较小的内存占用有助于避免难以维护的庞大、复杂的代码。

技术#3，`text-align: center`只对内联元素有效，所以只要我们的 HTML 已经是内联元素，我们就选择#3，并且我们避免了添加`display: inline` CSS 声明的额外需要。

技术#4，`margin: 0 auto`只对块元素起作用，所以每当我们的 HTML 已经是块元素时，我们将选择#4，并且我们避免了添加`display: block` CSS 声明的额外需要。

过于频繁地破坏文档流(例如过度使用`float`)会增加你的足迹，使你的布局更难理解。当有多种选择来编写 CSS 代码以获得相同的视觉效果时，我们更喜欢使用占用空间较小的技术。

## 错误 5:没有把设计和布局分开

CSS 的工作是提供样式，HTML 的工作是提供结构。一般来说，你应该首先以一种捕捉页面信息层次的方式编写 HTML，忽略任何设计上的考虑。之后，您可以添加 CSS 来使事情看起来更好。

虽然 HTML 提供了结构，但它不能总是将元素定位在页面上您希望它出现的确切位置。因此，您可以使用 CSS 来构建页面的正确布局，以便元素出现在正确的位置。一旦一个元素被放置在页面上的正确位置，之后就很容易修饰它，使它看起来很漂亮，而不用担心它的位置，所以**你应该把“布局 CSS”看作是不同于“口红 CSS”**的工作。

##### [](#mistake)🚫错误

```
.article {
  display: inline-block;
  width: 50%;
  margin-bottom: 1em;
  font-family: sans-serif;
  border-radius: 1rem;
  box-shadow: 12px 12px 2px 1px rgba(0, 0, 0, .2);
}
.sidebar {
  width: 25%;
  margin-left: 5px;
} 
```

```
<div class="article"></div>
<div class="article sidebar"></div> 
```

##### [](#correct)✅纠正

```
/* Layout */
.article, .sidebar {
  display: inline-block;
}
.article {
  width: 50%;
  margin-bottom: 1em;
}
.sidebar {
  width: 25%;
  margin-left: 5px;
}

/* Lipstick */
.card {
  font-family: sans-serif;
  border-radius: 1rem;
  box-shadow: 12px 12px 2px 1px rgba(0, 0, 0, .2);
} 
```

```
<div class="article card"></div>
<div class="sidebar card"></div> 
```

<small>这些代码样本来自[无痛 CSS](https://www.painlesscss.com) ，这是一个从基本原理开始教你 CSS 的书籍和视频课程。你可以在[课程](https://www.painlesscss.com)中下载包含所有 60 个代码样本和解决方案的完整报告。</small>

确保将“口红”工作与“布局”工作分开，也被称为*关注点分离*，这是一个常见的软件工程原则，有助于保持代码的可维护性和易于理解。

我们希望使用合适的工具来完成这项工作，因为以这种方式分离 CSS 可以很容易地将元素移动到页面的另一部分，而不会弄脏口红，还可以很容易地在不破坏布局的情况下更改口红的颜色。混合这两者迫使我们每次添加新特性时都要做两项工作，即使它只保证一项工作。

## [](#mistake-4-designing-for-desktop-before-mobile)错误#4:在移动之前为桌面设计

发展机构将手机网站作为额外的选择附加在发票上的日子已经一去不复返了。大多数人现在都用手机上网，通常会看到超过 50%的网络流量来自手机和平板电脑。

人们喜欢赞美“移动优先”的理念，但他们忘记了“移动优先”意味着“T2”台式机是二等公民。取而代之的是，“移动优先”的冠军们采取了相反的方式，首先为桌面编写代码，然后试图将网站塞进手机。他们使用`@media`查询来处理移动设备上的例外情况，但真正的*桌面*应该是例外情况。

在这个移动时代，你的用户首先通过他们的手机找到你，然后只有在后来，如果他们足够喜欢你的产品，他们才会升级到桌面体验。

但是移动真的是设计师、开发者和产品经理的首选吗？你是先编写移动网站的代码，然后再构建桌面版本吗？你的设计师在为桌面创建线框和模型之前，会为移动设备创建线框和模型吗？在测试桌面版之前，你会对你网站的移动版进行 A/B 测试并征求用户反馈吗？

##### [](#mistake)🚫错误

```
.container {
  width: 980px;
  padding: 1em;
}

@media (max-width: 979px) {
  .container {
    width: 100%;
    padding: 0.5em;
  }
} 
```

##### [](#correct)✅纠正

```
.container {
  width: 100%;
  max-width: 980px;
  padding: 0.5em;
}

@media (min-width: 980px) {
  .container {
    padding: 1em;
  }
}
/*
  Notice how this also avoids needing to
  write 979 instead of 980, so you don't
  need to search for two different values
*/ 
```

<small>这些代码样本来自[无痛 CSS](https://www.painlesscss.com) ，这是一个从基本原理开始教你 CSS 的书籍和视频课程。你可以在[课程](https://www.painlesscss.com)中下载包含所有 60 个代码样本和解决方案的完整报告。</small>

您的用户大多是移动用户，这并不是在桌面之前为移动设计和开发的唯一原因。主要原因是，在将设计升级到大屏幕之前，为小屏幕开发更容易，而不是相反。您总是可以将现有元素放大以填充大屏幕上的空白区域，但是要将元素从大屏幕上移除以适应较小的屏幕就比较困难了。

如果你发布了一个没有桌面网站的移动网站，现在你需要花费$X 来构建，然后再花费$X 来添加一个桌面网站。但是如果你提供一个没有移动网站的桌面网站，你现在需要花费 X 美元来建立，然后再花费 2X 美元来增加一个移动网站。这就是人们所说的“技术债务”:你今天付出了较低的成本，但从长远来看却付出了更大的代价。

现在，如果你是闪电式扩张，它可能是有意义的“低效”和只建立桌面网站，以证明产品市场适合的理解，移动网站将花费更多，但这是一个商业决定。工程师们更喜欢先建立移动网站，即使这并不总是有商业意义，所以根据你的成长阶段相应地调整这个建议。

## [](#mistake-3-not-using-a-naming-scheme)错误 3:没有使用命名方案

> 名称空间是一个非常棒的想法——让我们多做一些吧！
> —<cite>PEP 20:Python 之禅</cite>

不仔细考虑变量的命名方案就直接编写代码总是令人兴奋的。比如你叫它`img`还是`image`？`blog-img`还是`img-blog`还是`blog-image`还是`blog-img`？如果你叫它`blog-img`，你以后会同意`ad-img`和`thumbnail-img`吗？

这看起来很小，但 CSS 比其他语言更难重构，因为我们目前没有任何针对 CSS 的静态分析工具来让我们执行自动重构/重命名。CSS 类可以使用 JavaScript 动态地添加、删除和构建，所以不可能找到“未使用的”CSS 选择器——仅仅因为 CSS 的一个片段在一个页面的一个状态中没有被使用，并不意味着它从来没有被使用过*。如果你的团队不能就是否称之为`.center`、`.centre`、`.centered`、`.text-center`或`.align-center`达成一致，BEM 并不能解决这个问题。*

 *这个问题通过使用 CSS-in-JS 库得到了部分解决，因为这些框架移除了 CSS 的全局范围，并鼓励您将样式与其相关组件放在一起。然而，CSS-in-JS 尚未被广泛采用(截至 2019 年)，我的愿望清单上仍然有一个 CSS 的静态分析工具——理论上，CSS-in-JS 应该可以首次实现一个。

这也可以通过使用设计系统来部分解决，在设计系统中，你同意颜色、间距、字体等的一组固定变量。，但是未使用的类有持续存在的趋势，因为我们没有自动识别和删除它们的方法。

开发者理所当然地不愿删除任何可能破坏网站的东西，所以他们宁可小心谨慎，也不愿增加垃圾。更糟糕的是，除非你努力推广并清晰地传达你的设计系统，否则它可能会被忽视，因为你的同事会避免阅读你的文档，继续他们的旧习惯。

## [](#mistake-2-not-reading-the-documentation)错误 2:没有阅读文档

看起来很容易。只要安装引导程序，你就可以开始了，对吗？不幸的是，如果你没有真正理解 CSS 框架，并且不小心违背了它的原则，那么 CSS 框架经常会导致比它解决的问题更多的问题。无论你是遵循像 Bootstrap、Material Design、Foundation 这样的开源设计系统，还是你的公司自己开发的设计系统。

我职业生涯中做过的最好的事情之一是，每当我想使用一种新的编程语言或框架时，就留出 4-8 个小时来完整阅读官方文档*。也就是说，不是浏览文档或只搜索与我当前需求相关的关键词，而是像读书一样从头到尾地阅读。*

 *例如，在我开始使用 Bootstrap 之前，我花了 6 个小时来完整阅读它的文档。这为我以后的实现节省了无数的时间，尤其是避免了似乎普遍存在的网格嵌套问题，因为那些跳过文档的人没有意识到不允许嵌套引导容器。

认真阅读文档很重要，因为**许多框架要求你取消从以前的框架中获得的肌肉记忆**，并且直到你阅读了整个手册，你才能知道陷阱在哪里。例如，组件继承是早期前端框架(如 Backbone)中的常见模式。然而，当使用[组合而不是继承](https://reactjs.org/docs/composition-vs-inheritance.html)时，较新的框架如 React 工作得最好，但是如果你跳过文档，你将不会意识到这一点，因为你将只是默认使用你习惯的继承模式。

更一般地说，被你从旧技术中发展出来的肌肉记忆所困是很危险的，这让我们犯了最后一个致命的错误:

## 错误 1:没有按照系统的计划学习 CSS

技术是不断发展的，保持相关性的唯一方法是遵循一致的学习和改进计划。

从教条中死记硬背知识是很危险的。一些有经验的开发人员对听到“内联样式”有一种本能的负面反应，因为他们在职业生涯早期就被告知“永远不要使用内联样式”的最佳实践。

事实上，“永远不要使用内联样式”是很久以前的最佳实践，当时它使得 HTML 难以阅读、难以调试和难以更改。然而，这项技术不断发展，最终颠覆了导致这些“最佳实践”的假设。新的假设导致新的最佳实践。

只有当您手工编写内联样式时，它们才是令人痛苦的——但是如果您使用 CSS-in-JS 库来管理您的内联样式，这些缺点就会消失，您只剩下代码模块化、自动化测试和与前端 JavaScript 框架(如 React)协同工作的简单管理的优势。

如果你坚持“永远不要使用内联样式”的教条，你就不会有好奇心去探究当最初的假设改变时意味着什么。使用一个系统的、结构化的计划从头开始学习 CSS 是很重要的，而不仅仅是在谷歌上搜索只关注表面实践的教程。

不幸的是，我试图寻找免费的在线资源来帮助你以一种结构化、系统化的方式学习 CSS，但是根本没有什么好的资源。现有的 CSS 资源要么技术性太强，假设你已经知道浏览器 C++渲染引擎是如何工作的(这一点都不容易实现)，要么它们只提供快速而肮脏的技巧，而不是教给你基本的原则。

因为没有好的资源给你一个坚实的心智模型来说明 CSS 实际上是如何从头开始工作的，所以你只能复制/粘贴栈溢出的内容，并希望它能工作。这使得许多人对 CSS 知之甚少，只知道解决特定的 bug，但是面对一个看起来需要几个小时才能修复的难以理解的 bug 时，却感到完全无助。
