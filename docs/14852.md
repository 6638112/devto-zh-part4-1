# 实践 PDD -恐慌驱动的开发

> 原文：<https://dev.to/maurofrezza/practicing-pdd-panic-driven-development-2206>

在敏捷开发技术的第一波成功之后，有一种特殊的技术幸存了下来，并在这些年里不断涌现:PDD 恐慌驱动开发。这种技术分享了敏捷开发方法的核心概念，但是消除了所有降低团队速度的仪式和过程负担。
让我们更详细地看看这种方法论的原理是什么。

## 最近的问题有优先权

每当一个新的问题出现在冲刺阶段，它就优先于任何计划好的工作。新的总是更好，优先级更高。事实上，这也应该是敏捷开发的原则之一。对向客户交付价值的关注要求团队关注新项目并推迟之前定义的工作。

## 编写足够的代码来解决这个问题

开发人员以写代码为生。bug 只能通过代码来修复。设计和 UX 的讨论只能减缓发展。首先编写一个解决方案，用补丁验证你的假设。如果修复有效，您就解决了问题。

## 测试应该是后续任务

一旦实现了修复，就可以将测试计划为将来要完成的任务。测试是有用的，但不是重点。你可以以后再照顾他们。归档一张罚单，并将其放入待办事项中。手工测试应该足以证明你的修复。

## 相信你的直觉

编程是艺术，艺术有内在的本能成分。相信你的直觉。编写您的解决方案。部署它。只有勇敢的人才能成功。

## 流程灵活

任何开发、测试和发布软件的过程都只是一套约定和规则，并不是一成不变的。关键修复需要不同的方法。为了更快地行动，人们期望你改变流程。

## 这是一个经理驱动的过程

作为团队的一员，经理有权就发展问题发表意见。重构或良好实践可以也应该被业务需求所否决。工程师可以表达他们的意见，但他们最终应该致力于来自上面的任何需求。

### 结论

PDD 是一种可以在短时间内为每个项目带来巨大性能提升的技术。
世界上有几家公司都这么做。它代表了敏捷和极限编程的核心。
