# 测试有害

> 原文：<https://dev.to/jessekphillips/testing-doing-harm-1d4>

我开始想知道测试工作应该重新评估的一些一般指标是什么，因为测试很可能是有害的。软弱无力的测试是一个简单的方法，但我认为我所看到的肯定不止这些。这是在考虑自动化的情况下编写的，但是也适用于手工测试。

当我试图向一位同事解释一些测试挑战时，我意识到有害测试的关键指标。从足够高的角度来看，测试由输入和期望组成。当环境是输入的一部分时，或者当对输入存在多种期望时，这就变得复杂了。

如果你不能确定测试期望的输入或期望是什么，这可能是一个很好的改进，可能会导致一些剥落。

主要关注的是当你的输入或预期结果不断变化时。这是否改变了测试框架中的环境或 json 文件并不重要。我需要明确的是，我在这里通过举例来谈论指定的输入和结果。

有时你会失败，因为你的验证依赖于某种形式的内容，按钮或链接的文本。这对于很多测试来说是没问题的，但是如果文本从一个版本变化到另一个版本，那么是时候停止这些变化或者应用不同的验证了。(内容可能需要测试，但需要明确所有权和测试的内容)

我经常用 API 模拟和测试。有时候会发生变化，比如从 xml 到 json。我使用本地 API 格式来驱动我的测试。这意味着我的输入将从 xml 变成 json。然而，我并不关心它，因为这不是一个持续的变化。在没有我干预的情况下，我可以主要依靠测试来报告。然而，如果结构确实处于不断变化中，假设初始开发勘探已经完成，针对这种变化进行优化是很重要的。

为什么不断变化的输入和期望如此糟糕？您编写的测试旨在识别未能满足变更期望的情况。如果您的预期结果或输入需要修改，那么您需要评估您的测试是否需要更新或者是否有回归。我看到这导致了一个优化的工作流程，通过修改它来“修复”测试，从而导致通过。

现在，在这些情况下，系统正在发生变化，因此测试可以被视为提供对这些变化的洞察。但我想知道，这是额外的洞察力吗？不应该已经有关于这方面的变更记录吗？把时间花在深入了解系统如何运行、风险识别和覆盖系统不同部分的设置测试上不是更好吗？
