# 关于技术债务的思考

> 原文：<https://dev.to/benbirt/thinking-about-technical-debt-b72>

技术债务是软件工程师普遍关心的问题。但是我们并不经常从正确的角度去思考它。

# 货币债务

贷款后，你(通常)必须支付利息。

如果你不能全额支付利息，在短期内，你的债务会增加。如果你继续无法偿还利息，你可能不得不宣布破产。这意味着你可能会失去你的资产来偿还债务。

如果情况良好，你将能够既支付利息又偿还债务。在支付了足够的费用后，你就没有债务了。

# 技术债务作为货币债务

将技术债务引入代码库相当于贷款。

你现在必须连续支付贷款利息。兴趣可以有多种形式。一个是花费在修复错误或回归上的工程时间。另一个原因是在生产中运行系统会增加操作负载。

如果你没有足够的收入(工程时间)来支付成本，技术债务将会增加。由于没有时间进行适当的错误修复，工程师们将会给代码库带来更多的负担。

如果这种情况继续下去，你可能不得不宣布技术破产。这是一个可怕的情况:你现在必须放下所有其他工作来解决技术债务。在最坏的情况下，您可能不得不放弃对系统的所有支持，而从头开始完全重写它。

# 我们不应该害怕技术债

申请贷款通常有充分的理由。例如，它可能使我们能够进行一项在未来有很大回报的投资。

与贷款一样，技术债务也是如此。我们必须在技术债务和新功能投资之间做出权衡。这些功能可能会带来更多的利润，使我们能够雇佣更多的工程师。因此，偿还技术债务变得容易多了。

# 像管理贷款一样管理技术债务

正如真实债务一样，技术债务需要谨慎管理。

我们不应该为了技术债务而害怕技术债务。我们应该担心的是计划不周。

如果你打算用贷款进行投资，制定一个计划是明智的。该计划将模拟贷款价值和投资价值随时间的变化。

技术债务更难建模。但我们应该仔细考虑它的成本和不承担它的机会成本。我们还应该跟踪我们当前的债务，它让我们付出了什么代价，以及还有什么空间来承担更多。
