# [](#6-connect-with-other-developers)6:联系其他开发者

你听说过“铁磨铁”这句话。在软件开发行业，这当然是事实。我的大部分知识都归功于我的软件开发同事。

像 DEV、GitHub 和 Freenode IRC 这样的社区提供了许多与其他软件开发人员交流的机会。然而，成为社区的积极分子不仅仅是问问题。分享你的知识和观点，并接受反馈。多了解别人的背景和想法。*交朋友！*

通过这些关系，你会发现丰富的知识、支持和欢笑。

## [](#disagree-politely)委婉地表示不同意

既然我们谈到了这个话题，你绝对应该学习建设性的辩论技巧。同样，这似乎是编程的主题，但事实并非如此；标准、编码实践和项目决策都脱离了健康的辩论。我在正式的希腊式辩论方面有些背景，但你没必要走那么远。这里有一些建议:

*   **辩论思想，而不是人。(这是互联网讨论变得如此糟糕的最大原因。)当出现分歧时，*唯一应该质疑的是有争议的想法的真实性。所有政党仍然具有作为人的固有价值，它们的价值不会因为辩论的结果而上升或下降。攻击一个人作为摧毁一个想法的手段被称为*ad homine*(针对这个人)，这是一个肮脏的游泳池。***

*   核对你的事实。我们都有责任引用事实证明是不准确的。如果主题很重要，你应该花时间来确定你的主张。如果你*没有那么投入，你至少应该公开承认靠记忆工作。*

*   不要歪曲反对的观点来推翻它。这被称为“稻草人辩论”，是辩论中的另一种卑鄙手段。你应该假设任何对立的观点都至少有一点可取之处，其他人都是理性的人类。

*   **学会自嘲。你迟早会说错话的。当这种情况发生时，抑制住自卫的冲动。如果你是第一个对自己的错误轻描淡写的人，这会让每个人都更舒服。**

*   如果必须的话，走开。与其失去冷静，不如突然离开谈话。如果你发现自己过于情绪化，那就走开，直到你控制住自己。无节制的愤怒不会导致合理的行为。

# [](#7-ask-questions)7:提问

迟早，你会被难住的。这时你就可以向你的关系网寻求帮助了！

在你问之前，有几件事你应该先做:

1.  遵循“20 分钟规则”。尝试至少 20 分钟自己解决，然后再问。

2.  **分享详情。**只告诉我们“X 坏了”并不能帮我们帮你。我们需要看到您的代码、准确的错误输出、您期望看到的内容，以及您使用的编程语言版本和工具。

3.  告诉我们你尝试了什么。如果你先花了那 20 分钟，你应该会发现一些没用的东西。分享这些知识，这样我们就能更快地找到答案！

顺便说一下，如果你正在使用 IRC，*请耐心等待*。发布您的问题并等待。由于 IRC 的性质，有时可能要过几个小时才会有人看到它。如果你签字同意，我们就帮不了你了！同样，在问之前，不要问房间里是否有人认识...*就问*。回答一个关于`<X>`的问题不一定需要专家。

# [](#8-help-others)8:帮助别人

随着你了解得越来越多，你就能回答其他问题了...你绝对应该这么做！就我个人而言，我发现我从教别人那里学到的东西比我自己阅读学到的要多得多。有一些关于吸收知识，然后转身为别人重新包装它，这只是帮助它坚持下去。

当然，一开始回答问题是令人生畏的。“万一我做错了呢？”当然，你可以先研究这个话题，检查你的假设，但有时你会出错。

别担心。在最坏的情况下，你会给另一个年轻的开发人员错误的信息，但迟早他们会知道真相。最好的情况是，另一个更有经验的开发人员会过来纠正你，在这种情况下，你会学到一些东西。无论如何，不要为这样的错误责备自己。它们都是学习过程的一部分。

StackOverflow 可能是你想到“回答问题”时首先想到的地方，但绝不是唯一的地方。(如今，它甚至不是*最好的*地方。)DEV 的`#help`标签提供了许多机会，许多编程留言板和 Slack 频道也是如此。

你也应该考虑加入 Freenode IRC，在与你使用的技术相关的频道中帮助回答问题。我第一次加入那个网络上的`#python`房间是在我编码的第二年，从那以后我就一直在那里。(其实我的“死简单蟒”系列就是从我在那个房间的经历中走出来的。)

# [](#9-you-are-a-real-programmer)9:你是真正的程序员！

在这个过程中，你会对自己说:“如果我知道那个东西，我会成为一个真正的程序员。”

***停！*T3】**

如果你在生活中曾经写过任何代码，并且让它*工作*，你就是一个真正的程序员。永远不要质疑这一点。即使是已经写了 30 多年代码的开发人员也需要学习更多的东西。正如我之前提到的，这就是软件开发的美妙之处...你永远学不完东西。

作为一名程序员，你的合法性并不取决于你懂什么语言，你使用什么工具，或者你解决什么类型的问题。没有两个程序员有完全相同的经历，或者以完全相同的方式看待问题。每个人都有贡献。

你是一个真正的程序员。你没有假装。目标永远不是成为一个“真正的”程序员，而是成为一个更好的程序员。
