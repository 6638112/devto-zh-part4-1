# 在一个月内构建一个完整的应用程序。以下是我学到的一些经验。

> 原文:[https://dev . to/amckean 12/building-a-full stack-app-in-a-month-here-some-lessons-I-learned-bo](https://dev.to/amckean12/building-a-fullstack-app-in-a-month-here-are-some-lessons-i-learned-bo)

在过去的一个月里，我真的挑战了自己，在我的脑海中产生一个想法，并开发出一个产品。我想做一些不仅可以向潜在雇主炫耀，而且我最终可以使用的东西。该应用程序是一本食谱，用户可以在其中搜索食谱，将搜索到的食谱添加到菜单中，然后根据这些食谱生成购物清单。为了构建这个应用程序，我使用的主要工具是 React、Redux、Rails 和 PostgreSQL。在这个过程中，我学到了很多东西，但有三个主要的教训非常突出。

# [](#dont-be-afraid-to-break-things)不要害怕打碎东西

你会让你的应用崩溃。当这种情况发生时，不要惊慌，花时间解决问题。有些问题需要一个小时才能解决，有些需要几个星期。不要让一周内解决问题的恐惧阻止你实施新事物。最初，我让这种情况发生。我从我的应用中去掉了一些特性，因为它们缺乏清晰的实现路径。我的应用失去了让它变酷的核心东西，因为我不知道如何构建它们。当你去掉所有让你的应用程序变酷的功能时，你剩下的只是一个美化的待办事项列表，没有人愿意花一个月的时间来构建它。

创建一个新的分支并尝试/测试一些想法没有坏处。我学到了比固定课程计划更多的尝试和突破。最坏的情况是切换回你的稳定分支，不会造成任何伤害。

# [](#read-up-on-the-libraries-your-implementing-into-your-app)阅读你在应用中实现的库

每当你在你的应用中实现一个外部库的时候，仔细阅读它。我花了很多时间试图解决问题，这些问题本可以通过快速浏览回购问题页面来解决。对我来说，外部图书馆的文档质量是决定我是否使用它的一个重要因素。这是我研究某些图书馆的过程。

1.  阅读该库的官方文档
2.  查看回购问题
3.  查找与堆栈溢出库相关的问题
4.  阅读关于图书馆的文章

最初，我觉得这个过程有些过分。感觉我是在浪费时间阅读，而不是真正尝试用代码实现。我发现我对这个库有了更好的理解，当实际使用这个库的时候，我知道可能会出现的潜在问题以及到哪里去解决它们。

# [](#design-design-design)设计，设计，设计！！！

拜托，如果这篇文章能让你学到什么的话，那就是花时间去设计。我的设计流程是:勾画出应用程序的样子，然后直接编写代码。我很痛苦。没有适当的设计，你真的会失去对应用程序的控制。以下是我遇到的一些归因于糟糕设计的问题。

1.  重复组件
2.  太多基于类的组件，这些组件本可以发挥作用
3.  不必要的 CSS
4.  重复代码
5.  很多不必要的地方州
6.  一个太复杂的文件树

截至今天，我的应用程序正在运行和工作，但维护它肯定会很困难。对我来说，我认为一个很好的练习是使用 esLint 这样的工具和 airBNB style guide 这样的标准重写，强调设计和可维护性。
