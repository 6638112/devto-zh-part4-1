# 事后诸葛亮的问题是

> 原文：<https://dev.to/kayla/the-problem-with-a11y-after-the-fact-23pp>

人们似乎普遍认为，我们可以设计一个像素级完美的应用程序，经历整个开发过程，然后在一切都完成后，再挤进一点可访问性。这个过程可能会帮助你提高一点你的灯塔得分，但不会对你的用户有太大帮助。

可访问性不应该是事后才想到的！

## 自动化测试并不能捕捉到一切。

自动网页内容可访问性指南(WCAG)检查器通常只会捕捉“低挂的果实”，如颜色对比度不足，检查图像是否有`alt`文本等。您的应用程序可以通过自动检查器运行的每一次审核，但是不能发现对应用程序可用性有负面影响的东西。例如，这些工具无法检查图像的`alt`文本，以确保它们正确描述了每个图像中的内容或传达了意图。

WCAG 是可访问性测试的一个很好的起点，但是开发一个完整的应用程序并在其上添加 WCAG 樱桃并不能使它具有可访问性。

## 你在为自己做更多的工作。

老实说，尝试用可访问性来改造一个 web 应用程序比从项目一开始就结合可访问性的四个原则要耗时和令人沮丧得多。这使得您的工作(以及未来开发您的应用程序的开发人员的工作)变得更加困难。

我无法告诉你有多少次我不得不试图把可访问性的概念塞进一个现有的应用程序，试图使它比我接触它之前更有用一点。更糟糕的是，如果应用程序不是使用语义 HTML 编写的。😱

当你开始一个项目的时候，记住可访问性是对你时间的有效利用，并且会让你的用户受益。

## 关键要点

从一开始就考虑你的网站或应用程序的潜在用户是很重要的。早期为每个人设计是网站易访问的原因。

永远记住:在设计和开发应用程序时，考虑到可访问性有助于所有用户。
