# 选择学习什么

> 原文:[https://dev.to/remotesynth/choosing-what-to-learn-l2m](https://dev.to/remotesynth/choosing-what-to-learn-l2m)

作为开发人员，我们经常被新技术、API 和工具淹没。辨别什么值得学，什么不值得学实际上是相当困难的。当你有无数个方向可以选择，但时间有限时，你如何区分优先次序？在这篇文章中，我分享了我思考这个话题时记下的一些想法。我很想听听你是如何决定下一步要学什么，以及你是如何腾出时间来做这件事的。

因为我个人很好奇，所以有很多我为了工作必须学的东西和我想学的东西。为了展示如何思考这个问题，我提出了一些“类别”:

*   对于工作:
    *   东西我*得*学，我也*想*学。
    *   我*必须*学的东西，我*并不特别想*学
    *   让我在工作中做得更好/更有效率的事情
*   对于职业发展:
    *   在我的职业发展中，为了得到我的第一份/下一份工作，我需要学习的事情
    *   我想学的东西，因为它们可能会影响我的职业生涯。如果我想换一个方向，比如从后端开发到前端)
*   因为我很好奇:
    *   我想学的东西是因为它们看起来很有趣，但与我的工作或职业发展没有直接的关联(这并不意味着它们最终可能没有关联)

为了工作，我必须学习的东西显然是第一位的——专注于我不得不学但 T2 也想学的东西，因为这些可能是我能产生最大影响的领域(显然，我不会忽视我不想学的东西，但我认为降低它们的优先级是公平的)。还有你的工作是否给你时间专注于学习的影响——它们是否是你必须学习的东西，或者，希望，甚至是你应该探索以改进你的工作的东西。一般来说，工作应该提供时间来学习你*至少要*学的东西。这是不言而喻的，但遗憾的是没有。

职业发展领域和“因为我很好奇”往往是最难的，仅仅是因为时间的因素。就个人而言，我倾向于创造“强迫因素”，比如我承诺的博客帖子或演示文稿，以帮助将我想学的东西变成我 T2 必须学的东西。由于很少有时间立即做这样的事情，我也发现记录我感兴趣的事情很有用。我通过在一个特殊的文件夹中为博客帖子或 GitHub repo 等内容的相关 URL 添加书签来做到这一点。每当我找到时间来学习新的东西，我就会回到那个文件夹。

所有这些的另一个棘手的方面是决定什么是真正值得学习的。我们被新的工具和框架轰炸，特别是如果你关注社交媒体，它会让人感觉好像除了你，世界上每个人都已经知道并使用“技术 X”。这有时会让你很难区分你不得不学的东西和你想从 FOMO 学到的东西。以我的经验来看，你应该小心不要把趋势误认为你必须学习的东西，也要小心那些试图告诉你“你必须知道”的事情的社交媒体/博客帖子。

我希望我能提供一些明确的建议，告诉你哪些策略有效，哪些策略无效，但重要的是要注意，对我有效的策略不一定对你有效。那么，你如何决定重点学习什么呢？我很想听听你的策略。
