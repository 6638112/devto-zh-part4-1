# 2019 年塔林北欧测试日

> 原文:[https://dev . to/henrikwarne/nordic-testing-days-Tallinn-2019-4j 58](https://dev.to/henrikwarne/nordic-testing-days-tallinn-2019-4j58)

五月底，我参加了在爱沙尼亚塔林举行的北欧测试日。这是我第一次在瑞典以外的会议上发言，我度过了一段美好的时光。一天是辅导课，两天是工作坊和半小时的常规讲座。以下是我的观感: [![](../Images/c707f24bddc4c30d5432462db1e189c9.png)](https://henrikwarne1.files.wordpress.com/2019/06/ntd2.png)

## [](#talks-i-liked)我喜欢的谈话

为什么探索性测试应该成为主题演讲的主题？由 [Alex Schladebeck](http://www.schladebeck.de/) 所做的开场主题演讲是会议中最精彩的部分。探索性测试似乎经常被直觉所驱动——“也许我会试试这个”，或者“我想知道如果我那样做是否有效”。但是为了教会如何做好这件事，我们需要意识到*为什么*我们要做这些事情——直觉的基础是什么。做这件事的一些技巧是叙述你正在做的事情，思考你为什么采取你所做的行动，标记你所遵循的规则，练习回忆你所做的事情。

在演讲的第二部分，亚历克斯举了许多她使用的启发式的优秀例子。例子包括:***if 是 iffy***–现实生活中的 if 往往会导致代码中出现 if，而那正是 bug 的沃土，值得探究。 ***玫瑰的其他名字***——如果一个概念有重名(比如预订航班时婴儿与婴儿)，可能会有问题。*——对于 web 应用，尝试后退和前进，做和撤销，注意问题。 *****黄色很有趣***–当状态有两个以上的值时，比如绿色、黄色和红色，经常会有 bug，因为逻辑变得更加复杂。***

 *这些试探法非常有用，您可以马上在自己的测试中使用它们。亚历克斯还给它们起了朗朗上口的名字，这可能要归功于她在语言和语言学方面的背景。我真的很喜欢这个演讲，因为它让我以一种新的方式思考探索性测试——为什么我现在决定采取这个行动？–因为它给了我许多有用的测试启发。交付也是一流的。

[![](../Images/e91bb8665375165ce8c865b7db30f85e.png)T2】](https://henrikwarne1.files.wordpress.com/2019/06/dsc_1033.jpg)

利用组件测试进行超快速构建。 [Tim Cochran](https://twitter.com/timcochran) 谈到了如何划分和加速测试。他推荐在组件测试中测试功能和边缘案例(比如一个微服务)，用端到端测试测试组件(关键路径)和配置之间的集成。我认为这是相当标准的建议，但被提醒是件好事，它让我思考我们如何测试我工作的应用程序。

加速测试的想法包括并行运行组件测试，以及讨论进程内(函数调用)和进程外(API 调用)测试。他还提到了[加速](https://dev.to/henrikwarne/book-review-accelerate-457)对快速测试的重要性。我喜欢这个演讲是基于帮助公司建立 CI/CD 管道的实际经验。演讲中最好的引用是“可测试性是一个特性。不要害怕改变代码，使其更具可测试性”。对此我衷心赞同！

[![](../Images/1ba47bb5a7320d550bdef4897a2abd25.png)T2】](https://henrikwarne1.files.wordpress.com/2019/06/dsc_0978.jpg)

QA 后的生活。 [埃里克·卡朱](https://twitter.com/erikkaju)是 TransferWise 支付卡工程主管。他已经从测试角色转变为开发人员、技术主管和工程经理。在 TransferWise，没有专门的测试人员——这是每个人的责任。他继续谈论它们是如何工作的。我认为这真的很有意思，可以和我们在 [TriOptima](https://www.trioptima.com/) 的工作方式进行比较。以下是最有趣的事实:客户支持是产品工程团队的一部分(以确保收到反馈)。不同的角色经常互相影响，例如处理几天的客户支持。目前他们有 169 个微服务，每月发布 1500 次。他们在测试环境中使用连续部署，在生产环境中使用手动部署。他们保留所有公关和发布的统计数据——越小越快。开发人员可以部署到 prod，但不能访问客户数据。

[![](../Images/97bcfa53f0211c8296a7dd886d0b31d8.png)T2】](https://henrikwarne1.files.wordpress.com/2019/06/dsc_1049.jpg)

## [](#learning-from-bugs)向虫子学习

我演讲的题目是[从错误中学习](https://henrikwarne.com/2016/06/16/18-lessons-from-13-years-of-tricky-bugs/)，我是在最后一天午饭后做的。同时还有另外四个讲座和研讨会，我想大约有 90 人参加了我的讲座。我对事情的进展很满意。当我覆盖某些 bug 时，有几个人微笑着点头，这非常令人鼓舞。所有与会者都被推荐使用应用 Attendify 对他们所听的演讲进行评分。会后，我从我的演讲中得到所有的评价和评论。评分范围从 1 星到 5 星，我得到了 1 颗“2 星”，7 颗“3 星”，24 颗“4 星”和 18 颗“5 星”，平均评分为 4.18。真的很感谢能得到这个评分！16 个人也写了评论，大部分都挺正面的。这一条让我特别开心:“很好，轻松的谈话，午饭后的理想。你实际上在那些幻灯片中有一些内容，这是一件罕见的事情”。得到这样的反馈太棒了。我对我参加的所有演讲进行了评价，但现在我希望我也能对所有演讲写下评论，因为这是如此有价值的反馈。

这是我第一次在瑞典以外的地方演讲。我已经在二月份斯德哥尔摩的 Jfokus 会议和瑞典软件测试协会组织的 T2 本地测试会议上做了同样的演讲。在那之前，我已经在斯德哥尔摩的几个会议上发表过这个演讲的版本( [FooCafé](https://foocafe.org/stockholm/events/1757-learning-from-your-bugs) 、[斯德哥尔摩探索性测试会议](https://www.meetup.com/Stockholm-Exploratory-Testing-SET-Meetup/events/248500311/)和 [Sthlm Web Dev 会议](https://www.meetup.com/sthlm-web-dev-meetup/events/lngbfqyxmbjc/))。在集会上发言是在会议上发言的好练习。我在 Twitter 上发现了有关北欧测试日的信息，并决定提交一份提案。我已经提交了其他会议，但没有被接受。对于 Jfokus 和 Nordic 测试日，我花了相当多的时间来描述这个演讲。我查看了前几次会议的描述，以便对描述应该包含的内容有所了解。
