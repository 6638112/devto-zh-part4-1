# 克服自动化测试所需的加速延迟、成本和资源

> 原文：<https://dev.to/shrutiawake/overcoming-the-ramp-up-delay-cost-and-resources-required-to-automate-testing-18bg>

为自动化测试准备一个系统不仅仅是选择一个测试自动化工具，还包括设置几个有助于成功的测试自动化过程的元素和组件。

组成全功能测试自动化系统的各种组件是:

1.  正确的测试自动化框架，减少自动化工程师的脚本工作
2.  编写脚本的自动化测试工具
3.  一些机器来托管自动化测试用例——需要一台专用的机器来执行测试用例，这意味着额外的成本
4.  CI/CD 系统/集成，以确保测试用例在需要时被一致地执行，大部分是在检入之后
5.  收集关于测试覆盖(测试管理)和代码覆盖信息的工具，以确保所有编写的代码都被执行
6.  与协作工具、错误报告工具等的集成。
7.  测试数据管理，增加测试的覆盖面

测试自动化生态系统变得越来越复杂，涉及到与之相关的多个组件。尽管增量投资可以带来增量回报，但是测试自动化生态系统在早期阶段本身就是一项必要的投资。

将上述所有组件放在一起并确保它们按预期工作也不是一件容易的事情。自动化套件发挥作用之前涉及的各种步骤包括:

1.  决定自动化是否是一个可行的选择
2.  分析市场上可用的测试自动化工具
3.  确定自动化将涉及哪些资源
4.  列出测试矩阵
5.  测试用例创建
6.  测试结果的执行、报告和分析——与生态系统中的其他工具集成

## 1。决定测试自动化是否是一个可行的选择

如果您有非常频繁的发布并且测试资源较少，自动化是可以帮助节省手动测试时间的解决方案，同时每个测试运行的结果将在执行后通过电子邮件发送。但是，只有当以下条件成立时，自动化才会被证明是有益的:

有一大组回归测试用例需要非常频繁地运行，这样手动测试的成本就会超过自动化测试的成本
产品就是这样的，有多个模块不会经常改变。如果一直在产品中推进大量的变更，那么维护测试用例的成本将会比手工测试成本更高
大量的测试用例是可自动化的。如果产品/网站是 UI 密集型的，每个 UI 元素也应该考虑自动化。

## 2。分析市场上可用的工具

这是花费大部分时间的地方。在做出最终决定之前，最好使用多种工具进行概念验证。

一旦决定自动化是一个可行的选择，下一步就是分析市场上可用的自动化工具，并根据预算和产品的性质决定合适的工具。需要考虑的一个因素是产品采用何种技术构建，然后还应该考虑自动化工具的易用性。

拥有正确的自动化工具是最重要的，因为否则它会让你陷入许多问题，导致花费大量时间进行调试。查看这个指南来选择正确的测试自动化工具。

另一个不应该错过的因素是现有资源的技能，这将我们引向下一点。

## 3。确定测试自动化将涉及哪些资源

在决定了测试自动化工具之后，显而易见的步骤是决定如何进行自动化过程。

如果使用测试自动化工具需要特殊的技能和更多的资源，这将再次是一个耗时的任务，并且新雇佣的资源也意味着更多的成本。如果决定培训现有资源，培训旧资源需要一些时间和成本。

如果在这一点上，测试自动化工具需要对自动化测试人员进行最少的培训，那么这将是非常有益的。如果其他利益相关者也可以参与编写测试用例，这直接意味着时间和成本的大量减少。

这样的工具是那些允许无脚本测试自动化解决方案的工具，如**[【Testsigma】](https://testsigma.com)**高效，并允许使用 NLP 用简单的英语编写测试用例，并由人工智能维护。

此外，除了人力资源，还需要机器形式的基础设施来托管执行的测试用例。这又意味着成本增加。如果基础设施也可以作为服务使用，那么这一成本可以得到优化。像 AWS 和 Google 云平台这样的服务在市场上已经存在，它们可以托管你的测试用例，并以最低的成本执行它们。Saucelabs 和 *Testsigma* 提供的是测试自动化的特定服务，在云上托管测试用例，并提供执行和报告。

## 4。列出测试矩阵

这更像是为您的产品所支持的所有不同环境(操作系统、浏览器、设备)准备一份清单，必须通过自动化套件进行测试。如果您计划购买所有支持的设备，可能会非常昂贵。相反，订阅一个准备好所有设备进行测试和集成的服务在这里会被证明是有益的。同样，这种需求可以通过一些在线工具来满足，如[*【test sigma】*](https://testsigma.com)和 [*SauceLabs*](https://saucelabs.com/) 这些工具在多个设备上有各种浏览器可供测试，AWS Device farm 就是一个类似的产品。

## 5。测试用例创建

这是最耗时的任务。所花费的时间取决于测试用例的数量和复杂性。还取决于所选的自动化工具。如果选择一个易于使用的自动化工具，测试用例创建的复杂性可以降低。例如，如果选择无脚本自动化工具，这可以帮助节省大量的精力和时间。

## 6。测试结果的执行、报告和分析

如果自动化测试用例没有与 CI/CD 工具集成，并且在所有主要的检入之后运行，那么它们就没有意义。运行它们通常意味着更快的反馈。这可以在与 CI/CD、报告和分析工具集成时实现。

还提供与这些工具集成的云服务可以帮助节省大量时间和资源。

因此，尽管测试自动化可以被证明是一个主要的任务，如果正确的测试自动化工具和上面讨论的所有要素被选择的话，它可以避免花费大量的初始时间、成本和资源来开始自动化测试。
