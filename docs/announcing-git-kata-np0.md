# 宣布 Git 形

> 原文：<https://dev.to/slashgear_/announcing-git-kata-np0>

作为一名兼职计算机科学教师，我经常举办研讨会，学习计算机及其工具。有几次，我遇到一些团队已经使用 Git 和 Github 很多年了，但是遇到了很多麻烦。
基础是已知的，但在实践中。

> "等等，别动，我来做，你可以把它拿回来。"
> 
> "我一直在处理同一个文件。"
> 
> "把你的文件递给我，我会把它加到我的简历里."

我很确定你以前听过这句话！这听起来可能很有趣，但这是你在使用 Git 时缺乏实践和反应能力的表现。
用户通常知道如何管理，但做的事情却很晦涩难懂。

一个以前的同事更喜欢做几个`cherry-pick`来检索他的一个同事连续提交的工作，而不是使用一个`rebase`。

如果我设计一个研讨会，让团队在使用 Git 和 Github 时一起学习正确的反应，会怎么样？

我有机会参加了几次编码 Dojo/Kata 编程会议，我想从这个模型中得到启发。我设计的研讨会不是研究代码，而是关注 git 及其使用。

这里是研讨会的链接，可以在 Github 上免费获得。

| 持续时间 | 2 点 30 分 |
| --- | :-: |
| 所需供应品 | -(可选)故事立方体-视频投影仪-互联网 |
| 最大参与度 | 4 人团队，最多 12 人 |
| 所需技能 | 本次研讨会的目标是提高团队中 git 的使用，以限制 Git 可能遇到的日常问题(相互依赖，冲突，...) |

## 如果你是一名*形参与者*，请遵循以下说明

*   一个*开发团队*成员应该创建一个空的存储库并与其他成员共享 url
*   确保*每个参与者*都有可能提交和合并提交该存储库
*   存储库应该为空
