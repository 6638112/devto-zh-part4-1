# 选择最佳工具

> 原文:[https://dev.to/atila/choosing-the-best-tools-2j8i](https://dev.to/atila/choosing-the-best-tools-2j8i)

在 web 开发世界中，新工具和实践产生的速度可能有点令人生畏。如果你不完全接受所有的建议，你会发现自己转换工具和工作流的速度比交付项目的速度还要快。这肯定不好，尽管我们不应该害怕重构我们的代码，但在决定转换时我们仍然应该谨慎。

和我们大多数人一样，我也参与了讨论、<abbr title="Request for Comment">RFC</abbr>和其他决策过程。没有合适的轨道和明确的优先顺序，讨论往往会失控，我们会看不到真正重要的东西:交付最好的产品。在下面的文章中，我打算分享我是如何在提倡采用或放弃某个工具时引导自己的思维的。不管是在大团队，小团队，还是自言自语。

## [](#necessary-disclaimer)必要免责声明

下面的观点是我发现对我有用的，无论如何我建议任何人盲目地遵循它。如果有的话，你应该用你在你的项目/工作流程中的任何架构决策所需要的相同的批评观点来看待这些想法。

## [](#the-criteria)标准

作为一个心智模型，我倾向于将标准分为两类:

1.  当务之急:那些太重要而不能简单删除的，如果中途发生了什么，你会想要*前滚*，因为回滚会带来相当大的麻烦。例子。:React，Webpack，Babel。

2.  可撤销的:易于移除的工具。有时，我们认为有些事情超出了回滚的范围，因此能够用类似的工具替换工具将是决策过程的一个重要方面。例子:灯塔，斧头，时刻。

从这些例子来看，标准根据案例和要聚合的特性改变等级是非常合理的。我将在这里分享我自己的标准，但是请记住，即使是我最终也会改变列表中的顺序，以适应特定的用例。

### [](#1-reliability)1。可靠性

无论是转换工具、范式，还是完全改变方法，作为一名开发人员，关注新的方向会给你带来多大的轻松是非常重要的。你能相信这种新方法是可靠的吗？它至少能完成与前一种方法相同的任务。

如果添加一个完整的新功能，通常可靠性会和产品的硬需求联系起来。你通常可以从 <abbr title="minimum viable product">MVP</abbr> 需求或者描述你产品的基本功能中撇去这些。

### [](#2-user-experience)2。用户体验

在理想情况下，对应用程序的更改只会对 UX 产生积极的影响。然而，有时我们会面临一些情况，这些情况会影响用户的整体体验。这种情况可能来自法律要求，也可能直接来自客户，或者来自你的产品的一些固有的商业规则。

无论如何，我们的工作是确保*如果*变化的影响是负面的，减弱并考虑它是一个非常大的权衡。最终，没有什么比良好的体验更能留住用户/客户，从消费者的角度来看，“小问题”是如何成为交易破坏者的，这一点令人印象深刻。

### [](#3-usability)3。可用性

可用性是一个相当宽泛的术语，有人可能会说它与上面提到的**用户体验**有很多重叠。出于本文的目的，我决定将以下标准纳入可用性范畴:

*   使用直观性:在没有任何人事先了解或学习的情况下，你的特性使用起来有多容易

*   <abbr title="accessibility">A11Y</abbr> :它如何影响您的应用程序对辅助技术的支持。

*   用户故事:它如何积极地影响你的应用程序的整个用户旅程。

### [](#4-perceived-performance)4。感知性能

你的应用速度最终取决于你的受众如何看待它。感知性能不关心如何、做什么，甚至不关心是否优化了代码和连接。这都是关于用户将如何注意到。

换句话说:

> 在计算机工程中，感知性能指的是软件功能执行任务的速度。这个概念主要适用于用户接受方面。[在维基百科上的感知表现](https://en.wikipedia.org/wiki/Perceived_performance)

显然，没有必要指出这个基准对你的产品成功的重要性，因此你应该保护它免受任何会对它产生负面影响的变化。

### [](#5-unplugability)5。(不)可插拔性

可拔性是任何集成的一个重要方面，尤其是对第三方工具而言。整个行业发展速度惊人，一方面，尝试和学习的新选择多得惊人；但另一方面，更好的实践也在不断涌现。

在采用时，需要考虑从一个工具继续前进的能力，没有人希望被一段第三方遗留代码卡住，因为它太难摆脱了。

在其他情况下，一个人可能只是为场景的改变而后悔被收养。出现了所选工具不支持的新功能请求，工具的特定方面出现了问题，需要进行切换，等等。

你需要随时准备好与你无法控制的代码片段分道扬镳。

### [](#6-raw-performance)6。原始性能

即使最终用户无法察觉，这种变化是否会影响性能的其他方面？例如:

*   更少的请求或幕后处理
*   较少的外部依赖性
*   更快的处理

### [](#7-developer-experience)7。开发者体验

这种改变让开发人员的任务变得更愉快了吗？这个标准通常是一个很好的*平局决胜点*,每当其他权衡在竞争方法之间达到平衡时。大多数时候，开发人员的经验肯定会使事情向某个方向倾斜。
