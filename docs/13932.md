# 反应式编程不是一种趋势:为什么现在是采用的时候了

> 原文:[https://dev . to/this dot media/reactive-programming-is-a-trend-why-the-time-to-adopt-is-now-31e 8](https://dev.to/thisdotmedia/reactive-programming-is-not-a-trend-why-the-time-to-adopt-is-now-31e8)

2017 年是 JavaScript 领域革命性的一年，原因有很多。随着去年 Angular 2 的最终发布、React Fiber 的发布、RxJS 的最终发布以及 Vue.js 在 JavaScript 狂热分子中的大规模采用，行业朝着更好的现代 web 发展的步伐正在迅速加快。

趋势来来去去，框架也是如此，但有一种趋势似乎会一直存在。这种趋势就是反应式编程。

反应式编程并不新鲜。像 RxJS 这样的库从 2009 年就已经存在了，但在很大程度上，使用仅限于一小部分反应式和函数式程序员和学者。直到最近，由于糟糕的文档和缺乏广泛的知识，开发人员还没有办法理解它是什么。这个概念直到前几年才被前端框架广泛接受。

由于框架的采用，反应式编程、Observables 和 RxJS 都是我们在互联网上听到的时髦词汇。如今，它已迅速成为“最酷的事情”。但是，行业影响者采用反应式编程是有原因的，并不是因为它是“要做的事情”。采用和投资于反应式编程有很强的技术和商业理由。

**反应式编程—大图**

想象一下，如果你能让计算机科学中的任何东西与任何东西对话。使用反应式编程，您可以。你知道计算机科学中的任何东西都可以被观察到吗？这意味着我们已经有了一种通用的高级语言，它将计算机科学中存在的所有东西结合在一起。

**Rx 的实际使用案例**

联合所有计算机科学的梦想可能看起来有点夸张，但它离事实不远了。能够表示任何东西的设置和拆除、任何时间内的任何数量的值、动画、通过网络到达的消息、视图的整个生命周期，再加上您可以随时观察任何东西，这使得它们非常强大。更重要的是，因为所有的可观察物体都有相同的形状，而且几乎可以代表任何东西，所以把东西组合在一起，让它们互相交流和通知就变得很简单了。

反应式编程是一种更有效的编码方式，已经被行业领导者采纳为开发代码简洁的应用程序的新标准。这是一个真实的、有形的方法，它有很好的工具，可以在组织内部实现更好的开发标准。

一种跨越所有框架的语言

我们最终可以超越仅仅用反应式编程来增加数字。它是 JavaScript 之上的一种声明性语言，包含更高级的功能，这意味着您可以说“做这些事情”，并在一行代码中完成。能够在跨所有框架的语言之上进行交流是非常强大的。它对事件的作用就像 HTML 对布局的作用一样。想想添加一个 div 有多容易。在 Rx 中，您可以简单地添加一个操作符，行为就会改变，而无需修改任何一行代码。

**快速采用 JavaScript**

很明显，现在是采用反应式编程的时候了，这有两个主要原因:TC39 Observables 提案和 Angular (2.x)等框架通过 RxJS 等库采用反应式编程的确认。下面是 RxJS 2016 年每天 npm 下载量的图解。

随着 Observable 逐渐成为浏览器的原生类型，Observable 类型很快将被用于建模基于推送的数据源，比如 DOM 事件、定时器间隔和套接字。您可以遵循规范并在 [github](https://github.com/tc39/proposal-observable) 上找到更多关于特定用例及实现的细节。

尽管现代 web 框架已经采用了各种不完整形式的反应式编程，Angular (2)是唯一真正采用反应式编程的框架，它使用 Rx 作为异步原语。

支持取消(不同于承诺)的可观察性是单页面应用程序采用 Rx 的主要原因之一。Observables 的形状也允许你支持任何类型的异步，包括 Ajax 动画、websockets 和用户事件。

**更快地交付复杂功能**

像 RxJS 这样的库有助于开发人员用更少、更易维护的代码更快地交付复杂的特性。因为它是声明性的，所以它使您能够更快速地添加特性和透视。如果你想添加一个特性，你可以用几行代码来添加，并用一种与框架无关的、特定于领域的语言来编写。

了解 Rx 的开发团队拥有在应用程序中用几行简单的代码实现复杂功能所需的工具，例如多路网络套接字、拖放、自动完成、文本框、客户端速率限制和逐步向导模式。

**一个接口管理所有 API**

随着 web 开发由来自各种 API 的事件驱动，连接这些 API 的代码会变得冗长而复杂。但是，通过使用 RxJS 将您的代码包装在一个可观察对象中，您能够创建一个所有 API 都可以理解的接口，实现更快的 web 开发、更简单的组合，并大大减少将一种类型的数据适应另一种 API 所期望的时间。

**今天使用 Rx**

虽然 Rx 看起来很棒，但是很容易制造借口来解释为什么现在不是采用反应式编程的时候，比如代码重构可能需要做的大量工作。Rx 的伟大之处在于，你可以开始从面向 Rx 的重构中受益。Rx 的大多数实现都是编写几行代码来与现有代码进行交互。

**无功编程类型**

大多数反应式编程是基于推的。基于推送的接收对于前端 web 开发来说当然更理想，因为大多数交互都是推给你的事件，比如 DOM 事件和鼠标点击。然而，某些框架，比如 Ember，已经采用了基于拉的反应式编程。

对于基于拉取的接收，我们告诉系统给我们一个值，当需要时，它拉取值。基于拉的反应式编程在需要控制背压的情况下非常有用，在繁忙的系统中，您可能希望在获取信息时进行控制。您将得到一个值已经改变的通知，但是直到请求之后您才被告知值是什么。

相比之下，基于推送的接收方式通过一系列步骤推送值，并在每个时间间隔给出答案。基于推的 Rx 尽可能快地抽取值，这使得每个值的性能更高，但代价是无法控制何时获得值。

**投资于反应式编程**

反应式编程通过为 web 开发人员提供轻松完成复杂任务的能力，使他们能够更快地开发功能。越多的人理解反应式编程，我们作为一个现代网络将会变得更有生产力。采用的唯一障碍是不理解范式和它周围的语言。

通过为网飞、Slack、微软、脸书等公司的产品提供动力，反应式编程已经让我们的生活变得更加美好。
