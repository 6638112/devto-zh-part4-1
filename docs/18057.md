# 最小可行产品(MVP)

> 原文：<https://dev.to/al_karlssen/magento-or-woocommerce-fma>

一种特殊形式的启动精益方法被称为“MVP -最小可行产品”。MVP 应该帮助创始人、产品经理和成长型黑客，不要开发事后看来没人需要的产品。

# 如何培养一个真正的 MVP

当然，同样是尽早进入市场，目的是尽快获得真实的市场反馈。然而，作为精益方法论的[延伸，MVP 不仅](https://startupsstartup.wordpress.com/2018/10/03/lean-startup-startup-methodology/)[验证了初创企业](https://itechcraft.com/solutions/for-startups/)
，而且直接是产品的工作迷你版本。如果做得好，那么产品就可以根据用户的反馈进行扩展，不断增加新的功能和特性。
这样可以显著缩短初始开发时间。然而，分析工作增加了。我的产品有市场吗？客户真正需要的功能是什么？MVP 之后哪些特性最重要？几个 MVP 的例子。

产品开发人员面临的最大挑战是提前准确定义哪些功能是用户真正需要的，以产生足够的收益。收益不能太低，否则用户可能会立即失去，尤其是永远失去。

产品团队经常讨论的 MVP 特性的几个[例子。在第一个 MVP 中经常被忽略的特性，因为它们不会特别影响用户体验:](https://startupsstartup.wordpress.com/2018/09/12/how-much-does-mvp-cost-in-web-development/)

*   "重设密码"
*   支付功能
*   合法安全的数据保护和 GTC 文本
*   源代码的自动测试
*   高度可扩展的暂存环境
*   没有“好”的设计
*   装载时间长

根据我自己的经验，应该包含在 MVP 中的功能:

*   非常好和容易使用
*   提交用户反馈的能力
*   特征

"如果你对你产品的第一个版本不感到尴尬，那你推出得太晚了."里德·霍夫曼 Linkedin 的联合创始人

紧张的时间表经常强调一个好的 MVP 的实现，所以后来用户得到的是一个还没有完成的版本，而不是一个非常好的工作版本 1.0。产品可以做不完，但是用户千万不能注意到这一点，尤其不能在用户体验上。请不要忘记，特别是应用程序的加载时间属于用户体验。

这种现象可能更经常发生，尤其是在拥有多种产品的公司中。假设:市场发展迅速，竞争呈现良好态势。因此，产品开发人员必须跟进，尽快在竞争中领先。MVP 以创纪录的速度被定义、实现并投放市场。销售部门得到了祝贺，并被送上了销售 MVP 的征途。产品开发人员的工作似乎已经完成，你可以直接开始下一个项目，甚至下一个 MVP。从销售人员的角度来看，一切都是最佳的。他们得到了他们的特色，并再次赢得了销售。目前为止一切都好吗？不。从中长期来看，这种发展对一家公司来说是致命的，因为它完全忽略了客户对 MVP 的确认。对于一个公司来说，最糟糕的情况是，它有很多功能和产品只处于产品组合中的 MVP 地位，但没有一个真正打动用户。从长远来看，用户不会对“未完成”或“不成熟”的产品感到满意。至少在竞争对手只有一款产品上市的情况下不会，比这款产品的 MVP 版强太多了。
由于这个原因，就有了今天的 MLP 理论——即最小可爱的产品。当然，这不仅仅是市场上的一个小原型，它不会被进一步开发，而是用户已经喜欢甚至可能有点喜欢这个产品的核心需求。
