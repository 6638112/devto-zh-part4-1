# 框架、库和语言

> 原文：<https://dev.to/gtanyware/frameworks-libraries-and-languages-21n2>

本文对用于构建现代 web 应用程序的代码组件进行了分类，将它们分为框架、库和语言。我将描述其中的每一个，并提供一个关于为什么我们构建项目的方式经常导致事情不如预期的那样的观点。

框架是一种软件，类似于没有窗户、门或家具的新房子。结构都在那里，但你必须自己提供所有的细节；所有组成一个家的部分。很容易找到一个软件框架；它遵循好莱坞的原则“不要打电话给我们；我们会打电话给你”。框架在很大程度上处于控制之中；你不是。

图书馆则完全相反；它是由您的应用程序直接调用的代码，提供应用程序所需的“黑盒”功能。这里由你掌控。库是任何软件系统的重要组成部分，库的价值通常与定义它的 API 密切相关。

一种语言不是以上两种。这是一组约定好的单词和语法，用它们来表达意思。从结构上来说，这是一个完整的房子，有窗户、门、家具、摆放好的桌子和烤箱里的鸡肉砂锅。从某种有限的意义上来说，语法规则相当于一种框架，但它是一种让你处于控制之中的框架，而在英语等人类语言中，“库”的概念，在实现附加功能的意义上，是没有意义的。如果我们遇到一个需要详细描述的概念，我们把它封装在一个新的词汇后面，用它自己的语法，符合语言的一般规则，然后这就成为语言的一个无缝的部分。例如“激光”和“智能手机”。任何能让我们理解这些术语的东西都可以被视为一个隐式库函数。当 JavaScript 发布新版本时，就会发生这种情况；新的单词和语法规则与以前使用的一起出现。

现在来看看这三个类别的一些例子。

Angular 是一个框架，并自豪地宣称自己是这样的。虽然我没有使用它的经验，但我相信很少有人会把它当成别的东西。为了创造一个可行的角形产品，结构必须预先确定，然后一切都要符合该结构。这对于一些人来说过于死板，但对于另一些人来说却是一个优越结果的保证。

React 也是一个框架，但是——据我所知——也是一个库。与 Angular 相比，这个框架没有那么严格，对于任何在命令式编码环境中长大的人来说，它可能更容易学习。React 由脸书开发并为其服务；一些人认为它对任何事情都有好处，而另一些人认为它对小项目来说是多余的。

JQuery 是一个库，但是带有一些语言特性的提示。例如，选择器没有提供新的功能，只是提供了一种更简单的方式来表达 JavaScript 中已经完成的工作。JQuery 快速而简单，但是紧凑、有效且易于学习，并且不自命不凡。

JavaScript 是一种语言，就像人类语言一样，是不断变化和发展的。在早期，它缺乏结构化的特性，这为我们今天看到的框架和库的发展提供了最初的动力。有些人认为，最近对语言的改进，比如类的增加，使得对框架的需求不那么迫切了，现在大多数事情都可以在不需要框架的情况下完成。

那么，这三个概念与客户要求的产品类型之间的匹配是什么？

客户端编程的范围从添加到静态网页的非常简单的交互性到巨大的商业、工业或社交媒体网站。假设同样的工具在所有情况下都同样适用是相当愚蠢的，尽管看看潜在雇主所要求的技能，这往往是一种假设。我把这大部分归因于无知和群体本能。

和房子一样，改变一个大型软件系统的结构是一件昂贵的事情，所以框架在结构不太可能改变的地方工作得最好。矛盾的是，这使得库组件成为部署框架以获得最大效果的理想场所。一旦编写完成，组件的结构将很少改变，如果它是一个流行的库，它将有一个开发团队致力于它的长期维护。最好的例子之一就是谷歌地图。很少有用户对这个神奇的图书馆有丝毫兴趣，想知道里面发生了什么；他们所关心的只是它呈现给世界的 API。对于谷歌来说，不保持支持和继续开发地图所需的专业知识是不可想象的。

除了相对较少的例外，网站本身不会长久保持不变。当变化来临时，它通常是结构性的，往往不是由“看起来不同”的愿望驱动的，这就产生了一个问题。简而言之，软件工具旨在帮助开发人员变得更有效率。所有的重点都放在*创造*一个产品上，假设一旦开发和部署，同一个团队将继续提供维护。但这只适用于那些本身就是软件公司的公司。对于其他人来说，开发团队，无论是内部的还是签约的，都将继续前进。当你只是偶尔为他们工作时，很难证明维持昂贵的开发人员是合理的。

正如我上面所说的，结构性改变是昂贵的。在你开始之前，你必须让人们了解系统目前是如何工作的，以便让它做一些不同的事情。即使是最好的程序员也可能需要几个月的时间才能完全掌握一个复杂的系统，而在大多数情况下，我们没有几个月的时间。结果是必然的；变更是由缺乏对系统完全理解的人做出的，导致代码困惑未来的维护者，并且在下一次需要变更时更难处理。以此类推，一个膨胀、衰落和“千刀万剐”的螺旋，直到维护成本超过放弃产品重新开始的成本。通常在产品的原始预计寿命到来之前几年。

我不止一次亲眼见过这种情况。在一个案例中，我加入了一家大型连锁药店，它的主要软件产品可能只有 5 年历史，并且正在迅速变成一堆难以控制的意大利面条。只有 3 个人真正理解它；一个在我加入一年后离开了，另外两个是承包商。随着所有权的变更，该公司随后决定削减后者，因为成本的原因，所以那些留下来的人从来没有完全了解该系统是如何工作的，也没有人留下来问。自然没有软件文档——我们被期望成为能够“读懂代码”的专家。当然，我们不能；很少有人能像普通人一样阅读和理解法律文件，不管是用英语还是外语写的。最终，该产品被抛弃，取而代之的是标准的现成系统；价格昂贵，功能较少，但没有维护问题。

无论这个行业在创建新的结构实体方面多么努力，其结果永远不会被大多数负责维护产品的人所理解。它们实在是太复杂了，而且用来构建它们的工具过时得太快了。只要我们继续保持同样的心态，项目将继续在早期失败。

这让我想到了一个诊断。问题不在于结构；这是一种语言。我们说英语；电脑会说 JavaScript。我们人类的需求被翻译成计算机代码，在这个过程中，误解必然会发生。这在很大程度上是一个没有反馈回路的开放式过程。当我们想修改我们不完全熟悉的代码时，我们必须将 JavaScript 翻译成英语，这样我们就可以与我们的客户讨论，然后将同意的更改翻译回 JavaScript。这些翻译都不完美，会产生错误。有缺陷的系统不仅仅是不幸的；它们不可避免。

这个问题的唯一解决方案是让人类和计算机说同一种语言。由于人类永远不会说 JavaScript，这意味着计算机将不得不说英语，或者至少是某种形式的英语，作为一个共同的基础。在你认为这不切实际之前，请考虑:它已经被大规模成功地完成了。如果您是数据库专家(不一定是程序员)，您将熟悉 SQL，并且能够阅读甚至构建您想要实现的非常详细的描述。好消息是，电脑会理解你。总的错误率将会很低，因为所有相关方——客户、数据库专家、程序员和计算机——都使用同一种语言。我希望类似的说法也适用于金融业，在金融业，包含 Excel 宏的电子表格通常会绑定客户、金融专家、程序员和计算机。

这就是我呼吁为浏览器开发一种工业级语言的基础，这种语言使用类似英语的脚本来传达意思，并保持从网站所有者到运行它的浏览器的每个人的可读性。因为我相信实践我所宣扬的东西，所以我构建了自己的编译器，我用它得到了非常好的结果，但我不是那个领域的专家，也不属于当前开发框架(如 Angular 和 React)的人的智力水平。我很难想象，如果花那么多精力去设计和实现一种英语形式，让各种能力的人都能真正“编写”网站和网络应用，并持续运行十年甚至更长时间，会有什么样的力量和美丽，但我渴望有一天这样的系统会出现。
