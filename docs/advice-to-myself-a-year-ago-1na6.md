# 一年前给自己的建议

> 原文：<https://dev.to/kyam/advice-to-myself-a-year-ago-1na6>

嘿，是我- **未来一年的你**！我知道，你刚刚结束了你的编码训练营，以前你的编程经验并不比让你的笔记本电脑运行 Python 中的“Hello，world”重要多少，但是很快你将接受你作为程序员的第一份工作——并且你会爱上它的！你会做得很好，但你本可以在*做得更好*，所以我想我应该把后知之明的礼物传递给你，帮助你充分利用你在这个行业的第一年:

*   **陷入**——你不会在一家全新的创业公司找到工作，所以你必须掌握一些☠️ *遗留代码* ☠️.在记住你已经发现了什么和你正在寻找什么(以及你为什么要开始寻找)的同时，你可能很难找到自己的方向...).看着几百行代码*居然*让人望而生畏，而*完全*自然让人不知所措。克服这一点的唯一方法是陷进去——开始涉水前进(在你可以的地方，靠你自己),尽你最大的努力，你会习惯代码和它是如何被编写的——你很快就会意识到你知道的比你想象的要多。

*   **变得无聊**——在这里[阅读我的博客](https://dev.to/kyam/the-importance-of-being-bored-3fb4)关于为什么我认为允许你的思想开小差是重要的。TLDR 是:*试着让你的大脑恢复，每天给它一些时间，让你的大脑平静下来，不要被播客、twitter 或 reddit 弄得饱和。* ***无聊了，让自己放松一下。它对我的精神状态产生了奇迹。***

*   **从你的同事那里获得反馈**——你将加入一个团队，在那里你的同事比你有更多*的经验*。你应该充分利用这个机会，利用他们的知识，而不是为此感到沮丧或气馁。尝试通过结对编程向他们展示你自己的技能，通过定期询问来询问他们对你的方法的一些反馈，并且(比*任何事情*更重要的是)让他们审查你的代码。让你的代码被审查不仅给你信心，你不会破坏产品，它也让你知道你的不足在哪里，并且给你一个机会去知道你的同事如何以不同于你自己的方式处理问题。最初可能很容易被冒犯，或者将代码审查评论视为对您所创建的东西的批评，但是重要的是不要将代码视为您的孩子，而是视为您的整个团队正在共同努力创建和完善的东西。

*   **让自己休息一下** -编程很难，没有人知道所有的事情。尽早承认这一点真的很重要，因为你将与比你知道得多的人一起工作，这可能会让你感到自卑或你实际上不够好(也称为冒名顶替综合症——我不打算尝试进入这种情况，但[的这些](https://twitter.com/KyamLeigh/status/1096866850414170112)博客帖子可能会让你感觉好一点)。请记住，你是*对这一点全新的*，仍在学习，在过去的某个时刻，你团队中那些真正有经验的人也曾在这个位置上，他们做到了——你也可以。

*   **看起来很蠢**——所以，你讨厌问那些你知道其他人都已经知道答案的问题，你也不想说出来浪费大家的时间(或者看起来很蠢)。问题是，没人会记得你问了那个问题，而且可能会因为你有勇气说出来而更加尊重你。房间里的其他人也很可能不知道，你在帮助他们，也帮助你自己。所以，问一问，填补你的知识空白，并潜在地在不知不觉中帮助那些害怕说出来的人。如果我还是没有说服你，那就一整天都记下你不明白的地方，有机会的话就好好阅读一下。

*   不要推迟开始那个项目 -想出你想开始的酷项目真的很容易，但不能摆脱“我会等到我更了解*(在此插入技术)*”的想法。等到你更了解*那项技术*的时候，你也会想要完善你的*其他技术技能*，但你永远也不会达到那一步。就一头扎进去，开始做点什么；你可能会意识到，无论如何，你需要了解更多关于另一项技术的知识——但至少你已经开始做一些事情，并且知道你需要把精力和时间集中在哪里。

*   计划你的学习时间(T1)——我希望我早点做的一件事是计划我要学什么。我真的很幸运，我的工作场所每个季度都提供特定的培训时间，我现在做的一件事就是提前计划好我一年(或更长时间)要用什么。这意味着我必须提前一年知道我想去哪里，我想知道什么，也意味着我不会在学习可能对我的长期目标没有实际帮助的技术之间跳来跳去。这也让我可以计划任何我想做的业余阅读来支持这些季度会议。也不要陷入教程陷阱，在那里你所做的只是学习一些闪亮的新东西的基础知识——将这些技能付诸实践，然后**自己动手制作。**

所以，享受接下来的一年吧——这并不容易，但你将会学到比离开学校后任何一年都要多的东西，并且在这一路上会有一段不可思议的时光——充分利用它吧！
