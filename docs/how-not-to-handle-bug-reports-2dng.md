# 如何不处理错误报告

> 原文：<https://dev.to/codemouse92/how-not-to-handle-bug-reports-2dng>

本周，我在我最喜欢的电子邮件客户端中遇到了一个令人讨厌的错误，这恰好是一个相当受欢迎的项目，来自最古老和最大的开源开发社区之一。

事实上，我已经面对它几个月了，只是接受它作为新的标准，直到我想起“嘿，这是开源的！我可以提交一份错误报告。”

因此，像任何开源软件的忠实用户一样，我找到了该项目的官方公开问题跟踪器，并提交了一份详细的错误报告，包括环境规格和重现步骤。

对我的错误报告的回应以其唐突的无礼令我震惊，暗示不知何故问题要么是我自己的问题，要么是我明显怪异、非标准的操作系统(因为，说真的，什么样的疯子会用默认的包和设置运行 Ubuntu MATE？)问题不可能是他们的软件。

虽然最初我被这种断然拒绝弄得不知所措，很受伤，但现在我明白了，我的整个职业生涯都在做错误的软件。明明重点是疏远用户，得罪人！想想我因试图为*人*写项目而错过的所有乏味的自我建设机会。

受到这些勇敢的先驱者和许多像他们一样的人的启发，我收集了一些关于如何运行问题跟踪器的技巧，这样你就可以赶走所有讨厌的用户和贡献者，保留你的核心开发人员已经形成的精心策划的小团体。当然，用户的缺乏意味着你的软件在计算机历史上是一个相当无用的亮点，更不用说在任何地方都是免费和开源软件声誉上的一个污点，但对于那种比别人聪明的短暂感觉来说，这是一个很小的代价。

当然，如果你是那种怪人(...实际上像我一样...)谁真正想*建立*你的项目社区，并鼓励你的用户积极参与你的软件开发，你会想**颠倒**所有这些方便的提示。(但是你为什么想要这样呢？)

**注:通过一个讽刺的镜头阅读下面我的“建议”。**

## 分散问题跟踪

[![](img/f1ecf0ea49c1cd16e586c377cebb8c2b.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--kBdDpo2A--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/77b1qaatsqtswlbg3s1b.gif)

当然，如果你能在第一时间阻止错误被报告，那就更好了。最快的方法是拥有多个追踪器，每个追踪器都有一个特殊的、专用的目的，然后隐藏关于它的面向用户的文档(或者，你知道，根本不要写任何文档。)

想象一下，当你的用户试图找出他们应该在哪个网站上发帖的时候...

*   开发人员实际出现的空闲频道，
*   话语论坛有一个专门的`Bugs`部分，你的团队通常会忽略这个部分，
*   GitHub 发布了没人用的追踪器，你忘记关了，
*   很少有人意识到它的存在，
*   十年前的螳螂虫追踪器，你从没撤下过它的网站链接，
*   你去年安装的闪亮的新 bug 追踪器，但是上面只列出了几十个 bug。

自然地，如果他们猜错了，你和你的开发者应该居高临下地告诉他们，他们发布了错误的追踪器。也许你可以告诉他们去使用一个不同的(给他们多重定向的皇家借口加分)，或者只是简单地说“你应该知道这不是一个张贴错误的好地方。”

最重要的是，*永远不要*帮助他们将错误报告转移到正确的平台上！转介到另一个平台应该是模糊的，最好是贬低的，你不应该举手之劳迁移任何信息。他们打了一次，他们可以再打一次。再一次。

或者他们可以走开，这才是重点，不是吗？

## 许多铁环

[![](img/160d85b0863ca46ec90d2640db89a2a2.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--8YByw9R9--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/gy1lj880qahkeh90krm7.gif)

即使用户设法找到了正确的问题跟踪器，仍然有很多机会关闭它们。

许多精英项目最喜欢的是为一个好的 bug 报告创建一个长长的需求列表...然后惩罚违规者，因为最轻微的遗漏就立即驳回整个报告。

现在，即使那些令人作呕的面向社区的项目也有一些写好 bug 报告的指导方针，但是他们花时间用模板或其他提示来概述他们的期望。当用户没有击中所有目标时，他们会礼貌而温和地询问更多信息。自然地，他们将这样的要求保持在最低限度。他们要求这样的事情...

*   软件版本和环境详细信息。
*   期待什么。
*   发生了什么，还有
*   繁殖的步骤。

这些都很合理，因此不足以让你的用户产生恐惧和厌恶。您可以通过期望他们准确猜测您想要哪个库的版本来混淆这个过程，以包括大多数用户无法理解的高度技术性的细节(例如，让他们猜测用于找到他们的网卡固件版本的命令！)，或者具有其他不相关的信息(例如，编译器版本，当对发布的二进制文件进行错误报告时)。

您可以让 bug 报告者执行许多其他“事实调查”任务，这些任务远远超出了他们的技术能力，从而使这个过程变得更加痛苦。他们的错误报告被标记为“已关闭，信息不足”的威胁将促使他们尝试他们几乎没有希望成功的事情，比如要求非开发人员执行全面的`git bisect`来确定是哪个提交号创建了错误。

当然，尽管任何善意的项目可能会询问用户是否愿意在大量指导下尝试这些任务，但你的项目可以利用这一点作为疏远技术，只需期望用户已经熟悉这些任务，迫使他们承认他们不知道如何做，然后贬低他们。

## 在我的机器上能用！

[![](img/66dbecebe60958de5feca16dda0abbc6.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--L_5HTgi4--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/e8d9rk7pdq1s70tze1gd.gif)

让我们假设您的用户完美地描述了他们的环境，甚至包括您想要的库版本。现在怎么办？

自然，我们不想浪费一个绝佳的机会贬低我们的 bug 报告者！他们刚刚给了我们一堆弹药——我们现在可以消极地、积极地嘲笑他们使用了一个我们并不亲自花时间使用的系统。我们可以指责他们篡改库，安装奇怪的东西，以奇怪的方式改变设置，或者只是太愚蠢而不更新东西。

最重要的是，确保你*不要*试图在描述的环境中复制 bug！这可能会给错误报告带来合法性，并让用户感觉他们的努力得到了认可。相反，你可以通过尝试在不同的环境中复制它，或者甚至在一个与*相似*但缺乏一些简单关键共性的环境中复制它，来无休止地激怒他们。

显然，复制每个系统是不可能的，但是大多数用户不会期望你在一个 FreeBSD 实例上用一个定制的内核映像和一堆从不同的 Git 库构建的包来测试你的软件。所以，如果这是他们的环境，在这一点上你不会有太多的机会疏远他们；他们可能要对许多东西在他们的电脑上无法正常工作的事实负责。如果他们不愿意，即使是最心软的开发者也会认为他们不合理。

但是对于大多数使用主流的、受支持的、带有默认软件包和设置的操作系统的用户来说--------------------------------------------------------尤其是-----------------------------------------------------------------------------------------《10 分钟》你实际上是在说:“我懒得在虚拟机中加载实时映像并安装我们的软件。你的经历对我来说并不重要。”

现在，很明显，你有有限的时间，但是实际上没有什么*阻止你为调查 bug 设置一个低优先级的任务。你的用户可能知道这一点，所以不要含糊其辞。列举他们有多不讲道理(即使他们没有)。让他们感觉像昆虫一样，即使假设你的团队中有人会屈尊复制他们的环境。*

当然，如果您将 bug 公开太久，其他一些 Joe Schmoe 可能会出现并在最初描述的环境中确认它...或者更糟，在完全不同的环境中！因此，请确保尽快关闭“无法复制”或“WONTFIX”错误。这向世界其他地方传递了一个明确的信息，那就是你不在乎这个 bug 是不是真的；你只是不想解决它。

## 责怪别人

[![](img/be9bc05baeaa593defc2f7a2e51ad07b.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--ZPwRScdN--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/f85s2ko2s267qe055aot.gif)

当然，这里还有另一个问题:**由于环境而改变行为的 bug 的历史是巨大的，并且有很好的记录**。当然，你知道报告的错误可能*存在于你的代码中，它可能是某种阴险的未定义行为，某一天会以一种令人讨厌的方式爆发，但是不要让对好代码的渴望导致你错过一个摧毁另一个人或团体的绝佳机会！*

 *我对我自己的错误报告得到的部分回应印象特别深刻:

> 无论 MATE 或 Ubuntu 可能发布什么有趣的自定义输入法更改，这并不一定意味着[我们的代码]中存在 bug。

熟悉 Ubuntu MATE 的人都知道这是一个荒谬的说法，但这不是重点。你想不惜任何代价将这个错误归咎于其他人。通过将责任从你的项目中转移出去，你为你的团队节省了不得不承认错误的努力。它甚至可能有助于防止未来的错误报告！

当然，下一步是让用户去*其他项目*那里报告错误，在那里他们肯定会被告知“不是我们”。太棒了。你创造了一个皇家回避的情况，让你的用户感到绝望。任务完成。

## 责备用户

[![](img/e01094f33fe9b3aed72feb188780540e.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--3NriUF51--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/ilkic9jy08gwv3bnou50.gif)

也许“责怪别人”最典型的形式是你可以把责任转移到用户身上。这里有一些方法可以考虑:

*   “你应该知道标有[X]的按钮不做 X！”
*   "每个人都知道你应该做 A-B-C . "
*   “你知道怎么用电脑吗？”
*   "你对你的系统做了什么奇怪的黑客攻击？"
*   " RTFM n00b "

很自然，你会想让你的回答比例子更加消极主动，但是你明白了。只要你能暗示用户是 bug 的罪魁祸首，你就不必花力气去修改代码，澄清用户界面，或者改进文档。

## WONTFIX

[![](img/a8e514a37ae88c3e3d9ffe81a9531adb.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--M_N5coyb--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/zuj61pykw93jmgn7uavt.gif)

这个标签是你所有武器中最强大的一个，正是因为它的内涵。不像`Invalid`、`Expected Behavior`或`Cannot Reproduce`这样的短语，可怕的`WONTFIX`暗示着“我们知道这是个问题，而*我们不在乎*！”

一些软的、面向社区的开发人员可能会鼓励你把这个标签从你的问题跟踪器中完全去掉，用更具体、更合理的短语来代替它。

但是如果你想用你的问题跟踪器疏远用户和贡献者，你应该*绝对*保留它，并且自由地使用它！

顺便说一句，不要忘记当你遇到`Expected Behavior`的情况时，你的用户仍然会希望你浪费时间写更好的文档或者改进用户界面以防止混乱。不要落入他们的圈套。只是在这个问题上打一个`WONTFIX`然后继续前进。创造以人为本的软件体验只会吸引用户和贡献者，而这正是你想要避免的。

## 打开源，关闭门

[![](img/4ed9493d5b8a338eb2e895b16e0c3b11.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--cnlMINer--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/t5e8nwync70ibccvaqyk.gif)

仅仅因为你的代码是开源的并不意味着你需要对你的开发过程透明。你需要确保你的团队不会在公开的 bug 追踪器上报告他们发现的 bug 和问题；如果他们这样做，你会有三个问题:

1.  你的团队将会承认他们并不完美，这将会邀请更多想要帮助改进软件的贡献者，

2.  将现实世界中观察到的错误与您的开发人员已经报告的错误联系起来是很容易的，这使得关闭外部错误报告变得更加困难，

3.  用户会得到一个荒谬的想法，认为你会认真对待他们的反馈，或者你甚至希望他们在那里。

通过把你的内部问题跟踪器藏在远离你的用户和开源贡献者的窥探的地方，你创造了可信的抵赖，并且总体上散发出一种作为精英的、排外的俱乐部的感觉。你现在不觉得自己很特别吗？

不要将这与简单地为外部和内部错误提供不同的接口相混淆。一些做好事的项目实现了这种方法，以便更好地审查外部错误报告；当以某种方式完成时，它会导致项目开发人员更认真地对待用户！如果你真的想将这种分离武器化，你应该完全隐藏你的内部董事会。

请注意，如果您发现自己处在一个无法将开发团队的问题跟踪器设为私有的位置，您仍然可以使用这里的其他策略来防止任何外部错误报告泄漏到其中。当你的用户试图报告一个错误并发现一扇紧闭的门时，他们最终会放过你。

注意:这与向未授权的查看者隐藏安全漏洞没有任何关系，任何有经验的开发人员都会确认你应该这样做。

## 随意翻印

[![](img/bdfcfa5485372df9c4e6167bc7d82f8e.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--sMSWc6KS--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/2dnobwlfi0imbxvkil8i.gif)

如果你*正在*与其他封闭的 bug 一起开发一个面向公众的问题跟踪器，那么你的武器库中就有了一个新工具，标签`Duplicate`！当然，每个人都会使用问题跟踪器的这一功能来合并相关的问题，但这是一个完全不同的层次。

如果您因为任何表面上的相似性而反应性地将问题标记为`Duplicate`，您可以避免不得不一起解决许多不同的 bug！如果您针对一个旧的、已关闭的(或无效的)bug 进行标记，这尤其有效。当你的用户看到你认为他们的问题是“旧闻”(特别是当它非常真实和当前的时候)，他们会离开。

当然，你偶尔会遇到一个好战的 bug 报告者，他坚持认为`Duplicate`标签是不恰当的，然后去建立一个坚实的案例。*这并不意味着你必须让步。你总是可以消极-积极地质疑他们的逻辑和可信度。总之，他们说什么并不重要。你可以把`Duplicate`的标签留在原地，让他们对着风喊，直到他们厌烦了离开，最好是永远。*

## 杂记

在你的问题跟踪器上，你还可以使用其他一些策略来与用户互动，这会让他们感到沮丧和绝望:

*   问一些他们已经回答过的问题，让他们明白你并不认为他们的回答值得一读。你会惊讶地发现，问同样的问题，你能逃脱多少次！

*   使用消极攻击性的语言，尤其是那种读起来令人难以置信的伤害，但随后可以否认的语言:“一个人看不懂语气。我是在示好！”

*   当他们对你不屑一顾的行为感到不安时，暗示他们是不可理喻、爱发牢骚的小屁孩。双标牛逼！

*   当他们抱怨你对他们的错误不屑一顾时，指责他们心胸狭窄。

*   列举他们期望像你这样的大忙人花时间关注他们的问题是多么不合理。

你不需要在本文中使用所有这些技术。有时候，混合搭配会产生更好的效果。如果你可以通过很好地处理一些事情来引诱你的用户产生一种虚假的安全感，然后在另一个领域坚持给他们，由此产生的震惊和困惑会有效得多。

## 其实...

[![](img/76cb6c144c5bea58d9d80431536bc297.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--DO3e0UY6--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/qazxf84x8kwvoo25s7rv.gif)

*咳咳*

如果你还没明白，整篇文章充满了讽刺。当然你应该善待你的用户和贡献者！我甚至可以说大多数开发者相信这个前提，至少在理论上是这样。如果有人*希望*疏远用户和贡献者，我会感到惊讶。

然而，在实践中，问题追踪器已经成为软件开发世界中一些最应该受到谴责的行为的场所，现在是我们做些什么的时候了。你对行业和你的用户有责任，特别是如果你在自由和开源软件上工作，为礼貌的、建设性的问题跟踪者行为设定标准。

其次，如果你看到这种不良行为，你不应该原谅它！在我在文章开头提到的 bug 报告事件之后，我被几个长期的专业熟人公然嘲笑受到了伤害。我实际上被告知，在回应一份深思熟虑的错误报告时，我受到的轻视和漠视是*可以接受的*和*应得的*。如果我没有密切意识到免费和开放源码的好处，我可能会认为整个社区充满仇恨和精英主义，离开，永远不会回来。

令人欣慰的是，我有许多其他开发人员证实，我受到的待遇是不可原谅的。唯一为站不住脚的行为辩护的人是他们自己，他们对行为产生的项目是忠诚的，这就是为什么这种行为会被传播！

我也很高兴地说，就最初的错误报告而言，终于有人带着敬意做出了回应；尽管我的错误被证明是几乎不可能重现的，(这不是我第一次在编码中遇到这种情况)，但我还是得到了文明的回应。

我希望我可以说这个问题仅限于小的、无关紧要的项目，但是我在多个主要的软件社区都遇到过，包括几个 GNOME 社区。**什么时候疏远用户和贡献者变得可以了？**

更重要的是，你会怎么做？*
