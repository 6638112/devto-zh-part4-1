# 作为开发人员寻求帮助

> 原文：<https://dev.to/scrabill/asking-for-help-as-a-developer-5879>

啊哦。

你的代码不工作，你不知道为什么。

恭喜你！开发者成就解锁！

在你作为一名开发人员的旅程中，至少会有一次因为代码损坏而碰壁，并且(看起来)看不到解决方案。

这没关系。

碰到一个你无法解决的问题的想法并不是为了吓唬你。事实上，我的意思是正好相反。作为一名开发人员，知道如何排除故障、何时以及如何寻求帮助是一项很有价值的技能。

这里有一个简短的指南，告诉你如何以一种简洁的方式寻求帮助，让你找到解决方案。

## 具体说明你的问题

当提出你的问题寻求帮助时，要意识到你的代码库之外的人可能不知道你在做什么，这可能会使提供解决方案变得困难。

在提出问题之前，您需要问自己的一些问题包括:

*   你期待会发生什么？
*   发生了什么事？
*   你已经尝试了什么，结果是什么？

包括操作系统、浏览器或框架等相关细节。任何可能有助于其他人了解正在发生的事情。这样，他们就能很好地了解你在哪里，你在做什么，以及如何提供帮助。

## 分享你的代码

如果可以的话——请记住敏感信息——分享一段代码。对于试图提供帮助的人来说，确切地知道您正在使用什么代码是很有帮助的。有了代码示例，他们可以快速找到常见的问题，如拼写错误或语法错误。他们还可能使用您的代码尝试在本地重现问题，并测试可能的解决方案。

共享完整的代码是理想的。如果是一个更大的项目，把你的代码放到 GitHub 之类的地方，分享你的 repo 的链接。提及你目前遇到问题的文件、函数或代码行。

例如，你可能会说在`spooky.js`文件中`howManyDaysUntilHalloween`函数的数学是关闭的。这样，无论是谁在帮助你，都可以直接解决问题，而不是搜索你的代码库，希望他们也找到了他们认为你提到的那个。

对于较小的项目，共享一个文件或函数就可以了。如果可以，将代码作为文本文件附加，或者格式化为代码。这样，格式保持不变，代码可以很容易地复制粘贴。如果您收到任何特定的错误消息，您也可以将它们作为代码片段共享。

## 你已经尝试过什么？

你试过关机再开机吗？

为什么是的。是的，我有。

调试，就像知道何时寻求帮助一样，是作为开发人员的另一项关键技能。当你遇到意想不到的行为时，后退一步，在寻求帮助之前花一点时间尝试可能的解决方案。

如果你能自己找到解决办法，那太好了！你可以停止阅读本指南，继续编码。不然可以分享一下你尝试过的，以及结果。

例如，如果您的代码以前工作正常，那么最近添加了什么可能会破坏它？你想增加什么功能？

删除最近添加的代码——或者临时注释掉它——可能有助于您准确地看到它在程序中的哪个地方中断了。应该有输出的时候，你没有得到输出吗？或者，你得到了一个输出，但它不是你试图实现的结果？

当你在调试的时候，记下你已经尝试了什么，是否可行，工作的种类，或者不可行，当你寻求帮助的时候与你的代码分享这些。这样，你的助手可以跳过你已经尝试过的建议，引导你走向正确的方向。

## 提问

感谢您的帮助，您的代码现在可以工作了！酷毙了。

但是，现在它开始工作了…你明白最初是哪里出错了吗？或者修复程序实际上在做什么？

作为开发人员的一部分就是从你的错误中学习。如果可以的话，询问你的助手是否可以解释这个问题是什么，以便在将来的构建中可以避免这个问题。他们也许能够以一种在你的项目环境中更有意义的方式解释概念背后的细节。考虑询问他们是否有任何可以推荐进一步阅读的资源。

这也是记笔记的好时机。给你的代码添加注释，这样将来你就知道每一行代码在做什么以及为什么。

## 总结

遇到一个你无法解决的问题并不是世界末日，即使感觉如此。通过花时间分析问题，用支持代码和期望详细地呈现它，你可以立刻得到你需要的帮助。

当你的同事陷入困境并寻求帮助时，你会给他们什么建议？
