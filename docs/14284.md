# 无服务器是我们应得的抽象

> 原文:[https://dev . to/reactjstutorl/server less-is-the-abstraction-we-worth-4a 31](https://dev.to/reactjstutorl/serverless-is-the-abstraction-we-deserve-4a31)

整体式应用程序是一种反模式，尤其是在迁移到云规模部署时。多年来，微服务的采用一直与容器联系在一起，最近则与 Kubernetes 联系在一起。但是，从纯软件开发转向学习 Kubernetes 和生态系统中所有令人惊叹的工具和组件是令人生畏的。YAML 就是做不出一个很棒的用户界面。

虽然我们一直在平台上构建平台，但这种称为无服务器的东西已经成为现实。当然，它下面有一个平台，但它被抽象掉了。无服务器降低了准入门槛，允许开发人员似乎跳过了学习容器和启动 Kubernetes 实践的复杂性和负担。开发人员更关注他们的核心业务逻辑，而不是围绕它的所有事情。

然而，除了最初的几个功能，事情会变得复杂。分类问题和分析性能几乎是不可能的。这种抽象可能会成为一种负担。但是无服务器的存在是为了解决一个真正的问题，这是我们必须处理的抽象概念。在本次会议中，我们将讨论如何通过最初采用无服务器来完全接受它，采取“无服务器优先”的策略。
