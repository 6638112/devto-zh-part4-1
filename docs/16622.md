在这篇博文中，我将带你经历一个技术面试问题的例子，并告诉你 Abby 概述的步骤。

**我们的例题:**给定一个数组，去掉任何重复的值，返回一个包含唯一值的数组。

# [](#step-1-reiterate-the-question)第一步:重申问题

在深入问题之前，先花几分钟思考一下被问到的问题，然后向面试官重复一遍。你可能听过面试官问一件事，但他们问题的意图可能完全不同。

# 第二步:阐明你的假设

这是你提问澄清问题的时候了！不要假设你已经正确解读了面试官的问题和期望。

示例问题可以包括:

*   当您说删除重复值时，您的意思是每个元素只出现一次吗？或者是否应该将出现不止一次的值从数组中完全删除？
*   我的输入将永远是一个数组吗？
*   我会得到一个空的输入吗？
*   如果没有找到目标，我的代码应该返回什么？
*   我应该把 a -1 等同于 a 1 吗？
*   我的代码应该如何处理单个项目的输入？
*   输入排序了吗？如果是——降序还是升序？
*   该数组应该就地更新吗？

# [](#step-3-create-test-cases)第三步:创建测试用例

既然您已经了解了问题本身和您应该预期的输入/输出，那么就花时间写下一些测试用例，您可以利用它们来进行校对。理想情况下，您的测试用例将覆盖各种不同的输入(例如:预期输入、空数组、负数等)。

示例测试用例可以包括:
测试用例 1: [0，1，1，4，9]
测试用例 2: []
测试用例 3: [-1，0，1，2，2，2，3，10]
测试用例 4: [1]

# [](#step-45-discussexecute-a-brute-force-andor-optimized-solution)步骤 4/5:讨论/执行强力和/或优化的解决方案

在一个理想的世界里，你应该能够与一个强力解决方案对话，并在白板上执行你的优化解决方案。然而，你会遇到一些困难的问题。也许你只能想到强力解决方案——在这种情况下，执行强力解决方案，并讨论你将如何考虑在下一次迭代中以不同的方式处理或优化它。请记住，您应该只写下强力**或**优化的解决方案。您将没有时间或白板空间来完成这两项工作。

现在你可能会问——但是如果你马上就知道一个优化的解决方案，为什么还要讨论暴力方法呢？面试官想知道你知道什么，除非你开口，否则他们不会知道。能够用这两种方法说话表明你知道多少，而且你不仅仅是记住了问题的答案。

## [](#brute-force)蛮力

解决我们提供的示例问题的一种方法是遍历数组中的每一项，将其与数组中的每一项进行比较，并删除任何重复项。

这种方法肯定能满足问题的要求，但是对我们来说这不是最有效的方法。将每一项与每一其他项进行比较将导致 o(n^2 的时间/空间复杂度)，因为最坏的情况是要求计算机比较数组中的每一个排列。我们肯定能做得更好！

## [](#optimized-approach)优化方法

处理我们提供的问题的更好的方法可能如下所示:

*   创建一个默认值为 0 的哈希表(或字典)
*   遍历数组中的值
*   检查您当前正在查看的条目是否已经是哈希表中的一个键
*   如果是，将其设置为 nil(或任何其他值，以供将来删除时参考)
*   如果没有，将它作为值为 1 的键添加到哈希表中，然后继续！
*   在程序结束时返回调整后的数组

```
def remove_duplicates(arr)
    return if arr.length <= 1
    count = Hash.new(0)
    arr.each_with_index do |number, index|
        if count[number] != 0
           arr[index] = nil
        else
           count[number] += 1
        end 
    end 
    arr.delete(nil)
end 
```

这是解决这个问题的一个更有效的方法，因为它只需要在一个循环中检查每个值的重复，然后在另一个循环中查找删除。delete)给出 O(n)的时间/空间复杂度。

# 第六步:测试你的测试用例

既然您已经谈到了暴力方法并执行了优化版本，那么是时候参考那些您之前花时间进行头脑风暴的测试用例了。以每个测试用例为例，通过优化的解决方案进行测试，以此来发现或证明代码中的任何 bug！

# [](#step-7-discuss-timespace-complexity)第七步:讨论时间/空间复杂性

这可以是与面试官进一步讨论优化解决方案的时间/空间复杂性，以及是否有改进的机会。你的面试官甚至会问你如何调整你的代码来适应不同的时间/空间需求。
