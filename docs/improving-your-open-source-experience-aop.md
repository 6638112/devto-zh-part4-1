# 改善您的开源体验

> 原文：<https://dev.to/pascal/improving-your-open-source-experience-aop>

*本文最初发表于[我的个人博客](https://pascalprecht.github.io/posts/open-source-lessons-learned/)。在那里找到更多有趣的内容。*

在参与了几个开源项目并在过去做出了更多贡献之后，我认为是时候分享一些想法了，作为我自己的维护者以及其他项目的贡献者，这些想法非常适合我。

为开源世界中的每一个人创造一个健康和包容的环境可能很难，通常需要长时间的呼吸和耐心。我在这里分享的方法和技巧可能对你有用，也可能对你没用，但是我相信大多数时候会有回报，并且会对每个人产生积极的影响。

因此，如果你想知道如何增加你对 GitHub 上你最喜欢的库的第一个 pull 请求被合并的机会，或者如何让更多的贡献者参与进来并让他们参与到你自己的项目中，请继续阅读！👌

## 作为(首次)贡献者

无论你是人生中第一次做出贡献，还是第一次参与一个特定的项目，或者你是一个将你的改变引入他人代码库的老手。有一些简单的方法和技巧，不仅可以让你更熟悉你感兴趣的代码库，还可以在你与每个相关人员互动的过程中增加你的权威和来自同行开发人员的信任:

### 发现琐碎问题

幸运的是，在 GitHub 的存储库中给问题贴上“第一次计时”的标签已经成为一种好的实践和趋势。当你第一次参与一个项目时，这些可能是你最想看的。即使不存在这样的标签，花时间浏览现有的问题。看看能不能找到你认为可以修复或实现的东西。在你的第一篇文章中不介绍一个新的主要特性是可以的。

文档，小错误和错别字修复也不错！这是我对 IPFS 项目的第一个拉动式请求，只是为了让你有个概念。如果你找不到一个你觉得舒服的问题，试着联系项目的维护者，问他们是否能给你指出一个你可以开始研究的地方。

### 求婚早

一旦你找到了可以努力的东西，就开始尝试吧。第一次尝试不一定要完美。事实上，我相信你会更好地完成你的第一份变更草稿，并尽快提出一个拉取请求以获得早期反馈。这样，你就可以尽早表明你正在处理这个问题。事实上，您可能已经实现了它，这太好了！

### 敷衍了事

当某人编写特定代码时，很难理解其意图。当然，当使用 GitHub 这样的平台时，这可能会更容易一些，因为通常有一个问题可以在您的 pull 请求中引用，但是，通常有一些实现细节需要澄清。有时，你甚至会发现自己处于这样一种情况，你不完全确定是否应该以这种或那种方式实现你的事情，因为两者都有优点和缺点(就像生活中的所有事情一样)。帮你自己一个忙，现在只走一条路。在你的拉动式请求中附上评论，阐明并详细说明你做出某些决定的原因，并表明你也考虑过其他解决方案，如果需要，你很乐意相应地修改你的提议。

### 迭代

你已经走了很远了。您提出了更改建议，您做出了评论并详细阐述了可能需要考虑的事情，理想情况下，维护人员审查了您的 pull 请求，并要求您对一些地方进行调整。这是你可以大放异彩的地方。展示你的响应能力，浏览变更请求并更新你的拉动式请求。一直这样做，直到大家都开心为止。

我个人更喜欢在 PR 之上提交正在进行的工作提交，以获得更好的透明度，或者，如果 PR 相当小，我会立即重新设置提交的基础。无论哪种方式，在 PR 合并之前，最好将 PR 中的提交整理好，这样它们就有了正确的语义。然而，这也取决于项目和维护者如何偏好他们的提交历史。有的甚至不那么在乎。不过，表现出你在乎总是好的。

### 予以明确

最后但同样重要的是，对于您创建自己的问题的场景，无论是问题、错误报告还是特性请求，都要确保花费时间并尽可能明确。如果你发布一个问题，确保给出足够的上下文，这样即使是这个空间中的其他用户也会明白你从哪里来，你的问题是关于什么的。他们或许也能帮上忙。

如果是一个错误报告，解释导致你认为是错误的事情的确切步骤，最好附上一个最小的可重复的演示，这样维护人员可以更容易地跟踪正在发生的事情。这也表明你不只是把一个 bug 扔在那里，而是投入时间和精力来弄清楚为什么你认为某个东西坏了，以及它是如何被复制的。如果这是一个特性请求，请记住，很有可能你不是唯一一个使用这个项目的人。确保你的特性的用例足够通用，这样对其他人也有意义。理想情况下，提出新 API 的样子以及如何使用它。那些只覆盖你的特定用例而对其他人没有意义的特性被接受的机会非常低。

好的，这应该能帮助你把你的改变放到你最喜欢的项目中。当然，每个项目都是不同的，维护者也可能从他们的贡献者那里期望不同的东西，所以当你工作时，总是确保不断地检查期望的结果是什么。

这也把我们带到了这篇文章的下一部分。从维护者的角度做开源。

## 作为维护者

作为维护者，我们可以做很多事情来保持项目用户的参与和贡献者的满意。这里描述的一切应该适用于新开始的开源项目，以及已经存在社区的高活动项目。

### 尽可能简单

这听起来是一件显而易见的事情，但对于正确使用开源软件来说，这可能是最困难的任务之一。如果你的目标是让更多的人参与进来，使用你的代码，甚至为它做出贡献，那么最好的方法就是让它尽可能的简单。这意味着您需要提供完成任务所需的所有信息。换句话说，确保你有一个写得很好的文档来展示你的工具如何使用，不同的选项是什么，以及用户可能会寻找哪些用例。我第一个成功的开源项目是 [angular-translate](https://angular-translate.github.io/) 并把我的大部分时间投入到文档中(甚至是不同的语言！)对用户采用率的影响最大。

### 有求必应

这可能是让开源项目成功的最重要的事情之一。积极响应，向贡献者展示你是来帮忙的。没有什么比提出要求或问题几周甚至几个月都得不到回应更令人沮丧的了。我知道保持领先是非常困难的，尤其是当一个项目成功的时候。

如果你觉得你没有一个问题的答案，或者这个问题在 GitHub 这样的平台上可能被放错了地方，告诉你的贡献者在哪里张贴他们的问题最好，并参考聊天室和 StackOverflow。事实上，将问题委托给社区聊天室是一个很好的方式，可以让你的社区参与进来，让他们互相帮助，而不是让你成为唯一的答案来源。你也可以使用 GitHub 的[发布和拉取请求模板](https://blog.github.com/2016-02-17-issue-and-pull-request-templates/)来解决这些问题。

### 尽可能提供帮助

在一个不是你自己创建的代码库中找到自己的路是非常非常困难的。编码风格可以变化很大，有时甚至很难找到需要改变的地方。根据项目的不同，当涉及到许多组件和部件时，它也可能会变得非常复杂。当你的贡献者开始帮忙时，他们会有同样的感觉。他们需要时间来适应一切，并且很有可能他们不知道某些 API 可能会被更好地重用，而不是在他们的第一个拉请求中引入新功能。

作为一名维护人员，你的工作就是让这种体验尽可能愉快。向贡献者指出他们必须开始挖掘的正确位置，提高对可能涉及的 API 的认识，并给他们一个需要做什么的粗略想法。如果你想让你的贡献者留下来，甚至回来，让他们感到受欢迎，并显示你很高兴他们伸出援助之手，所有的时间！为了给你一个概念，[这里有一个评论](https://github.com/embark-framework/embark/issues/816#issuecomment-423136233)，我试着详细阐述了最初的问题是关于什么的。不必总是那么详细，但提供这种级别的支持会让你的贡献者知道你们是一伙的。

### 保持耐心

这与前一点是一致的。开源可能会非常令人沮丧，尤其是当你觉得你的同行没有为手头的任务付出必要的努力，并让你觉得你是最终必须解决和做事情的人。没关系，我也去过。然而，在 GitHub 这样的平台上不耐烦地做出反应永远不会改善这种情况，尤其是因为在书面交流中很难表达正确的情绪。给你的用户和贡献者时间去消化，即使你不得不重复你自己，也要继续伸出援助之手。从长远来看，这是值得的。相信我。

## 保持积极

无论你是维护者还是贡献者，如果我想让你从这篇文章中学到一件事，那就是在整个过程中保持积极是关键。当你做出贡献的时候，友善一点。思想开放。调整自己以适应你所参与的项目的其他部分，人们会很大程度上接受你。

当你坚持的时候，同样的事情也适用。善待你的贡献者。让他们对他们为你做的工作感觉良好。提供反馈时，尽可能使其具有建设性。这在审查拉请求时特别有用，在协作处理问题时也是如此。

以下是我遵循的一条非常适合我的规则:

> 对于每个变更请求，至少提出一个备选方案

这意味着，我们不应该只是说我们不喜欢什么以及为什么，还应该告诉我们的同事什么是好的替代方案，以及它如何改善这种情况。实际上，这通常会带来非常好的讨论，没有人会因为被拒绝而感到受伤。🙃

好吧，我就知道这么多。让我知道你过得怎么样！
