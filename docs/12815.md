# 我需要一个工具...不，你需要在系统上下功夫

> 原文：<https://dev.to/eviltester/i-need-a-tool-no-you-need-to-work-on-the-system-4il6>

有人通过电子邮件问了我一个问题，我把这个问题的实质解释为，“考虑到我们的项目问题，我们可以使用什么工具来自动提取所有这些数据，并自动化应用程序，让我们正确地测试它？”。

不幸的是，大多数时候，当我被问到这个问题时，答案并不是人们想听到的。

他们想听到的是“用工具 X”。

我经常要告诉他们的是...“没有工具能救你，我愿意做这些事情”。

## 工具什么时候能有帮助？

当您使用标准技术，并且需要以一种显而易见的方式与它进行交互时，工具可以迅速提供帮助

*   查询您有权访问的数据库
*   向 API 发送 HTTP 请求
*   比较文件
*   检查网站上的链接
*   等等。

当一个工具用于定义明确的任务时，它可以快速、轻松地工作，由开箱即用的功能实现，基于标准技术，如您所期望的那样使用。

## 工具什么时候没用？

在以下情况下，工具也无能为力:

*   我们需要访问数据库，但开发团队不给我们访问权限
*   我们希望看到应用程序中的数据，但不知道数据库
*   我们希望看到数据，但它是以自定义格式存储的
*   我们被告知有一个 GUI 使用的 API，但是没有记录下来
*   我们希望从 GUI 中删除数据，因为我们无法访问存储
*   我们想知道安装在服务器上的文件的版本，但他们不让我们访问服务器
*   等等。

以上都不是真正的技术问题。他们试图绕过社会动态问题。

## 工装作为变通方法

有时我们寻找技术解决方案来帮助我们绕过人的问题，因为我们有工作要做。

我确实这样做过，但是在找到工具之前需要知识和经验。和...实际上，黑客是有效管理的替代品。

例如

*   因为外部供应商不允许我访问他们工具的数据库来创建定制报告和分析。我使用 MS Access 和默认管理员密码访问了他们的工具数据库，我在网络共享的一个文档中找到了这个默认管理员密码，我用一些自定义脚本扫描了这个文档，以找到可能包含我需要的信息的文档。
*   因为外部供应商不会给我打开 excel 电子表格的密码，这些电子表格包含他们正在使用的报告的计算公式，并且我确信这些公式错误地计算了他们传递给我们的信息，所以我找到了一种方法来绕过密码，使用 OpenOffice 读取数据和用于计算数据的代码。
*   因为外部供应商不会解释他们的单元测试方法，也不会向我们展示应用程序的内部结构来查看是否存在任何架构风险或 API 访问，所以我使用反编译技术来让我了解应用程序，并进行风险评估和审查代码。
*   等等。

请注意，我重新表述了这些，这样实际问题就在前面了。

> *因为...一些社会交往和沟通问题...我利用我现有的技术知识，使用工具绕过这种情况，获得我需要的信息。*

和...

*   这可能适得其反，使社会互动和沟通问题变得更糟。这些不是我解决问题的第一个方法，我在这个“变通”活动的同时也在处理社会互动和交流问题。
*   在使用工具之前，我必须知道该做什么。我可以选择工具，因为我了解我所处理的问题的技术细节。
*   这不会解决我所面临的所有问题。当我没有获得生产环境的访问权限时，我不会使用工具侵入生产环境来获得访问权限。工具变通办法有一个法律和道德层面。

如果我们设法解决了交流和社会互动问题，事情会简单得多。

## 但是如果真的有工具呢？

比方说，你没有技术知识来破解你的社交方式。

如果真的有一个工具，我告诉你它是什么，你现在会有额外的活动和风险:

*   你需要学习如何使用这个工具
*   你需要学习工具和系统的技术来理解结果
*   您可能会产生新的维护问题
*   你可能需要找到一个预算

您现在必须花时间学习将该工具整合到您的流程中。您可能因为时间限制而需要该工具，并且希望该工具能够加快您的过程，但是，它只是减慢了您的速度。

如果没有技术知识来完全理解该工具、其工作原理及其功能，您可能无法识别该工具的技术限制，并会将信息报告为误报。当在项目中加入新的扫描技术(如安全扫描软件)时，这种情况经常发生。但是，如果没有评估和解释结果的知识，我们提出的问题实际上并不是问题。

如果我们不仔细选择一个工具，并且不有效地使用它，那么我们可能会为自己制造一个维护问题，这个问题可能不会立即出现，但是以后，在我们开始依赖它之后。

如果没有预算，我们经常只看免费的开源工具，这些工具可能没有我们快速入门所需的完美界面，或者没有支持作为我们缺乏经验的补偿机制。

工具需要知识和经验，以便我们理解它们，它们做什么，它们如何做，以及如果工具本身开始让我们失望，我们可以使用什么替代方案。

## 特征请求

我在电子邮件中被问到的许多问题并不需要工具来解决。

其中一些可能需要测试应用程序中的新特性来支持测试。

例如

*   附加报告
*   数据导出
*   控制数据的管理界面
*   等等。

额外的报告通常可以消除某人直接访问数据库或文件系统的需要。这些在实际环境中也经常被证明是有用的，因为我们在测试过程中需要的信息也经常需要在实际环境中进行协调。

数据导出通常可以消除从 GUI 抓取数据的需要。在实时环境中，这通常有助于支持临时请求和以后的报告。

控制数据的管理界面可以帮助将系统配置为受控状态，并限制环境中的数据量，使测试更加可控。

但所有这些都需要相当好的沟通和社会互动，否则可能没有意愿或动机来实现这些功能。

## 明面上并不是唯一的解决方案

当工具是解决方案时，正确的工具可以是您的过程的极好的补充。

工具不是唯一的解决方案。

工具会带来风险，所以我们在采用工具时必须小心。

工具化的一个风险是它会使社交和交流问题变得更糟。

工具似乎更容易引入。但是我不确定我是否见过工具作为解决社会和交流问题的有效替代品。但是这些问题的解决通常超出了寻求帮助的人的影响范围，所以他们寻求一种技术工具解决方案，希望有这样的解决方案。

社交和沟通问题可能是最难解决的事情之一，但一旦解决，它们会带来更好的工作环境和改善的长期状况。
