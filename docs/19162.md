# 千万不要在创业公司做软件工程师！

> 原文:[https://dev . to/veebuv/never-work-as-a-startup-4a4p](https://dev.to/veebuv/never-work-as-a-software-engineer-in-a-startup-4a4p)

明天我将在 200 人面前演讲，主题是创业公司的软件开发。关于这方面的书籍有数百种，所以我将试着从大多数书中总结我的经验。

尽管我们是一家位于塞纳里奥的初创公司，但我已经不再雇佣软件工程师了，尽管如此，我还是试图在这个过程中忘记和重新学习一些东西。

我知道这很令人困惑——我仍然不得不抓住它的全部，但诚实的事实是，仅仅是一名软件工程师就会让你很容易被解雇或在初创公司中失去价值。

你需要解雇自己，重新雇佣自己成为一名产品工程师。我在以前的文章中多次提到这一点，我真的袖手旁观这一点。

我认为这不一定适用于大公司招聘专家和 algo 重型工程师，但是在一家初创公司，你需要考虑产品、营销和最重要的客户。

在大公司中，从创造者(开发者)到最终用户，从层级到项目经理、产品经理、营销人员、高管等等，都存在明显的脱节——但在初创公司中，如果你把代码往上推...起来了。

那么，作为一名产品工程师，有什么特别之处是软件工程师做不到的呢？一些事情:

### [](#1-they-carry-a-get-shit-done-attitude)1。他们带着一种把事情做完的态度

当然，有些工程师也有这种想法，这些说法不是二元的，也不是排他的，而是针对绝大多数人的。当你看到 github 的讨论或人们分享他们的发现的会议活动时，这些都是围绕着工程师的，而不是围绕着客户的。

所以，是的，产品工程师有一种把事情做好的态度，记住他们需要把好的工作做出来，但是他们很快就能理解一些技术决策相对于其他决策会有多少债务。随着时间的推移，人们会更好地理解这一点，即使经过十年的编程，我也可以确定没有正确或错误的答案，这完全取决于具体情况。

### [](#2-business-first-software-second)2。业务第一，软件第二

你应该坚强起来，意识到建立在最新最伟大的技术上不会让你成为更好的工程师。作为一个失败创业公司的工程师，你几乎从来没有像一个成功创业公司的工程师那样享有好名声，即使你在失败创业公司的代码可能值得奖励，而你在成功创业公司的代码可能值得解雇。你看，这是与生俱来的——好的代码在好的公司中并不是巧合，这是因为公司做出了雇用小型 CTO 的明智决定，这些人明白他们的客户和他们的代码一样重要。

这并不意味着你要放弃所有的道德准则，建立在 PHP 之上(我是开玩笑的:p)，但事实确实如此。不是 PHP，而是任何因为流行与否而被认为不合适的语言。你做一个直接的风险分析，看看什么能让我尽快达到下一个目标。无论是更快的迭代、更多的特性还是模块化的代码库。

### [](#3-customer-first-business-second)3。顾客第一，生意第二

归根结底，当你为客户解决问题时，如何让他们的生活尽可能轻松。有时业务需求变成了业务需求，而不是客户需求，如果你只是一个软件工程师，你将做你被告知要做的事情，因为那是你的限制，至少是我几年前的限制。

跳出框框，理解如果业务需求超出了客户需求，你就可以发表自己的意见，更重要的是，通过用你的技术能力来证明你的“意见”是正确的，技术人员是坏蛋，我们是创造者，所以最终如果我们了解消费主义和执行，这将使我们刀枪不入。

所以，是的，如果你在创业，不要做软件工程师，做产品工程师。你的影响力将是 10 倍，我不骗你。

人们会更认真地对待你 ALOT，你会爬得更快，你的代码会更重要，影响会更大。你的工作很重要，没有理由不让更多的人体验你的 genius 代码，你可以通过专注于产品并确保你的客户拥有他们一生中最美好的时光来实现这一点。
