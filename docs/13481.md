# [](#clean-code-chap-2)清理代码第二章

啊干净代码的第二章。对于那些刚刚加入的人，我想最终读完[干净的代码](https://www.amazon.com/Clean-Code-Handbook-Software-Craftsmanship/dp/0132350882)，并把输入我的想法视为保持动力的一种方式。如果你有兴趣跟随我的想法，第一章被贴在[这里](https://davidemily.github.io/2019/clean-code-chapter-1/)。

第二章的标题是“有意义的名字”，充分暗示了这一章的内容。我发现这一章特别有趣的一点是，很多材料看起来都是常识，但也是我们应该更加关注的。正如罗伯特·马丁所说，

> “名字在软件中无处不在。我们给变量、函数、参数、类和包命名。我们命名包含 thm 的源文件和目录。我们将 jar 文件、war 文件和 ear 文件命名为。我们命名命名命名。”

我喜欢这句谚语。它完美地诠释了程序员的工作，命名事物。我听说许多人把编程更多地称为艺术，而不是科学，我认为这是非常正确的。也许一些英语和词汇的课程会比我不得不上的大量数学课更有帮助...也许吧。不管是谁，让我们继续学习如何成为更好的命名者，进而成为更好的程序员。这一章涵盖了许多不同的想法，我希望你们所有人最终都能读到它。我想涵盖一些要点，希望我记得在工作中使用它们。

#### [](#use-intentionrevealing-names)使用透露意图的名字

在所有的命名规则中，我发现这一条是最重要的。我非常清楚地记得在工作中使用了“records”或“recordsList”变量名。虽然在当时这很有意义，但几个月后回来时，却无法回忆起这是什么类型的记录。此外,“记录”列表如何与“记录列表”区分开来？这本书甚至进一步深入细节，抽象出一个有用的命名标准的比较，我认为这是一个很大的帮助。当然，将“recordsList”的计数与 5 进行比较在当时意味着完美的意义，但如果与“完整的应用程序记录”进行比较，就更容易理解了。

#### [](#avoid-disinformation)避免造谣

我发现这一部分非常有趣，但可能不会对我的日常工作产生影响。也许这只是我工作的环境——所有的微软技术人员，非常聪明的人，产品的特定部分——但是我发现在我们的代码中普遍缺乏虚假信息。我认为这是大公司和有特定目标的团队的优势之一，我团队的行话与我工作的几个团队的行话相吻合。我可以预见，在使用不同操作系统、第三方软件和工具的环境中，这将成为一个更大的问题。我认为这在很大程度上与注意到其他环境和工具的存在有关。

我确实记得有一段时间我们的团队会用“cron”这个词。由于来自 Linux 环境，我对我们的 cron 作业在哪里运行以及如何修改它感到困惑，结果却发现它是一个在计时器上运行的自制的预定程序。当有人向我解释时，这仅仅造成了 30 秒的延迟，如果这个软件被传递给另一个团队，它可能会更加有害。

#### [](#use-pronounceable-names)用可发音的名字

当我想到我问我的同事函数“Xyz23gh”是如何工作的时候，我咯咯地笑了，我很高兴我没有处理这个问题。这是我认为相当常识性的部分之一，实际上有点难以做到。也许这只是我与聪明人一起工作的经验，但是我不能想象写一个函数而不把它命名为容易发音的东西。将它命名为可发音的东西，这样做的好处是能够调出代码的特定部分，并让某人确切知道它在哪里，或者在哪里可以找到它。如果“GetRecordsAsync”中有问题，我可能会比试图拼写“RecPulAsy”更快地找到它。

#### [](#class-names-and-method-names)类名和方法名

> “类名应该是动词。方法名应该是名词。

有些东西读起来很有意义，但在编码时却很难记住。将它们分开可能比通常想象的要困难得多，这不是因为我们不都是聪明人，而是因为语言是如何工作的。如果我正在开发某种博客软件，变量名“reader”对我来说可能是一个非常好的名字，因为它是使用我的博客的人的类名。“Reader”作为反映我使用 API 的方法名可能也很有意义。为了避免犯这种错误，更好的选择(尽管我确信还有更好的选择)可能是调用类“Customer”或“Account ”,调用方法“SaveArticlesApi”或“AddEmployee”。

此外，坚持命名标准并且不要太有创意也很重要。如果你的整个代码库都在使用“检索”这个动作术语，那么继续这个模式是有意义的，而不是随机地用“拉”来开始这个方法。在团队内部笑话之后不命名类或方法也是有意义的。如果我读到一个以几个月前我的团队中发生的一件事命名的方法，我可能会窃笑；如果我读到发生在另一个团队的事件，我可能不会笑，我可能只是感到困惑，需要澄清。

#### [](#conclusion)结论

虽然这不是所有命名标准的完整列表，但是我发现这些是最有用的。我想把其他的提示放在一边，这样我就不会抢了罗伯特·马丁的风头，我建议通读这一章，哪怕只是为了扩展我提到的提示。这一章真的打动了我，不是因为它给了我改进的方法，而是因为它让我欣赏我工作的环境，以及我们通过(某种程度上)体面的命名标准避免的麻烦。我们海军有句谚语“每条规则都是用血写的”，我当然很高兴从我的前辈那里学到这些命名规则，而不是在工作中拔头发。
