# 迷失在翻译中——与技术人员合作

> 原文：<https://dev.to/bismuthlabs/lost-in-translation-cooperation-with-technical-people-18nn>

几年前，我经营着一家不起眼的广告技术公司。作为一个非技术型的创始人，我很快就面临着管理开发过程的极限。我当然想要这一切:不贵、速度快、质量好；不可能的三角。不知何故，与我一起工作的技术人员似乎有足够的信心承诺这一切，接下来的事情是可以预见的。

# 第一个问题和相互指责

第一个问题始于质量。大量的用户和大量的 bug。如果你有 100 个用户，其中 10%遇到了 bug，你将有 10 个会话需要维护。如果你有几千个用户…当市场购买和用户使用时，你有充分的理由对你的商业模式充满信心。无法相应交付的挫折感迅速增长。

我指责“无能”的开发者。他们对在他们之前从事代码工作的每一个人指指点点，然后把它像意大利面条一样放在一边。我理解他们。在最初验证商业模式时，我是一个吝啬鬼，你只能暂时逃脱，这很正常。你可以猜到发生了什么；重写。这一次代价很高。不仅在达成合作协议时成本很高，而且由于开发过程中的一些意外，完全交付时成本更高。我记得我对结果感到高兴，但对新事物的改变是如此僵化/昂贵感到震惊。我们终于有了一些美好的东西，但我们现在有点陷入了一段时间。在一个快速变化的环境中，尤其是作为一家初创公司，你希望能够进行 A/B 测试，进行 pivot，尝试这种和那种，但是如果你的代码不允许你这样做，你的小而灵活的竞争优势就会消失。这一变化非常重要，我没有明确地向开发人员提及。

# 沟通中的一课

很长一段时间，我的简单结论是好的开发人员很少。这是你会从许多非技术经理那里听到的说法。幸运的是，该公司在技术和非技术的艰难遭遇中幸存下来，我喜欢分享这一经验，以帮助其他人更快地发现和解决像我遇到的问题。毕竟，真正的教训不是好的开发人员很难找到，而是沟通方面的教训。

在我这边，我在判断一个我不理解的过程的结果。这个问题当然是在这个过程中发现的。我对短期和长期有什么期望？我真的沟通得那么好吗？我在这件事上质疑过自己吗？我是否充分说明了快速旋转、A/B 测试等..是必须的。从一个好的 MVP 比一个全功能的 bug 更有用的意义上来说，我提出了一个现实的要求吗？如果我对自己诚实，那么我不得不承认我很大一部分的天真。我只是不知道该要求什么或如何表达，因为我不知道在开发方有哪些权衡。例如，我不知道灵活性和架构是联系在一起的。这只是一个例子。

在开发方面，一些元素也值得一提。我得到了我想要的。我从来没有被问过我没有问过的问题。我并没有要求结构良好的代码或者符合命名约定之类的东西。我问了一个应该从某一天开始的结果。我从未真正面对灵活性/质量、价格和时间之间的权衡。我只是不知道它的存在或重要。软件架构和代码质量的意义离我的世界太远了。我对开发人员的工作缺乏兴趣可能是我最大的错误，我认为开发人员最大的错误是对我公司短期和中期的真正需求没有太多兴趣。

# 真正的共同利益

你可以期待一个创始人不幼稚，就像你可以期待一个开发者知道如何交付一样。但是在许多情况下，两者都是基于假设的；假设你知道对方想要什么或需要什么，但对对方的挑战没有真正的兴趣。如果我要求又快又便宜，我不知道会得到什么样的回报，也不知道改变的成本会有多高。

起初，我确实得到了我想要的一切:快速、随时可用、价格低廉。我为永无止境的重构付出了沉重的代价。这种重构不仅会带来财务成本，还会耗费时间；宝贵的时间，如果你学得快，不能快速转向。它麻痹了你作为创始人的野心。如果你依赖于不可预测的交付周期，它会使你的整个团队瘫痪。如果我知道代码质量和软件架构决定了代码对变化的容忍度，我会把它作为 KPI。

很长一段时间，我认为我的幼稚主要是因为我在毕业时开始创业，还没有在更大的公司工作过。然而，在过去的几年里，我在比利时的孵化器 Start it @KBC 为 70 多家初创公司担任商业教练，我惊讶地发现，拥有超过 20 年职业经验的人更经常犯同样的错误。我最近遇到了一位成功的连续创业者，他对这个话题显然很有主见，他说“哦，我和我所有的 CTO 都大吵过一架”。原来，即使是很多老牌公司、企业，技术人员和非技术管理人员之间的理解也很差。

# 结论

总之，当你不是技术人员，但你希望与技术人员进行良好的合作时，你需要对他们如何开展工作有真正的兴趣，并且你应该只与对你真正需要的东西有真正兴趣的人合作，而不仅仅是你认为你需要的东西。您想知道开发团队面临的挑战和权衡。守则的目标是什么？测试市场的快速 MVP 还是需要 2 年、3 年甚至 4 年产品生命周期的完整版本？灵活的成本是多少，你对灵活的定义是什么？您将使用哪些 KPI 来量化和评估期望值？这些 KPI 真的能告诉你你正在做的事情的本质吗？你确定其中一些不是虚荣的指标吗？你将使用哪些工具来跟踪、管理和改进？

我是作为 [bismuthlabs.io](https://bismuthlabs.io) 的创始人写这篇文章的。对于现有的代码，铋告诉公司什么是值得修复的。对于新的规范，铋保护了架构的符合性，因此保护了规范对变化的容忍度。目标是对变化有更大的容忍度，导致更少的重构，必要时更好的重构，以及更可预测的交付周期。作为一个有趣的副产品，一些公司把我们作为初级培训对象。
不用说，当我见到我的联合创始人时，我立刻明白了他们正在解决的客户难题的价值。

你们的科技-非科技合作进展如何？类似的误解还在继续吗？我是否指出了需要注意的地方？有兴趣借鉴其他经验。
