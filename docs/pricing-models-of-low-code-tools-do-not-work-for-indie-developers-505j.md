# 低代码工具的定价模式不适用于独立开发者

> 原文：<https://dev.to/jcabot/pricing-models-of-low-code-tools-do-not-work-for-indie-developers-505j>

我一直以为[低代码工具](https://modeling-languages.com/low-code-platforms-new-buzzword/)(以前称为 CASE 工具，模型驱动的工程工具，代码生成工具，...)是[独立开发者](https://www.indiehackers.com/)和[独立创业者](https://en.oxforddictionaries.com/definition/solopreneur)的完美搭档。

这些解决方案带来的[生产力和质量的提高](https://modeling-languages.com/benefits-modeling-or-how-convince-your-project-manager/)应该是那些试图推出他们的想法并独自围绕它建立业务的人的理想解决方案。我喜欢说**工具可以取代联合创始人**。

但我们看不到这一点。对于试图创业的人来说，低代码工具不是首选。我觉得这很奇怪。所以我决定做更多的调查，甚至考虑自己开发一两个应用程序，看看是否能为独立开发者找到这个主要的阻碍。事实证明，独立开发者不采用低代码工具有一个很好的原因:定价模型不适合他们。

## 低代码工具多少钱？

简而言之:很多。如果你的收入仍然很低，而你的主要目标是吸引更多的客户，那么用它们来开始一项新的业务就太多了。同样，我并不是说低代码工具绝对定价过高。我完全理解它们能给一家开发一些新的内部工具的大公司带来的好处，这些工具将为其运营成本节省数百万美元。我是在试图用新软件产品谋生的小企业的特殊视角下评估它们的。

是的，我肯定你知道一个工具，它更便宜(甚至免费)并且非常适合这个场景。如果是这样的话，请在下面留下工具名称的评论。我做了一个探索性的搜索，重点是我所知道的一些最著名的专业工具。这篇文章的目标不是要列出低代码工具的价格(事实上，出于同样的原因，我尽量避免给出太多具体的例子)，而是探索一种趋势。

### 低码工具提供免费计划吗？

是的。全部都是。他们可以很好地学习该工具的工作原理并进行实验，但是他们太有限了(就用户数量或“席位”而言)；例如， [OutSystems](https://www.outsystems.com/) 将免费计划限制在 100 个用户。我认为你甚至不能尝试你的想法是否符合产品市场。通常情况下，你的软件还会为你自己的用户提供一个免费计划，让他们在付费前测试你的工具。对于低代码免费计划来说，用户太多了。

免费计划的其他限制可能是应用程序的数量(1；同样，如果您正在尝试不同的想法，看哪一个更有吸引力，这还不够好)或者低代码特性(您可以使用建模环境，但是不允许部署生成的解决方案)。

### 低代码工具的价格范围

那么，如果免费版本还不够，如果我计划测试 2-5 个应用程序，并且我希望每个应用程序上有大约 1000 个用户，我应该为一个低代码工具支付多少钱呢？预计每月支付四位数的费用。例如，OutSystems 企业计划每月超过 5000 美元。而 Mendix Pro 计划也差不多是这个数字。当你按座位付费时(例如 [WebRatio](https://www.webratio.com/site/content/en/pricing) ，事情变得更糟，你最好直接与工具提供商协商！。

请注意，上述价格通常包括在工具提供商的云中部署和托管您的应用程序。因此，没有额外的成本支付亚马逊 AWS 或其他类似的基于云的解决方案。

然而，数学就是不起作用。在上面场景中的 1000 个用户中，只有一小部分(假设是 100 个，这已经很乐观了)是付费用户。如果你的服务收费是每月 20 美元(对于一个刚刚起步的软件服务来说，这是一个常见的数字)，这意味着你一个月只能赚 2000 美元。甚至不足以支付低代码工具的月成本。

此外，你也处于一种锁定状态。你不能只花一个月的时间(开发应用的那个月)购买低代码工具，然后就停止。你需要每月付费，这不仅是为了让你的应用程序继续运行，也是为了能够改进和发展它。而且，因为你开始了，这将是经常的情况。

## 有没有办法让独立开发者的低代码工具定价行得通？

上述定价模型对于低代码工具目前的目标客户来说是合理的。正如我所说的，当你开发一个公司从工具的每个用户中获益的内部应用程序时，按座位付费是有意义的，但当你发布一个有大量免费用户的软件时，就不是这样了。

我想请低代码供应商看看其他受众，并尝试提出一个更符合 solopreneurs 需求的定价模型。最近的运动，如 [Bubble](https://bubble.is/pricing) (具有更好的定价模型)或 [Generative Objects](https://www.generativeobjects.com/) (最近走上了完全开源的路线，只有当你想让他们托管你的应用程序时，你才需要付费，但你也可以自己托管它，许多其他开源项目如 [WordPress](https://livablesoftware.com/wordpress-content-management-democratize-publishing/) 使用的相同模型)都是令人鼓舞的例子，但我认为低代码工具提供商作为一个群体需要认识到，有一个巨大的未得到充分服务的独立开发者市场，他们正在寻找加快他们的开发过程的方法。一些提供商(如 [Betty Blocks](https://www.bettyblocks.com/pricing) )根据工具的复杂程度而不是用户数量向你收费(这种方式更好，因为当你有更多的用户时，你不会支付更多的费用，但当你几乎没有任何付费用户时，它在开始时仍然非常昂贵)。

我仍然相信，如果我们在定价模式上有点创意，低代码工具可能是独立制作人的完美匹配。我没有解决方案，但我认为有一些选项可以探索。例如，低代码工具**可以只对使用低代码工具开发的应用**的付费用户数量收费(达到免费/付费用户的一定比例)。或者**他们可以从软件的总收入中获得提成或百分比**，这样独立开发者赚得越多，低代码供应商赚得也越多。当然，在这两种情况下，对于低代码供应商来说风险更大，但仍然很小(“风险”只是赚不到钱，而不是真的失去钱，因为保持应用程序部署在其云中的成本非常低)，如果用低代码工具生成的软件产品最终成为[独角兽](https://en.wikipedia.org/wiki/Unicorn_(finance))，潜在的好处是巨大的。

不管怎样，今天平台开发者的关系更像一场政治斗争，而不是一场舞蹈。我们需要各方进行更深入的讨论来解决这个问题！
