# 开放/封闭原则:如何说服其他开发者？

> 原文:[https://dev . to/rvictorino/open-closed-principle-how-you-confident-other-devs-4 i33](https://dev.to/rvictorino/open-closed-principle-how-do-you-convinced-other-devs-4i33)

一些坚实的原则很容易理解，它们的好处也很容易解释:例如，单一责任原则就有一个明显的优点。软件实体每个只做*一件*事情。知道一段代码做什么变得非常容易，即使是在一个巨大的应用程序中，因为它做了*一件*事情。

另一方面，开/关原理有点抽象。

当我向老板建议我们可以在应用程序的某个地方实现某种模式时，我自豪地说这将更加尊重开放/封闭原则，*因此更好*。

他问为什么添加类会比编辑方法更好，因为我们仍然会做出改变；添加类会使我们的应用程序增长得更快，这可能会使它更难维护。

老实说，我没想到会这样。我意识到我可以证明我所知道的大多数编程原则的利益，但是不要在改变方法上增加类，因为两者都在某种程度上增加代码。

你如何描述尝试应用开放/封闭原则的好处？你觉得我误解了吗？

总的来说，除了它更容易维护，所以从长远来看我们会避免开支，当涉及到实现一些模式/任何与编程原则相关的东西时，你的理由是什么(特别是技术对/主要是如果它是一个领导/CTO/任何人的技术并且在你的公司层级中比你高)？
