# 开发人员宣传:常见问题

> 原文:[https://dev . to/di/developer-advocate-favouring-FAQ-questions-577k](https://dev.to/di/developer-advocacy-frequently-asked-questions-577k)

我收到了很多关于作为一名开发者拥护者(DA)的问题。我喜欢这些问题，因为每一个问题都是一个倡导开发者权益的机会！

最近，一位正在考虑 DA 职位的同事向我寻求建议，他们的问题很好地总结了我曾经被问到的关于我工作的每个问题，所以我想我应该在这里回答他们。

(此外，它对于电子邮件来说太长了)。

简单介绍一下我:我是 Google 的一名开发者倡导者，在那里我主要负责 Python 社区的宣传工作。我拥有计算机科学的学士/硕士学位，并在大学/政府研究实验室工作了多年，直到我决定离开学术界。在做了一些非软件的事情之后，我加入了一家软件咨询公司，在最终加入谷歌之前，我花了相当多的时间来开发定制软件。

另外，声明:我只在谷歌做过阿达。虽然我知道这个角色在其他组织中的意义有所不同，但我真的只能说它在我的组织中的意义。我也只做过我自己，所以我也不能代表谷歌的所有 da，但我不认为我的经历在我们当中是独一无二的。

首先，我要回答一个问题，这个问题不在最初的清单上，但却是我被问得最多的一个问题:

## [](#what-is-a-developer-advocate)什么是开发者倡导者？

我喜欢这样描述:我在谷歌帮助代表 Python 社区。当我们构建 Python 社区中的人们将使用的产品时，我帮助确保这些产品适合这些用户。

这意味着我的部分工作是深入参与 Python 社区。我做这件事的方式基本上完全由我自己决定，所以这通常表现为我喜欢在 Python 社区中做的事情，比如从事 [Python 包索引(PyPI)](https://pypi.org) 或其他打包工具，组织 [PyTexas](http://pytexas.org) ，以及参加和[在 Python 会议上演讲](https://di.codes/speaking/)。

作为倡导者也意味着我不想向你“推销”一种产品，但我确实想让你知道它的存在，但只有在我认为它可能对你有帮助的情况下。我真的不想向您展示您实际上并不需要的工具，如果我认为更合适，我更有可能告诉您使用更简单或更便宜的工具，即使它不是我们的产品之一！

更重要的是，如果你*正在*使用我的产品，我会很在意:我想知道你对它们的看法，你是否尝试过它们，你的第一印象是什么，你面临的挑战是什么，等等。而且，我想在未来成为你的资源。

## 离开一个传统的工程角色，你感觉如何？

我经常被问到这个问题。事实上，在我告诉一个朋友我将在开发关系部工作后，他们对我说的第一句话就是:“这是不是意味着你不再被允许写代码了？”

我实际上认为开发人员/用户支持是传统工程角色的一部分，只是一个经常被传统工程组织忽视的角色。因此，虽然这通常意味着“你对不总是编码有什么感觉”，但事实上这完全取决于我:如果我想，我可以像“传统工程”角色中的人一样编码。

在谷歌，我们目前有两个头衔:“开发者倡导者”(DA)和“开发者项目工程师”(DPE)。人们对阿达的刻板印象是，他们把时间花在制作内容、与客户交谈和发表演讲上。类似地，对 DPE 的刻板印象是，他们构建和维护我们的客户端库，编写代码样本，并致力于我们的开源项目。

但事实是，这两个头衔只是代表了单一角色“开发人员程序工程师”梯度的两个极端。这意味着一些 DAs 的人仍然编写代码，一些 dpe 的人在会议上发表演讲。

虽然每个人都有一套特定的责任，但这两个极端之间的“滑动”或多或少取决于我们。如果我想做比现在更多的工程，我有很多机会这样做，找到平衡或多或少取决于我。

## 你觉得如果你选择回到“传统工程”会有可能吗？

鉴于以上所述，我会说是的，我不仅觉得我可以回到一个“传统”的工程角色，而且我现在可能有一个更好的视角，会成为一个比以前更好的工程师。

成为阿达让我有能力与如此广泛和多样化的用户群建立联系，并不断让我重新评估一个“好主意”是我这个工程师想要的，还是用户*实际上*想要的。

## 你正在寻找一个好的工作/生活平衡吗？我知道助理检察官经常出差。

当然，但这受到谷歌文化的严重影响。与普遍的看法相反，我工作的好处在于，出差并不是成为谷歌阿达的必要条件。有经常出差的 da，也有一年才出差一两次的 da，两者都完全没问题，都会事业有成。

这意味着我想旅行多少就旅行多少，而且我随时都有能力调整我旅行的次数，以适应我的工作/生活平衡。

## 对于这样一个角色，你觉得最大的挑战是什么？

最大的挑战:衡量你的影响力。

衡量影响具有挑战性，因为很难将我们想要实现的结果与我们所做的事情联系起来。像“开发人员比一年前对我们的产品更有信心”这样的事情不仅是一个难以衡量的指标，而且即使我们能够衡量，也很难找到这种转变的根源。是我们所有的演讲吗？YouTube 上的内容？[吉尼斯世界纪录](https://www.blog.google/products/google-cloud/most-calculated-digits-pi/)？

这使得你很难知道什么是值得努力的，也很难证明你做得很好。博客帖子的浏览量是受欢迎程度的一个很好的代表，但是因为“拥有一个受欢迎的博客”不是我们的目标，除此之外很难量化。

## 在谷歌，阿达的一天是什么样的？你花时间做什么？

给人的印象是，阿达把所有的时间都花在了写演讲稿、参加会议和演讲上。这是有道理的，因为你可能看到最多的 da 就是做这些事情的 da，你看不到他们做其他事情。

考虑到我演讲的次数，以及他们在活动中见到我的频率，很多人惊讶地发现旅行和演讲实际上只是我工作的一小部分。就像我之前提到的，我这样做的主要原因是因为这是我成为社区一员的方式之一，也是我喜欢的事情。

我认为这种看法部分来自于倡导者和 T2 传道者的混淆。关于这些术语在开发者关系的上下文中意味着什么，有无休止的争论，但至少在谷歌的上下文中，区别是这样的:一个福音传播者希望用户知道一个产品的存在；倡导者希望产品知道他们的用户存在。

(在我看来，判断你是在寻找一个传道者的职位还是一个倡导者的职位的好方法是:你将向谁汇报？是工程总监吗？还是营销？)

例如，你可能会注意到，我在大会上的发言很少是关于我们的产品的。这是因为我的讲座只是与我的社区互动的一种方式，而不是向您介绍我们产品的场所。

所以，如果作为阿达，说话是可有可无的，我也是可有可无的，但只占用了我很少的时间，那我到底在做什么呢？嗯，很多:

**与客户交谈:**当我出差参加活动时，我通常会尝试与使用谷歌云的当地开发人员安排一些会议，或者在那里会见一些人。我偶尔也会旁听客户的电话会议，了解他们的体验如何，谈论某个具体问题，或者帮助他们决定如何最好地使用我们的产品。

与非客户交谈:我通常真诚地好奇每个人在做什么，他们在使用什么工具，这些工具是否有效。这有助于我对人们期望我们的产品能做什么、其他产品的优势和劣势以及行业的新趋势有一个大致的了解。

**与我们的产品团队交谈:**通常这是围绕一个新产品或新特性的发布，但有时是为 Python 社区设定总体策略。

**成为新产品或新功能的“零用户”:**这意味着尝试新事物，看看它们有多直观，以及是否有我们没有考虑的边缘情况，但可能是我们用户的有效用例，并在必要时提交错误报告。

**关注用户可能提出问题的地方:**我一直在跟踪 StackOverflow，并有各种方法关注 DEV、Twitter、黑客新闻等媒体。人们谈论他们使用 Google Cloud 的体验，然后接受反馈，并将其转化为我们工程团队的可行问题。

在开源领域工作:这基本上满足了我写代码的愿望，还有一个额外的好处就是让我参与到社区中，同时也回报社区。

**为 Python 开发者创造内容:**这可能是任何东西，从博客帖子、发布公告、YouTube 视频、播客、文档、研讨会/教程/代码实验室、[到像这样的特技](https://dev.to/googlecloud/ministry-of-silly-runtimes-vintage-python-on-cloud-run-3b9d)。

**为 Python 社区制定策略:**我帮助确定我们在活动中的存在和赞助，同时也为我们产品的未来制定更广泛的计划。

**作为谷歌关于 Python 和 it 社区的专业知识的一般来源:**我工作中最有趣的部分之一是，我被拉到谷歌的各个团队中进行讨论，这些团队希望了解如何与 Python 生态系统合作，以 Python 用户为目标，并以他们期望和欣赏的方式工作。

**领导一个致力于 Python 宣传的团队:**我不是谷歌唯一关心 Python 的人！为了协调我们的工作，我领导一个团队，他们做的事情和我一样(而且经常是更多)。

## 你对考虑这个角色的人有什么建议或忠告吗？

1)培养你的同理心。

至少，开始在你目前的公司做更多的宣传。参加或观察一些你的产品的用户体验研究，或者和一个从未使用过它的朋友坐下来，看看他们是否能通过你的快速入门。我保证，会让你大开眼界的。

挑选你的社区，并深入了解它。

也许你已经做了前半部分，但你做了后半部分吗？你需要在你的组织中为一大群人说话，你需要能够准确地代表他们想要什么。

请记住，组成你的社区的人并不是“每个使用我们产品的人”或“每个使用我们竞争对手产品的人”。那不是社区，是细分市场。

先考虑加入一家软件咨询公司。
