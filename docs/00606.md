# 开始我的计算机科学专业时，我希望我知道的四件事

> 原文:[https://dev . to/andyhaskell/four-things-in-coding-careers-I-wish-I-know-starting-my-cs-major-22d 0](https://dev.to/andyhaskell/four-things-in-coding-careers-i-wish-i-knew-starting-my-cs-major-22d0)

在我大学二年级的时候，我上了一门网络开发课程，并且非常喜欢这门课，有很多家庭作业，比如用 PHP 建立一个小型社交网站，用 Google Maps API 制作一个网络应用。有了这样的项目，看到你只用一学期的材料就能创造出如此多的东西，我立刻就知道 web 开发是我想从事的工作。所以在大三的时候，我宣布了我的计算机科学专业，并以这篇文章的形式加入了塔夫茨大学计算机科学系的脸书小组。

通过这门课，我已经学会了构建一些非常专业的东西，我想这个专业已经是囊中之物了。然而，我不得不在两年内加上暑期班赶上计算机科学课程。由于这一点，我不得不花更多的精力学习算法等更重理论的课程，这个专业与我的预期不同，我在大四的时候花了很多时间考虑如何将我即将获得的学位转化为一份工作，同时对仍然留在学校感到疲惫不堪。

现在我已经在科技行业工作了 6 年，所以我想分享 4 件我希望在进入计算机科学专业时知道的事情。在我们开始之前，有一点需要注意，这个建议主要是针对进入软件工程的人，因为我从来没有在博士轨道上做过学术计算机科学研究。所以，如果你更关注于 CS 学术，我希望你能发表评论，分享你对 CS 职业和你所做工作的看法！

## [](#1-know-what-impostor-syndrome-is-and-learn-how-you-challenge-it)1。知道什么是冒名顶替综合症，并学会如何挑战它

当我开始学习更高级的 CS 课程时，我周围有很多有才华的开发人员，他们在四年里都在这个专业。虽然我的课程成绩很好，但我看到我的同学在 GitHub 上与数百名明星一起做兼职项目，做令人兴奋的实习，在 StackOverflow 上解锁金质徽章。正因为如此，我感到非常落后，认为我毕业时将无法就业。高三的时候，我花了很多个晚上喝能量饮料，并试图组织一些业余项目来赶上我的同学。这让我精疲力尽，在我的实际课程中下滑，并感到沮丧，因为我看到了很多“10 倍工程师”的神话，让我觉得自己很普通。这一直伴随着我在麻省理工学院的计算生物学工程师的工作，我仍然经常处理它。

那种感觉是每个人都比你做得好，你觉得你欺骗了别人来获得你的成功，这种感觉有一个名字。这叫做[冒名顶替综合症](https://en.wikipedia.org/wiki/Impostor_syndrome)，在科技行业真的很常见。冒名顶替综合症擅长向人们撒谎，说他们能做什么了不起的事情，能为社区做出什么贡献，让他们觉得自己没有归属感。就我个人而言，这让我无法申请在会议上发言之类的事情，也无法使用最快的战略之一*到*学习新技术，尽早并经常地构建东西，因为我有很多完美主义。

然而，一些好消息是，因为这在技术领域非常普遍，即使是非常成功的人也有冒名顶替综合症，所以许多人公开讲述他们的故事。所以，如果你确实患有自闭症，要知道你并不孤独，这没什么不好意思的，也有一些自我护理的方法。你如何挑战冒名顶替综合症因人而异，所以在这篇文章的底部，我会分享一些我见过的关于 is 和在技术领域拥有更多自我同情的伟大博客文章和资源的链接。

对我自己来说，我做的主要事情是找到真正支持的社区，加入一种不居高临下的文化，这样就没有成为“10 倍”工程师的压力。我参与的一些项目有 [Go 社区](https://invite.slack.golangbridge.org)和# **CodeNewbie** 和#**coder stach**Twitter 聊天。此外，为了摆脱那种必须在一个周末在宿舍里建造一个完美架构的“下一个脸书”的固定思维模式，我尝试着从思考*我正在学习*和建造多少不同的东西的角度来重新构建我不知道的东西。这样我可以在学习的过程中有更多的成长思维。

无论你如何处理冒名顶替综合症，要知道你进入这个新的职业是很棒的，你会给科技社区带来很多，即使你感觉自己像个 00b。顺便说一下，一个秘密是，你实际上不需要知道人们谈论的每一项技术。如果一个项目使用所有这些，那听起来就像一组难以维护的微服务😉。

## [](#2-know-the-difference-and-relation-between-software-engineering-and-computer-science)2。了解软件工程和计算机科学之间的区别和联系

我在塔夫茨大学主修计算机科学，因为我知道我想开发应用程序，在大学里，我阅读了大量的科技 Twitter，以了解专业人士如何开发他们的 web 应用程序。所以令人惊讶的是，除了 COMP 40 -机器结构和汇编语言编程，核心类往往更侧重于数学，看起来不像我在推特上看到的工具。同样令人惊讶的是，当我开始在工作中编码时，我没有直接使用我在课堂上学习的许多算法技能。正因为如此，与我在网络编程中学到的东西相比，我常常要花更多的精力去理解数学的、抽象的计算机科学概念。

这些课程与在职编程如此不同的原因是因为这些核心课程教授计算机科学，但构建软件是基于软件工程的。那些领域是不同的，但它们是紧密相连的。CS 专注于使我们的计算机和软件工作的数学，例如我们如何表示数据和有效地访问它。这意味着计算机科学的工作包括设计算法和证明定理。同时，软件工程更专注于为建筑技术编写代码和架构；这是计算机科学的一个应用。

如果你发现与计算机科学的数学联系起来很有挑战性，那么学习它的用法可以帮助你获得更多关于数学概念可以解决的问题的背景知识。例如，图论是像[web pack](https://webpack.js.org)JavaScript bundler 这样的工具如何引入你的代码的依赖性，也是一些编程语言中的垃圾收集器如何知道它们可以释放哪些内存的。此外，当谈到数学时，线性代数是众多令人兴奋的领域的基础，如自然语言处理、计算机图形学和机器学习。

此外，软件工程的一个主要部分是知道如何*组合*不同的工具和代码来构建大型软件系统。因此，您正在使用软件的每个组件所使用的算法。因此，熟悉一项技术背后的计算机科学有助于您思考您的团队正在构建的系统的不同部分的数学保证和限制。在工程的大部分时间里，你不会直接证明一个定理或运行时大符号，但是计算机科学帮助你理解你的代码工作的**抽象**。如果你熟悉一个抽象，除了帮助你解决手头的问题，这种熟悉还可以将这个抽象应用到不同的问题中。

## [](#3-discover-how-you-learn-offroad)3。发现你如何学习“越野”

说到数学和代码之间的联系，因为计算机科学和软件工程都涉及到大量的抽象概念，所以你需要一种健壮的方法将每一个抽象概念和概念集成到你的整体知识中。

这在工作中尤其重要，因为在软件工程工作中，当你学习新的概念，如一个框架或一类算法时，学习通常不像在教室里那样正式和线性。不要像阅读教科书一样阅读框架文档，通常更好的做法是进入文档，了解什么问题会帮助你最快地应用你正在阅读的内容。其中一些问题可能是“使用这个包的核心概念是什么？”，“它的代码库中有哪些数据类型和接口？”，以及“我可以用这个包开始构建的最基本的东西是什么？”。

这种“越野”风格有助于使学习更加积极和集中，并让您在编写代码时更早地使用这个概念。这对工程团队来说很重要，你也可以把它应用到正式的课程中。尽管你在工作中学习的方式与在教室里不同，但在两个地方尝试如何学习会有回报。通过了解这一点，你可以利用你的学习方式更快地开始迭代你所理解的内容，从而积累更多的知识。你还可以避免被细节分散注意力，这样在你了解全局之后，你会学得更快。

以我自己的学习风格为例，我是一个非常直观、具体、动手的学习者。当我学习代码概念时，这对我意味着，一旦我看到抽象是如何在代码中实现的，以及它是如何在实践中使用的，它就真正地联系起来了。举例来说，帮助我理解 JavaScript 事件循环的是做实验来观察事件何时真正被处理，这就是本教程中的。此外，当我吸收一个新概念时，通过构建来学习对我帮助很大。当我学习在前端 web 应用程序的构建过程中使用 webpack 时，它极大地帮助了我开始为我的应用程序编写一个最小的 webpack 配置，然后看看我如何扩展它以支持在我的测试覆盖中使用 webpack 之类的事情。这就是我后来学习如何使用 webpack 的更高级功能的方法。最后，在我的笔记中，我会记录我的问题，以便找到我理解的“边缘”,这有助于我确定接下来该读什么或和同事谈什么。

## [](#4-keep-working-on-your-communication-skills-as-an-engineer)4。作为一名工程师，继续提高你的沟通技巧

在我看来，沟通是**作为一名初创软件工程师，对我帮助最大的单一技能。在开发团队中，每个人都在用不同的工具解决问题的不同方面。此外，每个人的工作都有一套复杂的权衡和细节，如果没有为什么代码库的一部分是以这种方式构建的上下文，很难理解。这就是为什么打破筒仓的沟通是重要的，特别是因为每个工程师在某个时候都必须对他们不经常工作的代码进行更改。**

除了与其他工程师交流，与产品队友交流也很重要。致力于向客户交付产品的人通常与用户工作得最紧密。所以用户反馈对决定你的产品需要支持什么功能和用户体验有很大的影响。这最终会转化为工程师们将要从事的工作。正因为如此，我经常参加工作中的产品会议以及工程会议，这样我就可以对我最终要构建的东西保持熟悉。这些信息对工程团队来说很重要，因为它让他们在解决同一问题的不同方法之间权衡利弊；您希望采用最容易支持客户所需功能的方法。

对帮助你的队友非常关键的一些交流方式是:

*   多种形式的文件；代码注释是典型的例子，但是也有描述性的 [Git commit 消息](https://chris.beams.io/posts/git-commit/)，READMEs，易于遵循的测试覆盖，以及标签中的工作描述。这些有助于解释你正在解决的问题和你正在做出的设计决策。
*   重构代码，让人们更直观地正确使用不同的函数和数据结构。
*   知道自己工作的哪些部分可以“解封”别人的工作。这有助于为更多的人完成工作，当你的队友开始使用你的代码时，你经常可以得到关于如何改进它的好的反馈。
*   在你的脑海中有一个高层次的描述，描述你正在做什么，它正在解决什么问题，而不需要太深入地研究实现细节。这在计划会议时尤其重要。

这些沟通技巧都有助于你和你的团队构建你的产品；成为 10 倍工程师的最好方法是让你的代码更容易被 9 个人使用！

如果你正在向科技行业过渡，虽然这可能会让人不知所措，但这是一个令人兴奋的变化。我希望我的建议对你成为工程师有所帮助。祝你转行好运！
