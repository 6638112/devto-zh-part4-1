### 单片 CMS

一般来说，内容，在我们的上下文中主要是博客帖子，过去是在同一个整体系统中创建、编辑和呈现的。Wordpress 是一个很好的例子，很可能你们中的很多人都曾经有过 Wordpress 博客。有一段时间，在 Wordpress 中编辑是一种相当不错的体验，但很快就发现它在展示你的内容方面做得并不好。主要问题是页面加载时间，但即使只是在同一个代码库上进行编辑和渲染，也会产生巨大的安全风险，世界各地的黑客会大规模利用这一点，并继续这样做。

如果我们看一下 Medium，我们可以看到另一个例子，说明该平台最初运行良好，但后来出现了不同类型的裂缝。Medium 提供了人们非常重视的出色的编辑体验，他们甚至承诺在某个时间点提供流量。然而，当 Medium 作为一个出版平台的利益可能与在平台上写作的大多数用户的利益不一致时，挑战就出现了。这开始于消除将你自己的域名绑定到你的博客的选项，接着是大的不要追踪弹出窗口，有些人会说结束于交通消防水管的堵塞。各种各样的媒体崩溃也凸显了对你的内容发布地拥有一定所有权的价值(例如，通过发布到你自己域名上的网站)，或者至少对你的数据拥有足够的控制权，这样你就可以用它做不同的事情。

你可以看到假设创建内容的最佳位置也是发布内容的最佳位置/方式的缺点。让我们看看近年来发展起来的一些现代替代品。

### 无头 CMS 的出现

有趣的是，我们已经见证了无头内容管理系统的出现，这是 OSS & SaaS 产品的一个类别，它纯粹专注于提供一个内容管理系统，而不关心或限制你对内容做什么。使用它来静态生成一个包含您的内容的站点，或者在移动应用程序中驱动内容更改——这对 CMS 来说都是一样的。这种拆分的最大好处之一是 CMS 不再是一个安全问题，你更接近于拥有你的内容，你可以在一个或多个你喜欢的地方发布你的内容。

无头 CMS 有两种主要风格——一种是将内容作为降价文件存储在 git repo 中，另一种更像带有 API 的数据库。如果数据所有权非常重要，您甚至可以自己托管操作系统 CMS。在这里了解顶级无头 CMS-[https://headlesscms.org/](https://headlesscms.org/)

如果您还不熟悉静态站点生成器，我们使用这些工具将来自无头 CMS 等来源的内容与模板结合起来，生成一个网站的静态副本，然后您可以将其部署到 Netlify 等服务中。您可能还想阅读 JAMstack，它是用于构建这些站点的架构。

您可以开始了解这些新的工作流程如何让您在完全不同的环境中创建内容的同时，以您喜欢的方式和位置发布内容。

### DEV 和 Stackbit

Stackbit 使得创建由各种数据源(如 Headless CMS)驱动的现代网站变得极其容易。与 DEV 的优秀人员一起，我们想尝试一些有趣的工作流，让您在编辑和发布内容的方式和位置方面有更多的选择。我们正在考虑的两种方法是:

1.  **DEV 作为一个无头 CMS**

    DEV 编辑器很棒，我们喜欢它，而且你们中的很多人已经用它在 DEV 上写内容和发布了。我们想让你有一个由你的开发内容驱动的个人网站，或者换句话说，把开发想象成无头的 CMS，在那里你管理你发布到你自己的网站和开发的内容。

2.  **从您的个人博客自动交叉发布到 DEV**

    你可以很容易地使用 Stackbit 来启动你自己的个人博客，它由现代技术和你选择的 CMS 驱动。我们希望让您在通过 CMS 创建或修改内容时，能够轻松地自动将内容交叉发布到 DEV。
