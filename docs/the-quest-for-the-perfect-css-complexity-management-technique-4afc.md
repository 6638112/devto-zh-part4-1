# CSS:一块空白的画布

对于一个完全成熟的编程语言明显占主导地位的领域，前端开发包含了数量惊人的积极竞争的软件包和方法。这对开发人员来说可能是令人沮丧的，因为了解它们的责任可能落在了他们的肩上。然而，这也是一件非常值得一看的事情，因为它证明了 javascript 作为一种语言的美丽和无限的灵活性，也证明了社区的勤奋精神。

然而，本文不是关于 Javascript 的。它是关于一个被类似的奇妙问题所困扰的实体:CSS。

CSS 规范为我们提供了一种声明性语言，本质上，理解和编写起来非常简单。但是和 javascript 一样，当这种语言被用于更大的项目时，如果不小心使用，它的简单性、灵活性和不单调会导致不合逻辑的、迷宫般的代码库。设计一个管理复杂性的策略是完全由开发人员或者开发团队和社区承担的责任。

这就把我们带到了出发点。

一个孤独的开发人员，或者开发团队，在为非平凡规模的项目编写 CSS 时，如何避免不可管理或者不必要的复杂性的出现？在技术不同的地方，每种技术的优点是什么？

直到最近，还没有正式的系统来帮助开发人员管理 CSS 的复杂性。在过去，样式表作者只受他们自己对 CSS 规范的看法的指导，以及由他们的同事培养的非官方的、自然产生的最佳实践的指导——通常在书籍、文章和聊天室中阐述。

今天，有几个正式的 CSS 组织系统存在。大多数同样是非官方的。其中包括 BEM、OOCSS、SMACSS、MCSS、AMCSS、Atomic CSS 以及被不敬地命名的 FUN CSS。BEM 和 Atomic CSS 是特别有趣的研究，因为它们是最流行的(或者最臭名昭著的，取决于你问谁)，也是最非传统的。BEM 与我们的调查特别相关，因为它是专门为驯服大量 CSS 代码库而设计的。

考虑到这一点，这篇论文将考察两种 CSS 组织实践——一种被归类为一般规则，另一种被归类为正式系统。
