# TDD 是救世主吗？

> 原文：<https://dev.to/d3zibel/is-tdd-the-savior-1620>

首先，我想让你在读完我的帖子后和我讨论一下我的观点，因为这只是我今天想法的一个片段。

在我参加的几乎每一个会议上，至少有一个关于 TDD 的讲座,“大公司”是如何处理它的，当你让所有的开发者都明白，他们真的生活在 TDD 中时，这就像是伊甸园。当我在初级开发人员阶段听到所有这些讲座时，我对这些人感到惊讶。我认为在一个热爱在实际代码之前编写测试的团队中工作一定很棒，因为代码不能有任何错误。

所以，当我从关于 TDD 重要性的第一堂课回来时，我写测试的动机增加了大约 1000%。但是当我在公司开始我的第一个项目时，写单元测试对我来说太难了。尽管我现在知道了如何做的理论...在实践中写它们是完全不同的事情。我试了又试，花了我那么长的时间才为非常简单的函数编写了几个测试。对于我写的每一个失败的测试，我都会问自己:“是测试错了，还是我的函数没有达到我的预期”。所以每次考试都伴随着挫折和漫无目的。

从那以后，我**讨厌**写测试。此外，我不再对 TDD 这个话题感到惊讶了。

随着我在不同语言方面变得越来越好，我的编程技能本身也变得越来越好，为我的函数和类编写一些测试似乎不再困难。语法和语义现在看起来很舒服。我不能单独测试最复杂的功能，但这没关系，因为我有我的高级开发人员在那里，我可以问。

虽然我现在是团队中最年轻的成员，但我似乎是对新项目中使用的新语言最有经验的一个。感觉就像我每天的任务一样，每次拉请求或者结对编程时，我都会问“你已经写好单元测试了吗？”。此外，由于我最近重构了很多代码，我写了很多测试，当我看到结果时，我真的起鸡皮疙瘩。我为自己感到非常自豪，现在我可以轻松地编写测试来防止我们的应用程序因为代码更改而出现错误。当然，我为我所经历的旅程感到自豪。

并且*确切地说*故事中的这个转折让我想到了我对项目中 TDD 的看法。我认为，告诉一个有初级开发人员、高级开发人员以及介于两者之间的人员的团队，TDD 是可行的方法是没有意义的。初级开发人员往往会感到沮丧，因为他们不能总是向高级人员求助。高年级学生感到沮丧，因为他们不能做他们通常能做的工作。

因此，在这样一个混合团队中，最好的方法可能是在编码**和**代码评审之后编写测试。因为那是通常你的代码应该做你期望的事情的时候。最重要的是，您可以防止代码中的错误，因为您的团队可以随时检查代码在大量测试中是否仍然有效。

在我看来，每个人都会经历和我写测试一样的旅程。关键是，对他们来说会有多难？

让我知道你对此的看法或者你在职业生涯中是否有过类似的经历！:)
