# 有人能帮我理解一下 React 设计模式的发展方向吗？

> 原文：<https://dev.to/omrisama/can-someone-help-me-understand-the-direction-react-design-patterns-are-moving-in-im2>

在钩子成为真正的东西之前，我在 React 中经历过的最权威的设计模式是“用封装表示组件的容器组件来还原”模式。基于类的(有状态的)组件和功能性的(无状态的，表示的)组件之间有非常明显的区别。你会被鼓励“提升状态”,并且——在我看来——在一个容器组件中存储尽可能多的状态，这样它最多可以向下钻取 2 或 3 层到表示组件。比这更多的层次，你可能会看到一个新的有状态组件。当然，所有的类/有状态组件都连接到 Redux store。

然后钩子出现了，“哪些组件应该或者不应该有状态”之间的界限变得非常模糊。我们甚至不要提背景。
题外话:我甚至不确定 Context 是否被认为是一个足够稳定的解决方案，可以完全取代 Redux 用于各种规模的应用程序。在这一点上，上下文使用看起来像是蛮荒的西部，人们试图像 Redux 一样使用它，人们围绕它构建像`reactn`和`constate`这样的库，并且将每个 Redux reducer 用作单独的上下文或其他东西。谁知道呢。

钩子应该鼓励我们通过编写更多有状态的组件来分散状态吗？或者它们只是被认为是编写有状态组件的新范例，大型集中式有状态组件的想法不会消失？
