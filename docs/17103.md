# 何时使用共识决策

> 原文:[https://dev . to/wlindley/when-to-use-consensus-decision-making-2bh 6](https://dev.to/wlindley/when-to-use-consensus-decision-making-2bh6)

我们应该如何做决定？几周前，我帮助我的团队决定了一个新的、更结构化的计划和工作流程。以前我们一直以一种相当特别的方式工作，希望我们在一起工作时有更多的结构和严谨性。它为我澄清了共识过程何时有效，何时无效。我想和你分享我学到的东西。

# [](#kinds-of-power)力量的种类

让我们从不同种类的权力的一些快速定义开始。这些不是唯一的种类，但它们是我要关注的。

*   角色权力是你从你在组织中的职位获得的权力和影响。随着你的升迁，你的角色权力也会增加。
*   关系权力来自于你和同事之间随着时间的推移而建立的人际关系
*   专业能力来自于你在该领域的经验。

# [](#setting-the-stage)设置舞台

我的团队在那一周实际上做了两个重大决定，它们之间的差异非常明显。让我们依次看一看:

第一，流程决定。我是一名高级工程师，加入我的团队不到一年，但我不是经理或技术主管，所以我在团队中没有角色权力。我不能，也不想，只是告诉我的团队我们将改变我们合作的方式。相反，我想让整个团队参与决定我们的新流程。我首先写了一份建议书，提前几天发给团队，这样每个人都有机会阅读。然后我主持了一次会议，我们在会上讨论了我们希望我们的流程是什么样子的。我们确保每个人都理解新流程的每个部分，并对此感到满意。我们还确保每个人都发言、提供反馈、提出他们的任何顾虑，并确保这些顾虑得到解决。一个小时后，我们定义了一个新的团队流程，每个人都参与其中。共识过程非常成功！

第二天，我的团队需要做出一个技术决策，决定我们要用哪两种方法来实现我们正在构建的系统的一部分。有人建议使用我们前一天使用的同样的协商一致程序。我们在白板上写了一个快速提案，并开始讨论利弊。我们大多数人都倾向于同一个方向，但没有人想做出坚定的决定。因为这个决定会影响其他团队，所以不清楚谁会将这个决定传达给他们，或者传达给我们团队的更大的组织。如果我们被拒绝，谁会回应呢？我们的时间表有什么变化吗？团队中的任何人可以做出改变交付日期的决定吗？有许多未回答的问题，在做出决定之前，谈话逐渐停止了。我们停滞不前了。协商一致进程让我们失望了。

这些决定有什么不同？为什么一个比另一个更容易制作？为什么一种情况会导致结盟和认同，而另一种情况会导致混乱和缺乏参与？

想到这些，我想起了我曾经工作过的一家初创公司。这家初创公司正在尝试扁平化的结构，以及在没有明确角色权力的情况下运营公司的方式。我们希望关系力量和专业力量足以将公司团结在一起，有效地工作。

我们花了很多时间思考在没有明确负责人的情况下，群体如何做出决策。通过大量的反复试验，我们发现以这种方式做决定总是缓慢而痛苦的。然而，这些决策经常会产生团队的长期认同和支持。其他时候，我们只是不停地讨论问题，从来没有真正做出决定。这导致了我们在原地打转时的浪费时间和麻痹。就像我的团队所经历的一样。

# [](#when-to-use-consensus)何时使用共识

当我回想我在以前公司学到的东西时，一些模式开始浮现。当您需要团队的高度支持，并且您有时间为成功的决策会议奠定基础时，当您在缺乏角色权力的团队中领导变革时，以及在团队不确定谁可以或应该做出决策的情况下，共识决策非常有效。

当团队不确定谁应该做决定时，达成一致的过程可以有效地决定谁应该做决定。这种方式很有效，因为团队成员通常都认为做出决策很重要，并且通常能确定一两个可以信任的人来听取每个人的观点。要做到这一点，团队需要支持这个人。

或者，当你没有角色权力或者不想使用角色权力时，达成一致的决策可能是从你的团队获得高水平支持的有效方法。这两种情况非常适合达成共识，也是我唯一推荐达成共识的情况。

# [](#how-to-use-consensus)如何使用共识

当您决定使用共识决策时，我建议遵循以下步骤:

1.  首先，写一份形势总结，几个选项，以及要做的决定。这减少了歧义和混淆的可能性。
2.  提前几天将总结发给小组成员。这给了那些喜欢私下处理信息的人一个思考和形成观点的机会。它也给人们一个机会去做他们想做的任何其他研究或准备。
3.  在决策会议期间，确保每个人的声音都能被听到。特别是，团队中可能有更安静的人，你可能需要明确地询问他们的意见。做笔记或实时更新文档以反映谈话内容。
4.  一旦对话慢下来，询问每个人是否支持这个决定。记录任何未回答的问题或未解决的问题。
5.  之后，跟进所有不同意的人，与他们一起更新文档，直到所有人都同意为止。
6.  最后一次发送文件，并要求每个人提供最后确认。这再次给了具有不同处理速度的人一个机会来回顾每件事，并对结果充满信心。
7.  最后，将决策记录在某种文档存储中，这样团队可以很容易地引用它。

如果以上听起来很多，确实很多。这只是简单的事情。此外，特权、阶级和其他类型的权力差异也会产生复杂的动力。如果一切顺利，这个过程可能需要一两周，如果不顺利，可能需要更长时间。这是一个开销很大的昂贵过程。但它也让每个人都参与到过程中，并认同结果。大多数决策无法证明这种投资水平的合理性，这也是我在大多数情况下不推荐这种投资的原因。但是，只有极少数决策可以真正从这种方法中受益。

# [](#when-to-avoid-consensus)何时避免达成共识

大多数时候，基于共识的决策是错误的选择。一般来说，由一个人负责决策会更有效。至关重要的是，这并不意味着这个人在真空中做决定，他们仍然应该从周围的人那里寻求建议和反馈，权衡选项，做出决定，与团队沟通，并在事情出错时对后果负责。此外，这允许其他人专注于其他工作，使团队更加有效。

相比之下，共识决策的成本。每个人都必须充分理解参与决策的背景。一旦每个人都有了这样的背景，他们必须保持同步，直到做出决定。而且决定做出后，还是要记录下来，用大家理解和认同的方式传达给大家。

由于共识过程的成本和复杂性，我建议在绝大多数情况下都有单独的决策者。当有疑问时，小组的结构应该有助于明确谁是决策者(行动中的角色力量！).请记住，决策可以授权，事实上这是经理的主要工作之一。

# [](#conclusion)结论

改变我们团队流程的决定受益于共识过程，因为我们愿意投入时间，并希望得到长期支持作为回报。而技术决策需要快速做出，并且已经有了明确的决策者。希望这能帮助你更好地理解什么时候使用共识过程，什么时候避免它们。如果你过去使用过共识决策，请在下面的评论中分享你的故事。
