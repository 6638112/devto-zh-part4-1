我准备了一些指导方针/问题，让你自己在评估新技术(框架、库、数据库等)的过程中有所帮助。)对于一个项目来说。

终极问题:*你真的需要这个新东西吗？*

# [](#questions-to-discuss-with-your-team)与你的团队讨论问题

1.  这真的是一个团队的决定，还是我们应该问问其他团队和公司内部的人关于使用它？
2.  你为什么需要它？你想解决什么问题？它是这项工作的合适工具吗？为什么值得？(错误答案:性感，只是尝试新事物)
3.  会带来什么商业价值吗？你能描述一下吗？
4.  它是否有助于更快地交付项目，或者使项目更易维护，或者质量更好？
5.  投资回报需要多长时间？
6.  为什么不能使用现有的堆栈来解决这个问题？
7.  写自定义库，当开源软件里有东西存在？为什么？
8.  如果因为性能更好而想用的话。你真的需要那场表演吗？你能出示数据证明它能证实更好的性能吗？(例如，反应式框架。我们是在为今天的流量做准备，还是在预测长期可能发生的可扩展性问题？)
9.  有多受欢迎？招人容易吗？是不是有大公司做后盾？很容易找到使用它的人吗？关于栈溢出有多少问题？LinkedIn 上的 CV 里有多少人？等等。
10.  你的公司里有其他人在使用它吗——他们的用例与你的用例相比如何？
11.  很容易找到帮助吗？还是你要自学？有多成熟？
12.  有学习曲线吗？好学吗？你的团队能够指定它吗？你认为和它一起工作的团队会舒服吗？
13.  你过去在生产中使用过它吗？
14.  还有其他公司在生产中使用它吗？
15.  您是否体验过它，创建过示例应用程序，并对此做了一些笔记？
16.  这是一个新事物吗，是你项目中的另一个新玩具吗？工具太多很难维护。
17.  你的团队已经对新工具有经验了吗？
18.  您在生产中运行的服务很少吗？在我们的堆栈中添加非常不同的工具真的有意义吗？
19.  它将如何影响你的团队生产力？在你的团队或公司内部寻找帮助容易吗？如果你是团队中唯一知道的人...避免它可能更好。
20.  成本估算，与当前解决方案相比，新解决方案更便宜/更贵吗？

记录下来。

# [](#good-3rd-party-libraries)好的第三方库

(深受来自[的](https://twitter.com/maneo)[演示](http://dudczak.info/presentations/2017_03_jugademy/#/29)的启发)亚当·杜扎克

1.  有文件证明
2.  可以访问源代码
3.  项目内部的大活动和稳定的发布周期
4.  有一个以上的维护者
5.  有集成的测试或 CI 工具(检查它们和覆盖率)
6.  将 lib 与您的测试堆栈集成起来很容易
7.  是关于 github stars 库的 top 5——建议查看其他类似的库，有时候 top 1 并不是最好的一个
8.  检查问题页面。有什么关键问题吗？(比如:可以用 uuid 库生成相同的 uuid)早就发布的 1.0.0 版本(还不如等别人给你查 bug)。已知的未知对未知的未知。小心使用新工具。或者准备为此作出贡献。
9.  通过 OWASP 依赖检查会很好，SonarQube 有它的插件[https://github . com/SonarSecurityCommunity/dependency-check-sonar-plugin](https://github.com/SonarSecurityCommunity/dependency-check-sonar-plugin)
10.  许可证(仔细看 GPL！)
11.  有官方 docker 图片
12.  很容易找到帮助、stackoverflow、github 问题等。
13.  它有 IDE 支持
14.  它与您的环境或框架(如我们的监控、linters、CI 等)集成得很好。)
15.  它有干净的 CVE 数据库

# [](#new-framework-new-database-new-language-etc)新框架、新数据库、新语言等。

1.  如果您想让它用于概念验证，请按照与生产相同的方式编写(通常情况下，概念验证是在生产中进行的...)-但无论如何不建议做生产用
2.  检查“良好的第三方库”部分的所有内容
3.  创建具有几个端点的示例应用程序，请记住:
    *   配置管理
    *   使用公司内部使用的工具进行监控、跟踪和记录
    *   像 OpenAPI 这样的 API 文档
    *   CI/CD，部署到沙箱
    *   添加棉绒
    *   建筑艺术品
    *   存储连接
    *   机密管理
    *   依赖性管理
    *   代码模块化
    *   编写任何类型的测试，单元，集成，e2e，验收，有外部服务
    *   尝试分析/调试您的应用程序
    *   尝试进行样本负载测试
4.  在开发期间，尝试创建“每小时 WTF”日志。(开发过程中发生了奇怪和令人困惑的事情)
5.  做一些笔记/结果，什么是好的，什么是坏的
6.  另一份清单。这个新事物:
    *   编写测试容易吗？有合适的工具吗？当您调用后端的一个端点并模仿外部服务时，编写一个测试应该很容易
    *   监测和跟踪代码很难吗？
    *   [“允许你写人类可读的代码？，生成针对其运行环境优化的代码？，有助于减少应用程序的大小？，允许代码可移植和/或可扩展？，确保我正在编写的代码的寿命？，提供方法来轻松测试它的 API 是否正常工作？”](https://dev.to/steveblue/why-i-ignore-the-hype-and-you-should-too-332p)作者 [@steveblue](https://dev.to/steveblue)
    *   有好的调试/分析工具吗？
    *   创建模块化代码容易吗？
    *   你能把负载测试和生产中运行的东西进行比较吗？
7.  在团队内部和团队外部讨论结果

# [](#adding-new-technology-to-production)将新技术加入生产

1.  成本估算，与当前解决方案相比，新解决方案更便宜/更贵吗？
2.  列出你想达到的目标，以便检查结果
3.  添加适当的监控、跟踪和良好的指标。做一些总结来检查你的假设。
4.  如果可以的话，创建一个 WTF /每小时/每天日志，尝试记录任何困难的情况-事件管理
5.  如果可以，创造“是工作吗？”记录证实或不证实你的假设的东西
6.  每季度做一些总结

# [](#the-end)结束

通常我们可以使用现有的工具来解决问题。向你的堆栈中添加任何新东西并不总是值得的。大多数框架都是非常相似的，但是细节很重要，通常你不想一个人在生产中发现新的错误。新技术应该证明将来会有很好的价值回报。
