# 开发者博客应该接受同行评审吗？

> 原文:[https://dev . to/dez fowler/should-developer-blogs-be-peer-reviewed-5071](https://dev.to/dezfowler/should-developer-blogs-be-peer-reviewed-5071)

通过个人博客或像 DEV 这样的网站与更广泛的开发人员社区分享您的想法、发现和观点是一件非常有趣的事情，而且从未如此简单。

权力越大，责任也越大，对于每一个最新的、经过充分研究的帖子来说，还有许多其他的不太有用的信息，往好里说是一些过时的信息，往坏里说可能包括一段代码，或者倡导一种方法，如果你对它采取行动或复制/粘贴错误的东西，可能会给你的代码留下严重的漏洞。

昨天在研究某个问题时，我不得不通读三篇文章，其中包含了一个问题的错误解决方案，然后才找到一篇包含正确解决方案的文章，在这个过程中浪费了很多时间。这是 15 年的经验，知道什么是正确的解决方案——对于初学者来说，这是一个真正的问题。更糟糕的是，在错误答案的帖子上，人们说“谢谢，这正是我要找的”的评论和正确答案的评论一样多。

这不仅仅是小博客，甚至在 MSDN 也曾经有(可能现在仍然有)许多帖子，其中包括含有 SQL 注入漏洞的样本，这些漏洞肯定会被复制/粘贴到全球的代码库中。新的[docs.microsoft.com](https://docs.microsoft.com)现在允许你在每个页面上建议编辑和反馈，这是一个巨大的进步。

其他人感受到同样的痛苦了吗？我们能做些什么来解决这个问题并提高我们共享信息的质量呢？

我在考虑一些自愿的行为准则，包括以下内容:

*   确保帖子包括日期和他们谈论的框架或库的版本号
*   作者的一些承诺，随着事情的变化保持帖子的更新——如果帖子过时了，可以有一个标准的方法在顶部标记它
*   多年后，旧帖子仍会出现在谷歌搜索结果中，因此保持评论开放，让人们可以标记一些东西
*   效仿其他职业，在发布前对帖子进行同行评审，就像对拉式请求进行代码评审一样

堆栈溢出已经很好地解决了这些问题。它有许多框架的版本特定标签，如果你想提出改进建议，允许你对其他人的问题和答案进行同行评审编辑，并且允许问题被标记为重复。

这样做的明显缺点是，它使事情变得更正式，更无趣，这也是为什么栈溢出被认为比 DEV 和其他人更不友好的部分原因。

你怎么想呢?
