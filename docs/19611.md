# 为什么大多数开发人员害怕 CSS

> 原文：<https://dev.to/steffenpedersen/why-most-developers-fear-css-3h99>

当我开始我的工作时，我们团队有两个前端开发人员。是我和一个资深前端开发者。后来那个资深前端开发者辞职了，我是我团队里唯一的前端开发者。一切都很好。我做了我的工作，没有伤害。我们参加了一些工作面试，但没有找到合适的人选。随着时间的流逝，我突然意识到:我是唯一一个了解我们前端的人。总线系数为 1，在这种情况下这不是一个好数字。为什么？如果我被一辆公共汽车撞了，这个项目的情况将会很糟糕。

然后我和我的老板谈了谈，我们达成了一致，我们都应该负责前端。这项工作可以结对编程来完成。至少在我们找到合适的配型之前。我觉得这是个好主意。我一个人做前端工作累坏了。

然后我们提出了这个想法，第一反应是来自其他开发者的恐惧。最令人担心的是 CSS。

## 为什么开发者害怕 CSS？

我和其他开发人员讨论了他们害怕 CSS 的原因。首先提出的是层叠😵如果他们对一个类做了改变，会影响到其他 500 个元素呢？

接下来提到的是对浏览器的支持💻如果他们实现了某些浏览器不支持的东西怎么办？

最后一件事是从头开始建造一些东西🏡如果他们做的东西不结实，看起来不好看怎么办？

### 层叠😵

这是最大的恐惧之一。其他开发者在说这个的时候，我猜他们在说很多东西。重要性、特异性、源顺序以及属性如何从不同的规则继承都很重要。样式可以在许多不同的地方指定，并且可以以复杂的方式交互。这就是 CSS 如此强大的原因，但也是它令人困惑和困难的原因。如果你不知道选择器如何否决其他选择器的规则，级联可能是一件可怕的事情😈

#### 重要性

这是一个讨论得很好的话题。我们都知道，使用这个小助手(在大多数情况下)是不好的。这是因为`!important`总是会赢🏆

#### 特异性

专一性基本上是衡量一个选择器有多专一的指标。就像，内嵌样式是最具体的，然后是`#id`，然后是`.class`，然后是`element`。由此，特异性可以用一百万种不同的方法来衡量。如果你有疑问，你可以使用特异性计算器。

#### 来源顺序

源顺序基本上是后规则会战胜前规则。如果你用一个属性和一个值写一个选择器，然后用同样的属性写同样的选择器——但是用不同的值。接下来会发生什么？🤓

### 继承

CSS 中的继承既简单又复杂。可以这样解释:*应用于一个元素的属性值有的会被该元素的子元素继承，有的不会*。听起来是不是很棒？😵像`font-size`和`color`这样的 CSS 属性会继承，而像`margin`和`padding`这样的属性不会继承。我想这一次你得运用常识了。如果`margin`被继承，将会引起巨大的混乱！您可以使用 [MDN](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference) 作为参考。

### 浏览器支持💻

除非你客户是一家只支持 Internet Explorer 6 的公司，否则你不必太担心。大多数用户使用的现代浏览器都非常支持 CSS。如果你对是否应该使用某处房产感到不确定，那么你可以在 [caniuse](https://caniuse.com/) 查看一下。如果一个浏览器不支持该属性，但你真的想使用它，那么大多数浏览器`@supports`同名的 CSS at-rule。然后你可以制定一个后备方案。

```
@supports (display: grid) {
  div {
    display: grid;
  }
} 
```

### 从地面建筑起某种东西🏡

这主要是设计师和开发者之间的交流。沟通并不总是那么好。有时候，相对于公司的钱包和开发人员的技能，或者实际可能的东西，设计师的野心可能太高了。

关键是，作为开发人员，我们应该更多地参与设计过程，这样我们才能对结果感到满意。也就是说，如果开发人员对设计感到满意，那么从头开始构建会更容易。
