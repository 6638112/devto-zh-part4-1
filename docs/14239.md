# [](#github-builtin-protection)GitHub 内置保护

如果您是存储库管理员，那么在存储库设置中，您将能够找到**交互限制**部分:

[![Interaction Limits Section](../Images/11f6ea5f3a8f51e97e1325460d1e3485.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--ykynxWsu--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://i.ibb.co/FwBhdGr/Screen-Shot-2019-07-10-at-11-41-45-AM.png)

在这个部分中，您可以为您的存储库配置临时交互限制:

*   限于现有用户
*   对以前贡献者的限制
*   仅限于存储库协作者

让我们逐一讨论。

## [](#limit-to-existing-users)限制现有用户

*限制现有用户*选项允许您禁止最近创建的用户与存储库交互。

在我的例子中，我们有许多不同的用户是在几个小时前创建的。根据他们的名字，我认为这是自动生成的用户。这就是为什么启用该选项停止垃圾邮件和打扰我。也作为所有看我们库的人 **~1000** 。

## [](#limit-to-prior-contributors)限于之前的贡献者

在这种情况下，我们可以禁止那些从未提交到存储库主分支的用户与存储库进行交互。这种选择是不可接受的，因为我们仍然需要允许我们的社区报告错误，并帮助我们处理他们的请求。

## [](#limit-to-repository-collaborators)限于存储库协作者

*限制存储库协作者*选项只是禁止不是该存储库中协作者的用户与该存储库交互(未授予他们推送访问权限)。这种选择对于开源项目来说也是不可接受的，因为我们需要允许我们的社区与资源库进行交互。

## [](#what-should-i-do-as-a-result)结果该怎么办？

显然，冷却垃圾邮件的最佳解决方案是启用对现有用户的*限制*选项，这将禁止最近注册的用户与存储库交互。

但是所有这些选项只能启用 24 小时。而且这是一个点，我们不能永远禁用最近注册的用户的交互。这就是为什么每当有人决定在我们的存储库中创建一些广告问题时，我们必须手动启用它。

好了，我们已经禁用了与这些垃圾邮件存储库的交互，但是存储库中仍然存在 **3000+** 垃圾问题。怎么处理他们？我们有以下选择:

*   关闭所有 **3000+** 问题
*   删除所有 **3000+** 问题

## [](#close-them-all)全部关闭

在这种情况下，关闭问题是不可接受的，因为我们将再次为所有存储库的观察者触发相同数量的电子邮件😳-糟透了。

## [](#delete-them-all)全部删除

删除问题不会触发任何电子邮件通知，因此，它可以很快完成。但这里我们有一个问题:问题只能通过 GitHub web 客户端逐个删除，我们不能对所有标记的问题进行批量删除。更重要的是，GitHub REST API 没有为问题移除提供合适的端点。

这就是为什么现在唯一的方法是向 GitHub 支持寻求帮助，然后他们会移除所有不合适的问题。
