# 通过自述文件启动项目

> 原文：<https://dev.to/vepo/start-a-project-by-the-readme-4c49>

在过去的几个月里，我花时间学习如何写一份好的文档。

我参与的几乎所有项目都缺乏文档。如果你已经尝试开始开发/使用任何项目，你可能不得不面对知道从哪里开始的问题。

因此，这里是我的方法，如何建立一个良好的记录项目

# 1。开始描述你的项目

要矮！用几行文字描述你的项目。

## 好例子

一个很好的例子是[动物园管理员](http://zookeeper.apache.org/)描述:

ZooKeeper 是一个集中式服务，用于维护配置信息、命名、提供分布式同步以及提供群组服务。

在第一行，你知道这个项目能解决什么问题。如果你继续阅读，会有更详细的描述。很好！

## 坏榜样

一个不好的例子是[卡夫卡](https://kafka.apache.org/intro)描述:

Apache Kafka 是一个分布式流媒体平台。那到底是什么意思？

文档很清楚，读者不会不理解！！如果你继续阅读，它将描述*实时流水线*、*集群*等...但是卡夫卡的主要目的是什么呢？

如果我不得不重写卡夫卡的文档，这将是它的第二个项目。第一部分将简短描述**什么是**卡夫卡和**它解决哪种类型的问题**。

## 提示

1.  避免使用自己定义的术语。使用公共术语
2.  简短些。少而短的线条。
3.  回答问题。

## 简述...

一个好描述是一个回答两个问题的简短文本。什么是解决什么类型的问题。

# 2。描述完你的项目后，解释隐藏的想法

我的读物的第二部分是理性的部分。在这里，我快速列出了我的代码背后的所有想法。

让我们想象我们正在建立一个新的 NoSQL 数据库。

用户首先想到的是:*我丢失了所有的 SQL 验证？*

开发人员首先想到的是:*我们可以添加某种验证吗？*

这两个问题都是受欢迎的，但在本次会议中可以用一句话来回答:

*   这实现了高吞吐量，数据库不会进行任何验证。

任何重要的想法**都应该**在这个环节中简短描述。如果需要更多信息，您可以添加另一个自述文件链接来解释它。

# 3。解释资源和术语

你的项目定义了术语吗？这里解释一下。

你的项目使用自己的文件格式吗？这里解释一下。

假设我正在构建一个业务规则引擎，所有业务规则都必须以特定的文件格式编写。您应该简要说明这是什么文件格式，并为其创建一个特定的自述文件:

*   BR 文件定义了一个业务规则。(更多信息链接)

# 4。解释行为和互动

解释你的代码是如何工作的。用户或开发者应该知道你的项目是如何工作的。如果您正在开发一个消息队列，这里是一个很好的地方，您可以在这里简单地描述一个消息将如何被传递到所有(或者不是所有)客户端。

# 5。描述你提供的 API

你们提供 REST API 吗？Java 库？一个 C 库？描述所提供 API 的要点。很快！

# 要短！

在主自述文件中，你**应该**很快就会。这不是你将深入解释一切的地方，这是展示关键概念的地方。

# 要一致！

如果对代码的修改改变了与自述文件相关的内容，该怎么办？这是软件开发中出现的主要问题。文件与规范不一致。

对于每个项目，你应该定义你自己的**完成**的定义。如果变更没有反映文件，那么变更就没有准备好！

# 总结性的

当你听到*文件是代码*时，不要碰这个代码！每一行代码都有一个隐藏的想法，有时没有表达出来，因为它不能用代码表达。
