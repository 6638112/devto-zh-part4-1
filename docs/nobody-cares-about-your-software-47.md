# 没人关心你的软件

> 原文：<https://dev.to/piotrmurach/nobody-cares-about-your-software-47>

我不想打断你，但是很可能没人关心你的软件。我不是想让你沮丧。这听起来很无情，但大多数程序员都很忙，没有时间阅读你的开源项目代码来理解它做什么，如何使用它以及它的意义。

大多数偶然发现你的项目的开发人员都受雇为他们的公司开发软件。他们不希望用严厉的评论把你的项目撕成碎片，或者向全世界宣布它有多糟糕。他们只是肩负完成任务的人。不管你是否同意我的观点，理解他们的观点可以帮助你欣赏使用你的软件的人，并把你的项目提升到另一个层次。

## 创造者心态

作为创作者，你在项目上投入的时间越多，就越难做到客观。你越来越喜欢你的项目，并相信其他人也应该如此。你如何使用变量、函数和模块会让你的同行嫉妒。你的工程技术在你工作的公司是无与伦比的。您的代码设计实现了美观和速度的完美平衡。

那又怎样？

许多软件程序员倾向于相信他们的工匠软件对编程大众来说是不言而喻的。我自己也曾为此感到内疚。为什么要麻烦文档呢？哼！教程是为门外汉准备的。任何让他们远离编写软件的事情都是浪费时间。但愿不会如此，推广他们的项目无疑是向资本主义上帝出卖自己。

拥抱不是一种非常有成效的态度。为什么要让这些工作漂浮在开源软件的汪洋大海中呢？如果你写了很棒的软件，但是没有人使用它，那么写它还有意义吗？

## 新鲜的视角

大多数开源项目的困境遵循一种模式。一个开发人员发现了一个像四叶草一样的软件项目。他们发现它很有用，但并不完全符合他们的需求。所以他们决定报告一个问题，要求软件做得更好、更少。你最初期望的欣赏和崇拜都没有实现。谈谈对自我的打击吧！

这些年来，我了解到无论你的项目看起来多么令人兴奋，大多数程序员关心的是它是否解决了他们的问题。他们希望看到立竿见影的效果。就是这样。如果你的软件几乎解决了他们的问题，那么他们可能会提出改进的想法。如果你非常幸运的话，他们甚至会抽出时间贡献一个补丁。这种认识帮助我改变了我的态度，从一个关注解决方案的开发人员转变为一个更加关注人的开发人员。

那么怎么做才能增加胜算，让大家关注呢？

## 解释和连接

在你自己的头脑中，你投入无数时间的项目是有意义的。但是其他人呢？如果这个项目对其他人有用，花时间在自述文件中解释它的效用似乎是一个好的起点。首要问题可能是:为什么会有人关心这个？在这里你可以说明你的想法的来源。目前的艺术水平是否让你感到沮丧？对我来说，使用其他(通常复杂得多)解决方案的挫败感是创建一个新项目的最有力的理由。同样，有些项目纯粹是为了满足我的好奇心而存在，开发起来很有趣。我用 Ruby 编写了 [lex 实用程序](https://github.com/piotrmurach/lex)来学习编译器 lexing 是如何工作的。不管什么原因，解释一下。

你还能讲什么故事？例如， [tty-command](https://github.com/piotrmurach/tty-command) Ruby 项目有一个激励部分，讲述了一个初创公司创始人的故事，他不得不处理数百个难以管理的 Bash 脚本，然后用一种可读性更好、更优雅、更有趣的语言 Ruby 重写它们。故事是你的项目和另一个开发者之间最快最强的沟通纽带。它使你的解决方案与众不同，并展示了其背后的深思熟虑。

一旦你解释了你的项目的背景，你可以接着问一个问题:我的软件有什么独特之处，别人应该跳过看黑客新闻？我经常从开发人员的角度用他们的语言写下一个特性列表，以帮助他们理解好处。例如，如果他们很可能熟悉活动记录模式，我可能会用它作为类比来解释使用某些 API 调用的简易性。

## 帮助开发者解决他们的问题

思考之前问题的答案会让你进入一个新的思维框架，提升你对项目的认识。这些答案可能会产生滚雪球效应，让你在启蒙树上走得更远。一旦你解释了你的项目存在的原因，以及它与其他已经存在的解决方案有什么不同，是时候考虑与你的软件用户的接触程度了。一些好问题包括:

*   我的项目如何向第一次使用的用户解释功能？
*   与高级开发人员相比，我如何向编程初学者解释事情？
*   我如何帮助开发人员成功使用我的项目？
*   我如何证明我的项目的感知效用？
*   我如何激励贡献和更高层次的参与项目？
*   愿景是什么？项目一旦成功，会是什么样子？

这些示例问题帮助并继续激励我为我的许多开源项目编写文档。例如，我喜欢添加一个内容菜单，让查找所有可用的 API 调用变得轻而易举。tty-command 项目将菜单分为基本的和高级的界面元素。第一次使用的用户可以找到他们需要的东西，而不会对所有的功能感到不知所措。同样，高级用户可以快速了解他们可以调节的所有额外旋钮。

不管他们的经验水平如何，告诉开发人员如何安装您的项目，并展示如何开始的最简单的例子，以便他们可以快速享受运行代码的荣耀。更好的是，录制一个演示。tty-reader 项目显示了我编辑文本行的动画，以展示可用的功能。因此，你的项目越简单，API 越直观，越容易理解，越好。我经常创建一个包含常见用例的示例文件夹，以便快速体验该软件的功能。当然，如果我说我知道如何完美地回答这些问题，那我就是在撒谎。潜在的问题很深，需要你去探索。

但是有些问题不值得回答。

## 把什么东西给漏掉了

通过贬低其他解决方案来解释为什么你的软件要好得多是一个失败者的游戏。你可以抗议。但是等等，其他项目要糟糕得多。人们当然应该知道。不尽然，原因如下。没有人试图构建复杂、错误和不可靠的软件。一系列在当时看起来很棒的设计决策使项目僵化。与我们的初衷相反，随着时间的推移和多次重构，我们的软件可能会变得面目全非。它的外表和行为不像一只在水上滑翔的雄伟天鹅，它的行为更像一只在垃圾箱里搜寻的浣熊；它正在工作，但不是很顺利。

无论你写什么文档，重要的是不要和人们玩捉迷藏，不要把关键信息隐藏在营销口号下。程序员是有洞察力的，你不想过早地打击他们。给他们省点麻烦。相反，考虑提供由基准和例子支持的公正信息。一旦程序员满足了他们的需求，他们可能会考虑将你的软件用于其他用途，并让他们的开发伙伴知道这一点。在程序员圈子里，口碑是一个强大而被低估的渠道。

如果你希望你的开源项目得到认可和重视，你需要努力解释它的目的。不要指望某个著名的开发人员会洒些星尘来保佑你的项目。它必须有自己的价值。我编写并发布了许多 Ruby 包。只是在最近，我注意到人们开始关注并在他们的代码中使用我的库。可悲的是，赢得许多程序员的奉承和芳心往往说起来容易做起来难。事实是，即使是最好的软件也不总是能吸引到应有的关注。

## 坚守流程

没人知道你的项目的好处是它让你有自由去做正确的事情。那么，做正确的事情会是什么样子呢？你以长远的眼光参与重构你的代码。你修好了所有的破窗户。这可能意味着用一连串简单的方法调用代替笨拙的对象继承。您花时间润色现有功能，以处理极端情况或无效输入。你着眼于移除与你的目标不匹配或者与项目愿景冲突的特性。您提高了您的测试覆盖率，确保测试快速而全面。基本上，你做了所有你应该做的事情来拥有一个维护良好的项目。随着时间的推移，维护的行为创造了一个好的软件项目。令人惊讶的是，这可能会导致人们开始关心你的软件。
