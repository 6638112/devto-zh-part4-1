# 按照你自己的标准

> 原文:[https://dev.to/jtenner/by-your-own-standard-29go](https://dev.to/jtenner/by-your-own-standard-29go)

自从我写了`as-pect`之后，我和这么多人谈论测试他们的软件，这真是令人惊讶。我可以和人们谈论*小时*关于他们如何只想使用有效的软件。几乎在每一次谈话中，我都会说得他们耳朵都掉了，但是更多的时候，当我们谈到软件开发中的个人责任这个话题时，我可以看到人们头脑中的齿轮在转动。

我们都听说过噩梦般的错误报告、0 天漏洞和代码审查的故事，这些故事可能会让任何人反胃。在这个问题上，人们真的很固执己见，并且讨厌不工作的软件。句号。每个人通常都同意这个简单的事实，这是不容争辩的。

这可能也是你个人关心的事情。根据定义，拥有一套有意义的规范和功能测试套件的软件是比推卸输出责任更好的软件实现。

那么为什么当我问人们他们打算如何测试他们的软件时，他们会给我借口呢？

一个典型的对话可能以这样的问题开始:“你现在在做什么软件？”

回答可能各不相同，但我不可避免地会问性能和测试。这就是不喜欢自己个人行为的人改变语言的地方。

他们可能会说，“我还没到那一步。”

他们可能会说，“我不需要测试，我的软件很小。”

大多数时候我听到的是:“我对测试还不放心。”

听说人们会选择这条路，尤其是当他们谈论这个话题时，他们用厌恶和愤怒的语言来描述自己时，这令人愤怒。声称想要经过正确测试的软件的同一批人会*跳过*测试他们自己的软件，因为*认为*他们更了解。所有这些借口中最糟糕的部分是它们说的都不是真的。

测试是你做的第一件事。先写你的测试，*然后*写软件。这可以一点点发生，也可以完全提前发生。最终结果是一样的。

我怎么知道这个？人们想让自己承担责任，但他们没有足够的勇气承认这一点。

让我换个说法。你想让自己承担责任，但却没有足够的勇气承认。

开始和完成软件测试的责任是你的，而且是你一个人的。那是因为除了*你*之外，没人以这个标准来捧你。你是主要的责任单位。永远不要给自己找借口。安装测试框架。从单个单元测试开始。找到一个边缘案例，并永远让自己对它做出的断言负责。

是时候开始*拒绝*任何人都不受考验的想法了。如果你的软件超过 5 行代码而没有编写一个单独的测试，你已经跳过了*非常*的第一步，这是你*个人* *认为重要的*单元测试。不是我。
