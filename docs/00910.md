# 开源解决方案架构模式

> 原文:[https://dev . to/chanakaudaya/open-sourcing-the-solutions-architecture-patterns-57nm](https://dev.to/chanakaudaya/open-sourcing-the-solutions-architecture-patterns-57nm)

解决方案架构师(sa)是将深奥的技术知识转换成技术人员和非技术人员都能理解的格式的人。他们是产品工程师和销售人员之间的桥梁。奇怪的是，在与解决方案架构相关的开源社区中，并没有共享的公共知识库(KB)。

这就是我构建这样一个知识库的动机，无论 sa 使用或推广哪种技术堆栈，它都可以被 sa 重用。我从小做起，创建了一个 GitHub 库，并开始贡献解决方案架构模式，这些模式是我从其他人那里学到的，也是我作为 s a 工作时学到的。今天，我发布了这个存储库的 0.3 版本，其中包含了一组新的解决方案架构模式。这是这个库的 GitHub 库链接。

[解决方案-架构-模式](https://github.com/chanakaudaya/solutions-architecture-patterns)

这里讨论的架构模式不是我自己的发明。相反，它们是我们作为解决方案架构师日复一日发现的常见模式。下面提到的图表是存储库中描述的一个示例架构模式。

[![Microservices-Security-Policy-Based](img/1d73d6b473b35b2bc31f1f507cd99144.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--HFVlLo94--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://github.com/chanakaudaya/solutions-architecture-patterns/blob/master/vendor-neutral/Micoservices-Security-Pattern-Policy-Based-OPA.png) 
图:微服务安全模式

这个存储库主要分为两类。这些是，
1)厂商中立
2)厂商特定

供应商中立模式类别包含解决方案架构模式，这些模式不局限于特定的供应商或技术堆栈。这里讨论的体系结构可以由多个供应商实现，也可以由一个供应商实现。但是在大多数情况下，对特定的供应商没有特别的说明。有些模式利用了特定的技术。但这种情况非常罕见。

特定于供应商的模式类别包含特定于给定供应商或技术的解决方案架构模式。这些体系结构被绑定到各自供应商及其产品。这是一个我们期待与这些供应商合作的人做出一些贡献的领域，因为我们没有与资源库中提到的所有供应商合作。

我们已经开始了这个项目，目标是围绕解决方案体系结构学科和一个人们可以分享他们的知识、想法和建议的公共场所建立一个开源社区。这仍然是早期阶段，我们有 65+生长激素星级和 15+生长激素分叉，因为这一写作。
