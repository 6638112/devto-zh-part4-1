# 简介

测试驱动开发(TDD)已经成为开发人员术语的一部分，我认为它对我来说是一门神秘的艺术。

作为一个**爱**或**恨**关系专一的开发商，你不需要这种态度。

这类似于组织采用敏捷软件管理实践。

当我参与帮助指导初级开发人员新加坡组织的开发人员健身房的 Python 开发人员时，我对它有了更多的了解。

除了试用教程或者看类似 [Clean Code](https://www.amazon.com/Clean-Code-Handbook-Software-Craftsmanship/dp/0132350882/ref=sr_1_1?keywords=clean+code&qid=1562471652&s=gateway&sr=8-1) 、[Python Test With PyTest](https://www.amazon.com/Python-Testing-pytest-Effective-Scalable/dp/1680502409/ref=sr_1_1?crid=1ACWNXBBZJ9AY&keywords=python+testing+with+pytest&qid=1562471624&s=gateway&sprefix=python+testing+%2Caps%2C422&sr=8-1)、&、 [Test Driven Python](http://www.obeythetestinggoat.com/pages/book.html#toc) 之类的书。

# 采用 TDD 的好处

以下是我在 developer gym 帮忙时得到的好处，它们帮助我采用了这种实践。

作为我个人开发人员工作流程的一部分，我目前正在。

我发现的好处如下:

*   健全性检查——允许您添加新功能、重构或升级，而无需怀疑自己
*   减少调试时间——通过花更少的时间找到错误的根本原因。
*   给你的代码注入信心——对于把你的代码库交给另一个开发者很有帮助。

除了您的文档，谁在维护它或将它集成到他们自己的文档中

# 收养中吸取的教训

## 它适用于各种编程语言

我将为你省去关于为什么 x 编程语言比 y 编程语言更好的宗教战争。

尽管测试框架或库与每种编程语言有很大的不同，但是通过了解 TDD。

它可以帮助你将你对编写测试用例以及 TDD 的理解传递给另一个开发人员，即使他/她使用的是不同的语言。

这是我在 developer gym 的一次会议上帮助指导开发人员时学到的。

## 加快发展进程

我相信通过学习 TDD，它可以帮助你更快地编码。

简单来说，当你编写代码而没有测试用例时，你会花多长时间来修复 bug？

修复一个 bug 需要 5 分钟到 3 小时吗？此外，您可能需要花时间手动一部分一部分地检查您的程序，以找出并解决 bug

通过编写测试用例，您可以合理地知道错误并没有发生在您的代码中。

相反，根据您的测试用例，您可以非常准确地找出哪些代码没有执行。

由于你之前介绍的代码。

## 挑战你的假设，变得有自知之明

在实践中，通过采用 TDD，你变得对问题更加有自知之明，并且通过测试用例来解决问题。

因为通过编写测试用例，它允许你在开始编写实际程序之前，获得你为解决问题而编写的代码的即时反馈。

这为您提供了许多您创建的测试用例，以检查您在编写实际代码时代码中的假设是否正确。

这意味着你需要不断地**重构&优化你的测试用例，以适应你想要解决的问题的解决方案。**

 **# 结论

总的来说，学习 TDD 帮助我减少了调试代码的时间。

相反，它迫使我专注于为测试用例制定一个解决方案，以获得理解，并挑战我的假设，以制定更好的解决方案。

通过**失败**我的方式与一个可行的解决方案，我得到的问题反映在我的代码中。

这让我对 TDD 有了更好的理解，并给了我很大的帮助。
