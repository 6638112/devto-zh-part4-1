# 什么是托管开源？

> 原文：<https://dev.to/tidelift/what-is-managed-open-source-2bfl>

让自己回到过去一分钟。现在是 2007 年，你刚刚完成了新的 J2EE 应用程序的开发。你在黑莓手机上收到一条来自采购部的消息，“你想让我把这些戴尔服务器运到哪里？”

废话。你意识到你还没有决定使用哪家医院。一个好朋友告诉你，在凤凰城有一个他非常喜欢的，但你对在盛夏每次服务器故障时飞 5 个小时到凤凰城毫无兴趣。

现在把自己带回 2019 年，同样的情况。只不过这次你不用担心那些东西。当你的 Node 应用程序准备就绪，输入几个命令，你就可以在几分钟内将它安装并运行在你最喜欢的云托管提供商处。没有飞机，没有硬件，没有在寒冷的房间里跑来跑去插拔电缆。没有采购。

当你的应用程序发布到世界各地时，事情比十年前简单多了。

然而，当涉及到通过集成开源组件来构建应用程序时，生活仍然是，嗯…困难的。让我们来数一数您浪费时间管理开源依赖项的方式:

*   保持最新的 bug 修复版本。
*   移植到新的、不兼容的框架主要版本——当上游项目没有带宽来支持旧版本时。
*   处理由缺失或不可靠的包维护者引起的问题:你会浪费团队的时间移植到一个替换包(最好的情况)或冒一个讨厌的木马的风险(最坏的情况)。
*   处理来自法律部门的请求，列出您正在使用的每个软件包及其许可证。
*   记录您为安全团队使用的一切，并解决实时漏洞。

我们的研究表明，在现代应用程序开发中，[几乎每个应用程序都包含开源依赖项](https://blog.tidelift.com/open-source-is-everywhere-survey-results-part-1)，开发人员花费多达 [30%的时间进行维护，其中 25%的时间与他们使用的开源组件相关](https://blog.tidelift.com/developers-spend-30-of-their-time-on-code-maintenance-our-latest-survey-results-part-3)。

就像云计算运动在 21 世纪初彻底改变了应用程序的托管方式一样，我们看到 2019 年应用程序的构建方式也将发生革命。

它叫做托管开源。

# 那么什么是托管开源呢？

简而言之，托管开源是一种将您和您的团队从目前花费在争论开源依赖上的时间中解放出来的方式。而且——就像你把你的托管责任外包给亚马逊、谷歌或微软一样，现在你可以通过 Tidelift 把你的开源组件的维护和供给外包给最了解它们的专家——创建它们的维护者。

这里有一个简短的视频，讲述了我对托管开源的看法，以及为什么我认为这可能是让您的团队摆脱处理开源依赖相关琐事的答案。有了托管开源，你可以回到真正重要的事情上——构建你自己的应用。
