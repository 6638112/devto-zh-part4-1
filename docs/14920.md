# 关于模式日会议的思考

> 原文：<https://dev.to/iamsteveworsley/thoughts-on-patterns-day-conference-187p>

今天我有幸去了趟布莱顿，和来自@AutoTraderLife 的一些开发者和设计师一起参加了#PatternsDay。如果有机会，这是一个值得参加的伟大会议。以下是当天的一些想法/主题:

*   请放心，不管你看到的大公司的设计系统有多专业，多团结，他们也在努力。无论你是一个人还是一个团队，你仍然要对变化做出反应，救火，并试图一次又一次地说服人们。

*   反应性是可以的。需要的时候再加东西。就我个人而言，我需要少一点“这应该被记录在案。”更多的是“没关系，这没有被记录下来，但现在我们需要它，我会添加它。”

*   您添加到文档中的每一行内容都是设计师或开发人员需要考虑的另一件事。

*   你可能从其他设计系统中获得的通用组件和模式需要在你的网站/应用/产品环境中进行测试。删除一个组件可能会增加一个应用程序的转化率，并对另一个应用程序产生负面影响。这个已经测试过了。

*   在将元件添加到设计系统之前，有必要记住它们对效率的潜在影响。

*   是时候停止使用一致性作为证明设计系统的拐杖了。它可能不会像我们最初认为的那样对用户产生太大影响。

*   当你激励同事使用一个设计系统时，你会一遍又一遍地进行同样的对话，当其他优先事项出现时，他们自然会失去兴趣。擅长这些对话。

*   GDS 提供了诚实的推理和研究，解释为什么一个组件会被这样使用或构建。它让我想起了架构决策记录，这是一种记录关于代码库的决策的方式，并且存储在代码库本身中。有助于回答诸如“我们为什么选择 React 作为此解决方案？”尤其是当做决定的人已经离开公司的时候。

*   我很想知道哪些公司已经为单个组件开发了不同的版本。我认为《金融时报》是这样开始的。意味着您可以发布组件的新版本，而不会阻止对其他组件的更新。但是随后增加了多个封装的复杂性。

会议本身组织得很好，有我在会场坐过的最舒适的座位，还有免费的咖啡师咖啡。你不会得到午餐，但有很多好吃的地方。不过，你可能会发现自己在排队上厕所。

布莱顿是个可爱的城市，非常值得一游，尤其是在阳光明媚的时候。😎
