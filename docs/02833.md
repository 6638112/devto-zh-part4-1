# 云原生上的大声叙述

> 原文：<https://dev.to/rasharm_/loud-narrative-on-cloud-native-3n9h>

为了理解 cloud native 的真正含义，我们首先需要做一个“匹配以下内容”的练习。

匹配以下陈述:

1.  云原生是关于如何创建和部署应用，而不是在哪里。
2.  云原生就是使用容器和 Kubernetes 来部署和操作应用程序
3.  云原生是指在公共云中利用完全托管的自动扩展服务
4.  云原生是指利用面向开发运维的方法来运营应用

与以下制造商合作:

A)一家云公司，尚未在容器领域取得突破，但有一个使用托管服务的流行替代方案

b)围绕开发运维流程和自动化提供工具和/或服务的公司

c)一家 IT 公司，在前云时代取得了惊人的业绩，但在云时代却难以保持相关性

d)一个强大的开源基金会，拥有强大的营销力量，支持许多容器技术 OSS 项目

如果你猜的是 1c、2d、3a 和 4b，那么你会对这篇文章的走向有一个相当好的想法。

在这篇文章中，会有夸张，会有嘲笑，会有花言巧语。不幸的是，当谈到“云原生应用”这个话题时，所有这些都是理所应当的。像科技行业的许多其他人一样，原生云也是一个被过度使用(甚至滥用)的术语。云原生应用的定义因供应商而异，取决于他们销售的产品；也因思想领袖而异，取决于他们的思想引领(或者应该是思想引领？我总是很困惑)。

让我们看看第一句话。

> 云原生是关于如何创建和部署应用程序，而不是在哪里。

因此“在哪里”并不重要—可能是云，也可能不是云。那么云原生应用到底是原生于什么呢？一个在内部运行的应用程序，如果使用“正确的架构、流程和文化”构建，似乎有资格被称为云原生的。这就好比说一个运行在 Android 或 Windows 上的应用程序是“iOS 原生的”。我认为，一个应用程序必须利用底层平台的一些特殊特征，并针对底层平台进行优化，才能被称为该平台的原生应用。

不管怎样，我们继续前进。

> *云原生应用使用容器构建，并使用 Kubernetes 进行编排*

我们到底是什么时候决定部署的打包格式和超级复杂资源编排器的使用使应用程序现代化的？这难道不是为了确保你的应用程序能够以预期的规模、可用性和可靠性完成它所设定的目标吗？那么，为什么会有使用包装格式和听起来像科幻小说的技术这样的特殊要求呢？无论如何，通过拥有一个与术语“云原生”相呼应的名称(更重要的是域名)，云原生计算基金会( [CNCF](https://www.cncf.io) )获得了 SEO 排名，提供了一个可信的定义。

所以，我们继续前进。

> *云原生是指在公共云中利用完全托管的自动扩展服务*

不用猜，这里指的是 AWS 和他们的 Lambda lovefest。这是一种有效的替代叙述——关注通过代码表达的业务逻辑，不要担心代码在哪里运行、如何扩展、数据库如何配置、消息如何到达等。就“云原生”的说法而言，它很接近——自早期以来，前所未有的规模和减少的基础架构管理一直是云的承诺。然而，遭受“冷启动”(这可以最好地解释为现在过度强加于客户认知的内部实现的副作用)之类的事情的“短命代码执行片段”，并不适合人们正在进行的所有计算。至少今天不会。

让我们继续前进。

> *云原生是指利用面向开发运维的方法来运营应用*

自动化、持续集成、持续交付/部署是人们提到 DevOps 时想到的一些东西。这些概念其实并不新，即使它们是新的，它们也可以(并且正在)用于交付与云无关的软件——无论是运行还是构建这样的软件。DevOps 仙尘到底是哪一部分，然后使这个软件云原生不清楚。

尽管这篇文章听起来像是在发泄，但它真正的意思是指出我们的许多陈述(“我们的”是科技行业)即使双击一下也似乎没有意义。所以，我们应该继续努力。我接受好的定义(50 字或更少)的建议。
