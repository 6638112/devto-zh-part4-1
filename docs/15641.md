# [](#design-patterns-in-real-life)现实生活中的设计模式？

我们来看看经典的[软件设计模式](https://sourcemaking.com/design_patterns)。

设计模式(通常)不是你在创建软件时预先使用的东西。它们是专门为你在软件的架构/结构、行为等方面可能面临的特定问题而设计的解决方案。

## [](#adapter)适配器

例如，当您有两个或更多不同的接口需要相互通信或集成时，可以使用适配器模式。

在现实生活中，这很像套筒扳手的适配器。

良好的....其实是一模一样的东西！

这其实是很重要的一点:**适配器模式不仅仅适用于代码/软件！**

事实上，所有这些模式不仅仅是为了软件设计！

我们看到它在物理意义上与套筒扳手一起使用，与 USB 适配器一起使用，让我们的手机与电脑通信，等等。

*注:这也类似于[领域驱动设计](https://amzn.to/2NoG2AW)中的[反腐败层](https://docs.microsoft.com/en-us/azure/architecture/patterns/anti-corruption-layer)概念。*

# [](#facade)立面

我们再拍一张。

当你想简化一个复杂的系统/界面时，使用 facade 模式。

例如，在一些 web 应用程序中提交订单的过程可能涉及一系列复杂的步骤。

我们可以将流程“包装”到一个方法调用中，这将给客户一个非常简单的方法来执行该流程。

就像亚马逊或易贝一样，他们可能有一个“一键购买”功能，可以自动引导你完成购买所需的步骤。

这是一个门面。

这种模式也可以应用到更“真实”的环境中，比如汽车。使用手动变速器包括知道如何使用离合器，如何用你的一只手同时移动变速杆，做这件事的时候眼睛要看着路面，等等。

然而，今天的大多数汽车会自动为你完成所有这些(我想这就是为什么它们被称为*自动变速器？*😜).

这是同一种模式，但是应用于非面向软件的环境中。

底层模式是关于简化客户如何与你的产品交互——不管它应用于哪个领域(软件、汽车等)。)

# [](#systems-thinking)系统思维

# [](#a-real-example)真实的例子

我最近参与的一次对话涉及到一些在. NET 中运行的遗留应用程序的性能问题。

我们讨论的一个问题是高负载下的系统性能下降。

总的问题是系统没有足够的“工作人员”来一次性处理许多 HTTP 请求。

一位同事给出的直接解决方案是“我们需要另一台服务器，这样我们就可以使用负载平衡。”

这个解决方案添加了一个新的节点/服务器，并在它们和客户端之间放置了一个中间人。有点像[调解人模式](https://sourcemaking.com/design_patterns/mediator)？

然而，我的回答是，最好的(也是最便宜的)解决方案是解决系统中所有数据库 I/O 都是同步的这一事实。

如果我们让所有这些 I/O 操作都是异步的，那么当数据库在后台工作时，这些线程将可以处理任何新的 HTTP 请求。

*这个解决方案在底层使用[未来/承诺模式](https://en.wikipedia.org/wiki/Futures_and_promises)。*

但是在更高的层面上，这和你在食谱中烘焙时使用的方法是一样的！

# [](#efficient-cooks-dont-need-a-master-chef)高效的厨师不需要大师级的厨师🤔

增加一个中间人和更多节点的解决方案，而不是让你的工人更有效率，就像给你的厨房增加一个主厨，并雇佣另一个你可以委派任务的人。

当一名工人盯着烤箱烘烤时，厨师会命令另一名工人制作糖霜。

听起来像你去过的一些公司？😂

然而，如果你的一个厨师已经在高效率地工作，那么你可能真的需要引入另一个工人，可能还有一个主厨来分配任务。

但是，如果一个工人工作效率不高，那么在雇用更多员工之前，您可能希望首先让他/她尽可能地高效(就像使用异步 I/O 让您的线程更高效一样)。

希望这个例子和厨房类型的类比可以帮助您了解这些支配性的模式和技术是如何应用于技术和非技术环境的。

# [](#with-knowledge-comes-great-power)知识带来巨大的力量

然而，在这种情况下，问题的一部分是理解根本问题。

有时，即使我们知道真正的问题，我们可能也没有意识到存在一种特定的模式来解决它。

例如，我的同事知道数据库 I/O 是同步的。但是也许没有意识到有一种更有效的方法。

在没有解决方案的情况下，思考过程必须上升到一个更高的层次，并应用一个效率较低的解决方案。

至少在你知道什么问题有解决方案的时候，学习设计和系统模式是必要的！

你不需要知道如何实现设计模式，但是你最好知道它们能解决什么问题。

一旦你遇到这个问题，你会记得，“哦，对了，我记得在书上读到过一个模式……”👌

# [](#some-more-deep-dives)更深一些的潜

让我们再看几个例子，看看哪些熟悉的模式适用。

## [](#example-requesting-a-restaurant-meal)*示例:请求餐馆用餐*

*[排队](https://docs.microsoft.com/en-us/azure/architecture/patterns/queue-based-load-leveling)* :
当你点菜时，服务员可能会把你点的菜排队，因为厨师现在正忙于其他的菜。

*[责任链](https://sourcemaking.com/design_patterns/chain_of_responsibility) / [管道](https://docs.microsoft.com/en-us/azure/architecture/patterns/pipes-and-filters)* :
可能会有一系列的厨师需要查看订单，完成他们的部分，然后将它传递给下一个厨师，下一个厨师可以调味，配菜等。

主仆关系 :
厨师可能会把下级厨师做的每一道菜组合成一顿完整的饭菜。

## [](#example-banking-transactions)*例如:银行交易*

*分类帐/仅限追加/ [事件来源](https://docs.microsoft.com/en-us/azure/architecture/patterns/event-sourcing)* :
当银行处理账户交易时，它们绝不会删除以前的事件或交易。

例如，如果 PayPal 错误地付给你 100 美元，他们不会“删除”那笔交易。他们将发出相应的反交易(例如，从你那里拿回 100 美元)。尽管双方的资金数额相同，但交易本身仍然存在。

这与大多数公司构建的典型 CRUD 类型的应用程序截然不同。这些通常只是存储系统的当前状态。

但是这种类型的仅附加模式解决了系统中的问题，该系统需要:

*   能够审计在系统上执行的所有操作
*   能够负责任地管理多个外部方之间的共享资源(如资金)
*   能够对系统中的趋势进行分析(因为您已经记录了系统中的每个事件)
*   能够在特定时间查询系统

*[传奇](https://blog.couchbase.com/saga-pattern-implement-business-transactions-using-microservices-part/) / [分布式事务](https://docs.microsoft.com/en-us/azure/architecture/patterns/compensating-transaction#example)* :
当你做一些类似于网上转账的事情时(例如，我想把钱寄给我母亲，她现在在另一家银行)，就涉及到一个内在的整体“事务”(类似于数据库事务)。

除此之外，这跨越多个系统(您的银行、另一个外部人员的银行、其他第三方等。)

如果你给你母亲寄去 100 美元，那么:

1.  它是从你的账户中“取出”的(至少，你没看到)。
2.  你母亲有 30 天(或更长时间)来接受这笔钱。
3.  如果她做到了，那太好了！
4.  如果没有，那么你的银行必须采取某种补偿措施(比如把钱存回你的账户。)

如果任何一步失败，你要确保钱被放回正确的地方。你不想丢失数据！那将意味着赔钱😯。

这涉及到一系列复杂的步骤，这些步骤并不局限于特定的数据库，甚至也不局限于特定的系统，而是需要成为一个一致的类似数据库的交易的一部分，以确保你的钱要么到达你的母亲那里，要么最终回到你的账户上。

被称为分布式事务(即跨越多个系统的事务)，有一些特定的模式可以帮助处理这类场景——比如[saga 模式](https://blog.couchbase.com/saga-pattern-implement-business-transactions-using-microservices-part/)。

这有点超前，但是如果你感兴趣的话，值得一试。
