这些文章你已经看过一千遍了:

他们谈论的是你应该学习`reactjs`或者`node`。构建 1.000.000.000 ToDo 应用程序。阅读`Python Crash Course`,你就会成为顶级开发者。

这都是(理论上的)技术知识。你需要学习技术知识，但是你认为一个能以技术上正确的方式拿剪刀的理发师是好的吗？每个职业都有更多的技巧需要考虑！

让我们来谈谈至少在我看来被忽视的技能。

#### [](#abstraction)抽象

作为一名开发人员，你必须实现一个会被别人使用的特性。这个人可以是你，一个客户，你的同事，网上那些你永远不会认识的人。

知道了这一点，你的工作就是为他们所有人考虑，并把特性归结到它的本质上。

你的管理层想知道人们点击网站上某个东西的频率。你需要明白他们是具体的思考者。

你的管理层考虑的是列表、数字和电子表格。现在，他们不关心和理解你的复杂软件的更大图景。他们不需要。那是你的工作！

回到`how often has a user clicked on the website`任务。我想象自己扮演着两种角色。作为用户和将看到这些数据并试图找出用户意图的人的角色。

对于最终用户来说，什么都不应该改变。也许是他/她必须点击一次的免责声明。就是这样。这些功能对于最终用户应该是不可见的。好吧，那很简单。始终首先考虑您的最终用户！永远！

现在，让我们考虑一下必须理解数据的人。那么他会看到什么呢？只是一个数字。比如`42`？但是这个数字意味着什么呢？也许更好的方法不是衡量他点击的频率，而是衡量他点击了什么？你回到你的产品团队或利益相关者那里，告诉他们，统计一下我们点击的频率和点击后的行为可能会更好。大概你会听到类似`Oh you can do that? Yeah, let's do that`的话。在这个话题上，我可以说得更抽象，但我希望你能明白我的意思。

#### [](#asking-the-right-questions)问正确的问题

从初级开发人员到高级开发人员，我一直看到这种情况。你得到一个任务，然后去做。我把这些人叫做`Code Monkeys 🐵`。

作为开发人员的一部分是提出问题，并深入到我们想要实现的本质(这又回到了抽象点)。
**一句话可以有一千种解释。**
你应该明白你为什么要实现这个特性。所以你可以更好地看到问题和未来的危险。

质疑**为什么公司里的**经常被视为**信任问题**。
你会听到这样的句子:

*   我们应该相信产品团队。
*   让我们相信他们，他们知道什么对公司最好
*   你不相信我吗？
*   我们先试一下，然后提问

问一个问题并试图理解为什么与信任毫无关系。作为开发人员，您知道系统的内部工作方式。你可以看到技术问题，并指出什么可以工作，什么不能工作。如果你曾经听过这些句子，这里有一个永远有效的回答:

*   “我信任你，我知道这很重要。”

#### [](#communicating-with-nontechnical-people)与非技术人员交流

这种情况在 slack 这样的公司聊天系统中发生的频率有多高:
你打开了整个公司的一个频道，你看到了一篇超级技术博客的链接，这篇博客是关于为什么在 javascript 中`forEach`比`map`快。

或者你说“不，我们不能这么做”，然后你开始解释 reactjs 没有这个特性，我们需要加载一个 npm 包。

如果你的产品经理不是前开发人员，那么他/她一个字也听不懂你在说什么。

相反，你应该试着从每个人都理解的领域中找到一个好的类比。就像我一开始对理发师做的那样。一个非技术人员可以理解这一点，并得出结论说你有道理。

#### [](#patience)耐心

你可以在 youtube 上看到这些教程，人们可以在 15 分钟内制作出视频中的东西，然后你尝试一下，这会让你花更长的时间！

你感到沮丧，因为你不能实现待办事项列表。这也是你第一次接触代码。Youtuber 有至少 10 年的经验，也准备了那段视频，并在之前至少一次实施了待办事项列表，现在只是按照脚本进行。

你知道开发人员是夜间生物这种陈词滥调是从哪里来的吗？因为我们喜欢？因为我们反社会？也许对于一小部分开发人员来说确实如此。最大的原因是编码需要时间！如果你尝试新事物的话，需要很长时间！

#### [](#strong-opinion)强烈的意见

在 web 开发方面，我是一个非常固执己见的人，我会告诉人们我的观点，即使我知道他们不会喜欢。我做那件事不是为了惹恼别人或让他们失望。比如，我的观点怎么会如此情绪化，以至于在听完之后，你怀疑你的整个存在？抱歉，但在这背后还有更重要的问题，你应该想出如何处理它们，因为这只会导致一件事:停滞。你在 18 岁、25 岁和 50 岁时都是一样的。我知道这写起来容易做起来难，但你需要知道:“你现在的状态只能让你走这么远”。

在开发团队中最糟糕的事情就是每个人都有自己的观点，但是没有人告诉他们！如果发生这种情况，你就完了。这是结束的开始。如果你不是一个`code monkey`人，你每天都会感到动力不足，更加沮丧，而且不仅仅是你。突然有一天，在那家公司工作了几年的人会离开，因为他们再也无法忍受了。

还有，我不是说你应该说你不喜欢这样。你应该告诉为什么，并给出一些例子。不要成为一个失败者，但也不要每天都感到沮丧。这对谁都没有好处。所以要么说出你的看法，没有看法做个代码猴，要么离开公司去找更好的工作，要么去做自由职业。我不知道，但是不要停滞不前！
