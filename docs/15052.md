# 一个新的架构师对 O'Reilly 软件架构会议的看法

> 原文:[https://dev . to/solace devs/a-new-architect-s-take-on-o-Reilly-software-architecture-conference-56p 6](https://dev.to/solacedevs/a-new-architect-s-take-on-o-reilly-software-architecture-conference-56p6)

作为一名相对较新的架构师，我欣然接受了参加最近在加州圣何塞举行的 [O'Reilly 软件架构会议的机会](https://conferences.oreilly.com/software-architecture/sa-ca)，并渴望尽可能多地吸收知识。

在我成为一名架构师之前，我认为这个角色主要是关于设计系统，以及评估和选择实现系统所需的技术。我没有充分意识到向他人有效解释这些设计和技术选择的重要性。因此，我选择在这次会议期间关注软技能，目标是提高我的设计和沟通技能。我的会议日程在第一天安排了两个半天的辅导课，在接下来的两天安排了简短的演讲和各种活动。

### [](#shaping-and-communicating-architecture)塑造和沟通建筑

在我参加的第一个教程中，Bounteous 的工程副总裁 Seth Dobbs 以简洁优雅的方式帮助我明确了一些关于建筑的发现。虽然许多演讲者和与会者哀叹定义架构师角色的困难，但我认为 Seth 对“在业务和技术之间转换”的描述是非常恰当的。他首先谈到了沟通的基础知识和一些常见的陷阱，比如倾向于假设即使存在脱节，也不存在脱节。这种沟通上的失败导致了责任的转移，比如“我告诉他们了”或者“在维基上”。正如赛斯所说，“交流是双向的，但我们拥有被理解的权利”。

第一步是认识到建筑很像销售，因为最终，“如果我们不能让其他人参与进来，我们的想法就没有意义”。Seth 详细介绍了参与决策的不同利益相关者，例如开发人员和项目经理。然后，他围绕这些角色形成原型，以更好地定义这些利益相关者提供什么(技术或业务见解)，他们想知道什么(如何构建一个功能或其成本)，他们的目标(开发的容易程度或沟通业务)，以及最后，他们可能发生的各种摩擦(怀疑或纸上谈兵)。

Seth 然后讨论了如何识别问题。本教程的核心原则是技术的商业方面，追溯到 Seth 对架构师角色的定义。因此，区分业务问题和技术问题非常重要。例如，数据库查询速度慢是一个技术问题，而客户等待很长时间而不返回是一个业务问题。试图解决前者而不是后者会有一些风险。第一，它可能根本不是一个值得解决的问题；也许它是一个按计划运行的后端查询，对用户没有影响。其次，你可能会错过更紧迫的问题；数据库查询可能只是速度变慢的一小部分，真正的问题是第三方 API 调用。在问题被识别之后，Seth 雄辩地描述了问题的解决是“从需求/目标到解决方案的映射，同时尊重约束”。

最后一步是展示架构。Seth 建议从问题陈述开始，然后是假设和约束等上下文，接着是高级解决方案。解决方案的细节可以随后提供，这些细节更适合不同的受众。

Seth 的教程是对会议的精彩介绍，虽然他的许多想法看起来很直观，但他形式化和展示它们的能力确实很有帮助。我参加的下一个教程从架构的协作方面转移到了内部方面。

### [](#thinking-architecturally)思考建筑

Pivotal 的解决方案架构师 Nathaniel Schutta 首先解释了每个解决方案都有其利弊。然后，他深入研究了如何通过[非功能需求](https://en.wikipedia.org/wiki/Non-functional_requirement)或者更可爱的“功能性”(像 scalab **功能性**或者可靠**功能性**这样的词)来识别和传达这些权衡。策略是通过对几个方面进行排序来确定架构的焦点，并在必须在它们之间做出冲突选择时使用该排序来打破束缚。有太多的特征是一个解决方案很复杂的标志，可能需要多次单独的调查。

这些课程通过[建筑形式](http://nealford.com/katas/)得到加强，在这些练习中，一个小组被给予一个理论项目，他们一起工作来创建一个粗略的设计。我们被要求为一个随机的形排列等级，并用它来指导我们的设计。这是一个引人入胜的练习，有助于练习建筑技巧(这是形的目的)。作为题外话，有人问了一个自从我成为一名架构师以来就一直萦绕在我心头的问题，“作为一名架构师，你如何继续成长？”Nathaniel 优雅地解释说，开发人员能够通过他们在工作中的职责来发展专业知识，而架构师可能没有太多的机会来设计主要的解决方案。这就是学习建筑形令我兴奋的地方，因为它们给了我一个作为建筑师发展的出口。

正如第一个教程有助于形成我对沟通的直觉想法一样，Nathaniel 能够描述我对决策中的权衡以及如何提高这些技能的理解。

会议的其余部分包括各种主题演讲、讨论和简短的演讲。

*   《构建进化架构》一书被许多演讲者大量引用。具体来说，通过测试代码的结构将适应度函数应用于架构。我有一部分有兴趣尝试一下，但是我总是犹豫是否要执行效力可疑的高级规则，因为这往往会导致最终对代码库更有害的变通办法。
*   围绕不同的代码分析工具和方法有一些相互竞争的想法。一个常见的例子是圈复杂度及其各种形式。然而，在主题演讲中，Adam Tornhill 基于复杂代码本质上并不坏的基础上，重点关注了另一种分析。Adam 的方法是使用 VCS 提交日志找到代码中耦合、陈旧或经常更改的区域，并将这些更改与错误或功能相关联。我计划尝试这两种方法，并利用我对代码库的了解来评估什么看起来更准确，看看我们的团队是否会发现这有所帮助。
*   几个讲座集中在社会方面，例如为不同的团队结构设计架构(而不是重组软件，考虑重组团队)以及将生物规模限制映射到团队规模限制(动物的大小受生物功能如何扩展的限制)。在我们目前的规模下，这些观察不太适用，但值得一提的是，自[以来，我们一直在不断增长](https://solace.com/careers/)。
*   James Thompson 的《超越偶然的建筑》我努力遵循建筑师的一些基本要求，例如建筑师扮演向导的角色(而不是独裁的领导者),并深入参与代码库以培养对技术技能的信心。我喜欢他将架构师的想法作为进一步的指导，并说开发团队真正拥有架构，因为他们拥有实现，而架构师只产生抽象形式的实现。
*   **测试和持续交付的架构设计**我对 Ken Mugrage 的演讲特别感兴趣，因为我们正处于 CI/CD 之旅的中途。我兴高采烈地发现，这个演讲与我最近在 Solace 内部发表的关于我们应该用来实现目标的策略(向后兼容的变更，使用并行变更模式)的演讲相似。听起来我们的方向是正确的！

这个会议是一个极好的机会，不仅从演讲者那里，也从其他与会者那里了解软件架构的无数策略和方法。我的直接行动是在 Solace 反思我们的架构过程，看看我们是否可以改进会议、文档和其他输出的结构，以利用我学到的沟通技巧。我们将尽一切努力让我们的团队继续创造令人敬畏的技术！
