# 测试 KPI

> 原文：<https://dev.to/jessekphillips/testing-kpi-1fen>

我通读了这篇文章，[质量发布的 8 个关键性能指标](https://www.softwaretestinghelp.com/key-performance-indicators-quality-releases/)，虽然它指定了一些重要的现代测试变化和需求...我的反应是，我不希望审查这些度量标准，也不希望管理层使用它们来审查测试进度。

我认为缺陷跟踪和识别模式是非常重要的，但是这个 DDE 有向管理层自我辩护的味道，因为他们不相信我们是有价值的。我知道这是 KPI 的要点，有些东西已经卖出去了，应该出示证明，甚至可以开账单。

我致力于将手动测试的效果整合在一起，以便提供跨产品的一致性和自动化。我理解这种炫耀价值的欲望(因为我们之前没有追踪，所以会错过)。我只是在想通过即席测试发现的所有小事。这个 KPI 如何产生了花时间放下所有东西的巨大欲望，这是我已经被施压来证明测试的事情。

我衡量成功的目标是提高团队发布的信心。考虑一个团队，一旦他们的自动化管道完成，他们就愿意交付生产。他们没有指标显示他们在开发中发现的 bug 比在生产中发现的 bug 多，他们没有自大，只是假设他们会在产品问题出现时处理它们。

该团队将通过参与、了解正在进行的测试工作、了解生产问题在管道中被可靠地复制，来获得这种信心。他们会很有信心，因为他们只改变了这一件事，而不是由三个不同的开发人员花六周时间开发的。

现在，当你的客户有一个月的发布周期和一到两个月的交付周期时，这是一个很高的要求。但是信心的进步仍然是最好的指标。
