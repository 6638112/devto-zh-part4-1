# 保持产品文档最新的最简单方法

> 原文：<https://dev.to/ionic/the-easiest-way-to-keep-product-documentation-up-to-date-22an>

文档。

成功和受人喜爱的产品的支柱，但是考虑到产品发展的速度，很难跟上时代的步伐。更不用说软件开发人员所害怕的了，他们会去写代码。

鉴于这些挑战，我发现保持产品文档最新的最简单方法是:

在开发一个产品的同时，更新它的文档。

从表面上看，这似乎很简单，但往往不是这样。

## 为什么采用这种方法？

当构建产品或创建演示时，您可能会专注于某个特定的领域或“主题”这包括安装/设置、托管、与 CLI 交互、调试、UI 主题化、后端 API 集成等。当相关的产品概念在您的脑海中记忆犹新时，同步更新文档会快得多。这通常是开发人员回避文档更新的原因——它被留到当前任务或 sprint 的末尾，很容易半途而废。或者更糟:“当我们开始的时候”，剧透一下，通常是永远不会。

随着时间的推移，堆积如山的产品变更，使情况变得难以控制。因此，让您的团队将更新文档作为其常规工作流程的一部分的诀窍是减少相关的摩擦。

## 如何？

当你坐下来开始一项新任务时，花几秒钟找出与该工作领域相关的文件。浏览一遍，让自己进入文档思维。你会开始注意到过时或完全不正确的细节。

接下来，开始产品工作。偶尔在每个主要步骤上暂停一下，将现有文档与您正在执行的实际工作进行比较。需要注意的事项包括:

*   这一页事实上仍然正确吗？
*   还有其他细节要补充吗？也许第三方工具的工作流程发生了变化，或者最近增加了新的步骤。
*   进一步澄清会有帮助吗？例如，文档在技术上是正确的，但是根据经验，您已经知道用户应该小心问题 X 和 y。

对页面进行更新。产品或演示任务完成后，提交文档更改。

> 根据我的经验，当创建产品演示/教程并且该领域/主题的文档已经以某种形式存在时，这种方法最有效。这样，您可以避免与管理相关的工作(又名 [yak shaving](https://seths.blog/2005/03/dont_shave_that/) ),这些工作会分散对更新内容的最初目标的注意力，例如创建新页面、向目录添加新链接、处理样式/设计问题等等。

## 具体例子

我正在构建一个展示其所有插件特性的[电容器演示应用](https://github.com/ionic-team/capacitor-plugin-demo)。我决定将它作为一个 web 应用程序托管在 Firebase 上，让我的同事们有机会在他们的移动设备上测试后提供反馈。在开发演示应用时，我打开了 Progressive Web App hosting 上的 [Ionic docs 页面，并将这些说明与我需要采取的实际步骤进行了比较。应用完成部署后，我向](https://ionicframework.com/docs/publishing/progressive-web-app)[提交了以下变更](https://github.com/ionic-team/ionic-docs/pull/733/files):

*   向 Firebase web 控制台添加链接。Ionic 用户需要在完成其他托管步骤之前创建一个新的 Firebase 项目。
*   更新`firebase init`命令的用户提示。一个重要的问题是关于包含编译后的 web 应用程序的公共目录。对于 Ionic 应用程序，标准的构建输出目录是`www`(而不是 Firebase 建议的`public`)。
*   提供完整的`firebase.json`配置文件，用户可以进行比较。

## 结果

更新 Firebase 托管页面只给我的演示应用托管任务增加了 30 分钟。我澄清并扩展了部署的一些更棘手的方面，这将在相当长的一段时间内持续存在。

遵循这种方法，您的文档保持最新，并且改进可以以可持续的速度继续，同时消除开发团队的负担。
