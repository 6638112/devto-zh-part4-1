# 我的结对编程经验

> 原文：<https://dev.to/healeycodes/my-experience-with-pair-programming-4c0a>

当我们在大学学习敏捷方法时，我学到了结对编程。当时，它看起来非常全面——几乎令人害怕。我想象着自己被键盘绊倒，而我的同事脸贴在我身后。事实上，结对编程会议帮助我发展成为一名开发人员，而且我发现两人一组处理事情更加轻松。

> 你介意我开车吗？

*驾驶*就是控制鼠标/键盘。*导航*是观看，根据两人的风格，可能会口述要写的内容，或者提出问题、难题，以及对设计的思考。点击这里，可以找到更详细的角色划分[(最早由乔丹·波尔顿撰写)。](https://gist.github.com/healeycodes/5acc53131957f6a96a281c89890c7706)

我看到一项研究声称，在谈话过程中，当轮到我们回应时，我们给对方的回应时间是我们给自己的三倍。我认为结对编程也是如此，因为我有时会感到压力。我记得我自己犯了无数的错别字(直到它们被指出来我才意识到),忘记了代码的基本结构，担心导航员在等我完成代码思考的时候变得无聊。然而，我不记得我的任何同事曾经做过这样的事情。每个人都很聪明。

对于门外汉来说，两个开发人员处理一个工单可能比他们分别处理两个工单更有效率，这似乎很奇怪。其中一个原因是专注的程度，这可以由驾驶员来保持。有人看着你写代码，会发现你的 linter 可能不会发现的愚蠢错误。这节省了构建和测试时间。你可以把这些简单的问题抛在脑后，留在你的 IDE 中，而不是搜索简单的问题(和上下文切换)。能够讨论想法有助于创造性地解决问题。我发现我的思绪很少游荡，因为我的伙伴总是在那里指引我回到我们的设计。兔子洞可以被扼杀在萌芽状态——例如，我们中的一个人说“我们先不要优化这个部分”。

对我来说，这是一个舒适的氛围，很有趣。我见过它被称为“自动双向橡胶鸭子”。

调试时，最好的方法是将问题分解成最小的部分——排除假设，找出你 100%确定有效的部分。然而，很难摆脱你自己的假设，随后你会发现你认为正确的事情实际上是解决问题的途径。结对编程时，更容易用更多的逻辑来处理问题，因为有第二双眼睛必须确信每个假设。

配对适用于以下情况:

*   你卡住了！
*   你是新来的，需要指导
*   团队需要共享知识
*   有一个时间敏感的特征

### 有些票不需要

我读过一些人在公司的经历，在那里每一行代码都是成对编程的。我很好奇这样的要求有什么好处。我想象一个团队的[巴士系数](https://en.wikipedia.org/wiki/Bus_factor)会低得令人难以置信。但是我正在思考我所熟悉的代码库中的故事和错误。其中大部分工作是设置样板代码和一些测试。

到目前为止，我最喜欢的设置是结对编程留给开发人员的团队。如果我们觉得一张票需要结对编程才能有效，那么我们所要做的就是让我们的经理知道这就是计划。

### 担任导师

在过去的一年里，我一直在教我的妻子如何编码。当我们以两人一组的方式处理一个功能或项目时，她的知识会有最大的飞跃。这让我有机会观察她如何着手解决问题，并让我引导她走上正确的道路(或至少接近正确的道路)。当我开车时，她报告说看到的最有帮助的事情是那些不是编码而是更基于知识的步骤，例如设置开发环境、适当的版本控制、不同的测试套件和 CI/CD 集成。

### 注意

这读起来有点像 Pair Programming Institute (tm)的宣传。有些人不能很好地适应结对编程，可能会发现它令人苦恼且效率低下。这一点都没有错！
