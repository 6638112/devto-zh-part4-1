# 为什么 DEV 在 Heroku 上托管(以及我们如何配置我们的服务)

> 原文:[https://dev . to/dev team/why-dev-hosts-on-heroku-and-how-we-configure-our-service-1 caj](https://dev.to/devteam/why-dev-hosts-on-heroku-and-how-we-configure-our-service-1caj)

披露:Heroku 是 DEV 的赞助商，但是这篇文章本身并没有被“赞助”。

我选择写这篇文章，因为我想提供一些关于我们如何在 DEV 思考这些事情的想法。我收到很多关于这种事情的问题，我的回答通常是“看情况”。也就是说，我已经是 Heroku 的忠实用户近十年了，所以一定有什么东西让我(和 DEV)留在身边。

有很多很棒的云选择，包括我们的另一个赞助商: [DigitalOcean](https://dev.to/digitalocean) 。对于开发者来说，托管/云的前景越来越好，但这不是苹果对苹果的讨论。这些不是商品，它们是具有不同服务、不同 UX 关注点和许多权衡的特定服务。

我们不以任何方式使用 Heroku *独占*，但是由于该平台提供的开发人员经验，我们在一些核心工作中一直使用它。

### [](#first-of-all-what-does-heroku-do-exactly)**首先，Heroku 到底是做什么的？**

Heroku 将自己描述为“平台即服务”，这实质上意味着它是一种比其他云服务更加集成的体验。我从来没有确切地确定在阐明这些事情的时候在哪里划线；但原则上，Heroku 捆绑了一个运行时环境，一个插件生态系统，以及对集成开发人员体验的普遍关注。

### [](#whats-the-big-benefit)**有什么大好处？**

简而言之，我们已经使用 Heroku 很多年了，因为它*让我们的时间倒流*，而且真的*不需要被教*。也就是说，Heroku 的入职流程很大程度上可以直观地完成，而不是为了使用该服务而必须通过学校。他们做了很好的工作，用优秀的向导勾勒出了我们需要走的几乎每一条路。

Heroku CLI 和 web 界面都提供了与服务交互的有效方式。我发现不同的团队成员在这方面做出不同的选择，而不会导致团队内部的沟通不畅。

以不同的方式与 Heroku 交互，比如与附加生态系统交互，通常是通过命令行来完成的，比如:

```
heroku addons:create rediscloud:100 
```

### [](#integration-without-lockin)**集成无锁定**

也许我们坚持使用 Heroku 的最大原因是我们从来没有发现自己被特别锁定在 Heroku 的生态系统中。我们需要构建的东西的第一个版本几乎总是可以在 Heroku 附加组件生态系统中获得，但是当我们出于任何原因需要改变时，总是很容易删除这些捆绑，因为集成几乎总是可以通过应用配置来定义，从而实现交换服务

### [](#our-heroku-config)**我们的英雄配置**

我们在单个[性能 L Dyno](https://www.heroku.com/dynos) 上运行，autoscale 配置为在流量下降时并行化至 2+。我们使用 Puma 派生应用程序，在一台 Heroku 机器(即 dyno)上运行并发实例。早期，我们使用标准的 2x Dynos，直到我们最终在一个大的 Dynos 上运行，看到了更高的效率。

我们的核心数据库使用 Heroku 管理的 Postgres，我们的大多数其他服务是非 Heroku 产品，但通常仍通过 Heroku 提供的附加系统进行管理。然而，当附加系统在访问仪表板的最简单方式等方面满足我们的需求时，我们已经能够摆脱它。附加系统是一个很好的默认设置，但同样重要的是，我们可以在需要的时候改变我们的方法。
